# fix204 — Instrument the source-refresh DB-write phase (measurement only, no behavior change)

> **Version:** build `+204`; patch = next available at ship time (read live pubspec — patch and fix numbers drift; do NOT assume). Tag = next available `v1.23.X` (immutable; never reuse/delete/recreate).
> **Why:** fix194 raised `bulkInsertRows` 60→1000 on a theoretical statement-count argument. Measurement (v1.23.20 log) shows the Emjay refresh did NOT get faster — wipe→restorePreserve went ~393s (at 60) to ~637s (at 1000). The refresh has **no insert/commit timing logging**, so the 637s is a black box: we cannot tell whether it is the bulk inserts, the trailing `restorePreserve`/`updateGroups`, per-row index/trigger maintenance, or run-to-run variance. This runbook adds timing logs ONLY so the next refresh log pinpoints the cost. No logic changes — do not "fix" performance here.
> **Bug:** none. Pure instrumentation.

## What we know going in (context, not assumptions to encode)

- v1.23.20 Emjay refresh: fetched 271,092 rows; `wipeSource deleted 270973` at 12:59:56; `restorePreserve total=14035` at 13:10:33 → ~637s gap. `ChannelSearchCache` rebuild was separate and fast (~3s).
- `getXtream` (`lib/backend/xtream.dart`) builds a `statements` list (wipe + per-batch insert closures + `updateGroups` + `restorePreserve`) and runs it all via `Sql.commitWriteBatched` — that single call is the unmeasured window.
- `channels` carries ~12 indexes plus AFTER INSERT/UPDATE/DELETE FTS triggers; `restorePreserve` does per-row UPDATEs matched on `name` (which IS indexed — `index_channel_name`, `channels_unique`). These are *hypotheses* for where the time goes; the point of fix204 is to measure, not to assume.

## Fix 204.1 — per-batch + total timing in `commitWriteBatched`

**File:** `lib/backend/sql.dart`

### Current code (verbatim — match exactly)
```dart
    if (commits.isEmpty) return;
    final shared = memory ?? <String, String>{};
    for (var i = 0; i < commits.length; i += batchSize) {
      final end = (i + batchSize < commits.length) ? i + batchSize : commits.length;
      await commitWrite(commits.sublist(i, end), memory: shared);
      onBatchCommitted?.call(end);
    }
  }
```
### Replacement code (verbatim)
```dart
    if (commits.isEmpty) return;
    final shared = memory ?? <String, String>{};
    // fix204: instrumentation only (no behavior change). Time the whole batched
    // commit and each transaction batch so a refresh log shows where the time
    // goes (bulk inserts vs the trailing restorePreserve/updateGroups closures).
    final swTotal = Stopwatch()..start();
    var batchIndex = 0;
    for (var i = 0; i < commits.length; i += batchSize) {
      final end = (i + batchSize < commits.length) ? i + batchSize : commits.length;
      final swBatch = Stopwatch()..start();
      await commitWrite(commits.sublist(i, end), memory: shared);
      swBatch.stop();
      AppLog.info('Sql.commitWriteBatched: batch ${batchIndex++} '
          'closures=${end - i} (through $end/${commits.length}) '
          'took ${swBatch.elapsedMilliseconds}ms');
      onBatchCommitted?.call(end);
    }
    swTotal.stop();
    AppLog.info('Sql.commitWriteBatched: DONE ${commits.length} closures '
        'in $batchIndex batches, total ${swTotal.elapsedMilliseconds}ms '
        '(bulkInsertRows=$bulkInsertRows, batchSize=$batchSize)');
  }
```

### Key changes
- Adds a per-batch `Stopwatch` (logs ms per transaction batch and cumulative progress) and a total `Stopwatch` (logs total commit ms + the active `bulkInsertRows`/`batchSize`). No control-flow or data change.

## Test plan / how to use the data
1. Install the build, refresh the large source (Emjay) and the smaller one (Aniel3000), then export/share the debug log.
2. In the log, read:
   - `getXtream: committing N statement-closures … totalRows=…` — how many closures and rows.
   - `Sql.commitWriteBatched: batch K closures=… took Xms` — per-batch timing. **If the early insert batches are fast and one late batch is huge, the cost is `restorePreserve`/`updateGroups`, not the bulk inserts.** If every batch scales with size, it is the inserts (index/trigger maintenance).
   - `Sql.commitWriteBatched: DONE … total Xms (bulkInsertRows=1000 …)` — the authoritative commit time to compare against future changes.
   - `getXtream: DB commit phase … took Xms` — the phase total.
3. Bring that log back. The measured breakdown decides the next step: revert/retune `bulkInsertRows`, or move to the index/trigger bulk-load pattern (drop non-essential indexes + disable FTS trigger during insert, rebuild after) — whichever the data implicates.
