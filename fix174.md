# fix174 — channel import collapses to the distinct-name set (upsert keyed on `name`); slow ~270k-row import; no real progress %; plus "Loading Channels" every launch is `refreshOnStart`

> **Version:** 1.23.9+172 → **1.23.10+174** (phone/even build; patch bump from shipped 1.23.9+172).
> **Evidence:** `Free4Me-IPTV-1.23.9` source zip + `free4me_log-20260530-195256.txt` (Pixel 8) + `free4me-backup-stamp.json` (schemaVersion 3, appVersion 1.23.9, `refreshOnStart: true`).
> **Bugs:**
> 1. **A full Xtream import of 55205 live + 170908 movies + 44440 series ends with only ~46 channels in the DB.** The channels table is `UNIQUE(name, source_id)` and `insertChannel` does `INSERT … ON CONFLICT(name, source_id) DO UPDATE`. Any two streams that share a display name within one source collapse onto a single row — so a catalog with heavy name reuse (divider rows, `24/7` blocks, dup HD/4K names) dedups down to the count of *distinct names*. The 46 surviving rows match exactly what `getChannelsPreserve` reports after the import.
> 2. **"Loading Channels to the database" runs on every cold launch.** This is the `refreshOnStart` setting (currently `true` in the backup). `main.dart` passes `refresh: widget.settings.refreshOnStart` into `Home`, and `home.dart` fires `showSourcesRefreshDialog` whenever `widget.refresh` is true. Working as designed; documented here so it isn't re-diagnosed as a bug. **No code change** — settings guidance only.
> 3. **The DB write of ~270k rows takes ~4 minutes.** The fetch is fast (≈13 s in the log). The slow part (`19:32:11`→`19:36:07`) is the insert loop: `insertChannel` runs **one `tx.execute` per row plus a `SELECT last_insert_rowid()` per row** — ~540 000 SQL round-trips — and `commitWriteBatched` `await`s each closure individually. There is no multi-row insert.
> 4. **The progress bar shows no real progress.** `value` is `sourceIndex / sourceTotal`, so with a single source it sits at 0 % the whole time then jumps to done. There's no row-level percentage, count, or ETA.
> **Root cause (bug 1):** the uniqueness key (and the upsert conflict target, and the FTS mirror key) is the human-readable `name`, not the provider's stable `stream_id`/`series_id`. Xtream providers do not guarantee unique names; they guarantee unique stream/series IDs.
> **Root cause (bug 3):** per-row statements + a redundant `last_insert_rowid()` read on the import path (its result, `memory['lastChannelId']`, is unused by the Xtream/M3U bulk path). **Root cause (bug 4):** no numeric progress is plumbed from the insert loop to the dialog.

## What the log proves

The single session in `free4me_log-20260530-195256.txt` is a backup-import refresh, and it is internally consistent with bug 1:

```
19:32:11  SourcesRefreshDialog: "mj" — Loading 55205 live channels…
19:32:11  SourcesRefreshDialog: "mj" — Loading 170908 movies…
19:32:11  SourcesRefreshDialog: "mj" — Loading 44440 series…
19:32:11  Xtream: fetched source="mj" live=55205 movies=170908 series=44440
19:32:11  Sql.wipeSource: sourceId=1 deleted 0 channels
19:36:07  Sql.restorePreserve: sourceId=1 total=0 epgRestored=0 manualRestored=0
19:52:46  Sql.getChannelsPreserve: sourceId=1 total=46 favorites=7 lastWatched=6 …
```

- The fetch **succeeded** — 270 553 items parsed, `failCount` well under the `>= 3` abort threshold (no `Xtream: fetch failed` line), and `Saving to database…` was reached.
- `wipeSource … deleted 0` confirms the table was empty going in (fresh import), so the post-import row count is purely what the insert produced.
- After the import, only **46** rows exist for `sourceId=1`. 270 553 inserts → 46 rows is a ~99.98 % collapse. The only mechanism in the insert path that silently discards rows is the `ON CONFLICT(name, source_id) DO UPDATE` upsert: duplicate-named rows update an existing row instead of adding a new one.
- `restorePreserve … total=0` is expected here (fresh DB had nothing to preserve); it is *not* the cause of the low count.

This is not a fix170/fix172 regression — fix172 was the multi-view compile/focus work. This is a pre-existing schema design choice that only becomes catastrophic with a large, name-colliding provider.

## Root cause — bug 1

`lib/backend/db_factory.dart` declares (migration 1):

```dart
await tx.execute(
    '''CREATE UNIQUE INDEX channels_unique ON channels(name, source_id);''');
```

`lib/backend/sql.dart` `insertChannel` upserts against that key:

```dart
INSERT INTO channels (
  name, image, url, source_id, media_type, series_id, favorite,
  stream_id, group_name, epg_channel_id,
  catchup_type, catchup_source, catchup_days
)
VALUES (?, ?, ?, ?, ?, ?, ?, ?, ?, ?, ?, ?, ?)
ON CONFLICT (name, source_id)
DO UPDATE SET …
```

So uniqueness is per **`(name, source_id)`**. The provider's stable identifier is carried in `stream_id` (live/VOD) or `series_id` (series) — `xtreamToChannel` sets `streamId: int.tryParse(stream.streamId ?? "") ?? -1` for live/VOD and stores the series id in both `url` and `series_id` for series. Names are decorative and frequently repeat; IDs do not. Keying on name is the defect.

The fix is to make uniqueness (and the conflict target) include the stable id and the media type, so two genuinely-different streams that share a display name can both persist, while a true re-import of the *same* stream still upserts in place.

A correct composite key is **`(source_id, media_type, stream_id, series_id)`**:
- live/VOD rows: `stream_id` is the real key, `series_id` is NULL.
- series rows: `series_id` is the real key, `stream_id` is `-1`.
- `media_type` disambiguates the rare case where a live `stream_id` numerically equals a VOD `stream_id`.

SQLite treats `NULL` as distinct in a `UNIQUE` index, which would *break* upsert on series rows (every NULL `stream_id`/`series_id` is "unique", so re-imports would duplicate). To get deterministic upsert we index **expressions that coalesce the nullable id to a sentinel**, and use the identical expressions as the `ON CONFLICT` target.

## Part B — `lib/backend/sql.dart`: change the upsert conflict target to match the new key

**Locate** `insertChannel` (≈ line 55). Current verbatim conflict line:

```dart
        ON CONFLICT (name, source_id)
        DO UPDATE SET
```

**Replacement:**

```dart
        ON CONFLICT (source_id, media_type, COALESCE(stream_id, -1), COALESCE(series_id, ''))
        DO UPDATE SET
```

**Key changes:** the `ON CONFLICT` target must list the *exact* expressions of the partial/expression unique index for SQLite to use it as the upsert arbiter. Everything in the `DO UPDATE SET` clause stays as-is (it already refreshes `url`, `group_name`, `media_type`, `stream_id`, `image`, `series_id`, the `COALESCE`-guarded `epg_channel_id`, and catchup columns; `engine_override` intentionally omitted). The bound parameter list is unchanged.

> **Why this is safe for upsert semantics:** a true re-import of the same stream has the same `(source_id, media_type, stream_id, series_id)` → conflict → UPDATE in place (favorites/EPG preserved via the existing `COALESCE`). Two different streams sharing a name now differ in `stream_id`/`series_id` → no conflict → both inserted. This is the intended behavior.

## Part C — FTS mirror: no change required, but verify

The FTS triggers (`channels_ai` / `channels_au` / `channels_ad`) mirror by `rowid = channels.id` and index `name`. They key off the channel **row id**, not the name uniqueness, so they continue to work unchanged: every inserted row gets its own `id` and its own FTS entry. `channels_au` fires `AFTER UPDATE OF name` (fix170 migration 11) — re-imports that upsert and change `url`/`group` but not `name` won't churn FTS, which is correct. **No edit.** (Listed so the build machine confirms rather than assumes.)

## Part D — `restorePreserve` matching: leave as-is (documented)

`restorePreserve` (sql.dart ≈ 816) updates by `WHERE name = ? AND source_id = ?`. With the catalog now fully populated, a favorited name that legitimately appears on multiple streams will have its `favorite`/`last_watched` applied to **all** rows sharing that name within the source. That is acceptable and arguably desirable (the user favorited "US: HBO HD"; all entries with that name get the flag). Keying preserve on name avoids needing the user's old `stream_id`, which the backup's `preserve[]` list does not carry (it stores `name`/`favorite`/`lastWatched`/EPG only — see `getChannelsPreserve` SELECT). **No edit; intentional.**

# Fix 174.3 — bulk multi-row insert for imports (kills the ~4-minute write)

The goal: replace ~270k single-row `tx.execute`s (each followed by a `SELECT last_insert_rowid()`) with **multi-row `INSERT … VALUES (…),(…),…`** batches and **drop the per-row rowid read** on the import path. One round-trip per ~500 rows instead of two per row takes the write from ~4 min to well under a minute on this device.

## Part A — `lib/backend/sql.dart`: add `insertChannelsBulk`

Add a new static method near `insertChannel` (after its closing `}` ≈ line 105, before `updateGroups`). It builds one parameterized multi-row upsert per call. The conflict target is the **same fix174.1 id-based key** so semantics match `insertChannel` exactly.

```dart
  /// fix174.3: bulk upsert for large imports (Xtream/M3U). One multi-row
  /// INSERT per call instead of one statement + last_insert_rowid() per row.
  /// Same ON CONFLICT key as insertChannel (fix174.1). Does NOT read
  /// last_insert_rowid() — the import path does not use it.
  static Future<void> Function(SqliteWriteContext, Map<String, String>)
      insertChannelsBulk(List<Channel> channels) {
    return (SqliteWriteContext tx, Map<String, String> memory) async {
      if (channels.isEmpty) return;
      final sourceId = int.parse(memory['sourceId']!);
      const cols = 13;
      final rowPlaceholder =
          '(${List.filled(cols, '?').join(', ')})';
      final values =
          List.filled(channels.length, rowPlaceholder).join(', ');
      final params = <Object?>[];
      for (final c in channels) {
        params.addAll([
          c.name,
          c.image,
          c.url,
          c.sourceId == -1 ? sourceId : c.sourceId,
          c.mediaType.index,
          c.seriesId,
          c.favorite,
          c.streamId,
          c.group,
          c.epgChannelId,
          c.catchupType,
          c.catchupSource,
          c.catchupDays,
        ]);
      }
      await tx.execute('''
        INSERT INTO channels (
          name, image, url, source_id, media_type, series_id, favorite,
          stream_id, group_name, epg_channel_id,
          catchup_type, catchup_source, catchup_days
        )
        VALUES $values
        ON CONFLICT (source_id, media_type, COALESCE(stream_id, -1), COALESCE(series_id, ''))
        DO UPDATE SET
          url = excluded.url,
          group_name = excluded.group_name,
          media_type = excluded.media_type,
          stream_id = excluded.stream_id,
          image = excluded.image,
          series_id = excluded.series_id,
          epg_channel_id = COALESCE(channels.epg_channel_id, excluded.epg_channel_id),
          catchup_type = excluded.catchup_type,
          catchup_source = excluded.catchup_source,
          catchup_days = excluded.catchup_days;
      ''', params);
    };
  }
```

**Key changes / verification:**
- 13 columns, 13 `?` per row — matches `insertChannel`'s column list and bind order exactly (read from sql.dart 58–104).
- `favorite` binds the `Channel.favorite` value as-is, exactly as `insertChannel` does (it passes `channel.favorite` straight through; the column is INTEGER and the existing code relies on the same coercion — do not change typing here, match the proven path).
- **SQLite variable limit:** modern SQLite (`SQLITE_MAX_VARIABLE_NUMBER`) defaults to 32 766 (≥ 3.32). 500 rows × 13 = 6 500 params — safely under. Keep the per-call row count at `importBatchSize` (500). If the build machine's bundled sqlite is older (limit 999), it would error — see Part B note; 500×13 exceeds 999, so **the batch size for bulk insert must be chosen against the params limit, not row count.** Set a dedicated constant (Part B) so this is explicit and safe.
- No `last_insert_rowid()` read → no second round-trip.

## Part B — `lib/backend/sql.dart`: add a param-aware bulk batch size

At the top with the other consts (≈ line 23, beside `importBatchSize = 500`), add:

```dart
const int importBatchSize = 500;
// fix174.3: bulk insert binds 13 params/row. Keep rows × 13 under SQLite's
// variable limit on every supported engine (older builds cap at 999).
const int bulkInsertRows = 60; // 60 × 13 = 780 params (< 999, < 32766)
```

> Rationale: 60 rows/insert is still a ~60× reduction in round-trips (270k → ~4.5k inserts) and is safe even on a 999-variable SQLite. If the build machine confirms the bundled `sqlite_async`/`sqlite3` engine uses the 32 766 default (it almost certainly does on current Android), `bulkInsertRows` can be raised to ~500 for a further speedup — but 60 already removes the bottleneck and carries zero risk. **Build machine: verify the engine's limit before raising.**

## Part C — `lib/backend/xtream.dart`: feed channels through the bulk path

Currently `processXtream` appends one `Sql.insertChannel(channel)` closure per stream into `statements`, and `getXtream` later calls `Sql.commitWriteBatched(statements)`. Change `processXtream` to **chunk the channels and append `insertChannelsBulk` closures**.

**Replacement:**

```dart

  for (var live in streams) {
```

> **M3U path — do NOT migrate to bulk as-is:** `lib/backend/m3u.dart` interleaves `insertChannel` with `insertChannelHeaders` (m3u.dart ≈ 140), and the latter reads `memory['lastChannelId']` set by `insertChannel`. `insertChannelsBulk` sets no rowid and writes no headers, so swapping it in would break per-channel headers. M3U keeps the existing path here; a separate runbook can speed it up by inserting channels in bulk, reading back ids, then batching headers.

# Fix 174.4 — real progress: percentage + count + ETA

The dialog currently can't show row progress because none is plumbed. Add a numeric progress callback from the import loop up to the dialog, and render `%`, `done/total`, and a rough ETA.

## Part A — `lib/backend/xtream.dart`: emit numeric progress

`getXtream` already knows the totals (`liveCount`, `movieCount`, `seriesCount`) and calls `onProgress?.call('Saving to database…')` once. We want progress *during* the save. The cleanest approach that doesn't disturb the existing `onProgress(String)` channel: add an **optional numeric callback** `onRowProgress(int done, int total)` to the import functions and call it as bulk batches commit.

Because the inserts are appended as closures and committed later in `commitWriteBatched`, thread the callback into the **commit loop** instead of the build loop, so progress reflects actual DB writes (what's slow), not buffer-building.

**Add a parameter to `getXtream`** (xtream.dart, the function signature that currently ends with `void Function(String)? onProgress,` ≈ line 25):

```dart
  void Function(String)? onProgress, [
  void Function(int done, int total)? onRowProgress,
]) async {
```

> Verify the exact current signature head and preserve its existing positional/optional structure; add `onRowProgress` as a trailing optional positional (or named) to avoid breaking the other callers (`refreshSource` → `getXtream(source, wipe, onProgress)`). A trailing optional positional is non-breaking.

Compute the grand total and pass a counting wrapper to the commit. Replace the tail of `getXtream`:

Current verbatim (xtream.dart ≈ 125–131):

```dart
  statements.add(Sql.updateGroups());
  if (preserve != null) {
    statements.add(Sql.restorePreserve(preserve));
  }
  onProgress?.call('Saving to database…');
  await Sql.commitWriteBatched(statements);
}
```

Replacement:

```dart
  statements.add(Sql.updateGroups());
  if (preserve != null) {
    statements.add(Sql.restorePreserve(preserve));
  }
  onProgress?.call('Saving to database…');
  final totalRows = liveCount + movieCount + seriesCount;
  await Sql.commitWriteBatched(
    statements,
    onBatchCommitted: onRowProgress == null
        ? null
        : (committedClosures) {
            // Each bulk closure carries up to bulkInsertRows rows; this is an
            // upper-bound estimate of rows written so far. Capped at total.
            final approxRows = committedClosures * bulkInsertRows;
            onRowProgress(approxRows > totalRows ? totalRows : approxRows, totalRows);
          },
  );
}
```

## Part B — `lib/backend/sql.dart`: `commitWriteBatched` reports batch progress

Add an optional `onBatchCommitted(int committedClosures)` callback. Current verbatim (sql.dart 41–53):

```dart
  static Future<void> commitWriteBatched(
    List<Future<void> Function(SqliteWriteContext, Map<String, String>)>
        commits, {

    Map<String, String>? memory,
```

```dart
String _etaSuffix(DateTime? start, int done, int total) {
  if (start == null || done <= 0 || done >= total) return '';
  final elapsed = DateTime.now().difference(start).inMilliseconds;
  if (elapsed < 500) return '';
  final perRow = elapsed / done;
  final remainMs = (perRow * (total - done)).round();

  if (s < 1) return '';

}
```
