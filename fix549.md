# fix549 — Import progress: surface the silent index-recreate phase (+ dead divider-filter cleanup)

**Version:** 2.0.49 (`2.0.49+549`)
**Base:** v2.0.48 (`2729b6450ec2115a895e0a09903d56ba68fca647`)
**Branch from:** `v2.0.48`

---

## Why

On a phone re-import, the dialog sat on a static **"Saving to database…"** for
minutes and looked frozen. Root cause: everything after that one string —
group sync, FTS rebuild, and the **per-index recreate loop** in
`Sql.withDroppedBrowseIndexes` — emitted nothing to the UI. The index recreate
is the dominant tail (each `CREATE INDEX` over the ~1.17M-row catalog is a
multi-second external merge-sort on eMMC; ~13 indexes rebuilt per import).

This fix threads the **existing** `onProgress` callback into the recreate loop
and emits a per-index counter (**"Building index N/M…"**) to the dialog plus a
per-index elapsed-ms line to the log (field-diagnostics: identifies the slow
indexes). It does **not** change import speed — it makes the slow phase legible.

Also removes one now-dead branch: the in-memory search cache's
`if (e.isDivider && e.hideDividers) continue;`. Since fix546 discards divider
rows at import and `runPendingDividerCleanup` purges existing ones,
`e.isDivider` is uniformly false and the filter never fires. The cache's
`isDivider`/`hideDividers` fields are **left in place (inert)** to avoid
reshuffling the `getAllChannelNamesForCache` tuple positions consumed by
`BrowseOrder` — deliberately conservative (Option 1).

### Note on the dead divider INDEX
`idx_channel_divider` needs **no code change**: it is already in the fix537
`dead` drop-list, so `runPendingIndexMaintenance` drops it on first launch for
every upgrading device, and `withDroppedBrowseIndexes` recreates from
`sqlite_master` (live), so once dropped it is never rebuilt by import. Verified
on the real onn seed (1.21M channels, 2,523 dividers): the deferred maintenance
drops the index and purges the dividers cleanly. Fresh installs build-then-drop
it once (mig 22 is immutable; not touched). No action here.
