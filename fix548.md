# fix548 — Enable v1 (JAR) APK signature alongside v2 (reduce Play Protect warning)

**Base tag:** `v2.0.47`
**Version:** `2.0.47+547` → `2.0.48+548`
**Scope:** `android/app/build.gradle` release signing config only. No Dart/schema change. Full suite green.

---

## Why

A `2.0.42 → 2.0.47` sideload update showed Google Play Protect's harsher **"This app tries to bypass Android's security protections"** dialog. Investigation ruled out a code regression:

- **Cert is identical** across v2.0.42 / v2.0.46 / v2.0.47 — same SHA-256 `d8d34d5a…69ea2`, same key (RSA 4096, `CN=Free4Me-IPTV`). Not a signing/key change.
- **No android/manifest/gradle/native/permission/declared-dependency changes** between 2.0.42 and 2.0.47 — only Dart source deltas, none touching anything Play Protect inspects.

Conclusion: the warning is Play Protect's heuristic re-classification of a **new APK hash** under tightened rules, keyed on the app's long-standing profile (`REQUEST_INSTALL_PACKAGES` + cleartext + sideload self-update) — not anything 543–547 introduced. `apksigner` showed the release is **v2-only (no v1 JAR signature)**; some Play Protect heuristics weight v1 presence as a "known-app" signal.

## Change

Add `v1SigningEnabled true` (and explicit `v2SigningEnabled true`) to the release `signingConfig`. Strictly additive — same key, **cert SHA-256 unchanged**, so updates still install over any prior release. v1 is safe on all supported API levels. This is mitigation #1 of the agreed three (see also: the Play internal-test-track route in `PLAY_PROTECT_NOTES.md`).

> This only affects builds where the release keystore is present (CI). Unconfigured local `flutter run --release` still falls back to the debug signer unchanged.

## Verification

```
flutter analyze    # expect: No issues found!

```
Authored result in sandbox: analyze → **No issues found!**; full suite **259 passed**. Gradle block brace-balanced and well-formed. (Signing behaviour can only be fully verified by the CI build.)

**Post-build check (on the build machine):** after CI produces the 2.0.48 APK, confirm BOTH schemes are present:
```
apksigner verify --print-certs --verbose <2.0.48.apk> | grep -i "scheme\|v1\|v2"
# expect: "Verified using v1 scheme (JAR signing): true" AND "v2 scheme: true"
# and the cert SHA-256 must still equal d8d34d5a…69ea2 (unchanged key)
```
Then install over an existing release install and confirm the Play Protect dialog is softer/absent. If the cert SHA-256 differs, STOP — the keystore wasn't used.
