## Root cause (S24 2026-06-15 09:01 log: first paint = 5844ms, search=5833ms)
The very first browse after launch — `view=All mediaTypes=livestream sources=2` — took 5.8s for 36 rows (`map=0ms`, so pure query). Investigation (EXPLAIN at 700K-row scale) found TWO compounding causes:
1. **Structural:** the fix365 index `idx_channels_browse_enabled` leads with `source_id`. The browse `ORDER BY tier, name` has no `source_id` lead, so with **multiple sources** (`source_id IN (1,2)`) SQLite can't satisfy the sort from the index — it builds a **`USE TEMP B-TREE FOR ORDER BY`**, materializing and sorting every enabled row across both sources (~116K) to take the top 36. Single-source pins `source_id` and avoids this; that's why earlier single-context loads were fast.
2. **Cold cache:** this was the first query on a freshly reopened DB (version-change to 1.34.1), so the temp-b-tree build faulted its pages from disk cold.

## Fix (both, per Rich)
**Structural — migration 30, new index `idx_channels_browse_mt`:** same partial tier/name expression as fix365 but **led by `media_type`** instead of `source_id`. For a single-media-type browse (Live / VOD / Series — the home views) `media_type` is pinned and `(tier, name)` then matches the ORDER BY directly → **no temp b-tree**, stops at LIMIT 36, `source_id IN (...)` residual. EXPLAIN-proven for single-media multi-source AND single-source (both index-served, no temp b-tree). The mixed-media "All" view (`media_type IN (0,1,2)`) still can't pin media_type and keeps a temp b-tree — covered by the warm-up below. The fix365 index is retained (other code paths/orderings use it).

**Warm-up — `Sql.warmBrowseCache(settings)`, called unawaited in `main.dart`** after `hasSources`: runs one representative default browse (livestream, page 1, all sources) in the background, discarding the result, to fault the hot index + data pages into cache before the user's first Home.load. Best-effort (errors swallowed), off the render path, skipped when no sources exist. Logs `main: browse cache warm-up complete (Nms)`.

## Rule 8 / EXPLAIN (executed)
700K-row, 2-source seed: the emitted browse with the new index → `SEARCH c USING INDEX idx_channels_browse_mt (media_type=?)`, **no TEMP B-TREE**, for single-media multi-source and single-source. The mixed-media All case still shows temp b-tree (documented; warm-up handles its cold cost). New `test/browse_mt_index_test.dart` (+2) asserts the index is used and no temp b-tree for the multi-source single-media case.
