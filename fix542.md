# fix542 — Stop the on-open black screen: defer the heavy fix537 index work off the cold-start path

**Base tag:** `v2.0.41`
**Version:** `2.0.41+541` → `2.0.42+542`
**Scope:** migration 39 made trivial; heavy index maintenance + VACUUM moved to a deferred, gated background task. No query-path change. Full suite green.

---

## Problem (reproduced on device)

After the 2.0.35→latest update, both the phone and the onn box showed a **multi-minute black screen on open** (the onn eventually reached "What's New"; it was slow, not crashed). Cause: fix537's migration 39 ran its heavy work **inside the migration, on the cold-start path before `runApp()`** — on the full ~1.43GB catalog that is **~27s of index rebuild + a full-file VACUUM** on the main thread, blowing far past Android's startup window.

Measured on the real field DB: migration-39 index work alone = **26.8s**; the dead-index drops alone = **8.4s**; VACUUM ≈ **20s**. All of it was blocking startup.

## Fix

1. **Migration 39 is now trivial** — it only creates the `app_meta` marker table (~instant; ~3s one-time on a 1.43GB file for the schema write, well within the startup window).
2. **All heavy work is deferred** to `Sql.runPendingIndexMaintenance()` — drop the 5 unused indexes, rebuild the 7 browse indexes without `cat_enabled`, then VACUUM — called **unawaited from `main()` after first frame**, gated once by an `app_meta` marker. Until it completes, the OLD cat_enabled-partial indexes remain and browse works normally (enable-source is merely slow — the pre-fix537 behaviour), never a black screen.
3. **Legacy-marker skip:** devices that already pushed through the OLD blocking migration (marker `fix537_vacuum_done`, e.g. the onn that reached What's New) are already in the target state, so the deferred pass detects that and **skips the redundant ~50s rebuild**, just recording the new marker.

Idempotent and crash-safe: every step is `DROP IF EXISTS` / drop-then-create, and the completion marker is written only after the whole pass succeeds, so an interrupted run simply retries next launch. Bounded memory pragmas (`temp_store=FILE`, 32 MiB cache) keep the rebuild OOM-safe on a 1–2 GB box.

Validated on the real 1.43GB DB: trivial migration ~3s; deferred pass produces the identical end state (1426→1135 MB, **zero** indexes mention `cat_enabled`, enable-toggle fast) — now entirely off the startup path.
