# fix335 — Live TV on ExoPlayer falls back to libmpv despite playing fine → v1.26.51+335

## Why (onn 4K Plus, build 334 section of the log)
On live `.ts` (ESPN UHD), ExoPlayer opened and was genuinely rendering — `ExoEngine: SURFACE[mid-playback+2s] initialized=true size=1280x720 playing=true buffering=false` — but **no `buffering=true/false` transition was ever emitted**. The Player's startup watchdog only cancels on a buffering event, so it timed out at 15s ("open succeeded but no frame") and fell back to libmpv via fix316 — even though Exo was fine. VOD was unaffected because VOD does emit buffering true→false.

(Note: the uploaded log spanned three versions — 331/332/334 — because the debug log is NOT being wiped on upgrade. Analysis used the latest, build 334. The stale-log-wipe is a separate issue; see below.)

## What this does
- `ExoEngine` now emits a one-shot liveness signal (`buffering=false`) the first time the controller reports `isInitialized && isPlaying`, driven from BOTH the value-changed listener and the 1s poll (so it fires within ~1s even if no further value event arrives). The watchdog's cancel path accepts any buffering signal, so it cancels and Exo is kept. One-shot (`_signalledPlaying`) so it doesn't fight the real buffering stream.
- libmpv path untouched; VOD behaviour unchanged (it already signalled).
