# fix194 — Speed up Xtream source refresh (bulkInsertRows 60→1000) + fix two escaped-`$` log lines

> **Version:** build `+194`; patch = next available at ship time (read live pubspec — patch and fix numbers drift; do NOT assume). Tag = next available `v1.23.X` (immutable; never reuse/delete/recreate).
> **Evidence:** Refreshing a large Xtream source is very slow. Log `free4me_log-20260602-005316.txt`: "Emjay" (live=55,249 movies=171,214 series=44,510 ≈ 270,973 rows) took ~393s in the wipe→reinsert window (00:46:27 → 00:53:00); "Aniel3000" (~48,720 rows) took ~117s. The same log shows `max_connections=$n` printed literally (an escaped-`$` bug).
> **Bug:** (1) The bulk-insert batch constant `bulkInsertRows = 60` forces ~4,516 separate multi-row INSERT statements for a 271k-row refresh — statement count is the dominant cost of the reinsert window. (2) Two log statements in `fetchXtreamMaxConnections` use `\$n` / `\$e`, printing literal text instead of the value.
> **Root cause:** `processXtream` buffers ALL channels (live, movies, AND series) through the bulk path `Sql.insertChannelsBulk` (which does NOT read `last_insert_rowid()`), flushing every `bulkInsertRows` rows. At 60 rows/INSERT the per-statement overhead dominates. The two log lines have a stray backslash before `$`.

## What the evidence proves

- Fetch (5–11s) and post-refresh search-cache rebuild (~2s) are negligible; the cost scales with row count in the reinsert window — i.e. it is the INSERT work, not the network or the wipe. (`Sql.wipeSource` uses an index-backed `DELETE … WHERE source_id = ?` — `index_channel_source_id` exists — and completes sub-second; SQLite has no TRUNCATE, and a whole-table truncate would be wrong here anyway since wipe is per-source.)
- The refresh insert path is entirely the **bulk** path: `processXtream` (`lib/backend/xtream.dart`) buffers channels and calls `Sql.insertChannelsBulk`; the per-row `Sql.insertChannel` (the one reading `last_insert_rowid()`) is only used by `getEpisodes` (on-demand when opening one series), NOT during refresh. So raising the bulk batch size is safe and has no `last_insert_rowid()` interaction.
- `max_connections=$n` appearing literally in the log proves line 187 was reached with the guard `n != null && n > 0` true — so detection succeeded and returned a real value; only the log text was wrong. (fix192's display is unaffected.)

## Pre-verified facts (checked against the uploaded `v1.23.17+192` source)

- **`const int bulkInsertRows = 60;`** at `lib/backend/sql.dart:25`, referenced by `xtream.dart:229` (flush threshold) and `xtream.dart:143` (progress math) — both read the constant, so they scale automatically with no other edit.
- **`insertChannelsBulk`** (`sql.dart:108-146`) builds one `INSERT … VALUES (…),(…),…` with `channels.length × 13` params and does NOT read `last_insert_rowid()` (`sql.dart:107` comment).
- **SQLite param ceiling:** `SQLITE_MAX_VARIABLE_NUMBER = 32,766` (default since SQLite 3.32). At 13 params/row, 1000 rows = **13,000 params** — safe with large margin (ceiling ≈ 2,520 rows).
- **Seeded Rule 8 test run in-sandbox (sqlite3 3.45.1):** created the `channels` table with its `channels_unique (name, source_id)` UNIQUE index, executed a **1000-row** (13,000-param) `INSERT … ON CONFLICT DO UPDATE`, then re-ran it with changed URLs. Result: 1000 rows inserted, re-insert kept the count at 1000 (no duplicates) and updated the URL — ON CONFLICT upsert behaves correctly at 1000 rows/statement.
- **Log lines:** `xtream.dart:187` `…max_connections=\$n'` and `xtream.dart:193` `… — \$e'` — `n` (parsed int, `xtream.dart:184`) and `e` (catch var, `xtream.dart:191`) are both valid in-scope identifiers to interpolate.
- **Compile status:** all three edits applied to the uploaded `1.23.17+192` tree; `flutter analyze --no-fatal-infos` returned **exactly the 2 pre-existing tolerated INFOs** (`lib/settings_view.dart:2151/2739`). Rule 6 grep (`\$`, `\u`) on the edited blocks: clean.

## Test plan
1. **The headline test — large source refresh.** On a device, refresh a large Xtream source (e.g. the ~271k-row one). Compare the wipe→reinsert window in the debug log to the ~393s baseline.
   - **Expected:** substantially faster reinsert (statement count drops ~16×; the window should fall well under 2 minutes). The catalog total after refresh must match the fetched counts (no lost/duplicated rows).
2. **Correctness across a re-refresh.** Refresh the same source twice; favorites/EPG matches preserved as before (ON CONFLICT path unchanged in behavior, only batch size changed).
3. **Log check.** After a refresh, the log shows `max_connections=<number>` (e.g. `max_connections=2`), not literal `$n`. Force a detection failure (e.g. offline) → log shows the real error after `—`, not literal `$e`.
4. **Memory/stability.** Watch for any OOM or "too many SQL variables" error during the large refresh (not expected at 13,000 params; the on-device `sqlite_async` engine is the final confirmation of the param ceiling).

## Notes
- **Independent of fixes 188/190/192** (different files). No migration; safe to ship alone.
- **Secondary lever NOT taken here (future work):** `channels` carries ~10 indexes; every insert updates all of them. A drop-indexes → bulk-insert → recreate-indexes pattern could further cut large-refresh time, but it is structurally riskier (must recreate every index exactly and must not disturb the `channels_unique` index the ON CONFLICT relies on) and warrants its own design + number. Do not bundle it into a batch-size change.
- **`importBatchSize = 500`** (closures per transaction) was left unchanged; only ~9 transactions occur for a 271k-row refresh, so it is not the bottleneck, and larger transactions hold the write lock longer.
