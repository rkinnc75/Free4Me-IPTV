# fix420 — ship a single arm64-only APK

**Version:** 1.35.18+419 → **1.35.19+420**
**Type:** Build/packaging only. No Dart code change.
**Tag:** v1.35.19
**Base:** current origin/main (fix418+fix419, 1.35.18+419). Applies directly.

## What
Restricts the release APK to a single ABI, **arm64-v8a**, by adding `--target-platform android-arm64` to the `flutter build apk --release` command in **both** build paths:
- `.github/workflows/release.yml` (the CI release build)
- `scripts/build_and_release.sh` (the local build-machine release)

This drops the `armeabi-v7a`, `x86`, and `x86_64` native libraries (libmpv, media_kit, sqlite3, etc.) from the APK. Every device in the report/log set is arm64, so nothing supported loses coverage. The output path is unchanged — a single `--target-platform` without `--split-per-abi` still emits `app-release.apk`, so the existing `APK_SRC` path in `build_and_release.sh` is unaffected.

`flutter analyze` clean, 166 tests pass (no code touched). `release.yml` re-validated as YAML. **Build machine: re-validate the workflow with `ruby -ryaml` before pushing** (the sandbox lacks ruby; validation here was via python).
