# fix390: Search method auto-set to LIKE substring on first run for low-RAM devices

**Repo:** rkinnc75/Free4Me-IPTV · **v1.34.19+390** · **APK target:** onn 4K Plus + similar 1.9 GB TV boxes

## What this fixes

On a low-RAM device (< 2300 MB total, matching `ChannelSearchCache._minRamMbForCache`), the in-memory search cache is never built (see `ChannelSearchCache.cacheSkipped` at `lib/backend/channel_search_cache.dart:117`). Before this fix, the default `Settings.searchMethod` was `inMemory` everywhere — so a user on a 1.9 GB onn 4K Plus would see "in-memory search" in Settings while the actual search silently fell through to FTS. The setting and the actual behavior disagreed, which is confusing and makes diagnosis hard.

This fix moves the device-aware default from the engine-side safety check to the settings layer: on first run (no persisted `searchMethod` key) on a low-RAM device, the default is `likeSubstring` — the LIKE scan path, no FTS triggers, no cache build. The user-visible setting now matches the actual search path.

A persisted value always wins — the auto-set only fires on first run. A user who explicitly chose `inMemory` (or `ftsAnd`, or `ftsTrigram`) on a low-RAM device keeps that choice. The runtime safety net (`ChannelSearchCache.cacheSkipped`) still routes `inMemory` to FTS internally for that case; the user-visible setting matches the user's explicit choice.

## How the fix was validated

- **Extracted helper** `SettingsService.resolveSearchMethod` is a pure function — the production code calls it from `_readFromDb`, and the unit tests call it directly with explicit `totalMb` so they don't depend on the host's `DeviceMemory.totalMb`.
- **Threshold (2300 MB) matches `ChannelSearchCache._minRamMbForCache`** — same gate the runtime safety net uses. Devices at the boundary (exactly 2300 MB) get the recommended `inMemory` default; the check is strict `<`.
- **Preserved existing fallback chain** — unparseable persisted strings still coerce to `ftsTrigram` (the first enum value, index 0); the trailing `?? SearchMethod.inMemory` only fires for out-of-range indices. Test pins both.
- **`DeviceMemory.totalMb = 0` safety** — if `DeviceMemory.init()` hasn't run, the auto-set is skipped and the recommended default applies. Production calls `DeviceMemory.init()` at `lib/main.dart:51` before `SettingsService.reload()`, so the safety is only relevant in tests.

## What I did NOT change (deliberate)

- **`ChannelSearchCache.cacheSkipped` and the `cacheSkipped` branch in `Sql.search`** — kept as a runtime safety net for users who explicitly chose `inMemory` on a low-RAM device (e.g. moved from a high-RAM phone to a low-RAM TV). The persisted-value gate means the auto-set doesn't apply to them, but `cacheSkipped` still routes their `inMemory` choice to FTS internally to prevent OOM.
- **`Settings.optimisedFor`** — keeps the existing `isTV` device-class checks. They're used for timing/buffer decisions (cache seconds, timeouts) which don't lie to the user.
- **Other engine-side overrides** (`hwDecode` for TV, `preWarmOnFocus` for TV, etc.) — out of scope for this fix. Same pattern (move to settings layer) would apply, but shipping one fix at a time per the user's review-driven approach.
- **`safeMode` and `debugLogging` defaults** — explicit user choices; not auto-set.

## Risks

- **First-install on a 1.9 GB device**: `searchMethod` is now `likeSubstring` instead of `inMemory`. The `_searchLike` path is a SQL scan with no FTS triggers and no cache build — for low-RAM devices, this is faster end-to-end (the in-memory cache was never built). No risk of OOM, no slower insert path.
- **First-install on a 4 GB phone**: `searchMethod` stays `inMemory` (the recommended default). No change.
- **Upgrade from a prior version**: persisted `searchMethod` is preserved. The auto-set only fires on first install (no persisted key). Existing user choices are honored.
- **The threshold is `< 2300`, so a 2300 MB device gets `inMemory` (not auto)**: matches `ChannelSearchCache.cacheSkipped` (also `< 2300`). The boundary case is consistent.

## Followups (not in this fix)

- **`Settings.hwDecode` for TV** — same pattern (engine overrides at `mpv_engine.dart:567-568`; user setting says `true` but actual is `hwdec=no`). Move detection to settings layer, auto-set on first run.
- **`Settings.preWarmOnFocus`** — `Settings.optimisedFor` already encodes `isTV ? true : false`; could be auto-applied on first run.
- **All `Settings.optimisedFor` timing/buffer fields** — `liveCacheSecs`, `vodCacheSecs`, `openTimeoutSecs`, etc. Already encoded; could be auto-applied on first run instead of requiring the user to call `optimisedFor` (rare in practice).
