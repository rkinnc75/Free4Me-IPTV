## 1. Where the silent-stall arc actually stands

The recovery logic for a frozen live channel has failed three times, always for the same reason: **we keep trying to infer, in Dart, why playback stopped.**

- **fix747** assumed a silent stall keeps `playing=true`. The S938U wedge reports `playing=false`, so the watchdog bailed — *and refreshed its own baseline every 3s*, making it structurally unable to ever fire. (ABC30 Fresno froze >2 min.)
- **fix749** keyed on pause *intent* instead (`_pauseRequested`, set at 5 call sites). But intent cannot see an **OS-induced** pause, so a phone call would have reconnected a live channel. Withdrawn.
- **Cache depth** (`demuxer-cache-duration`) was tested against the field log and **rejected** — it reads `0` during a wedge *and* during ordinary non-playing samples.
- **`cache-speed`** was measured in fix750 and is **also dead**: during a real buffering stall it froze at a stale non-zero value (`676011`) instead of dropping to 0. It is not a liveness signal.

Every attempt added another guess to an unbounded allowlist of "reasons playback might legitimately stop" (user, OS focus, Bluetooth, doze, media_kit internals…). Each missing entry is a new bug.

## 2. The property that may end the arc

mpv already tracks the distinction we keep hand-rolling:

| Property | True when… |
|---|---|
| `core-idle` | not producing frames — **paused OR wedged** (cannot discriminate) |
| **`pause`** | playback was **actually paused** — by the user, by us, or by the OS/media-session layer |

If `pause` behaves as documented, the stall rule becomes simply: **position frozen + `core-idle=yes` + `pause=no` → wedge.** Every pause cause is exempt automatically, *including causes we could never enumerate*. The real fix then **deletes** `_pauseRequested` and its five instrumented call sites rather than adding them.

**This is a hypothesis with two load-bearing unknowns that must be measured before any watchdog is rewritten:**

- **(a)** Does an **OS/audio-focus steal** (incoming phone call, another app taking audio) set `pause=yes`? If Android stops playback *without* telling mpv, `pause` stays `no` — and fix749's spurious-reconnect regression comes straight back. **This is the one that kills the design if it fails.**
- **(b)** Is `pause=no` during the **wedge**? If a wedge reads `pause=yes`, it is indistinguishable from a pause and this direction is dead.

`getProperty` accepts any mpv property name (verified), so this is a one-key addition.

## 3. What this changes

One property (`pause`) added to the existing debug STATS poll. Same 1s tick, full-screen + debug only. The property map is built by zipping `keys[i]: values[i]` and read **by string key** in both consumers (verified), so inserting a key cannot corrupt existing fields. **No watchdog, no reconnect, no player behavior is touched.**

## 7. Data collection (the point of this fix — two captures, one session)

Enable the debug overlay. Every STATS line now ends with `pause="…"`.

**Capture A — the audio-focus steal (this is the decisive one).**
While a live channel is playing normally, trigger an OS interruption: **have someone call the phone**, or start a video/music in another app, for ~20 seconds. Then return to the app.
*Question answered: does `pause` flip to `yes`? (If it stays `no` while playback stops, the design is dead — better to learn it here.)*

**Capture B — the wedge.**
Play until a channel freezes on its own (frozen frame, position stuck, no buffering spinner — as on 2026-07-14 morning). **Leave it frozen ~60s** so several samples land.
*Question answered: does `pause` read `no` throughout? (It must, or a wedge looks like a pause.)*

Note: an ordinary **buffering stall** (spinner, `buffering=true`, existing watchdog reconnects at 12s) is NOT the wedge and is not useful here — the wedge is the silent one with no spinner and no recovery.

Send the log. Both answers together determine whether the real fix gets written or the approach is discarded.
