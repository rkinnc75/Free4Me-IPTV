# fix677 — Scheduled Recording never starts: insertRecording returned id 0 (pooled-connection last_insert_rowid)

**Build:** `3.0.19+677` · **Base tag:** `v3.0.18` (fix676) · **Ship order:** after 676
**Files touched:** `lib/backend/sql.dart`, `test/recording_insert_rowid_test.dart` (new), `lib/whats_new_modal.dart`, `pubspec.yaml`
**This is the real fix** for the SR-never-records bug (fix675/676 were the diagnostics that found it).

---

## Root cause (confirmed on device, not hypothesised)

The fix676 `[SRDBG]` trace on the Samsung showed:

```
[SRDBG] SR-CB: entered recordingAlarmCallback id=0
[SRDBG] SR-CB: got row id=0 rec=NULL
```

So: the alarm fires, the callback runs, plugins/DB work in the isolate — but it looks up recording **id 0** and finds nothing (`getRecordingById(0)` → null), logs "not found", returns. No capture. Row stays `scheduled`. This matches every symptom across the last several tests (notification could appear, but never a file, row never leaves `scheduled`).

**Why the id is 0:** `Sql.insertRecording` did the INSERT and then read the id in two separate calls:

```dart
await db.execute('INSERT INTO recordings ...');
final row = await db.getAll('SELECT last_insert_rowid()');   // <-- separate call
```

`DbFactory.db` is a **`sqlite_async` `SqliteDatabase`, which pools connections**, and `last_insert_rowid()` is **per-connection**. The `SELECT` could resolve on a *different* pooled connection than the one that ran the INSERT — one that never inserted anything — returning **0**. On the Samsung it did so consistently. That 0 flowed straight into `AndroidAlarmManager.oneShotAt(when, id=0, …)`, so the callback was handed 0. `sql.dart` already documents this pool hazard (the note near `_getAllHinted`: `db.execute` routes through the reused write connection but `getAll` does not).

The recordings schema is fine (`id INTEGER PRIMARY KEY AUTOINCREMENT`, migration 44). The capture engine (fix668) is fine — it was simply never reached, because no recording ever had a valid id.

## Fix

Run the INSERT and its `last_insert_rowid()` read inside **one `db.writeTransaction`**, pinning both to a single connection — the exact idiom already used by `insertSource` and `insertChannel` elsewhere in `sql.dart`. Now `insertRecording` returns the true row id, the alarm is scheduled with it, and the callback finds the row.

Adds `test/recording_insert_rowid_test.dart`: seeds the verbatim migration-44 `recordings` DDL via raw sqlite3 FFI and asserts an INSERT allocates a non-zero, monotonically increasing rowid that the row is then findable by — documenting the contract so a revert to the two-call form is caught in review. (sqlite_async's pool isn't exercisable in the sandbox; the test guards the invariant `insertRecording` must honour.)

The fix675/676 `[SRDBG]` diagnostics are intentionally LEFT IN for one confirming on-device pass; a later fix strips them once SR is verified end-to-end.

+// fix677: guards the insertRecording rowid contract.
+//
+// The device bug: Sql.insertRecording did `db.execute(INSERT)` then a SEPARATE
+// `db.getAll('SELECT last_insert_rowid()')`. Under sqlite_async's connection
+// pool those two statements can land on DIFFERENT connections, and
+// last_insert_rowid() is per-connection — so the read returned 0 for a row that
+// was inserted on another connection. The 0 became the alarm id, and
+// recordingAlarmCallback(0) then found no row (confirmed on device by the
+// fix676 [SRDBG] trace: "got row id=0 rec=NULL"). Every scheduled recording
+// died before capture.
+//
+// The fix runs the INSERT and its last_insert_rowid() read inside ONE
+// writeTransaction (single connection). sqlite_async's pool isn't exercised in
+// the sandbox, so this test asserts the underlying SQLite contract against the
+// verbatim `recordings` DDL: an INSERT that omits the INTEGER PRIMARY KEY
+// AUTOINCREMENT column allocates a non-zero rowid, and last_insert_rowid() read
+// on the SAME connection returns it, monotonically increasing across inserts.
+// If someone reverts insertRecording to the two-statement form, the reviewer
+// still has this documenting the id must be captured on the inserting
+// connection.
+import 'package:flutter_test/flutter_test.dart';
+import 'package:sqlite3/sqlite3.dart' as s3;
+
+bool _sqliteAvailable() {
+  try {
+    s3.sqlite3.openInMemory().dispose();
+    return true;
+  } catch (_) {
+    return false;
+  }
+}

+      markTestSkipped('sqlite native lib unavailable in this environment');
