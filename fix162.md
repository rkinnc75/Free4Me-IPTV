# fix162.md — Playback analyzer "does nothing": session metrics never accumulate correctly + data gate too high (verified against source)

> **Version:** Free4Me-IPTV 1.23.3+160 build.
> **Report:** Tapping "Analyze playback & suggest settings" appears to do
> nothing. **Evidence:** `free4me_log_1780153065764.txt` — a 3m37s session
> (10:52:08→10:55:45), 2 streams. Debug logging IS on; playback markers present.
>
> **What's actually happening:** the analyzer DID run and showed the "Not enough
> data yet" dialog — the gate requires **≥20 min AND ≥3 streams** and this
> session is 3 min / 2 streams. So it's "working," but it feels broken, and
> there are TWO real bugs that prevent the rolling history from ever clearing
> the gate. All three issues fixed here.

---

## Root causes (verified in source)

**Bug 1 — `sessionStart` is permanently `DateTime.now()`; rows can't dedup → the
SAME session gets re-inserted as NEW rows on every app-pause, corrupting the
aggregate.**
`playback_analyzer.dart`:
```dart
final m = PlaybackMetrics(DateTime.now());      // line 113 — always "now"
...
if (m.sessionStart == DateTime.now() && firstTs != null) {
  // retroactively set session start to first log line   <-- EMPTY no-op
}
```
`sessionStart` is `final` and is set to `DateTime.now()` at construction; the
block that was supposed to set it to the log's first timestamp does NOTHING.
The capture hook (`main.dart` on `AppLifecycleState.paused`) calls
`insertPlaybackMetrics`, which dedups via
`DELETE FROM playback_metrics WHERE session_start = ?`. Because `session_start`
is "now" (different every capture), **the dedup never matches**, so each pause
inserts a fresh row for what is really the same ongoing session →
double/triple-counting and a meaningless rolling history.

**Bug 2 — data gate is high and the "not enough data" outcome reads as
"nothing happened."** `hasSufficientData => totalMinutes >= 20 && totalStreams
>= 3`. A user testing the feature with a few short sessions never clears it, and
the terse dialog feels like a no-op.

**Bug 3 — the feature is effectively undebuggable.** The handler logs nothing
about its own run; the metrics insert logs only on success. When it bails at the
gate there's no trace, so "it does nothing" can't be distinguished from a crash.

# Fix 162.1 — Set `sessionStart` to the real first timestamp (enables correct dedup/accumulation)

**File:** `lib/backend/playback_analyzer.dart`

Make `sessionStart` mutable and assign it from the parsed first timestamp.

**Field + ctor (lines 7, 26):**
```dart
class PlaybackMetrics {
  final DateTime sessionStart;
  ...
  PlaybackMetrics(this.sessionStart);
```
**Change to:**
```dart
class PlaybackMetrics {
  DateTime sessionStart;            // fix162: settable so we can pin it to the
                                    // log's first timestamp (stable dedup key)
  ...
  PlaybackMetrics(this.sessionStart);
```

**In `_parseSession`, replace the no-op block (lines ~120-122):**
```dart
      if (m.sessionStart == DateTime.now() && firstTs != null) {
        // retroactively set session start to first log line
      }
```
**With:** (remove that block entirely; instead set it once where we finalize
firstTs/lastTs at the end, lines ~202-205)
```dart
    if (firstTs != null && lastTs != null) {
      m.sessionStart = firstTs;                       // fix162: real session start
      m.sessionMinutes = lastTs.difference(firstTs).inSeconds / 60.0;
    }
```
Now `session_start` is the session's actual first-log epoch. Re-capturing the
same ongoing session (app paused multiple times) produces the SAME
`session_start`, so `insertPlaybackMetrics`'s
`DELETE WHERE session_start = ?` correctly REPLACES the row instead of adding
duplicates — the row grows with the session, and distinct sessions accumulate
as distinct rows.

> `parseLatestSession`'s fallback `PlaybackMetrics(DateTime.now())` (for an
> empty log) is fine — it has `streamsOpened == 0` and is never inserted (the
> caller guards `if (m.streamsOpened > 0)`).

# Fix 162.2 — Lower the data gate + clearer "keep watching" messaging

**File:** `lib/backend/playback_analyzer.dart`

Drop the threshold so the feature is usable after a realistic amount of viewing,
while still requiring enough to be meaningful.

**Current (lines 70-71):**
```dart
  bool get hasSufficientData =>
      totalMinutes >= 20 && totalStreams >= 3;
```
**Change to:**
```dart
  bool get hasSufficientData =>
      totalMinutes >= 10 && totalStreams >= 2;   // fix162: lower bar, still meaningful
```

**File:** `lib/settings_view.dart` — update the "Not enough data yet" dialog text
to be encouraging and show progress toward the (new) bar:

**Current content text:**
```dart
            'Need at least 20 minutes of logged playback across 3+ '
            'streams. Current: '
            '${agg.totalMinutes.round()} min, ${agg.totalStreams} streams.'
            '\\n\\nWatch a few more channels and try again.',
```
**Change to:**
```dart
            'Building a picture of your playback. So far: '
            '${agg.totalMinutes.round()} of 10 minutes, '
            '${agg.totalStreams} of 2 streams.\\n\\n'
            'Keep watching normally — this analyzes in the background and '
            'will have suggestions once there\\'s enough history. '
            'Check back after a bit more viewing.',
```

> Reframes the dialog from "you failed a threshold" to "it's working, keep
> going." Shows progress toward the bar so the user knows it's accumulating.

# Fix 162.3 — Log the analyzer's own run (make it debuggable)

**File:** `lib/settings_view.dart`

Add a couple of `AppLog.info` lines in `_runPlaybackAnalysis` so the log shows
what the feature decided. Cheap, and it turns "it does nothing" into a
diagnosable trail.

After the snapshot/aggregate (just before the `hasSufficientData` check):
```dart
    AppLog.info('PlaybackAnalysis: aggregate '
        'minutes=${agg.totalMinutes.round()} streams=${agg.totalStreams} '
        'sessions=${agg.sessionCount} sufficient=${agg.hasSufficientData}');
```
After computing recs:
```dart
    AppLog.info('PlaybackAnalysis: ${recs.length} recommendation(s)');
```

> Now a log review shows exactly why a given tap produced "not enough data,"
> "healthy," or N suggestions — no more guessing.

## Test plan

### Accumulation across sessions (the core fix)
1. Logging ON. Watch a channel ~5 min, background the app (triggers capture),
   reopen, watch another channel ~5 min, background again.
2. Run "Analyze playback".
3. **Expected log:** two `Sql.insertPlaybackMetrics` lines with DIFFERENT
   `session=` timestamps (distinct sessions), and `PlaybackAnalysis: aggregate
   minutes=~10 streams=2 sufficient=true`. The analyzer proceeds past the gate.

### No duplicate-counting within one session
1. Watch one channel, background/foreground the app 3 times without changing
   channel, then analyze.
2. **Expected:** the `playback_metrics` table has ONE row for that session
   (same `session_start`, updated), not three. Aggregate minutes reflect real
   elapsed time, not 3×.

### Gate + messaging
1. With <10 min logged, tap Analyze.
2. **Expected:** progress dialog "So far: N of 10 minutes, M of 2 streams. Keep
   watching…" — reads as in-progress, not failure.

### Suggestions appear
1. Accumulate ≥10 min / 2 streams on a connection that rebuffers.
2. **Expected:** `PlaybackAnalysis: N recommendation(s)` in the log and the
   review dialog with Apply all.

### Healthy
1. ≥10 min / 2 streams, clean playback.
2. **Expected:** "Playback looks healthy," `0 recommendation(s)` logged.

## Notes for the implementer
- **162.1 is the core bug** — `sessionStart` was hardwired to `DateTime.now()`
  (the "retroactively set" block was an empty no-op), so the dedup key was never
  stable and the rolling history mis-counted. Pinning it to the first log
  timestamp fixes accumulation.
- **Verified against actual source** (this build: analyzer present, capture hook
  wired in main.dart, insert logs on success).
- **162.2** lowers the bar to ~10 min / 2 streams and reframes the dialog so the
  feature stops feeling like a no-op during accumulation.
- **162.3** makes it debuggable — future "it does nothing" reports are
  answerable from the log.
- Phone/TV identical — no device gating, by design.
- **Files:** `playback_analyzer.dart`, `settings_view.dart` (+ pubspec +
  fix162.md). Build `1.23.4+162`, tag `v1.23.4`, even = phone.
