## Bug
On a fresh install, the welcome/setup screen offered only **"Import settings backup"** (a local FILE via the picker, `_importBackup` → `SettingsIo.importFromFile`). The QR / Wi-Fi push flow — the same `ExportServer` LAN portal the running app exposes in Settings — was unreachable until AFTER setup completed. So a brand-new device couldn't be provisioned by uploading a backup from another device's QR; the user had to first finish manual source entry, then import. The report: "first install import doesn't allow QR loading of the settings/sources files."

## Fix
Add a **"Receive via QR / Wi-Fi"** button to the first-run welcome screen (only when `showAppBar == false`, i.e. genuine first run — matching the existing file-import affordance; not shown when Setup is opened from Settings → Add Source, where Backup & Restore already exists). It runs a new `_receiveViaQr()`:
- Starts a **receive-only** `ExportServer(const [], …)` — empty export items means the portal serves the import form only, no export payloads.
- Shows the QR code + LAN URL(s) in a dialog (reusing `qr_flutter`'s `QrImageView`, same package Settings uses).
- `onImportSources` → `SettingsIo.importSourcesOnly(bytes)` (the same call Settings' portal uses — carries sources + EPG + favorites + category state + positions per fix355/358).
- On a successful import (n > 0): stop the server, run `showSourcesRefreshDialog`, then `navigateToHome` — identical landing to the file-import path.
- Server start failure and "closed without importing" are handled gracefully (dialog / stay on welcome).

No backend changes — reuses `ExportServer` and `importSourcesOnly` unchanged; the token/auth (fix347) and source-level field carry (fix358) apply automatically.
