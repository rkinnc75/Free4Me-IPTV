# fix616 — Preserve the log across a bad refresh: move the clear-on-version marker out of db.sqlite

**Target:** v2.2.27+616 (diff against live tag **v2.2.26**, verified `git ls-remote --tags`)
**Apply check:** `git apply --check` PASSED on a fresh v2.2.26 clone.
**Analyze:** run on the build machine — no Flutter toolchain in the authoring sandbox, so the 0-issue gate is NOT asserted here. (Manual: raw paren counts balanced original 156/156 → patched 170/170, +14/+14; braces balanced; `dart:io`/`File` already imported; `HashMap` still used elsewhere; `lastLogClearedVersion` const is library-public so not flagged as unused.)

## Why
On the sms938u, a sources refresh hung at "Preparing…" and left `db.sqlite` with ~36 channels. On reopen, the **log had been cleared on a SAME-VERSION restart** (2.2.26 → 2.2.26), destroying the stuck-session log we needed.

Root cause: `maybeRotateLogOnVersionChange()` decided whether to clear by reading `lastLogClearedVersion` from the **settings table inside db.sqlite**. When the bad refresh left db.sqlite wiped, that read came back absent → the guard thought this was a new version → it cleared the log. So the log-clear was collateral damage from the same refresh problem, and worse, it wiped the evidence of that problem.

This is a diagnostics-robustness fix: the log must survive exactly the failures we most need to investigate. It does NOT fix the refresh hang itself — that is the next task, and it is now diagnosable because the log will persist.

## Fix
- The clear-on-version guard now lives in a **sidecar file** `{appSupportDir}/.log_cleared_version`, next to `app_log.txt` (which is already a plain file independent of db.sqlite). A db.sqlite wipe can no longer make the guard misread and re-clear.
- `AppLogger` gains `readClearedVersionMarker()` / `writeClearedVersionMarker()` (best-effort; a read failure means "not yet cleared", a write failure is non-fatal).
- `maybeRotateLogOnVersionChange()` reads/writes the file marker instead of the db.sqlite settings key. `clearPlaybackMetrics()` still writes db.sqlite — that's metrics DATA, not the guard, and wiping it is harmless.

## Behaviour notes
- Genuine version upgrades still clear the log once (the file marker won't match the new version) — unchanged intent.
- First boot after installing THIS build clears once (they're upgrading to it — correct).
- The old `lastLogClearedVersion` db key is left defined (library-public const, harmless); its doc comment is updated. It is simply no longer the guard.

## Files
- `lib/backend/app_logger.dart` — marker file read/write + path.
- `lib/backend/settings_service.dart` — guard switched to the file marker; doc updated.
- `pubspec.yaml`, `lib/whats_new_modal.dart` — version bump + changelog.

## Validation plan (build machine)
1. `flutter analyze` → expect "No issues found!".
2. Fresh upgrade to 2.2.27: confirm the log clears ONCE on first boot, then a same-version restart does NOT clear it (a `.log_cleared_version` file appears in the app dir).
3. Simulate the failure: with 2.2.27 installed and the marker written, wipe/replace db.sqlite (or reproduce the stuck refresh), then restart — confirm the log is PRESERVED (no "Log cleared" on a same-version boot).
4. Confirm version-change clear still works: bump to a later build, confirm one clear.
5. Then re-attempt the stuck-refresh repro and capture the now-surviving log for the real refresh investigation.
