# fix403 — TV log-clear + What's New on version change (bundled with the decode-probe label)

**Target version:** `1.35.2+403` (from `1.35.1+402`)
**Branch:** `main` · **Base:** `53abc97` (v1.35.1+402)
**Toolchain (verified):** Flutter 3.44.0 / Dart 3.12.0 (CI pin), `libsqlite3.so` symlinked.

> Bundles two file-disjoint changes per request (fix403 was authored but not shipped):
> **Part A** — the decode-state probe label (the original fix403). **Part B** — the TV
> version-trigger fix found while reviewing the smarttvpro log.

## Part B — Android TV never cleared its log / never showed What's New

**Evidence:** the TCL Smart TV Pro log (`smarttvpro`) accumulated **5 app versions**
(1.33.2 → 1.35.1) with no `--- Log cleared ---` marker anywhere, and the user confirmed
release notes never appear after a TV update.

**Root cause:** the once-per-version startup tasks — `maybeRotateLogOnVersionChange()`
(clears log + playback metrics) and the What's New check — live **only inside
`Home`, gated on `widget.firstLaunch`** (`home.dart:114` and `:145`). `firstLaunch: true`
is set in exactly one place: `main.dart:183`, the **phone** entry. On a TV
(`forceTVMode || isTV`), `main.dart:177` returns **`TvHome`** instead, which pushes
`Home(firstLaunch: false)` (`tv_home.dart:40`). So the TV path trips neither gate — the
log never rotates and What's New never shows. Both bugs, one root.

**Fix:** extract the two tasks into a shared `runVersionStartupTasks(context)` helper and
run it from the **`TvHome` root** (`initState`, `!nested`, after first frame). The phone
path is left exactly as-is. Both steps stay idempotent (`lastLogClearedVersion` /
`lastSeenVersion`), and the What's New modal already marks itself seen on close, so there
is no double-show: TV → `TvHome` runs the tasks; the `Home` it later pushes has
`firstLaunch:false` and skips them.

## Part A — decode-probe state label (original fix403)

The `DECODE[...]` probe logged `hwdec-current` raw. At the first frame the hardware
decoder may still be initializing (mpv shows the first frame in software while mediacodec
spins up), so it reads `"no"` transiently — which is exactly what was misread as a
permanent software fallback during the fix402 investigation. A pure helper
`hwdecDecodeState(tag, req, current)` now appends `decodeState="…"`:
`hardware` / `software` / `initializing(transient; trust [+3s])` / `initializing` /
`unknown`. Diagnostics only — no decode/playback behaviour changes.

+/// fix403: a human-readable decode-state label for the `DECODE[...]` probe.
+///
+/// The probe samples `hwdec-current` directly. At the first decoded frame the
+/// hardware decoder may still be initializing — mpv decodes the first frame(s)
+/// in software while mediacodec spins up, so `hwdec-current` reads "no" (or "")
+/// *transiently* even when hardware decode is about to (and does) engage. Only
+/// the settled `[+3s]` sample is authoritative.
+///
+/// This labels the sample so a first-frame "no" cannot be misread as a
+/// permanent software fallback (which is exactly what happened during the
+/// fix402 investigation). Returns one of:
+///   • "hardware"    — a mediacodec decoder is active
+///   • "software"    — settled on software (a "no" at/after the [+3s] sample,
+///                     or a "no" when software was actually requested)
+///   • "initializing(transient; trust [+3s])" — a "no" at an early probe while
+///                     a hardware decoder was requested (do not trust yet)
+///   • "initializing" — no value yet
+///   • "unknown"     — the property read failed
+String hwdecDecodeState({
+  required String tag,
+  required String req,
+  required String current,
+}) {
+  const hw = {'mediacodec', 'mediacodec-copy', 'mediacodec-ndk'};
+  if (current == '?') return 'unknown';
+  if (current.isEmpty) return 'initializing';
+  if (hw.contains(current)) return 'hardware';
+  if (current == 'no') {
+    final settled = tag.contains('+3s');
+    if (hw.contains(req) && !settled) {
+      return 'initializing(transient; trust [+3s])';
+    }
+    return 'software';
+  }
+  return current; // some other hwdec (vaapi, vdpau, …)
+}

+/// fix403: the once-per-version startup tasks — rotate (clear) the debug log and
+/// playback metrics on a version change, then show the What's New dialog.
+///
+/// These previously lived ONLY inside `Home(firstLaunch: true)` (the phone
+/// entry constructed in main.dart). The TV entry routes through `TvHome`, which
+/// pushes `Home(firstLaunch: false)`, so on a TV NEITHER task ever ran: the
+/// debug log never cleared across versions (it just accumulated every build's
+/// sessions) and release notes never appeared after an update. This shared
+/// helper lets both entry points run the same tasks.
+///
+/// Both steps are idempotent — gated on `lastLogClearedVersion` /
+/// `lastSeenVersion` inside [SettingsService] — so calling this on every entry
+/// is a cheap no-op once it has run for the current version. The What's New
+/// modal marks `lastSeenVersion` itself when it closes (see whats_new_modal),
+/// so this helper does not need to.
+Future<void> runVersionStartupTasks(BuildContext context) async {
+  await SettingsService.maybeRotateLogOnVersionChange();
+  final version = await SettingsService.shouldShowWhatsNew();
+  if (version == null) return;
+  if (!context.mounted) return;
+  await showDialog(
+    context: context,
+    builder: (context) => WhatsNewModal(version: version),
+  );
+}
