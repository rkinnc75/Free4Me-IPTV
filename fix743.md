# fix743 — Persist hwdec-probe failures per stream (skip the doomed probe)

## 1. Why (follows fix742, S938U logs 2026-07-13)

fix742 made streams like "(d) USA | YES Network" survive the hw-probe failure — but the probe itself still re-runs on EVERY open: the 16:47 log shows the identical `Could not open codec` → software-fallback cycle on the initial tune, on the buffering-watchdog reconnect at 13:21:11, and again on the next session. Hardware capability for a given stream is a property of the provider's encoding; re-discovering it every open is pure startup latency on exactly the channels that reconnect most.

## 2. Design (settled with Rich)

- **Persisted per-URL, standalone table** `hwdec_blocklist(url PK, failed_at_utc, app_version)` — NOT a channels column, because source refresh is wipe+reinsert (sql.dart fix619 machinery) and would destroy a column flag on every refresh, auto-refresh included.
- **Write** only at the moment fix742's software fallback is CONFIRMED by a decoded frame (`_codecErrorLogged && !_codecFallbackConfirmed` in the firstFrameStream listener), and only when hardware was actually requested for that open (new `MpvEngine.appliedHwdecMode` gate). Unrelated errors can never pin a channel.
- **Read** in the fix395/505 hwdec block: if the URL is blocklisted, apply `hwdec=no` and skip the probe. Log gains ` blocklist=hit (fix743)`.
- **Self-healing:** entries from a different app version, or older than 30 days (`Sql.hwdecBlocklistTtlDays`), are treated as absent — a libmpv/ffmpeg bump (pending n6.1 arc) or provider re-encode re-probes automatically. No coupling to source refresh.
- **Escape hatch:** `forceHwDecode` (fix505 advanced override) bypasses the blocklist read.
- Preview/multi-view branches never set `appliedHwdecMode`, so preview opens can neither read nor write the blocklist.
