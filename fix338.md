## Why (S24 2x2 log on fix337/1.26.53)
fix337's texture-attach detection + restart worked (textures recovered: ids climbing 3,4,6…28), but the cells THRASHED — generations climbed to 14+ over 3.5 min. Two causes interleaved:
1. The +4s texture check fired even on a cell that ALREADY had a provider open-failure in flight (log lines 150→151: `Failed to open …` immediately followed by `TEXTURE-ATTACH-FAILED` on the same cell) → double restart.
2. The texture error was classified `[permanent]`, so it restarted immediately with no backoff — hammering a connection-limited provider (Dino, 4 concurrent `.ts`).

Separately, the user asked to GUARANTEE the once-per-version debug-log clear happens whenever the What's New screen appears. Audit found the log rotation (`maybeRotateLogOnVersionChange`, key `lastLogClearedVersion`) and the What's New trigger (`shouldShowWhatsNew`, key `lastSeenVersion`) were INDEPENDENT — both fired under firstLaunch but tracked different keys, so a path showing What's New without rotating would leave the prior version's log. (Note: the log-clear mechanism is pre-existing, fix180-era — not added here.)

## What this does
1. **Self-gate the texture check (`mpv_engine.dart`):** a new `_emittedError` flag — once the engine has emitted ANY error (provider open-failure routed through a single `_emitError` sink), the +4s texture-attach check is suppressed, so a cell already restarting is not hit a second time.
2. **Backoff instead of permanent (`multi_view_cell.dart`):** `"video texture failed to attach"` is now classified TRANSIENT, so it draws on the existing 3s-cadence retry budget rather than an immediate permanent restart.
3. **Couple log-clear to What's New (`home.dart`):** `showWhatsNew()` now awaits `maybeRotateLogOnVersionChange()` before showing the dialog. Idempotent (guarded by `lastLogClearedVersion`), so "What's New shown" and "log cleared for this version" are now inseparable.
