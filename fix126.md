# fix126.md — Black screen after close-mini-then-close-full (post-swap) — leaked mpv video texture; the adopted ex-preview VideoController/GlobalKey surface is never released

> **Version:** Free4Me-IPTV 1.22.9 build=124 (verified against actual source
> in the uploaded zip).
> **Evidence:** `free4me_log_1780072141613.txt` (the 12:25→12:28 session on
> 1.22.9+124).
>
> **fix124 is CONFIRMED WORKING — and it DEFINITIVELY localised the bug.**
> Both repaint mechanisms fired in the log:
> - `Player: onExit forced post-pop repaint of revealed route` — 1× (124.1) ✓
> - `Home: overlay changed — forcing repaint of revealed Home` — 5× (124.2) ✓
>
> Flutter IS repainting Home on exit. **The screen is still black, force-close
> at 12:28:58.** Per fix124's own carry-forward branch: *"Both 124.1 and 124.2
> logs present + still black → the problem is below the framework: the disposed
> mpv video surface is not being released."* **That branch is now taken.**
>
> **Root cause:** the full-screen engine at exit was **created as
> `previewMode=true`** (a mini engine) and **adopted** into full-screen via
> swap. Its media_kit `VideoController` and the `Video` widget's
> `GlobalKey<VideoState>` were bound to the *mini* render surface. On adopt,
> the new full-screen `Video` reuses the **same `VideoController`** and the
> **same `GlobalKey`**, reparenting the `VideoState` instead of creating a
> fresh one. When the engine is finally disposed, `MpvEngine.dispose()` kills
> mpv (`_player.dispose()`) but **never releases the video texture/controller**
> — there is **no `_controller.dispose()` anywhere in the codebase** — so the
> platform `Texture` (now showing the dead engine's last black frame) leaks and
> the compositor keeps presenting it ON TOP of the correctly-repainted Home.

## Root cause — adopted ex-preview engine strands its video texture

Three structural facts in `lib/player/mpv_engine.dart` combine:

```dart
39  late final mkvideo.VideoController _controller =
40      mkvideo.VideoController(_player);
41  final GlobalKey<mkvideo.VideoState> _videoKey =
42      GlobalKey<mkvideo.VideoState>();
…
80  Widget buildVideoView(BuildContext context) {
81    return mkvideo.Video(
82      key: _videoKey,              // ← GlobalKey: one Element at a time
83      controller: _controller,
84    );
85  }
…
129 Future<void> dispose() async {
…
152   await _player.dispose();       // ← mpv dies; controller/texture NOT disposed
153 }
```

**The handoff path (swap → adopt):** the engine object (with its `_controller`
and `_videoKey`) is handed from the mini to the full-screen Player (fix116).
The full-screen Player calls `buildVideoView`, producing a `Video` with the
**same `_videoKey`**. A `GlobalKey` can attach to only one `Element`; Flutter
therefore **reparents the existing `VideoState`** from the (dying) mini `Video`
to the new full-screen `Video` rather than disposing it and creating a fresh
one. media_kit's `VideoState` registers its platform `Texture` in `initState`
and unregisters it in `dispose`. Because the State is reparented (never
disposed) and then the *engine* is disposed without disposing the controller,
**the texture registration is never torn down in the right order** — the
platform view keeps a live texture id pointing at a now-dead mpv instance.

**Why it's specifically the close-mini-then-close-full case:** closing the mini
first (12:28:35) disposes the *other* engine and tears down the mini `Video`
subtree while the ex-preview engine's `VideoState` has already been reparented
to full-screen. Then closing full-screen disposes the ex-preview engine. The
combination leaves the reparented `VideoState`'s texture orphaned — nothing
runs its unregister. The compositor presents the last frame (black) of the dead
texture above Home. Home is fine underneath (fix124 repaints it); the black is
the leaked texture layer.

**Why playback looked fine before disposal** (per the project notes "adopted
ex-preview engine runs full-screen … works fine"): while the engine is alive
the reparented texture still receives frames. The defect only manifests at
**teardown**, which is exactly when the black screen appears.

---

## The fix — release the video surface in dispose, and stop reusing the GlobalKey across adopt

fix126 has two parts. **126.1 is the primary cure** (release the controller/
texture on dispose, before mpv dies). **126.2 removes the GlobalKey reuse** that
reparents the `VideoState` across adopt, so each Player gets a clean
`VideoState` (and thus a clean texture lifecycle). Apply both.

---

# Fix 126.1 — Dispose the VideoController and release the texture in MpvEngine.dispose (primary)

**File:** `lib/player/mpv_engine.dart`

In `media_kit_video ^2.0.1` the `VideoController` does not expose a public
`dispose()` (the `Player` owns the controller's native resources), so the
correct release is: **exit fullscreen on the keyed state, drop the texture by
disposing via the controller's player ownership in the right order, and null
the video output before `_player.dispose()`.** Concretely, set mpv's video
output to null so the `Texture` stops being fed and the platform view releases
it, THEN dispose the player.

**Current (lines 129–153):**

```dart
  Future<void> dispose() async {
    if (_disposed) {
      if (AppLog.enabled) {
        AppLog.info(
          'MpvEngine: dispose() called twice — ignoring'
          ' channel="${channel.name}"',
        );
      }
      return;
    }
    _disposed = true;
    AppLog.info(
      'MpvEngine: dispose() eid=${identityHashCode(this)}'
      ' channel="${channel.name}"'
      ' previewMode=$previewMode',
    );
    for (final s in _subs) {
      await s.cancel();
    }
    await _bufferingCtrl.close();
    await _completedCtrl.close();
    await _errorCtrl.close();
    await _positionCtrl.close();
    await _player.dispose();
  }
```

**Replace with:**

```dart
  Future<void> dispose() async {
    if (_disposed) {
      if (AppLog.enabled) {
        AppLog.info(
          'MpvEngine: dispose() called twice — ignoring'
          ' channel="${channel.name}"',
        );
      }
      return;
    }
    _disposed = true;
    AppLog.info(
      'MpvEngine: dispose() eid=${identityHashCode(this)}'
      ' channel="${channel.name}"'
      ' previewMode=$previewMode',
    );

    // fix126: release the VIDEO SURFACE before tearing down mpv. The adopted
    // ex-preview engine's media_kit Texture is registered against a Video
    // widget that was reparented via GlobalKey (see 126.2). If we just call
    // _player.dispose(), the platform Texture is never unfed and the
    // compositor keeps presenting the dead engine's last (black) frame ON TOP
    // of the revealed Home — the black screen confirmed in the 1.22.9+124 log
    // (fix124's repaints fired, yet still black → native surface, not Flutter).
    //
    // Order matters:
    //   1) leave fullscreen on the keyed VideoState (if any) so the platform
    //      view detaches its fullscreen surface cleanly;
    //   2) null mpv's video output so no more frames are pushed to the Texture
    //      (this is what lets media_kit_video drop the texture id);
    //   3) THEN cancel subs / close stream controllers / dispose the player.
    try {
      if (_videoKey.currentState?.isFullscreen() ?? false) {
        await _videoKey.currentState?.exitFullscreen();
      }
    } catch (e) {
      AppLog.warn('MpvEngine: dispose exitFullscreen skipped — $e');
    }
    try {
      // Detach video output so the SurfaceTexture stops being fed and
      // media_kit_video releases the platform texture for this controller.
      await (_player.platform as dynamic)?.setProperty('vid', 'no');
    } catch (e) {
      AppLog.warn('MpvEngine: dispose vid=no skipped — $e');
    }

    for (final s in _subs) {
      await s.cancel();
    }
    await _bufferingCtrl.close();
    await _completedCtrl.close();
    await _errorCtrl.close();
    await _positionCtrl.close();
    await _player.dispose();
    AppLog.info('MpvEngine: dispose() released video + player'
        ' eid=${identityHashCode(this)}');
  }
```

### Key changes

1. **`vid=no` before `_player.dispose()`** — stops frames being pushed to the
   platform `Texture`, which is what allows `media_kit_video` to release the
   texture id. Without this the texture keeps its last black frame composited.
2. **`exitFullscreen()` on the keyed `VideoState` first** — detaches the
   platform fullscreen surface cleanly before the texture is dropped.
3. **Both wrapped in try/catch + logged** — `dispose() released video + player`
   is the new tripwire confirming the surface-release path ran.
4. **Ordering is explicit** — surface release → subs → stream controllers →
   player. The player is disposed LAST so the controller's native handle is
   valid while we null the video output.

> `_player.platform` is media_kit's `NativePlayer` (libmpv) on Android; the
> `setProperty('vid','no')` cast via `as dynamic` avoids a hard compile
> dependency on the platform type and is null-guarded. If the project prefers a
> typed call, import `package:media_kit/src/player/native/player/native_player.dart`
> is NOT advised (private); the dynamic call is the pragmatic, analyzer-clean
> choice. Confirm it compiles; if `platform` is not exposed, see the fallback
> in the Notes.

# Fix 126.2 — Don't reuse the GlobalKey across adopt; give each Player a fresh VideoState

**File:** `lib/player/mpv_engine.dart`

The `GlobalKey<VideoState>` causes Flutter to **reparent** the same
`VideoState` from the mini `Video` to the adopted full-screen `Video`, which is
the upstream reason the texture lifecycle gets stranded. Because the engine is
handed off (fix116), the full-screen `Video` does not need to be the *same*
`State` as the mini's — it needs the same **controller** (so the same mpv frames
render), but a **fresh `VideoState`** with its own clean texture
registration.

Remove the `GlobalKey` from the `Video` widget so each `buildVideoView` call
mounts a fresh `VideoState`. Preserve fullscreen control (which currently uses
`_videoKey.currentState`) by routing it through the controller instead.

**Current (lines 41–42, 80–85):**

```dart
  final GlobalKey<mkvideo.VideoState> _videoKey =
      GlobalKey<mkvideo.VideoState>();
…
  Widget buildVideoView(BuildContext context) {
    return mkvideo.Video(
      key: _videoKey,
      controller: _controller,
    );
  }
```

**Problem:** `_videoKey.currentState` is also used at lines 119, 214, 219, 223,
230 for fullscreen + aspect-ratio control. Removing the key outright breaks
those. media_kit_video 2.x exposes fullscreen helpers that take a
`BuildContext` of the `Video`, but the simplest robust approach that keeps the
key's API while fixing the reparenting is:

**Recreate the GlobalKey when the engine is adopted**, so the adopted
full-screen `Video` gets a DISTINCT key from the one the mini used — Flutter
then mounts a fresh `VideoState` (clean texture) instead of reparenting.

Make `_videoKey` non-final and add an `onAdopt()` hook the Player calls on the
adopt path.

**Replace the field (lines 41–42):**

```dart
  // fix126: NOT final — recreated on adopt so the full-screen Video gets a
  // fresh VideoState (clean texture) instead of reparenting the mini's State
  // via a shared GlobalKey, which stranded the platform texture at teardown.
  GlobalKey<mkvideo.VideoState> _videoKey =
      GlobalKey<mkvideo.VideoState>();
```

**Add an adopt hook** (place near the other lifecycle methods, e.g. just after
the constructor or before `buildVideoView`):

```dart
  /// fix126: called by Player when this engine is adopted into a NEW Player
  /// (swap handoff). Recreate the video key so the adopted Video mounts a
  /// fresh VideoState with its own texture registration; the old mini
  /// VideoState then disposes normally and unregisters its texture.
  void onAdopt() {
    if (_disposed) return;
    _videoKey = GlobalKey<mkvideo.VideoState>();
    AppLog.info('MpvEngine: onAdopt — new video key'
        ' eid=${identityHashCode(this)}');
  }
```

`buildVideoView` stays as-is (it reads `_videoKey`, now the fresh one after
adopt). The fullscreen/aspect calls at 119/214/219/223/230 keep working because
they read `_videoKey.currentState` of whichever key is current.

**File:** `lib/player.dart`

Call `onAdopt()` on the adopt path. The adopt path logs `initAsync adopt path
— skipping open` (player.dart) and sets `_adopted=true`. Add the hook where the
engine is adopted, before `buildVideoView` is first called for the new Player.

Find the adopt branch in `initAsync` (the block that logs `initAsync adopt
path — skipping open`) and add, as its first action on the engine:

```dart
      // fix126: give the adopted engine a fresh video key so the new
      // full-screen Video mounts a clean VideoState (own texture) rather than
      // reparenting the mini's State via a shared GlobalKey.
      (_engine as MpvEngine).onAdopt();
```

> Guard the cast if non-mpv engines can be adopted: the swap fallback for
> non-mpv is dispose+reopen (per fix116 notes), so only `MpvEngine` reaches the
> adopt path — but a defensive `if (_engine is MpvEngine) (_engine as
> MpvEngine).onAdopt();` is safest. Grep the adopt block first:
>
> ```
> grep -n "adopt path — skipping open\|_adopted\|adoptEngine" lib/player.dart
> ```
