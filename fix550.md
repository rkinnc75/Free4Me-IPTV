# fix550 — Report-issue dialog: fix Android TV D-pad focus trap (can't reach Cancel/Submit)

**Version:** 2.0.50 (`2.0.50+550`)
**Base:** v2.0.49 (fix549). Independent of fix549's files — touches ONLY
`lib/settings_view.dart` — so it also applies clean directly on v2.0.48 if the
queue order shifts. Verified both standalone on v2.0.48 and stacked after fix549.
**Branch from:** latest version tag (v2.0.49 once tagged; else v2.0.48).

---

## Bug

On Android TV, the in-app **Report an issue** dialog was unusable by remote: the
D-pad could not move past the multi-line **Details** field to reach **Cancel** or
**Submit**. The report could be typed but never sent.

**Root cause:** `_showReportIssueDialog` used raw `TextField`s. Flutter's stock
`TextField` absorbs every arrow key for caret movement once focused. The Details
field is `maxLines: 5`, so D-pad DOWN moved the text caret between the five lines
instead of moving focus to the action buttons — a focus trap. (The Subject field
is single-line so it was less obviously stuck, but it too never yielded focus.)

The codebase already solved this exact problem with `DpadFocusEscape` /
`DpadTextField` (`lib/widgets/dpad_text_field.dart`), used by every other TV text
input (search, EPG URL, channel picker, EPG mapping). The report dialog simply
never adopted it.

## Fix

1. Wrap both `TextField`s in **`DpadFocusEscape`** (already imported in
   `settings_view.dart`). It intercepts arrow up/down and calls
   `previousFocus()` / `nextFocus()` before the field sees the key, so focus can
   leave the field. Chosen over swapping to `DpadTextField` specifically to
   **preserve `maxLength: 100` + `textInputAction.next` on Subject and
   `maxLines: 5` on Details exactly** — touch soft-keyboard and char counter
   unchanged.
2. **`autofocus: true` on the Submit `FilledButton`.** The dialog now opens with
   Submit focused; the user D-pads UP into the fields to type, then one D-pad
   DOWN returns to Submit. Matches the proven primary-action-focus pattern from
   the working EPG-URL dialog in the same file.

No reordering of Cancel/Submit (touch convention preserved). Touch behavior is
unchanged throughout.

## On-device check after CI

1. On the Onn (remote only), open Settings → Report an issue.
2. Dialog opens with **Submit** highlighted. D-pad UP reaches Details, then
   Subject; type via the on-screen keyboard.
3. From Details, **D-pad DOWN moves focus out of the field** (no longer stuck on
   caret) and reaches Cancel / Submit. Submit sends; Cancel dismisses.
4. Touch a phone: fields, char counter, and 5-line Details still behave as before.
