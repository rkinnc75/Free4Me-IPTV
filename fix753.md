## 1. The failure (measured — S938U, ABC30 Fresno, logs 20260714)

A live channel freezes on one frame — no spinner, no error — and sits frozen until the user exits (>2 min observed). Trigger, captured end-to-end at 17:13: the provider resets PTS (position jumps 36002ms → **-10031ms**), mpv chases the discontinuity back up to ~2136ms, then wedges at the seam: `playing=false`, `pause=yes` (mpv self-sets it), position frozen, **buffer FULL (95.6s cached)**, `cacheSpeed=0`. Confirmed on **both providers** (A3000 `478170.ts` and Trex `PRIME| ABC30 ᴿᴬᵂ`) — a shared upstream feed defect, not a one-provider quirk.

fix747's watchdog cannot fire on this: it bails on `!_enginePlaying` — and the wedge reports `playing=false` — while the bail branch ALSO refreshes the baseline every 3s, erasing its own evidence forever.

## 2. Why intent, not sensors (five falsified theories)

Each candidate discriminator was tested against field data and killed:

| Theory | Killed by |
|---|---|
| mpv `pause` flag (`no`=wedge) | wedge reads **pause=yes** — mpv self-pauses at the seam |
| `eof-reached` | 30s of `eof=no` at wedge onset — identical to a pause |
| `demuxTime` advancing during pause | advances in a pause, yes — but frozen in ordinary buffering stalls too; and classifier runs showed frozen-demuxTime MIXED states while playing |
| `cacheSpeed==0` = starved | a settled pause also reads 0 once its buffer fills (33/75 pause samples); and the wedge buffer is FULL, so "starved" is factually wrong (F1) |
| cache depth | 0 during wedge AND during ordinary non-playing samples |

A settled user pause is **sensor-identical** to the wedge. The only reliable separator is **intent**: did we ask for this pause? (Rich's design.) Coverage is exhaustive and verified: this app has **no** MediaSession / audio_service / headset-button handling (F3) — nothing outside the app's own call sites can request a pause. And the feared OS case was measured: **a ringing call does not stop playback at all** on the S938U (2 calls → voicemail, playing=true throughout).

## 3. The fix

- **`_pauseRequested`** (primary): set at the four intentional-pause sites (user toggle, cast handoff, sleep timer, overlay handoff); cleared on user resume, DVR resume, and **every open()** (zap/reconnect), so a stale intent can never disable the watchdog.
- **`_lifecycleResumed`** (F2, required by review): stall time accumulates only while `AppLifecycleState.resumed`. An **answered** call — the one OS-pause case field data couldn't capture — backgrounds the app; this closes that class without enumerating pause causes.
- **Guard:** `!_enginePlaying` is REMOVED as a bail (F8: never consult mpv's playing/pause flags — the wedge sets both misleadingly). New exemptions: `_pauseRequested`, `!_lifecycleResumed`, `_engine.dvrActive`. Buffering/grace/cast/reconnect exemptions unchanged.
- **Position rule (measured constraint):** ANY change — forward, backward, or negative PTS resets — refreshes the baseline (`pos != _stallLastPos`, already fix747's comparison, now pinned in tests). Only a truly-unchanged position accumulates. Micro-jitter evasion is an accepted limitation (F5): measured wedges are exactly frozen for minutes.
- **Firing-time heuristic (F1, demoted from "confirmation"):** at threshold, the watchdog reads `cache-speed` **itself** via new `MpvEngine.readCacheSpeed()` (F6: STATS polling only runs with debug logging on — a debug-only watchdog would fail silently in production). `null` (unavailable/disposed) or nonzero → hold fire, keep accumulating. This only delays a false fire during the first seconds of a hypothetical unflagged pause; the intent+lifecycle gates are the real protection. Post-await revalidation (mounted/intent/lifecycle/position) before firing.
- **Adopt-path note (F4):** promote-to-fullscreen adopts an engine WITHOUT open(), so the per-open reset doesn't run there; the flag construct-initialises false, and an engine adopted mid-stability-pause would be reconnected after the threshold — acceptable, and currently the only recovery an orphaned stability pause has.
- **Known behavior (F7):** ABC30-class channels wedge every ~30–60s, so the fixed app will show a periodic auto-reconnect cycle on them instead of a permanent freeze. The reconnect counter resets on each 30s-stable window, so it never gives up. The upstream cure (mpv PTS-discontinuity tolerance) is a separate future arc.
- **Test contract rewritten:** the fix747 test pinned the falsified assumption (`'!_enginePlaying) {'`) as a literal; it now pins the corrected guard, asserts (negatively, guard-scoped) that mpv's playing flag is never consulted, and pins the intent-flag set/clear sites, the lifecycle gate, and the self-read heuristic.
