## Root cause
Migration 15 (fix178) created `channels_unique_series ON channels(source_id, series_id) WHERE series_id IS NOT NULL` — one row per series. But series PARENT tiles never set `series_id` (`xtreamToChannel` stores the provider id in `url`, `stream_id=-1`, exempt from both partial indexes). The only rows that ever matched the index were EPISODES, every one of which carries the parent's `series_id` — so `getEpisodes`' N inserts all collided on `insertChannel`'s bare `ON CONFLICT DO UPDATE`, each overwriting the last. Exactly one episode survived per series. The collapse was also the only re-fetch dedupe, so the fix must give episodes a real identity.

## What this does
- **Migration 28 (`db_factory.dart`, ceiling 27 → 28):** drop and recreate the partial index as `(source_id, series_id, url)`. Episode `url` embeds the provider's unique episode id + container extension, stable across fetches — episodes coexist, re-fetches upsert. Parents unaffected (series_id NULL, outside the partial index). Existing collapsed data (1 row/series) trivially satisfies the wider key — **no migration-time conflict, no table rewrite** (index recreate only; cheap even on the 737K-row onn).
- **Self-healing:** the per-session `refreshedSeries` guard resets each launch, so the next open of any series re-runs `getEpisodes`; the stale row upserts onto its matching episode and the rest insert cleanly. No data repair pass needed.
- **No Dart logic changes** — fetch and upsert code were correct; only the key was wrong.
- `test/series_episodes_test.dart` (NEW): permanent Rule-8 test — pre-28 schema reproduces the collapse (3 inserts → 1 row); migration SQL verbatim; post-28 re-fetch → 3 rows; second re-fetch → still 3; parent row untouched (1 row, media_type=serie); cross-series coexistence.
- `whats_new_modal.dart` — 1.29.1 changelog. `pubspec.yaml` — 1.29.1+353.
