# fix686 — Re-mux skip guard fix: convert MediaStore recordings (v4.0.2)

**Builder:** desktop (Mac). **Live base:** v4.0.1. **Ships as:** v4.0.2+686.

**Why:** on-device log (SM-S938U, v4.0.1) shows the fix685 re-mux was skipped for
every recording — the file stayed `.ts`. Tag re-verified at start and again
immediately before this diff — latest remote tag = **v4.0.1**.

---

## Root cause

fix685 wired the FFI re-mux, but `RecordingRemux._processOne` gated on:

```
!src.toLowerCase().endsWith('.ts')  →  skip
```

Captured recordings are **MediaStore** entries whose `output_path` is a
`content://` URI, which never ends in `.ts` (only the display name does). So the
guard was true for every recording and remux was skipped 100% of the time.

**Log proof (v4.0.1, id=17):**
- `id=17 capture done; status=done (remux=true, deferred to Dart)` — capture OK, 84 MB.
- `remux: id=17 skip (status=done path=content://media/external_primary/video/media/1000062227)` — the skip.
- No `remuxCreateOutput`, no `fd:` output open, no `matroska`/`mp4` mux — remux never attempted.

## Fix

Remove the `.ts`-suffix test. `RecordingStatusJournal.drain()` already calls
`process()` **only** for ids the native service flagged `"remux":true` on a fresh
capture, so `status==done` + non-null `output_path` is the correct and sufficient
guard. Container choice is codec-probed (not extension-based), and `_deleteTs`
deletes the `content://` URI — nothing downstream assumed a `.ts` suffix. No
re-remux risk: after success the row's path becomes the new mp4/mkv URI and the
id is no longer journaled with `remux:true`.

   With re-mux OFF it stays `.ts`. Both play. (No playback-regression risk —
   engine unchanged since v4.0.0.)
