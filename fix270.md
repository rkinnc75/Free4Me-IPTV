# fix270 — Release remediation: commit the correct version.json for 1.26.4 (recurring "stale" guard)

> ## ⚠️ BUILD MACHINE — APPLY VERBATIM
> GIT + version.json remediation. NO Dart/code changes. Write the version.json bytes EXACTLY, run the git commands in order. See BUILD-MACHINE-RULES.md.

> **Recurring failure:** the CI guard (`.github/workflows/release.yml` → "Verify version.json is up to date") runs `python3 scripts/update_version_json.py` on the tagged commit and fails if the committed `version.json` differs from what the script regenerates. It has now failed for 1.26.2 (fix264), and 1.26.4 (this), with "version.json on the tagged commit is stale."
> **Root cause:** on the failing tagged commits, `pubspec.yaml` + the `_changelog` entry advanced to the new version, but the committed `version.json` was left at the PREVIOUS version — i.e. version.json was regenerated locally but not actually committed into the tagged commit (or the release was cut via a path that didn't stage it). 1.26.3 (fix266) passed because that release committed version.json correctly.
> **Goal:** put the exact 1.26.4 `version.json` on `main` and move `v1.26.4` to that commit. Then a process change so this stops recurring.

## Process fix — stop this recurring (do this once)
The failures happen when version.json is not in the SAME commit that gets tagged. Two durable options:

**Option 1 (preferred): always cut releases with `scripts/build_and_release.sh`.** It regenerates version.json (step 3b) and `git add -A` + commits BEFORE pushing/tagging. Do NOT cut releases with manual `git commit-tree`/partial-add paths or `commit_and_release.sh` if that path can omit version.json.

**Option 2: add a pre-tag guard to whatever script cuts the release**, mirroring CI, so it fails locally before pushing a bad tag:
```bash
python3 scripts/update_version_json.py
git add version.json
git diff --cached --quiet version.json || git commit -m "version.json: sync for $TAG"
# now tag the commit that includes version.json
```

> RULE (also in RELEASE-PROCEDURE.md): version.json MUST be regenerated AND committed in the tagged commit. The CI guard compares the tagged commit's version.json against `update_version_json.py` output — they must be identical.
