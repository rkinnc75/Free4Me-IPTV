# fix124.md — Black screen after close-mini-then-close-full (post-swap) — RouteObserver never fires for the `home:` root route; force a repaint another way

> **Version:** Free4Me-IPTV 1.22.8 build=122 (verified against actual source
> in the uploaded zip).
> **Evidence:** `free4me_log_1780062913079.txt` (the 09:49→09:55 session on
> 1.22.8+122).
>
> **fix122 is CONFIRMED — build is unblocked** (log banner: `version=1.22.8
> build=122`). **fix120.2 is CONFIRMED LANDED** — the swap now logs
> `_swap pop+push` (line 102), no more `pushReplacement`.
>
> **The black screen STILL happens**, and the diagnostic fix120.1 added is
> now conclusive:
>
> **`Home: didPopNext` NEVER appears in the log — zero times, the entire
> session.** Per fix120's own branch logic ("ABSENT → Home not revealed"),
> Home is not being told it was revealed. But the deeper finding is *why*:
> **fix120.1's `RouteAware`/`didPopNext` mechanism is structurally incapable
> of firing for Home, because Home is mounted as `MaterialApp.home` (the
> implicit root route) and a `RouteObserver` never delivers `didPopNext` to
> the first/root route.** The subscribe call succeeds but no callback can
> ever arrive. So the intended repaint never happens → revealed root renders
> the popped Player's retained black layer → force-close.

route (Home) to repaint, because the chosen trigger can never fire.**

## Root cause — RouteObserver does not fire for the `home:` root route

The wiring fix120.1 added is *correct in isolation*:

- `main.dart:157` registers `navigatorObservers: [playerRouteObserver]`.
- `home.dart:84` `didChangeDependencies` → `ModalRoute.of(context)` →
  `playerRouteObserver.subscribe(this, route)`.
- `home.dart:94` `didPopNext()` logs + `setState`.

But Home is mounted here (`main.dart:208`):

```dart
      home: _RootPage(            // ← MaterialApp.home == the implicit "/" root route
        …
      ),
```

`_RootPage.build` returns `Home(...)` directly (no nested Navigator, no
`push`). So Home lives on the **root route** that `MaterialApp` installs at
startup.

A `RouteObserver` only delivers `didPopNext` to a subscriber whose route it
saw **pushed** (it pairs a later `didPop` of the route *above* with the
`didPopNext` of the route *below*). **The root route is never "pushed" through
the observer** — it is the initial route, installed as the navigator is built,
before/as the observer is attached. The observer therefore has no record of
the root route in its route→subscribers bookkeeping for the *previous-route*
relationship, and **`didPopNext` is never dispatched to Home.** This is a
long-standing Flutter behaviour, not a bug in our subscribe call.

Net: fix120.1 subscribes successfully (no error, no crash) but its callback is
unreachable for a `home:`-mounted screen. The repaint it was supposed to
trigger never happens.

---

## The fix — force the repaint from the side that DOES run: the Player's pop

We already have a code path that provably runs at exactly the right moment:
`Player.onExit` pops the route (`player.dart:1041`, confirmed in the log every
time as `onExit popping route`). Instead of relying on Home to notice the pop
(it can't), **drive a frame from the pop itself**, and additionally **mark Home
dirty directly** via the root navigator context so the revealed root repaints
with fresh content.

fix124 does three things, in order of decisiveness:

- **124.1 (primary):** in `onExit`, after `navigator.pop()`, request a frame
  and mark the revealed route's element tree dirty via the root navigator's
  context. This runs from the path the log proves executes.
- **124.2 (structural):** replace fix120.1's dead `RouteAware` mechanism on
  Home with a `WidgetsBinding` post-pop repaint **driven by the overlay
  controller** (which Home already listens to), so Home repaints on return
  regardless of route observership.
- **124.3 (keep as harmless diagnostic / remove):** the `RouteAware` hooks can
  stay (they cost nothing and the `didPopNext` log remains a useful tripwire if
  the app is ever restructured to push Home), but they are no longer the fix.

# Fix 124.1 — Force a frame + mark revealed tree dirty after the pop (primary)

**File:** `lib/player.dart`

**Current (around lines 1039–1041):**

```dart
    AppLog.info('Player: onExit popping route'
        ' channel="${widget.channel.name}"');
    navigator.pop();
```

**Replace with:**

```dart
    AppLog.info('Player: onExit popping route'
        ' channel="${widget.channel.name}"');
    navigator.pop();

    // fix124: the revealed route is Home, mounted as MaterialApp.home (the
    // root route). A RouteObserver never delivers didPopNext to the root
    // route, so fix120.1's Home.didPopNext can NEVER fire (confirmed: zero
    // occurrences in the 1.22.8+122 log) and Home is never prompted to
    // repaint — the compositor keeps the popped Player's black layer.
    // Force a repaint from HERE, the path the log proves runs every time:
    //   1) schedule a frame so the compositor re-rasterises the revealed tree
    //   2) mark the navigator's context subtree dirty so Home rebuilds
    WidgetsBinding.instance.scheduleFrame();
    WidgetsBinding.instance.addPostFrameCallback((_) {
      try {
        // ignore: invalid_use_of_protected_member
        (navigator.context as Element).markNeedsBuild();
      } catch (e) {
        AppLog.warn('Player: onExit post-pop repaint skipped — $e');
      }
      WidgetsBinding.instance.scheduleFrame();
      AppLog.info('Player: onExit forced post-pop repaint of revealed route');
    });
```

### Key changes

1. **`scheduleFrame()` immediately after the pop** — asks the engine for a new
   frame so the revealed route is re-rasterised rather than the retained black
   layer being shown.
2. **`markNeedsBuild()` on the navigator's context element** in a post-frame
   callback — forces the subtree under the navigator (which contains the now-
   top root route = Home) to rebuild, then schedules another frame. This is the
   repaint fix120.1 wanted, driven from a path that actually executes.
3. **Wrapped in try/catch + logged** — `Player: onExit forced post-pop repaint
   of revealed route` is the new tripwire. If the black screen ever recurs and
   THIS line is present in the log, the problem is below the framework layer
   (engine surface), not the route reveal.
4. **`navigator` is the already-captured `final navigator = Navigator.of(
   context)` from fix118** — no new context lookup, safe after the widget
   unmounts.

> `markNeedsBuild()` is a protected member; the `// ignore:` comment keeps
> `flutter analyze` clean. If the project prefers not to call a protected
> member, fix124.2 (controller-driven Home repaint) achieves the same result
> without it — apply 124.2 and you may drop the `markNeedsBuild` block here,
> keeping only the two `scheduleFrame()` calls.

# Fix 124.2 — Home repaints on return via the overlay controller (structural, robust)

**File:** `lib/home.dart`

Home already depends on `OverlayPlayerController` state changes elsewhere; the
controller is notified on `unregisterMain` (log line 09:54:56
`OverlayController: unregisterMain`) which fires exactly when the full-screen
Player exits. Use that signal to repaint Home, instead of the dead RouteAware
path.

Replace the fix120.1 `didChangeDependencies` subscribe block with a controller
listener that repaints Home when the full-screen player is torn down.

**Current (around lines 80–96):**

```dart
  @override
  void didChangeDependencies() {
    super.didChangeDependencies();
    // fix120: subscribe so we get didPopNext when the full-screen Player
    // is popped on top of us — needed to force a repaint of Home,
    // otherwise the compositor can retain the Player's black layer.
    final route = ModalRoute.of(context);
    if (route is PageRoute) {
      playerRouteObserver.subscribe(this, route);
    }
  }

  /// fix120: a route on top of Home (the full-screen Player) was popped —
  /// Home is visible again. Force a rebuild so the compositor repaints
  /// Home rather than retaining the popped Player's black layer.
  @override
  void didPopNext() {
    AppLog.info('Home: didPopNext — revealed, forcing repaint');
    if (mounted) setState(() {});
  }
```

**Replace with:**

```dart
  // fix124: fix120.1's RouteAware/didPopNext can NEVER fire — Home is
  // MaterialApp.home (the root route) and a RouteObserver does not deliver
  // didPopNext to the root route (confirmed: zero occurrences in the
  // 1.22.8+122 log). Instead, repaint Home when the overlay controller
  // signals the full-screen player was torn down (unregisterMain), which
  // fires exactly when the Player exits and Home is revealed.
  bool _overlayListenerAttached = false;

  void _onOverlayChanged() {
    // When no full-screen player and no mini is active, Home is the visible
    // surface again — force a repaint so the compositor re-rasterises Home
    // rather than retaining the popped Player's black layer.
    if (!mounted) return;
    AppLog.info('Home: overlay changed — forcing repaint of revealed Home');
    setState(() {});
    WidgetsBinding.instance.scheduleFrame();
  }

  @override
  void didChangeDependencies() {
    super.didChangeDependencies();
    if (!_overlayListenerAttached) {
      OverlayPlayerController.instance.addListener(_onOverlayChanged);
      _overlayListenerAttached = true;
    }
  }
```

> **Confirm the controller accessor.** This assumes
> `OverlayPlayerController.instance` (a singleton) with `addListener` /
> `removeListener` (it extends `ChangeNotifier` — it already calls
> `notifyListeners`, per `registerMain` / `unregisterMain` / `muting …` log
> lines). If the controller is exposed differently in this codebase (e.g. via
> an InheritedWidget or a top-level field), wire the listener to that instead —
> the goal is simply: *repaint Home whenever the overlay controller notifies.*
> Grep before editing:
>
> ```
> grep -n "class OverlayPlayerController\|instance\|notifyListeners" lib/player/overlay_player_controller.dart
> ```

**Update Home's `dispose()`** (around line 281) to remove the listener and the
now-unused RouteAware unsubscribe.

**Current:**

```dart
  @override
  void dispose() {
    playerRouteObserver.unsubscribe(this); // fix120
    _scrollController.dispose();
    searchController.dispose();
    _debounce?.cancel();
    _scanProgress.dispose();
    super.dispose();
  }
```

**Replace with:**

```dart
  @override
  void dispose() {
    // fix124: detach the overlay-controller repaint listener (replaces
    // fix120's dead RouteAware unsubscribe).
    if (_overlayListenerAttached) {
      OverlayPlayerController.instance.removeListener(_onOverlayChanged);
      _overlayListenerAttached = false;
    }
    _scrollController.dispose();
    searchController.dispose();
    _debounce?.cancel();
    _scanProgress.dispose();
    super.dispose();
  }
```

**Remove the now-dead `with RouteAware`** on the class declaration (line 45)
and the unused `playerRouteObserver` import (line 19) — leaving them is a
`flutter analyze` WARNING (unused), which is FATAL to the build gate.

**Current (line 45):**

```dart
class _HomeState extends State<Home> with RouteAware {
```

**Replace with:**

```dart
class _HomeState extends State<Home> {
```

And delete the import at line 19:

```dart
import 'package:open_tv/models/app_navigator.dart' show playerRouteObserver;
```

> If `app_navigator.dart` exports anything else Home still uses, keep the
> import but drop the `show playerRouteObserver` clause. Grep to confirm:
> `grep -n "playerRouteObserver\|app_navigator" lib/home.dart` — after the edit
> there should be ZERO references to `playerRouteObserver` in home.dart.
