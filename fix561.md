# fix561 — Diagnostic: log search D-pad focus events to find the "needs 2 presses" cause

**Version:** 2.0.61 (`2.0.61+561`)
**Base:** v2.0.60 (live CI tag). Single ```diff block, byte-exact against
v2.0.60. Verified clean on a fresh v2.0.60 clone.
**Branch from:** v2.0.60.

---

## Why this is diagnostic-only

After fix560 shipped, on-device testing found: the FIRST Up press immediately
after a cross-section escape jump only scrolls the target into view — it does
NOT move focus to a new tile. A second Up press is needed to actually move up
one row within the new section. After that, every other Up press works fine
(1 press = 1 move).

My first theory (fix561 draft, not shipped) was a frame/microtask timing race
between the escape's `requestFocus()` and the `Scrollable.ensureVisible`
animation. **That theory is now disproven**: waiting **3 seconds** between the
escape and the next Up press still requires 2 presses. A genuine timing race
would disappear with any meaningful delay; this doesn't. The cause is a real
logic bug I haven't yet pinpointed, not a frame-timing issue.

Rather than guess a third time, this ship adds precise logging at every
decision point so the next field log shows exactly what happens, frame by
frame, on that "wasted" first press:

- `ChannelTile.onKeyEvent`: logs every Up key event received — which tile,
  whether it has an `onFocusUpEscape`, its `FocusNode` hash, and whether it
  currently has focus.
- `ChannelTile`'s focus listener: logs every focus-GAIN — which tile, its
  node hash — so the log shows the exact sequence of which tile received
  focus after each press.
- `TvSearchView`'s escape closure: logs the resolved target column/node hash
  and whether that target already had focus at the moment of the call.

**Reverted:** the unverified `Duration.zero` change from the unshipped fix561
draft (restored to the original 200ms animated `ensureVisible`), since the
3-second-gap test proved shortening the animation could not have been the
real fix — no reason to keep an unverified change in.

No other behavior change.

## Verification

- `flutter analyze` → **No issues found!**
- `flutter test` → **259 passed**
- `git apply --check` → clean on a fresh v2.0.60 clone

## What to do on-device

Reproduce the "2 presses needed" sequence once (Series → Movies escape, then
the immediate next Up), then send the log. Look for: does the log show the
escape firing once and the target node correctly gaining focus, but the very
next key event still being routed to the OLD (pre-escape) node? Or does the
target node never actually report `hasFocus=true`? That distinction tells us
exactly where to fix it next.
