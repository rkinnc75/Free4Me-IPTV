# fix376 — Populate source counts on first-time Xtream Add Source

**Target version:** `1.34.5+376` (from `1.34.4+375`)
**Branch:** `main`
**Toolchain (verified in sandbox):** Flutter 3.44.0 / Dart 3.12.0 (CI pin), `libsqlite3.so` symlinked.
**Source:** external handoff `fix-handoff-xtream-first-add-counts.md`, **corrected** (the handoff's proposed code would not compile and would crash first-add — see "Correction" below).

## What & why

On a **first** Xtream add (first-run setup or Settings → Add Source), `getXtream` queued `Sql.getOrCreateSourceByName(source)` into its batched `statements` list and **never read `memory['sourceId']` back**, so `source.id` stayed `null`. The count-persistence block — `if (source.id != null) { … Sql.updateSource(source); }` (xtream.dart ~257-270) — was therefore skipped, leaving `last_live_count` / `last_movie_count` / `last_series_count` null until the user manually refreshed. `m3u.dart` (lines 43-44) already does the right thing.

**Fix:** commit the create-source statement up front (m3u pattern), read the id back, assign `source.id`, and **thread the shared `memory` map into the batched import commit** so the bulk channel inserts still resolve correctly.

## Correction to the handoff (caught in code review — would have shipped broken)

The handoff's proposed snippet referenced a `memory` variable and only changed the create call:
```dart
await Sql.commitWrite([Sql.getOrCreateSourceByName(source)], memory: memory);
source.id = int.parse(memory['sourceId']!);
```
Two problems, both verified against the real code:
1. **`getXtream` has no `memory` map.** Unlike `m3u.dart` (which declares `final memory = <String,String>{}` and threads it everywhere), `getXtream` relied on `commitWriteBatched` creating its *own* internal map. The snippet as written does not compile.
2. **Channel inserts resolve `sourceId` from that map.** `insertChannelsBulk` does `final sourceId = int.parse(memory['sourceId']!)` and writes `ch.sourceId == -1 ? sourceId : ch.sourceId` (channels are built with `sourceId: -1`, xtream.dart:538). Today the first batched closure (`getOrCreateSourceByName`) populates the batch's internal map for those inserts. **Pulling it out of the batch removes that population** — so unless we declare our own `memory` and pass it into `commitWriteBatched`, the first-add channel inserts hit `int.parse(null!)` and **crash the entire import** (strictly worse than the silent-count bug).

This fix therefore makes **three** coordinated edits, not one: declare `memory`, commit+assign up front, and add `memory: memory` to the `commitWriteBatched` call. The handoff's Open-Q3 gestured at this but its proposed code did not implement it.

Behavior note (accepted, matches m3u): the source row now commits in its own transaction ahead of the bulk write, rather than as the first closure of the single import transaction. Same tradeoff `m3u.dart` already ships.
