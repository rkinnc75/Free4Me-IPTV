# fix402 — Hardware decode engages on phones (route phone → mediacodec-copy)

**Target version:** `1.35.1+402` (from `1.35.0+401`)
**Branch:** `main` · **Base:** `fe2bf84` (v1.35.0+401)
**Toolchain (verified):** Flutter 3.44.0 / Dart 3.12.0 (CI pin), `libsqlite3.so` symlinked.

## What & why

A device log (S24/sms938u, v1.35.0) showed hardware decode requested but **not engaging**: `hwdec-req="mediacodec"` → `hwdec-current="no"` (software) on a stream the S24 decodes trivially in hardware. Settings confirm `hwDecode = true`, so it isn't a settings issue.

**Root cause:** `androidFullscreenHwdec` routed phones to surface-mode `mediacodec`. Surface mode decodes into its own Android `SurfaceTexture`, but media_kit renders through the **libmpv render API into a Flutter texture** (`vo="null"`, `gpu-context=android`) — there is **no native Surface** for surface-mode mediacodec to bind to, so mpv silently falls back to software. This is the *same* failure the app already documents for TV (`fix108`: "surface-mode `mediacodec` binds a SurfaceTexture and fails silently"), which is why every TV/Tegra class already uses `mediacodec-copy`. The phone branch was the one place still assuming surface mode works; the S24 log proves it doesn't — it's a render-API limitation, not Tegra-specific.

**Fix:** route phones to `mediacodec-copy` as well (hardware decode + a cheap CPU copy). On a flagship phone the copy overhead is negligible, and it's strictly better than the current silent software fallback: real hardware decode, lower CPU and battery. After this, **no device class** uses surface-mode `mediacodec` — consistent with it never working under media_kit's render API.
