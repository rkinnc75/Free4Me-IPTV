# Free4Me-IPTV — On-Device (ADB) Test Guide

Reproduce-and-verify procedures for the perf / native / device-dependent findings —
the ones that **cannot be proven from code alone** and need a connected device. Each
procedure gives the build type, the exact `adb` steps to trigger the bug on the CURRENT
(unfixed) code, and the measurable pass criterion after the fix. Pair with the fix specs
in `docs/IMPLEMENTATION_SPEC.md` (each device-testable spec's *Acceptance* field points
here). Generated read-only from the live working tree; commands quote real log strings.

## Global ADB toolkit

- **Package / launch:** `me.free4me.iptv` — `adb shell monkey -p me.free4me.iptv -c android.intent.category.LAUNCHER 1` (or `adb shell am start -n me.free4me.iptv/.MainActivity`).
- **Logs:** the app logs via `AppLog → debugPrint`, tag **`flutter`**. Clear + capture: `adb logcat -c && adb logcat -s flutter:V AndroidRuntime:E`. Enable Settings → **Debug logging** first so `AppLog.info` lines are emitted. Grep the message substring only (lines are `[ts] [LEVEL] <msg>`).
- **Native crashes:** `adb logcat -s AndroidRuntime:E` → `FATAL EXCEPTION` / `IllegalStateException`.
- **D-pad:** `adb shell input keyevent <code>` — UP=19 DOWN=20 LEFT=21 RIGHT=22 CENTER=23 BACK=4 HOME=3 MEDIA_PLAY_PAUSE=85 FF=90 REW=89. ⚠️ adb sends one KeyDown+KeyUp — it CANNOT reproduce physical-remote **key-repeat** or **long-press**; findings that need held keys are flagged and require a real remote.
- **Network drop:** `adb shell svc wifi disable` / `enable`, or `adb shell cmd connectivity airplane-mode enable` / `disable`.
- **Memory (2 GB scale):** `adb shell dumpsys meminfo me.free4me.iptv` → read **TOTAL PSS**. Sample before/during/after the action.
- **Jank:** `adb shell dumpsys gfxinfo me.free4me.iptv` → **Janky frames %** and 90/95/99th percentiles.
- **Storage growth:** debug build → `adb shell run-as me.free4me.iptv du -sh <dir>`; release → `adb shell dumpsys diskstats` (coarse).
- **Build types:** a **release** APK is enough for UI / crash / logcat findings but is NOT debuggable (no `run-as`, no DB pull). Findings marked **debug-required** need a debug/profile build to read app-internal storage or pull the SQLite DBs. `~/development/flutter/bin/flutter build apk --debug` then `adb install -r build/app/outputs/flutter-apk/app-debug.apk`.
- **PiP support check (native category):** `adb shell pm list features | grep -i picture` — if `android.software.picture_in_picture` is ABSENT the box is a positive repro target for the PiP crash; `adb shell getprop ro.build.version.sdk` for API level.
- **Scale data:** seed a large provider (the onn box already carries a ~149k-channel Xtream source) to exercise the perf findings; a 5×450k setup is the worst case named in the review but any 100k+ source surfaces the query/memory behavior.

## Native / Android (crash · PiP · Cast)

**Category setup:** applicationId = me.free4me.iptv. Launch: `adb shell monkey -p me.free4me.iptv -c android.intent.category.LAUNCHER 1` or `adb shell am start -n me.free4me.iptv/.MainActivity`. App logs go through AppLog -> debugPrint -> logcat tag `flutter`. Standing log filter for these findings: `adb logcat -c` (clear) then `adb logcat flutter:V AndroidRuntime:E *:S`. Native crashes surface under tag `AndroidRuntime:E` as `FATAL EXCEPTION` / `IllegalStateException`.

BUILD TYPES: release APK is NOT debuggable — no `run-as`, no DB pull. PiP crash and PiP/logcat findings are all observable on a RELEASE APK because they emit either a native FATAL EXCEPTION (logcat) or an AppLog `flutter` line. The Cast findings (F3/F4/F5) require a working Google Cast environment: a device WITH Google Play Services (the onn 4K Plus / Shield have GMS; a bare AOSP box does not) AND at least one Chromecast/Google-TV cast target on the SAME Wi-Fi/LAN. If no cast target exists, `getState` returns `no_devices` and cast UI never enables — you MUST have a real Chromecast (or a Google TV with cast-receiver) on the LAN to exercise F3/F4/F5. For F5 (mimeType) a debug build is preferable so you can inject/observe the classification, but it can also be exercised release-only by observing whether the Cast affordance appears for an HLS channel whose URL contains the substring `mpeg`.

PiP SUPPORT FACT: onn 4K Plus and Shield are leanback TV boxes. Many such boxes do NOT declare `android.software.picture_in_picture` even though SDK_INT >= 26. Confirm the device's PiP support first: `adb shell pm list features | grep -i picture` — if `android.software.picture_in_picture` is ABSENT, the device is a positive repro target for F1/F2 (the bug fires); if PRESENT, PiP actually works and F1/F2 will NOT crash on that box (note this in the report and, if possible, run F1/F2 on a box lacking the feature). `adb shell getprop ro.build.version.sdk` confirms API level.

DPAD KEYEVENTS: navigate the D-pad UI with `adb shell input keyevent 19|20|21|22|23` (UP/DOWN/LEFT/RIGHT/CENTER), BACK=4, HOME=3. adb keyevents CANNOT reproduce physical-remote key-repeat/long-press; none of these five findings depend on held keys, so that limitation does not apply here.

To reach a playing stream (precondition for F1/F2/F3/F4): launch app, D-pad to a Live channel, CENTER to play, confirm playback with `adb logcat` showing player activity or `adb shell dumpsys window | grep mCurrentFocus` on MainActivity. Dart calls `PipController.setPlaying(true)` on playback start, which sets native `isVideoPlaying=true` — required for the onUserLeaveHint path.

#### MainActivity.kt:258 — onUserLeaveHint() calls enterPictureInPictureMode() guarded only on SDK>=O, not on FEATURE_PICTURE_IN_PICTURE; crashes on TV boxes lacking PiP support
- **Build:** release-ok
- **Reproduce (bug, current code):** 1. Pick a device that LACKS PiP: `adb shell pm list features | grep -i picture` — proceed only if `android.software.picture_in_picture` is ABSENT (typical onn 4K Plus / many leanback boxes). If present, this box cannot repro; note and try another.
2. Install/launch the release APK: `adb shell monkey -p me.free4me.iptv -c android.intent.category.LAUNCHER 1`.
3. Start clean logs: `adb logcat -c` then in a second shell `adb logcat AndroidRuntime:E flutter:V *:S`.
4. D-pad to a Live channel and press CENTER to start playback: `adb shell input keyevent 20` (repeat to move down list) then `adb shell input keyevent 23`. Confirm playback started (video on screen; Dart has called setPlaying(true) so native isVideoPlaying=true).
5. Trigger onUserLeaveHint by leaving to the launcher: `adb shell input keyevent 3` (HOME).
6. Read logcat. On the unguarded path the native call throws — watch for `AndroidRuntime` `FATAL EXCEPTION` with `java.lang.IllegalStateException` referencing `enterPictureInPictureMode` / `Device doesn't support picture-in-picture mode` and the process `me.free4me.iptv` dying. Confirm the crash: `adb shell dumpsys activity activities | grep -i me.free4me.iptv` shows the task gone, or relaunch is required.
- **Pass (after fix):** After a correct fix (guard on ctx.packageManager.hasSystemFeature(FEATURE_PICTURE_IN_PICTURE) and/or try/catch around enterPictureInPictureMode), step 5 (HOME during playback) produces NO `FATAL EXCEPTION`/`IllegalStateException` in `adb logcat AndroidRuntime:E`, the app process me.free4me.iptv stays alive (still listed in `adb shell dumpsys activity activities`), and it simply backgrounds to the launcher without PiP. On a box that DOES declare PiP, behavior is unchanged (enters PiP, EventChannel emits and Dart logs `PipController: entering PIP`).
- **Notes:** onUserLeaveHint fires on HOME/recents. isVideoPlaying must be true (playback started) or the branch is skipped. This is the highest-severity finding: a hard native crash on a primary target class (boxes without PiP).

- **Build:** release-ok

On a PiP-capable box, isSupported()=true and entering PiP still emits `PipController: entering PIP` and dumpsys shows the activity in PiP.
- **Notes:** Two coupled defects at line 136 and 138-141. isSupported over-reports capability; enterPip over-reports success. Both observable via the flutter-tag AppLog strings quoted above and dumpsys PiP state.

#### cast_controller.dart:124 — mimeTypeFor uses u.contains('mpeg') to classify MPEG-TS; this substring matches HLS/other URLs containing 'mpeg' (e.g. 'application/x-mpegURL'-style paths or any host/path with 'mpeg'), misclassifying them as video/mp2t and making isCastable() wrongly return false
- **Build:** either
- **Reproduce (bug, current code):** RELEASE-observable path (behavioral): 1. Device WITH GMS + a Chromecast on the LAN so cast affordances are enabled. 2. Add/select a Live channel whose stream URL contains the substring `mpeg` but is actually a castable HLS stream — e.g. a URL like `http://host/live/mpegdash/stream.m3u8` or any real provider URL whose path/host contains `mpeg` and ends in `.m3u8`. 3. Play that channel. 4. Compare with a plain `.m3u8` URL (no 'mpeg' substring): Cast is offered. 5. Confirm the misclassification directly on a DEBUG build: run `flutter test` or a scratch call — CastController.mimeTypeFor('http://x/mpeg/a.m3u8') returns 'application/x-mpegURL' expected but yields 'video/mp2t'; isCastable of that URL returns false.
DEBUG-required path (precise): build a debug APK, add a unit assertion or use `adb logcat flutter:V` after adding a temporary log, feeding URLs: 'a.m3u8' (expect application/x-mpegURL, castable), 'host-mpeg/a.m3u8' (BUG: classified video/mp2t, not castable), 'x.ts' (video/mp2t, not castable).

- **Notes:** The `.contains('mpeg')` also matches the correct HLS mime string 'application/x-mpegURL' conceptually, but here the ordering saves the `.m3u8` case FIRST — so the real-world break is URLs that contain 'mpeg' in host/path yet are m3u8/mp4/etc, OR any URL with 'mpeg' and no earlier match. This is a pure-Dart logic bug; exercisable on release via UI behavior but cleanest on a debug/test build where the function output can be observed directly.

## Player lifecycle · reconnect · seek · keys

**Category setup / prerequisites.** Enable AppLog file logging so the exact strings below reach logcat: in the app, Settings → enable "Debug logging" (`settings.debugLogging`; it makes `AppLog.info` reach `debugPrint` → logcat tag `flutter`). AppLog lines are formatted `[ts] [LEVEL] <message>` — grep the message substring only. Start capture: `adb logcat -c && adb logcat -s flutter:V AndroidRuntime:E`.

- **Launch:** `adb shell monkey -p me.free4me.iptv -c android.intent.category.LAUNCHER 1`
- **Reach a LIVE item (D-pad):** from Home, navigate the browse/guide grid with `adb shell input keyevent 22` (RIGHT) / `20` (DOWN) to a live category, then `23` (CENTER) to open a livestream. Confirm live via the DVR/transport log and the live branch in `MpvEngine: options applied` (live uses `demuxer-max-back-bytes 0`; VOD uses `64MiB`).
- **Reach a VOD item:** navigate to a Movies/VOD category and CENTER a movie. VOD emits `startPosition=<Ns>` in `MpvEngine: open()` when a resume position exists.

(a) DEBUG build — import a source with one entry whose URL is a black-holed host (`http://10.255.255.1/dead.mp4`) or a 200 that never sends media; open it as a movie. - **adb caveat (throughout):** `adb shell input keyevent` synthesizes one KeyDown+KeyUp; it CANNOT produce Android key-REPEAT or a physical long-press. Findings that depend on held keys (#5, and the acceleration ladder of #10) need a real remote.
