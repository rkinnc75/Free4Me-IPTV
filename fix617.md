# fix617 — Make the hang observable: per-line log flush + suppress log delete + stage logging

**Target:** v2.2.28+617 (diff against live tag **v2.2.27**, verified `git ls-remote --tags`)
**Apply check:** `git apply --check` PASSED on a fresh v2.2.27 clone.
**Analyze:** run on the build machine — gate not asserted here. (Manual: raw braces/parens balanced; `dart:async`/`unawaited` already imported.)

## Why
Every captured log of the phone refresh hang was tiny and showed nothing of the stall. Two reasons, both fixed here:
1. `AppLogger.log()` writes to a buffered `IOSink` that only flushes on close — a HUNG (alive but stuck) process never flushed, so force-closing it lost the buffered lines describing the stall.
2. The version-change log rotation cleared the file, compounding the loss.

This build makes the stall observable. It does NOT fix the refresh hang itself (still open) — it's the instrumentation step so the next phone log finally shows where "Preparing…" stops.

## Changes
1. **Per-line flush** (`app_logger.dart`): fire-and-forget `_sink?.flush()` after each write (NOT awaited; synchronous `log()` API unchanged). Shrinks the buffer-loss window to near-zero.
2. **Suppress the log delete — THIS BUILD ONLY** (`app_logger.dart`, `clearLog`): instead of deleting the file (and sweeping Xtream dumps), it appends a divider line `=== LOG CLEAR REQUESTED — would delete from line N (fix617: delete suppressed) ===` and keeps everything. **Marked clearly for revert** once the hang is understood.
3. **Stage logging in the refresh start** (`utils.dart`, `processSource`): logs around `sourceNameExists`, entering/leaving `withDroppedBrowseIndexes`, and the fetch dispatch — so the log pinpoints the exact stall stage. These flush immediately too.

## Files
- `lib/backend/app_logger.dart` — per-line flush + clearLog delete suppressed (revert later).
- `lib/backend/utils.dart` — `fix617` stage logs in `processSource`.
- `pubspec.yaml`, `lib/whats_new_modal.dart` — version bump + changelog.

## Revert note
After the hang is diagnosed, restore the original `clearLog()` (delete file + `_clearXtreamDumps()` + `_close()`/`_ensureOpen()`), and optionally keep the per-line flush and stage logs.

## Validation plan (build machine)
1. `flutter analyze` → expect "No issues found!".
2. Confirm a restart/version event no longer empties the log — it shows the `LOG CLEAR REQUESTED … would delete from line N` divider with all prior lines intact.
3. **Reproduce the phone Dino hang**, force-close while stuck, reopen, pull the log: it should now end at the LAST `fix617 processSource:` stage reached (name-check / index-drop / fetch dispatch). Send that log.
