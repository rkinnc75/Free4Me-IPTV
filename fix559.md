# fix559 — Fix the fix558 D-pad escape (was landing on Settings/tabs, not the section above)

**Version:** 2.0.59 (`2.0.59+559`)
**Base:** v2.0.58 (live CI tag). Single ```diff block, byte-exact against
v2.0.58. Verified clean on a fresh v2.0.58 clone.
**Branch from:** v2.0.58.

---

## fix558's escape mechanism never actually fired — root cause

fix558 tried: *"try Flutter's `focusInDirection(up)` first; only call our
explicit escape if it returns false (found nothing)."* On-device, this made
things **worse** — Up from Channels went to Settings, then Series, Live TV,
Search, back into Movies at the wrong column, etc.

**Why:** `FocusScope.focusInDirection()` almost never returns `false`. Per
Flutter's own `DirectionalFocusTraversalPolicyMixin.inDirection` source: if
nothing is found "in-band" (same horizontal lane), it falls back to **the
closest out-of-band node anywhere in the scope by distance to the center
line** — which can be the Settings gear or a tab button clear across the
screen — and **still returns `true`**. There is no "nothing found" case in
this layout. The `if (!focusInDirection(...))` check could never be true, so
`onFocusUpEscape` never ran; the wrong long-range jump won every time.

## Fix

1. **`ChannelTile.onFocusUpEscape` no longer probes `focusInDirection` at
   all.** When set, arrow-up calls it **unconditionally**. The caller is now
   responsible for only setting it on tiles that truly are a section's top
   row — no ambiguity left to resolve at the tile level.
2. **`tv_search_view.dart` computes the REAL column count** via
   `LayoutBuilder`, using the identical formula
   `SliverGridDelegateWithMaxCrossAxisExtent` uses internally
   (`ceil(width / (maxExtent + spacing))`), and only wires `onFocusUpEscape`
   to indices `0..columns-1` — the tiles that are actually in row 0. Every
   other row's tiles get `onFocusUpEscape: null`, so Flutter's normal (and
   here, correctly-working) in-section traversal handles them exactly as it
   did before fix558.

Net effect: up from any non-top-row tile behaves exactly as it always did
(unchanged, was never broken); up from an actual top-row tile jumps straight
to the previous section's last tile, every time, deterministically.

## On-device check after CI

In Search with 3+ sections populated (e.g. "fox"): from each section's actual
top-row tile, press Up — must land on the previous section's last tile, never
on Settings, a tab button, or skip to the wrong section. From a NON-top-row
tile, Up must behave as before (moves up one row within the same section).
