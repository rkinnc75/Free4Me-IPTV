# fix140.md — Multi-view: restore-saved-cells must skip non-livestream channels (enforce live-TV-only invariant)

> **Version:** Free4Me-IPTV 1.22.11 build (fix130/132).
> **Invariant:** Multi-view shows LIVE TV only — never movies or series.
>
> **Status:** The picker already enforces this (`mediaTypes: const
> [MediaType.livestream]` in `channel_picker_screen.dart`). The GAP is the
> restore-saved-cells path (`MultiViewScreen._restoreChannels`): it fetches each
> saved cell ID via `Sql.getChannelById(...)` and assigns it to a cell WITHOUT
> re-checking media type. A saved ID can become a movie/series after a source
> refresh (provider reassigns that stream_id) or come from an older/imported
> backup — restoring it would load non-live content into a multi-view cell,
> violating the invariant.

---

## Root cause

`lib/multi_view_screen.dart`, `_restoreChannels()` (~line 152–156):

```dart
    for (final entry in toFetch.entries) {
      final ch = await Sql.getChannelById(entry.value);
      if (!mounted) return;
      if (ch != null) _channels[entry.key] = ch;   // ← assigns ANY media type
    }
```

`getChannelById` returns whatever the persisted ID now points to, with no
`mediaType` filter. The picker that ORIGINALLY saved these IDs is
livestream-only, but persisted IDs are not guaranteed to still be livestream at
restore time. This is the only restore path that fetches by saved ID (the
`initState` → `_restoreChannels()` call at line 53; the comment near line 187
referring to "_restoreSavedCells" describes this same method — there is no
separate one).

# Fix 140.1 — Guard restore to livestream-only

**File:** `lib/multi_view_screen.dart`

**Current (~lines 152–156):**

```dart
    for (final entry in toFetch.entries) {
      final ch = await Sql.getChannelById(entry.value);
      if (!mounted) return;
      if (ch != null) _channels[entry.key] = ch;
    }
```

**Replace with:**

```dart
    for (final entry in toFetch.entries) {
      final ch = await Sql.getChannelById(entry.value);
      if (!mounted) return;
      // fix140: multi-view is LIVE TV only. A persisted cell ID can become a
      // movie/series after a source refresh or arrive from an older backup;
      // skip anything that isn't a livestream so non-live content never loads
      // into a cell. The cell stays empty (ready to assign).
      if (ch != null && ch.mediaType == MediaType.livestream) {
        _channels[entry.key] = ch;
      } else if (ch != null) {
        AppLog.info(
          'MultiViewScreen: restore skipped non-livestream cell ${entry.key}'
          ' channel="${ch.name}" mediaType=${ch.mediaType}',
        );
      }
    }
```

### Key changes
1. **Only `MediaType.livestream` channels are restored into cells** — movies/
   series are skipped, leaving that cell empty (ready to assign).
2. **Skips are logged** — `restore skipped non-livestream cell N` so a stale
   saved ID is visible in the log rather than silently vanishing.
3. **`MediaType` is already imported** (via `channel.dart` at line 8) — no new
   import.

> The persisted ID list (`multiViewCells1x2`/`multiViewCells2x2`) is NOT
> rewritten here — the skipped cell simply opens empty this session. If the user
> assigns a new channel, the existing persist path updates it. (Optional
> follow-up: prune non-livestream IDs from the saved list on restore so they
> don't linger — not included; the runtime skip is sufficient to honor the
> invariant.)
