# fix210 — Revert fix206 (drop/rebuild-index refresh) — it regressed refresh badly on multi-source catalogs

> **Version:** build `+210` (or highest in release); patch = next available at ship time (read live pubspec). Tag = next available immutable `v1.23.X`.
> **Evidence:** fix206 shipped in v1.23.22. A single-source refresh of Aniel3000 (48,720 rows) took **874,216ms (~14.5 min)** — vs ~145s before fix206 (~6× WORSE). Instrumentation (fix204) log: `Sql.commitWriteBatched: DONE 56 closures … total 874216ms`; the same log shows `ChannelSearchCache: rebuilt 319819 entries` — i.e. the full multi-source catalog is 319,819 channels.
> **Bug:** fix206 drops the table-global non-unique indexes + FTS triggers, bulk-inserts the one refreshed source, then **rebuilds the FTS index and recreates every index over the ENTIRE channels table** (all sources, 319,819 rows) — not just the 48,720 refreshed rows. The old code paid incremental per-row cost only on the inserted rows. So fix206 traded 48,720 rows of incremental work for a 319,819-row full-table reindex + full-table trigram FTS rebuild.
> **Root cause:** indexes and the FTS5 index are table-global, but a refresh is single-source. A whole-table drop→rebuild therefore scales with TOTAL catalog size, not refresh size — the opposite of what's wanted. fix206 was validated only on a single-source test table, which could not surface this.

## What the evidence proves

- `commitWriteBatched … total 874216ms` for 48,720 fetched rows = the cost is in the commit, and it ballooned.
- `restorePreserve` (21,794 rows) is NOT the cause — its `UPDATE … WHERE name=? AND source_id=?` is served by the kept `channels_unique` index (verified: 21,794 such updates take ~89ms in a seeded test).
- The `319819` total-catalog figure is ~6.6× the 48,720 refreshed rows; fix206's post-insert FTS `'rebuild'` and index recreation operate on all of it.
- Reverting fix206 returns refresh to the pre-206 ~145s behavior.

## Decision

- **Revert fix206 entirely.** Its core operations (external-content FTS `'rebuild'`, `CREATE INDEX`) are inherently whole-table; there is no per-source variant, so no partial keep is salvageable.
- **Keep fix204 instrumentation** (the timing logs) so refresh stays measurable.
- The original perf goal is parked: the ~2-3 min large-source refresh is largely inherent to maintaining an indexed, full-text-searchable 319k-row catalog. (FTS is the default search engine — `channels_fts MATCH`, `SearchMethod.ftsAnd` — so it cannot simply be removed without regressing search.)

## Fix 210.3 — `lib/backend/sql.dart`: remove the two fix206 helper methods

Delete the two helper methods. They sit immediately before the `insertChannelsBulk` doc comment.

### Current code (verbatim — match exactly)
```dart
  /// fix206: bulk-load optimization. Before a large reinsert, drop the
  /// NON-UNIQUE secondary indexes on channels and disable the FTS sync trigger,
  /// so each inserted row does not pay per-row b-tree maintenance + a trigram
  /// FTS insert. The dropped indexes' exact CREATE SQL is captured from
  /// sqlite_master and stashed in [memory] for verbatim recreation afterward.
  /// UNIQUE indexes (channels_unique + the partial unique indexes) are KEPT —
  /// the bulk INSERT's ON CONFLICT depends on channels_unique, and the partials
  /// guard integrity. Measured ~2.6x faster on 60k rows (seeded test).
  static Future<void> Function(SqliteWriteContext, Map<String, String>)
      dropChannelsBulkIndexes() {
    return (SqliteWriteContext tx, Map<String, String> memory) async {
      final rows = await tx.getAll(
        "SELECT name, sql FROM sqlite_master "
        "WHERE type = 'index' AND tbl_name = 'channels' "
        "AND sql IS NOT NULL "
        "AND name NOT IN (SELECT name FROM sqlite_master "
        "  WHERE type = 'index' AND sql LIKE 'CREATE UNIQUE%')",
      );
      final captured = <String>[];
      for (final r in rows) {
        final name = r.columnAt(0) as String;
        final sql = r.columnAt(1) as String;
        captured.add(sql);
        await tx.execute('DROP INDEX IF EXISTS $name;');
      }
      memory['fix206_droppedIndexSql'] = jsonEncode(captured);
      await tx.execute('DROP TRIGGER IF EXISTS channels_ai;');
      await tx.execute('DROP TRIGGER IF EXISTS channels_au;');
      await tx.execute('DROP TRIGGER IF EXISTS channels_ad;');
    };
  }

  /// fix206: after the bulk reinsert, recreate the non-unique indexes verbatim
  /// (from the SQL captured in [memory]), rebuild the external-content FTS index
  /// in one pass, and restore the FTS sync triggers.
  static Future<void> Function(SqliteWriteContext, Map<String, String>)
      restoreChannelsBulkIndexes() {
    return (SqliteWriteContext tx, Map<String, String> memory) async {
      final raw = memory['fix206_droppedIndexSql'];
      if (raw != null) {
        final List<dynamic> sqls = jsonDecode(raw) as List<dynamic>;
        for (final sql in sqls) {
          await tx.execute(sql as String);
        }
      }
      await tx.execute(
          "INSERT INTO channels_fts(channels_fts) VALUES('rebuild');");
      await tx.execute('CREATE TRIGGER IF NOT EXISTS channels_ai '
          'AFTER INSERT ON channels BEGIN '
          'INSERT INTO channels_fts(rowid, name) VALUES (new.id, new.name); END;');
      await tx.execute('CREATE TRIGGER IF NOT EXISTS channels_ad '
          'AFTER DELETE ON channels BEGIN '
          "INSERT INTO channels_fts(channels_fts, rowid, name) "
          "VALUES('delete', old.id, old.name); END;");
      await tx.execute('CREATE TRIGGER IF NOT EXISTS channels_au '
          'AFTER UPDATE OF name ON channels BEGIN '
          "INSERT INTO channels_fts(channels_fts, rowid, name) "
          "VALUES('delete', old.id, old.name); "
          'INSERT INTO channels_fts(rowid, name) VALUES (new.id, new.name); END;');
    };
```

## Notes
- **Lesson recorded:** a single-source refresh must not trigger whole-table index/FTS work; validate refresh perf changes against a MULTI-source catalog, since indexes/FTS are table-global. (fix206 passed single-source tests but regressed badly with other large sources present.)
- The FTS5 index is NOT removable as a perf shortcut: it is the default search engine (`SearchMethod.ftsAnd`, `channels_fts MATCH`), and the LIKE fallback is a multi-second full scan on large catalogs.
- fix204 instrumentation stays so any future refresh-perf attempt is measurable from the start.
- Independent of fix208 (whats_new_modal). Touches only `sql.dart` + `xtream.dart`.
