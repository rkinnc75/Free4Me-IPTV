## Root cause (the tail of the guide-empty bug)
fix709 added an in-isolate chained-Future mutex `_serializeMatch` (`_matchGate`)
so two EPG channel-matches can't collide on the db.sqlite writer + WAL **TRUNCATE
checkpoint** (needs an EXCLUSIVE lock) — a collision exhausts the 5× SQLITE_BUSY
retries, `matchChannels` throws, the per-source catch swallows it, and that
source's channels are left silently **unmatched** ("guide empty on every
channel"). fix712 serialized the whole multi-source refresh. BUT the settings
**"Re-match all"** button called `EpgService.matchChannels` **directly**
(settings_view.dart ~1333), bypassing the gate — so a user Re-match could still
collide with the 2-hour scheduled refresh (or another gated match) in the same
isolate and hit exactly the SQLITE_BUSY-→unmatched failure it was meant to fix.

## Change
- `lib/backend/epg_service.dart`: new PUBLIC `matchChannelsSerialized(source,
  channelMap, {forceAll, onProgress})` = `_serializeMatch(() => matchChannels(…))`
  — the single, only entry point external callers use. `refreshSource` now calls
  it too (was inline `_serializeMatch(() => matchChannels…)`), so there is ONE
  gate source of truth and `matchChannels(` has exactly two occurrences (its def
  + the wrapper's call).
- `lib/settings_view.dart` (~1333): Re-match loop → `matchChannelsSerialized(…)`.

The gate is a single static field, so every gated caller across the isolate
serializes against every other, regardless of trigger (scheduled / setup /
source refresh / Re-match). `finally { done.complete() }` releases the gate even
if a match throws, so one failed source can't wedge the next or the scheduler.

## Scope / limits (verified by adversarial review)
SAME-ISOLATE only (`_matchGate` is a per-isolate static). This closes the
realistic window — a foreground Re-match vs a foreground launch/stale
`refreshIfStale` (both run on the main isolate; `BackgroundTaskService.run` does
NOT spawn an isolate). It does NOT serialize a Re-match against a Workmanager
BACKGROUND refresh (separate isolate). And unlike `refreshAllSources`, the
settings Re-match neither sets nor checks the `epg_refresh_in_progress` app_meta
marker, so that cross-isolate window is guarded ONLY by the SQLITE_BUSY retries.
The narrower original bug (same-isolate concurrent multi-source refresh) is fully
closed by fix709+fix712; this adds the same-isolate Re-match path.
