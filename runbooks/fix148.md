# fix148.md — Repo transfer follow-up: build/release script still looks up the GitHub token under the OLD keychain account (rkinnc75)

> **Version:** Free4Me-IPTV 1.22.12 build (post-fix146, post repo transfer to
> `rkinnc75`).
> **Context:** Checking the build machine's work against fix146. The four
> fix146 code/CI URL changes all landed correctly (verified: no `rkinnc75` in
> `lib/`, `version.json`, `.github/`, `android/`). **But the release script
> `scripts/build_and_release.sh` has one straggler the URL-sweep missed**, and
> it's functional — it will break the GitHub-release step.

---

## The bug

`scripts/build_and_release.sh` was updated almost entirely to `rkinnc75` — the
git remote (line 151), every `api.github.com/repos/...` and `uploads.github.com`
endpoint, the `releaseUrl`, the handbook link, and the setup instructions
(line 19: `security add-internet-password ... -a rkinnc75`). **One line was
not updated** — the Keychain token LOOKUP:

```bash
# line 165 (current):
GITHUB_TOKEN=$(security find-internet-password -s "api.github.com" -a "rkinnc75" -w 2>/dev/null)
```

This is internally inconsistent with the rest of the script:
- **Line 19** (setup doc) tells you to STORE the token under `-a rkinnc75`.
- **Line 168** (this block's own error message) tells you to
  `security add-internet-password -s api.github.com -a rkinnc75 -w <token>`.
- **Line 165** SEARCHES for it under `-a rkinnc75`.

So if the token was (re)stored under the new account `rkinnc75` — as every
instruction says to do — **line 165 finds nothing**, `GITHUB_TOKEN` is empty,
and the script fails at the release step (line 167 prints "token not found", or
the subsequent `api.github.com` calls 401). The build + git push (SSH) succeed;
only the GitHub *release create / APK upload* breaks. Classic miss that passes a
"did the URLs change" check but fails at release time.

---

## fix148.1 — Look up the token under the new account

**File:** `scripts/build_and_release.sh`

**Current (line 165):**
```bash
GITHUB_TOKEN=$(security find-internet-password -s "api.github.com" -a "rkinnc75" -w 2>/dev/null)
```

**Replace with:**
```bash
GITHUB_TOKEN=$(security find-internet-password -s "api.github.com" -a "rkinnc75" -w 2>/dev/null)
```

Now the lookup (165), the setup instruction (19), and the error message (168)
all agree on account `rkinnc75`.

### Verify
```bash
grep -n "rkinnc75" scripts/build_and_release.sh   # expect: no output
grep -n 'security find-internet-password' scripts/build_and_release.sh  # should show -a "rkinnc75"
```

> **Keychain entry must actually exist under the new account.** If you haven't
> already, store the token under `rkinnc75` (one time, on the build Mac):
> ```bash
> security add-internet-password -s api.github.com -a rkinnc75 -w <token>
> ```
> If the old `rkinnc75` keychain entry still holds a VALID token, you can also
> just re-store that same token value under `rkinnc75`. Either way, after
> fix148 the script reads from `rkinnc75`.

## Token / access sanity (same as fix146's note, restated for the release path)

A classic PAT is account-scoped, not repo-path-scoped, so the token value still
works for API calls **as long as the owning account has write access to
`rkinnc75/Free4Me-IPTV`**. After a transfer the new owner (`rkinnc75`) has full
access by definition. If the token was issued by `rkinnc75` and `rkinnc75` is no
longer a collaborator, the `api.github.com/repos/rkinnc75/...` release calls
will 401/403 — issue a fresh token under `rkinnc75` and store it per above.
