# fix618 — Fix the StreamSink crash from fix617: safe synchronous log append

**Target:** v2.2.29+618 (diff against live tag **v2.2.28**, verified `git ls-remote --tags`)
**Apply check:** `git apply --check` PASSED on a fresh v2.2.28 clone.
**Analyze:** run on the build machine — gate not asserted here. (Manual: raw braces/parens balanced; removed now-unused `dart:async` import; no `_sink` writes remain in `log()`.)

## Why (regression introduced by fix617)
fix617 added `unawaited(_sink?.flush())` after every log write. `IOSink.flush()` shares in-flight stream state; calling it again while a prior flush/write is pending throws **`Bad state: StreamSink is bound to a stream`**. Under rapid logging (a source refresh emits many lines fast) this corrupted the sink, which broke:
- **Settings import** (the visible `Import failed: Bad state: StreamSink is bound to a stream` error), and
- **Source refresh** (stalled).

So fix617's flush is the cause of the import failure and the refresh stall ON THIS BUILD. (The ORIGINAL phone refresh hang that prompted fix617 is a separate, still-open issue — fix617's instrumentation never got a clean run because its own flush broke things.)

## Fix
- `log()` now writes via **`_file.writeAsStringSync(line, mode: append, flush: true)`** — a synchronous atomic append that reaches disk immediately (keeping fix617's "capture a hang" goal) but has NO shared in-flight stream state, so it cannot enter the bad state. Wrapped in try/catch so an I/O error never throws out of the synchronous `log()`.
- The buffered `IOSink` is no longer opened by `_ensureOpen` (two write handles to one file is unsafe); `_file` is now the "open" sentinel. `_close()` still defensively flushes/closes any stray sink.
- Removed the now-unused `dart:async` import (`unawaited` is gone).
- `clearLog()` delete-suppression from fix617 is RETAINED (still investigating the original hang; logs stay preserved). Revert that together with fix617's note once the hang is solved.

## Files
- `lib/backend/app_logger.dart` — synchronous append write path; drop IOSink open; remove `dart:async`.
- `pubspec.yaml`, `lib/whats_new_modal.dart` — version bump + changelog.

## Validation plan (build machine)
1. `flutter analyze` → expect "No issues found!".
2. **Settings import** works again (no `StreamSink is bound to a stream`).
3. A source refresh runs without the sink-corruption stall; log fills normally with per-line entries.
4. Confirm the log still writes immediately (kill mid-operation, reopen, lines up to the kill are present).
5. THEN resume the original phone-hang investigation on a clean build — reproduce the Dino refresh, capture the now-reliable log, and read where it actually stalls.
