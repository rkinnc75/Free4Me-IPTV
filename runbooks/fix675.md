# fix675 вЂ” DIAGNOSTIC: trace why a scheduled recording never starts

**Build:** `3.0.17+675` В· **Base tag:** `v3.0.16` (fix674) В· **Ship order:** after 674
**Files touched:** `lib/backend/recording_scheduler.dart`, `android/app/src/main/kotlin/me/free4me/iptv/RecordingCaptureService.kt`, `lib/whats_new_modal.dart`, `pubspec.yaml`
**Behavior change:** none вЂ” instrumentation only.

## Why

After fix674 (alarm now registers), an on-device "Record now" still produces no file and the Recordings row stays on **`scheduled`** вЂ” it never even flips to `recording`. That means the alarm callback (`recordingAlarmCallback`) is either not firing, or throwing before its first status write. But the exported report log shows **nothing** from the callback вЂ” only the `RecordingScheduler: scheduled recording вЂ¦` line.

Root reason the callback is invisible: it runs in the alarm plugin's **background isolate**, and `AppLog` is an in-memory singleton whose `_enabled`/`_file` are **per-isolate**. In that isolate `AppLog` is a fresh instance with logging disabled and no file handle, so every `AppLog.info/warn` in the callback silently no-ops. We're blind exactly where the failure is.

Three candidate causes remain, and the real fix differs between them:
1. The alarm isolate has no plugin registrant в†’ the first plugin call (path_provider via `Utils.appDir`, or the `me.free4me.iptv/recording` MethodChannel) throws `MissingPluginException`.
2. The callback dispatcher was never registered в†’ the alarm has no Dart callback to invoke (nothing runs at all).
3. The callback runs but throws somewhere downstream.

Per the standing rule, we do not ship a network/isolate-layer fix on a guess. This runbook adds **plugin-free** logging so the next single test discriminates all three.

## What it does

- **Dart (`recordingAlarmCallback`):** `debugPrint("SR-CB: вЂ¦")` breadcrumbs вЂ” pure Dart, no plugin, always emitted to logcat. The FIRST line (`entered вЂ¦`) is emitted **before any plugin/DB call**, then after each step (`got row`, `status->recording`, `RecordingCapture.start returned OK`). The `catch` now captures the stack and prints `SR-CB: вЂ¦ THREW <e>\n<stack>`. `AppLog` lines are left as-is (harmless).
- **Native (`RecordingCaptureService`):** `Log.i(TAG=SRCapture, вЂ¦)` at `onStartCommand ACTION_START` and at `runCapture START`, so `adb logcat -s SRCapture` on the onn shows whether the MethodChannel start even reached the platform side and whether the capture thread started.
- No control-flow change. No manifest change. Version + changelog bump only besides the logging.

## How to read the result (after this ships)

Trigger a 2-min "Record now", then collect logs:

- **Samsung (no adb):** export the in-app report log. The `SR-CB:` breadcrumbs appear **only if plugins are alive in the alarm isolate** (they route through `debugPrint` в†’ the app's debug log capture). If you see `SR-CB: entered вЂ¦` followed by `SR-CB: вЂ¦ THREW MissingPluginException`, that's cause #1. If you see NO `SR-CB:` line at all, that points at cause #2 (dispatcher/callback never invoked).
- **onn (adb), when available:** `adb logcat -c` then trigger, then `adb logcat -d -s SRCapture:* flutter:* > sr.txt`. This is the definitive trace вЂ” it shows the `SR-CB:` breadcrumbs, the throw+stack, and whether `onStartCommand ACTION_START` / `runCapture START` fire.

Send whichever log you can and it will pin the exact cause; the real fix follows.
