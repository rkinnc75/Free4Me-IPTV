# fix240 — Analyze & Suggest can recommend values outside the slider's range; clamp to the real bounds

> **Version:** build = highest fix number in release; patch = next available. Tag = next immutable.
> **Builds on:** the live 1.25.x source (touches playback_analyzer.dart + new setting_bounds.dart). Independent of other pending fixes.
> **Bug:** "Analyze playback & suggest settings" can recommend a value the user cannot actually set, because `PlaybackAnalyzer.recommend()` clamps to its OWN hardcoded numbers, not the settings sliders' min/max. Worst case: `liveCacheSecs` could be suggested up to **120** while the slider maxes at **60**; `startupGraceMs` could be suggested as low as **0** while the slider floor is **100**.
> **Fix:** introduce a single source of truth (`SettingBounds`) holding each setting's min/max (matching the current sliders) and clamp every recommendation to it. (Per the chosen scope: the analyzer is wired to `SettingBounds` now; the sliders keep their inline literals for a later pass — the constants already match them, so migrating the sliders later is drift-free.)

## The mismatches (analyzer clamp vs actual slider)

| Setting | Slider min/max | OLD analyzer clamp | Problem |
|---|---|---|---|
| `liveCacheSecs` | 5 / 60 | `clamp(10, 120)` | could suggest up to 120 — **2× the slider max** |
| `startupGraceMs` | 100 / 3000 | `clamp(0, 2500)` | floor 0 < slider 100; max 2500 < slider 3000 |
| `bufferingWatchdogSecs` | 5 / 60 | `clamp(5, 30)` | max 30 under-recommends (slider allows 60) |
| `openTimeoutSecs` | 5 / 60 | `clamp(5, 30)` | max 30 under-recommends (slider allows 60) |
| `bufferSizeMB` | 16 / `maxBufferSizeMb` | `clamp(16, maxBufferSizeMb)` | already correct — switched to the constant for consistency |

After the fix all five clamp to exactly the slider bounds (verified).

## Pre-verified facts (checked against 1.25.x, compiled)

- New file `lib/backend/setting_bounds.dart`; edits in `lib/backend/playback_analyzer.dart` (import + 5 clamp sites).
- The constant values MATCH the current settings_view.dart sliders: liveCache 5–60, startupGrace 100–3000, bufferingWatchdog 5–60, openTimeout 5–60, bufferSize 16–`DeviceMemory.maxBufferSizeMb`.
- `bufferSizeMax` is a getter returning `DeviceMemory.maxBufferSizeMb` (device-dependent), matching the slider's `max: DeviceMemory.maxBufferSizeMb`.
- `DeviceMemory` import in the analyzer is still needed (used in a rationale string), so it stays.
- **Compile:** `flutter analyze --no-fatal-infos` → 2 tolerated `settings_view.dart` INFOs. Rule 6 clean.

## Fix 240.1 — new file `lib/backend/setting_bounds.dart`

Create verbatim:
```dart
import 'package:open_tv/backend/device_memory.dart';

/// fix240: single source of truth for the min/max of each tunable playback
/// setting. The settings_view.dart sliders and PlaybackAnalyzer.recommend()
/// must agree on these bounds — previously the analyzer clamped to its own
/// hardcoded numbers (e.g. liveCacheSecs up to 120) while the slider maxed at
/// 60, so it could recommend a value the user could not actually set. The
/// analyzer now clamps to these constants. (The sliders still use inline
/// literals for now; when they are migrated to reference this class the two
/// can never drift again. The values below MATCH the current sliders.)
class SettingBounds {
  const SettingBounds._();

  // liveCacheSecs — slider min 5, max 60.
  static const int liveCacheMin = 5;
  static const int liveCacheMax = 60;

  // startupGraceMs — slider min 100, max 3000.
  static const int startupGraceMin = 100;
  static const int startupGraceMax = 3000;

  // bufferingWatchdogSecs — slider min 5, max 60.
  static const int bufferingWatchdogMin = 5;
  static const int bufferingWatchdogMax = 60;

  // openTimeoutSecs — slider min 5, max 60.
  static const int openTimeoutMin = 5;
  static const int openTimeoutMax = 60;

  // bufferSizeMB — slider min 16, max is device-dependent.
  static const int bufferSizeMin = 16;
  static int get bufferSizeMax => DeviceMemory.maxBufferSizeMb;
}
```
