# fix190 — Editing a source wipes its detected max_connections (preserve it on save)

> **Version:** build `+190`; patch = next available at ship time (read live pubspec — patch and fix numbers drift; do NOT assume). Tag = next available `v1.23.X` (immutable; never reuse/delete/recreate).
> **Evidence:** A source's connection limit (used to gate multi-view layouts) is auto-detected and stored, but disappears after the user opens Edit on that source and taps Save — multi-view gating then behaves as if the limit is unknown.
> **Bug:** `EditDialog` builds a new `Source(...)` for `Sql.updateSource(...)` but omits `maxConnections`, so it defaults to `null`. `updateSource` writes `max_connections = ?` from `source.maxConnections`, so the stored value is overwritten with NULL on every edit.
> **Root cause:** In `lib/edit_dialog.dart`, the `Source(...)` passed to `Sql.updateSource` carries the unchanged fields (`enabled`, `epgUrl`, `defaultEngine`) through from `widget.source`, but `maxConnections` was never added to that list. The edit form has no max-connections input, so the field is simply dropped.

## What the evidence proves

`maxConnections` is populated automatically (`fetchXtreamMaxConnections`, fix184/186) and read by `lib/multi_view_picker_dialog.dart` to gate multi-view layouts. After an edit it reads back NULL, which the picker treats as "unknown," so gating regresses. The value is correct until the first edit-save — proving the loss happens in the edit path, not in detection or storage.

## Pre-verified facts (already checked against the uploaded `v1.23.16+186` source)

- **File:** `lib/edit_dialog.dart`, the `Source(...)` argument to `Sql.updateSource(...)` inside the Save button's `onPressed`.
- **`Source` constructor** (`lib/models/source.dart`) declares the named param `this.maxConnections` (type `int?`). So `maxConnections:` is a valid argument. (source.dart line 31; field declared line 18.)
- **`Sql.updateSource`** (`lib/backend/sql.dart`) SQL is `UPDATE sources SET url = ?, username = ?, password = ?, default_engine = ?, max_connections = ? WHERE id = ?` with `source.maxConnections` bound to the `max_connections` placeholder. So carrying the value through the constructor is sufficient to persist it; no SQL change needed.
- **`widget.source.maxConnections`** is the in-memory current value loaded for the dialog (`EditDialog` receives the full `Source` via `widget.source`), so it is the correct value to preserve — identical pattern to the sibling fields `enabled` / `epgUrl` / `defaultEngine` already carried through unchanged.
- **No migration / DB-schema change** (the `max_connections` column already exists, migration 16). No Rule 8 migration test required.
- **Compile status:** applied to the uploaded `1.23.16+186` tree, `flutter analyze --no-fatal-infos` returned **exactly the 2 pre-existing tolerated INFOs** in `lib/settings_view.dart` — no new ERROR/WARNING/INFO. Rule 6 grep (`\$`, `\u`) on the edited file: clean.

## Fix 190.1 — carry `maxConnections` through `EditDialog` save

**File:** `lib/edit_dialog.dart`

### Current code (verbatim — match exactly)

```dart
                      enabled: widget.source.enabled,
                      epgUrl: widget.source.epgUrl,
```

### Replacement code (verbatim)

```dart

                      // Without this it defaults to null and updateSource writes
                      // NULL, wiping the connection limit on every source edit.

```

### Key changes
- Adds `maxConnections: widget.source.maxConnections,` to the `Source(...)` passed to `Sql.updateSource`, preserving the auto-detected connection limit across an edit-save.
- Same proven pattern as the adjacent preserved fields (`enabled`, `epgUrl`, `defaultEngine`).
- No SQL change, no model change, no new symbol, no import.
