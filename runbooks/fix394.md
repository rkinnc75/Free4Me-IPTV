# fix394: Developer / libmpv advanced tunables + Live DVR section

**Repo:** rkinnc75/Free4Me-IPTV · **v1.34.22+394** · **APK target:** advanced users / power tuners

> ## ⚠️ Review corrections applied before release (2026-06-17)
>
> The draft below shipped fields backed by libmpv properties that **do not exist**
> or have the **wrong type**, plus two **wrong defaults** that broke the
> "no-op until opt-in" guarantee (the engine sets every dev property on *every*
> stream open, so these affected all users, not just opt-in tuners). Verified
> against the mpv manual; corrected as follows. **Shipped set: 13 fields, 5 enums**
> (was 18 fields / 6 enums). The embedded patch + prose below are the original
> draft; the actual shipped diff is `fix394.patch` in this folder.
>
> **Removed — non-existent / wrong-type libmpv properties:**
> - `demuxer-cache-wait` — it is a `<yes\|no>` flag, not a seconds value (was modeled as a `double` slider passing `"0.0"`).
> - `demuxer-max-wait-keepalive`, `demuxer-backward-buffer-secs`, `demuxer-dont-buffer-secs` — no such mpv properties exist.
> - `target-colorspace` (+ the `TargetColorspace` enum) — no such property; the real option is the `target-colorspace-hint` yes/no flag plus `target-prim`/`target-trc`. The 6 `bt.709/hdr10/...` values mapped to nothing.
>
> **Corrected enum values to libmpv-valid choices only:**
> - `video-sync`: dropped `display` and `audio-desync` (not real modes); kept `audio`, `display-resample`, `display-resample-vdrop`, `display-vdrop`, `desync`.
> - `framedrop`: dropped `yes` (not a current choice); now `no`/`vo`/`decoder`.
> - `audio-spdif`: it takes a comma-separated codec list — `no` now means "don't set the property" (sentinel → null) and `all` expands to `ac3,eac3,dts` (was the literal, rejected, `"no"`/`"all"`).
>
> **Corrected wrong defaults to true libmpv upstream:**
> - `audio-buffer`: `0.0` → **`0.2`** (mpv default is 200 ms).
> - `framedrop`: `no` → **`vo`** (mpv default; `no` silently disabled mpv's normal VO frame-drop on every open).
>
> Gates after correction: `flutter analyze` → No issues found!; `flutter test` → **127/127**.

## What this fix delivers

A new **Developer** folded menu at the very bottom of Settings exposing 18 advanced libmpv tunables across 4 sub-headers (Refined buffering, Demuxer/cache, Sync/image quality, Audio/network). Every default matches libmpv's upstream value exactly so the section is a no-op until the user opts in. Plus a new **Live DVR** folded menu split out of Buffering per user decision #1, leaving Buffering with just the everyday knobs.

**Menu order after fix394:** Playback → Buffering → **Live DVR** (new) → Multi-view → Content → EPG → Backup/Restore → Reset → Diagnostics → **Developer** (new, last).

## Design decisions (user-confirmed)

- (1) **One fix, not split.** Section-order change in `settings_view.dart` is a single edit; splitting would force a second PR for a structurally-related change.
- (2) **No isTV branching on `devVideoSyncMinFps`.** Both TV and phone default to 30. PAL/50Hz-specific tuning is a future fix once a PAL device is reported with an A/V issue.
- (3) **`devAudioBufferSecs` default is 0.0**, not 0.2. 0.0 is libmpv's upstream default; the subtitle says "Defaults match libmpv upstream; adjust only if a specific provider or device needs it."
- (9) **6 enums in a new file** `lib/models/dev_mpv_options.dart`, mirroring the `multi_view_*.dart` pattern. Keeps `settings.dart` readable.

## Engine wiring (`lib/player/mpv_engine.dart:_applyMpvOptions`)

Three changes, all preserving the existing behavior:

1. **Replaced hardcoded `network-timeout=30`** with `s.devNetworkTimeoutSecs.toString()`.
2. **Replaced the per-source `ignoreSsl` block** with: `if (ignoreSsl) tls-verify=no; else tls-verify = s.devTlsVerify ? 'yes' : 'no'`. Per-source `ignoreSsl` STILL wins for sources with self-signed certs — do not invert.
3. **Appended the Developer block** before the final `AppLog.info('MpvEngine: options applied ...')`. Reads each dev field and calls `np.setProperty(...)`. Skips `target-colorspace` and `hwdec-image-format` when their user-visible value is the "no override" sentinel (`auto` / `defaultFmt`). The log line is extended to confirm `netTimeoutSecs`, `tlsVerify`, `videoSync`, `tscale`, `audioSpdif` reached libmpv.

The `_bufferSlider` helper in `settings_view.dart` got a new `decimals` parameter (default 0 = integer display, 1 = 1-decimal, 2 = 2-decimal) so 0.5 / 0.20 / 1.5 style fields render correctly. Existing call sites pass no `decimals` arg, so behavior is unchanged for them.

Added `test/settings_dev_defaults_test.dart` (3 tests, +3 to baseline):

- `Settings.defaults() — 18 dev fields match libmpv upstream` — pins every default to a hard-coded value. Catches accidental default drift: if a future change moves a default away from libmpv upstream, the section is no longer a no-op until the user opts in.

## Out of scope (deferred)

- **Stats / observability** — `observeProperty("demuxer-cache-state", ...)`, `decoder-frame-drop-count`, `display-fps`, `video-params`, `audio-params`. Needs a code refactor (call `player.stream` events on the NativePlayer, not just the high-level `buffering`/`completed`/`error`/`position` streams). Not a Settings fix.
- **Subtitle / track selection** — `aid`/`sid` property set + `track-list` observation. Needs the existing "Select subtitles" dialog in `lib/player.dart:877` to actually wire up.
- **NativePlayer.command(...) for `frame-step` / `screenshot` / `cycle audio` etc.** — These need a control-bar / button surface, not a Settings tile.
- **Demuxer-cache tweaks that don't have a UI knob** (e.g. `demuxer-cache-wait` when value is a bytes string not seconds): the user-side knob is the seconds-form; the bytes-form is a future extension.
- **`ao=opensles` / `iccc-profile` / `target-prim` / `target-trc`**: deliberately omitted per the "no useless, no read-only" rule.
- **PAL/50Hz-specific `devVideoSyncMinFps` default** — deferred until a PAL device is reported with an A/V issue.

## What I did NOT change (deliberate)

- **The `_helpWatchdog`, `_helpDvr`, `_helpVodDemuxerMB` help tuples** — kept where they were; the help text doesn't move when the tile moves.
- **The `_bufferSlider` signature** — added `int decimals = 0` as a defaulted parameter so existing call sites compile unchanged. New dev call sites pass `decimals: 1` or `decimals: 2`.
- **Existing `Settings` fields** — only added 18 new fields; did not rename, retype, or remove any existing field. Older backups and v3 schema continue to work (v4 reads fall through to constructor default when keys are missing).
- **Existing libmpv options outside the dev block** — `cache=yes`, `cache-secs`, `demuxer-max-bytes`, `force-seekable`, `profile=low-latency`, `cache-pause-initial`, `cache-pause-wait`, the hwdec branch, the audio downmix path. All unchanged.
- **Tests for the engine wiring** — the engine's new `np.setProperty(...)` calls mirror the existing pattern at lines 494/510/etc., and the dev fields have no observable engine effect when they're at their libmpv-upstream defaults. The test guard on `Settings.defaults()` and the `Enum sentinels` test is the right level for this fix.

## Risks

- **First run on any device**: defaults match libmpv upstream exactly, so the section is a no-op. No behavior change.
- **Upgrade from 1.34.21**: persisted settings are preserved; the 18 new dev keys default to libmpv-upstream values when missing. No migration needed.
- **Per-source `ignoreSsl` source behavior**: preserved. Self-signed-cert sources still force `tls-verify=no` unconditionally, even if the user has set the dev toggle to ON.
- **`network-timeout` change**: 30 (was hardcoded) → 30 (default, persisted as `devNetworkTimeoutSecs`). No change unless the user opts in.
- **`tls-verify` change**: was never set on the engine path (libmpv's default `yes` applied). Now explicitly set to `s.devTlsVerify` (default `true` = `yes`) on the non-`ignoreSsl` path. Same effective behavior unless the user toggles to OFF.
