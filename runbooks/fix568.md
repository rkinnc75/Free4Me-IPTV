# fix568 — 30 fps cap via the `select` filter (fps filter is absent from libmpv)

Base: **v2.0.66** (origin/main; fix567 was a never-released dry-run iteration).
Ships as **v2.0.68+568**.

## Why fix565–567 all failed (each confirmed on the onn 4K Plus, not guessed)

The cap tried to use libavfilter's `fps` filter via mpv's `vf` property:

| build | filter string | device result |
|---|---|---|
| fix565 | `fps=30` | `Option vf: fps doesn't exist` (bare name, no bridge) — cap inert + reconnect storm |
| fix566 | `lavfi=[fps=30]` | accepted, then `No option name near '30'` → **stall** |
| fix567 | `lavfi=[fps=fps=30]` | `No such filter: 'fps'` → **stall** |

Root cause: **the `fps` libavfilter filter is not compiled into media_kit's
bundled `libmpv.so`** (confirmed by binary inspection + the runtime "No such
filter" error). When the `lavfi` filter fails to create, mpv **deselects the
video track** → no first frame → permanent black-screen stall. So a missing
filter is catastrophic, not benign — fix566's `isVfOptionError` suppressor
can't help (the set succeeds; the failure is a deferred create that breaks the
pipeline and surfaces as "video texture failed to attach"). Recompiling libmpv
with `--enable-filter=fps` was rejected: it's a prebuilt pub dependency
(`media_kit_libs_android_video`); NDK cross-compiling ffmpeg+mpv per media_kit
bump is disproportionate for one filter.

## Fix

The `select` filter **is** present in this libmpv. Use it to pass every other
frame, halving the frames reaching the VO (the texture-upload bottleneck the
fix564 overlay identified):

```
vf = lavfi=[select=not(mod(n\,2))]
```

Dart raw string `r'lavfi=[select=not(mod(n\,2))]'` so the backslash is literal.
The comma in `mod(n,2)` MUST be escaped or libavfilter splits the graph on it
(`No such filter: '2))'`). Validated on **mpv 0.41** locally: escaped form
creates the filter + reconfigures the VO cleanly; unescaped form fails exactly
as predicted. Because `select` exists, the filter always CREATES — worst case
the stream plays uncapped, it can NEVER stall like the absent `fps` filter.
fix566's reconnect-suppression is retained.
