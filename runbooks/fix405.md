# fix405: aspect-ratio button — revert the dynamic icon swap from fix404

**Repo:** rkinnc75/Free4Me-IPTV · **v1.35.4+405** · **APK target:** TV + phone player surface

## What this fix delivers

fix404 introduced a 3-state ZoomMode cycle (fit → stretch → crop) and tried to make the aspect-ratio button's icon and tooltip swap with the active mode, so a user could see at a glance which mode they were in. On-device testing showed the icon and tooltip did NOT change between modes — they were frozen on the values captured when the player first opened.

**fix405 reverts the dynamic icon and tooltip to a const `Icons.aspect_ratio_outlined` and a generic `'Aspect ratio'` tooltip.** The 3-state cycle itself is preserved — the engine's `setZoomMode(BoxFit)` call updates the video frame on every tap. The user discovers the active mode by the visible frame change.

## Why the icon was frozen

`MaterialVideoControlsTheme` (from media_kit_video) reads the `bottomButtonBar` parameter **once at mount** and does NOT re-read it on subsequent parent rebuilds. When the user tapped the aspect-ratio button, `_PlayerState.setState(() => _zoomMode = next)` triggered a parent rebuild, but media_kit's internal `MaterialControls` widget was still holding the bar that was passed in at first mount — and the icon inside that bar was the `Icon(_zoomMode.icon, ...)` from the first build, frozen on `Icons.aspect_ratio_outlined` (the default `ZoomMode.fit` icon).

This is the same class of bug fix367 hit on `primaryButtonBar` for the DVR transport row. fix367's workaround was to key the `MaterialVideoControlsTheme` subtree on `_engine.dvrActive`, forcing media_kit to remount and re-read the bar. The proper fix for the zoom icon would be a similar key trick — `ValueKey('mvct-dvr-${_engine.dvrActive}-${_zoomMode.name}')`. We chose not to do that because:

1. The remount cost is small (the `mkvideo.Video` child has its own stable `_videoKey` and is preserved across remounts), but it does re-run all the media_kit internal initialization.
2. The affordance was nice-to-have, not load-bearing. The 3-state cycle is the real feature — users can tell which mode is active by looking at the video frame, which DOES change on every tap (the engine's `setZoomMode` call is independent of the media_kit button bar).
3. Reverting to a const icon and a generic tooltip makes the code honest: the button no longer claims to do something it doesn't.

## What did NOT change

- `lib/models/zoom_mode.dart` (the `ZoomMode` enum with `next()` cycle, `boxFit` mapping) — kept as-is. The enum is still used by `toggleZoom()` and `MpvEngine.setZoomMode(BoxFit)`.
- `lib/player/mpv_engine.dart` (`BoxFit _zoomFit` field, `setZoomMode(BoxFit)` method, `fit: _zoomFit` on `mkvideo.Video`) — kept as-is. The 3-state cycle still routes through the engine.
- `test/zoom_mode_test.dart` (9 tests on the `ZoomMode` cycle, BoxFit mapping, distinct icons, non-empty tooltips) — kept as-is. The tests still pass and are still useful — they pin the production cycle and the `BoxFit` mapping that the engine consumes.
- The `updateAspectRatio(double)` method on `MpvEngine` — kept for backwards compatibility (no other call sites use it).

## Behavior

Before fix405: tap aspect-ratio button → cycle fit/stretch/crop → video frame changes → **icon does not change** (frozen on `Icons.aspect_ratio_outlined`).

After fix405: tap aspect-ratio button → cycle fit/stretch/crop → video frame changes → **icon does not change** (now honestly a const `Icons.aspect_ratio_outlined`).

The user-visible behavior is identical. The difference is that fix404's changelog claimed a feature that did not actually work, and fix405 corrects that claim.

## Risks

- **No regression.** The 3-state cycle still works (engine updates the video frame on every tap). Users who never tapped the aspect-ratio button see no change. Users who did tap see the same behavior as 1.35.3+404.
- **The `Icon(_zoomMode.icon, ...)` widget was technically `const`-eligible if the parameter were a const, but the `_zoomMode` field is non-const. Reverting to `const Icon(Icons.aspect_ratio_outlined, ...)` makes the icon tree fully const, which is a tiny perf win for the build phase.

## Tests

- `test/zoom_mode_test.dart` — 9 tests, all pass. The `ZoomMode` enum's `next()`, `boxFit`, `icon`, and `tooltip` getters are still tested. The `icon` and `tooltip` getters on `ZoomMode` are now dead code (no production caller), but the tests still pin them so future changes that try to bring back the dynamic icon have a known-correct cycle to lean on.
