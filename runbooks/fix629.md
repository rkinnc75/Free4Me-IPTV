# fix629 — Kill the last full-scans: EPG staleness check, re-match, and 3 dead indexes (v2.2.40)

**Builder:** desktop coder+builder. **Live base:** v2.2.39 (fix628).
**Ships as:** v2.2.40+629. Follow-on to the audit in
`docs/CHANNELS_SQL_INDEX_MAP.md` + `docs/APP_SQL_INDEX_MAP.md` — "why do we
still have full-scans; either index them or drop the unused index."

## What was still scanning (grounded in the audit + onn logs)

1. **`EpgService.isStale()`** — `SELECT count(*) FROM programmes WHERE
   start_utc <= ? AND stop_utc > ?`. No `source_id`, so `idx_programs_time_range
   (source_id, start_utc, stop_utc)` is unusable → **full aggregate over ~1M
   programme rows on every launch**, just to answer "is anything airing".
2. **`Sql.getChannelsNeedingEpgMatch()`** — the per-source unmatched-channels
   query the "Re-match all channels" job runs. The planner mis-picked
   `index_channel_source_id` / `idx_channels_epg_id` and full-scanned the
   per-source slice (**13-21 s each** on the onn, private log 2026-06-30). A
   purpose-built partial index `idx_epg_unmatched` already existed but nothing
   forced it, and ANALYZE can't fix the pick (averaged stats can't model the
   5-source, 149k-450k-row skew — the documented fix531 failure).
3. **Three bare single-column channels indexes** that no query can seek on —
   pure write-amplification on every channel insert during a refresh.

## The fix (three parts — corrected after `/rr`)

### 2a. `isStale` → per-source, index-only, early-exit
Take the eligible source IDs from `refreshIfStale` and probe each:
```sql
SELECT 1 FROM programmes
WHERE source_id = ? AND start_utc > ? AND start_utc <= ? AND stop_utc > ?
LIMIT 1
```
- `source_id =` seeks the leading column of `idx_programs_time_range`; the
  `start_utc` window (`now - 2 days`, `now]`) is a tiny range on the 2nd column;
  `stop_utc > now` is the 3rd column → **index-only, no table access**.
- Early-exit: returns "not stale" on the FIRST source with something airing (the
  common case touches one source, not the whole table).
- The `start_utc` floor is `now - 3 days` = the MAX EPG retention window
  (`epgPastDays` caps at 3). `deleteStalePrograms` never prunes a row still airing
  and no retained airing programme can have `start_utc` older than that window, so
  the floor is provably equivalent to the old unbounded `count(*)` within the data
  the DB actually holds (adversarial-review Q2 hardening).

### 2b. `getChannelsNeedingEpgMatch` → forced `INDEXED BY idx_epg_unmatched`
```dart
final hint = await _indexExists('idx_epg_unmatched')
    ? ' INDEXED BY idx_epg_unmatched' : '';
```
The query WHERE is **identical** to the index's partial predicate
(`media_type=0 AND epg_manual_override IS NULL AND epg_channel_id IS NULL`), so
the index contains exactly the rows the query returns → forcing it is a valid,
full-result path (same logic as fix627). Gated on `_indexExists` because
`withDroppedBrowseIndexes` drops it mid-refresh and a forced `INDEXED BY` on a
missing index is a hard crash (fix526). `getChannelsForEpgMatching`
(`source_id=? AND media_type=0` only) is deliberately **NOT** hinted — its WHERE
does not imply the partial predicate, so forcing the index there would drop rows.

### 2c. Drop 3 dead indexes (migration 41 + remove from the self-heal map)
Dropped: `index_channels_stream_id`, `index_channels_group_name`,
`index_channel_last_watched`. All bare single-column. Verified dead by grepping
every query:
- `stream_id`: zero WHERE/ORDER/JOIN use.
- `group_name`: used only in `source_id`-scoped `GROUP BY` / correlated joins,
  which drive off `index_channel_source_id` + the `groups` unique index — a bare
  `(group_name)` index can never be the chosen plan there.
- `last_watched`: the only ORDER BY (History view, sql.dart ~1170 & ~3378) always
  adds `AND last_watched IS NOT NULL` (~1127 & ~3349), so the kept PARTIAL
  composite `idx_channel_lastwatched_media (last_watched, media_type) WHERE
  last_watched IS NOT NULL` fully covers the sort — the bare index is redundant.

Implemented as **migration 41 `DROP INDEX IF EXISTS`** (db.sqlite chain) +
removal from `Sql._canonicalChannelIndexes`. Historical migration 1 is left
untouched (never rewrite history); on a fresh install it creates the 3 and
migration 41 immediately drops them (trivial, empty table). Removing them from
the canonical map stops the fix628 startup self-heal from resurrecting them;
`withDroppedBrowseIndexes` reads `sqlite_master` dynamically, so once dropped and
never recreated they are gone for good.

## Verification (onn 4K Plus) — measure AT REST, not mid-refresh

1. Sideload v2.2.40, launch. In the log confirm migration 41 ran once and
   `PRAGMA index_list(channels)` no longer lists the 3 dropped indexes
   (`adb shell "run-as ... sqlite3 db.sqlite 'PRAGMA index_list(channels)'"`,
   or grep the startup self-heal line — it should now report 16 canonical
   indexes, not 19).
2. Launch when the EPG is stale → confirm `isStale` is fast (no multi-second
   programmes scan) and, when a source has current programmes, it does NOT
   trigger an unnecessary foreground refresh.
3. Run **"Re-match all channels"** → each source's unmatched-channels fetch is
   now sub-second (was 13-21 s). If a query is slow the TimedDb harness logs
   `SLOW SQL` + its EXPLAIN QUERY PLAN — confirm the plan shows
   `SEARCH channels USING INDEX idx_epg_unmatched`.
4. Open the **History** view → still ordered most-recent-first, fast (served by
   `idx_channel_lastwatched_media`).
5. Run a full refresh → confirm it still completes and browse stays fast
   (regression check on the dropped indexes; the fix628 self-heal must report
   16, not resurrect the 3).
