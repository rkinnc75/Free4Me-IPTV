# fix676 — DIAGNOSTIC: write SR alarm-callback trace to the in-app log (no PC needed)

**Build:** `3.0.18+676` · **Base tag:** `v3.0.17` (fix675) · **Ship order:** after 675
**Files touched:** `lib/backend/recording_scheduler.dart`, `lib/whats_new_modal.dart`, `pubspec.yaml`
**Behavior change:** none — instrumentation only.

---

## Why (fix675 couldn't be read on the Samsung)

fix675 added `SR-CB:` breadcrumbs via `debugPrint`. On the latest device test (3.0.17+675) the exported report log showed the `RecordingScheduler: scheduled recording …` line and then **nothing** from the callback. That does NOT prove the callback didn't run: `debugPrint` only goes to Android **logcat**, and the in-app issue reporter transmits **`app_log.txt` only** (confirmed in `issue_reporter.dart` — it sends `AppLog.readLog()` and nothing else). So on a device without adb, fix675's breadcrumbs are unreadable by construction.

To make the trace visible on the Samsung, the breadcrumbs must be written **into `app_log.txt` itself**, from the alarm background isolate — where `AppLog` is a per-isolate singleton with logging disabled, so `AppLog.*` silently no-ops.

## What it does

Adds `_srDebug(msg)` — a plugin-free breadcrumb writer that **appends directly to `app_log.txt`** via `dart:io` (`File.writeAsStringSync(..., FileMode.append, flush: true)`), so the line lands in the exact file the reporter already sends. It:

- resolves the dir from `Utils.appDir` (`getApplicationSupportDirectory()`), and if that throws (path_provider unavailable in this isolate — a leading suspect), falls back to the fixed Android path `/data/data/me.free4me.iptv/files` (on Android `getApplicationSupportDirectory()` == `context.getFilesDir()`, verified: the native capture service already opens `db.sqlite` there);
- still calls `debugPrint` too (so adb on the onn keeps working);
- swallows every error — the diagnostic never changes behavior.

Each `SR-CB:` breadcrumb in `recordingAlarmCallback` is switched from `debugPrint(...)` to `await _srDebug(...)`. The FIRST (`entered …`) fires before any plugin/DB call. No control-flow change; no manifest change.

## How to read the result (Samsung, no adb)

Trigger a 2-min "Record now", wait ~30s past the scheduled fire time, then export the in-app report log and send it. In `app_log.txt` you'll now see `[SRDBG]` lines. Interpretation:

- **No `[SRDBG]` line at all** → the callback truly never runs (dispatcher/registration issue) — cause is in the alarm-manager wiring, not the callback body.
- **`SR-CB: entered …` then `SR-CB: … THREW MissingPluginException…`** → the alarm isolate has no plugin registrant; the first plugin call (DB path via `path_provider`, or the capture MethodChannel) fails.
- **`entered` → `got row` → `status->recording` → `THREW …` at start** → the callback works; the capture MethodChannel call is the failure — points at the native service / registrant for `me.free4me.iptv/recording`.
- **All breadcrumbs incl. `RecordingCapture.start returned OK`** → Dart handed off fine; failure is inside the native service (use the fix675 `SRCapture` logcat lines on the onn when available).

Send the log and it will pin the exact cause; the real fix follows.
