# fix180 — clear Analyze/Suggest history (`playback_metrics`) on new-version boot, alongside the debug-log clear

> **Version:** 1.23.12+178 → **1.23.13+180** (phone/even build; patch bump from shipped 1.23.12+178).
> **Evidence:** `Free4Me-IPTV-1.23.12` source zip (`version: 1.23.12+178`).
> **Request:** the app already clears the debug log file the first time it boots on a new version; it should, at the same moment, truncate the table that backs the Analyze/Suggest feature, because performance suggestions computed from sessions recorded on an *older* version can mislead (they reflect problems the new version may have already fixed).
> **What the data is:** the Analyze/Suggest feature ("Suggested settings" in Settings) aggregates `playback_metrics` rows. Each row is one analyzed playback session (first-frame ms, rebuffers, reconnects, etc.). `getAggregatedMetrics()` reads **all** rows and weights them into one summary that drives the suggestion. So pre-upgrade rows bias the aggregate — exactly the concern raised.
> **Fix:** wipe `playback_metrics` inside the existing once-per-version hook `maybeRotateLogOnVersionChange()`, right after `AppLog.clearLog()`. Same trigger, same idempotency guard (`lastLogClearedVersion`).

## Why this hook is the right place

`SettingsService.maybeRotateLogOnVersionChange()` (settings_service.dart) already does exactly the once-per-new-version work we want to extend:

```dart
static Future<void> maybeRotateLogOnVersionChange() async {
  final String version = (await PackageInfo.fromPlatform()).version;
  final settingsMap = await Sql.getSettings();
  if (settingsMap[lastLogClearedVersion] == version) return;   // idempotency guard

  await AppLog.clearLog();
  AppLog.info('Free4Me-IPTV $version — log cleared on version change');
  final HashMap<String, String> update = HashMap();
  update[lastLogClearedVersion] = version;
  await Sql.updateSettings(update);                            // remembers the version
}
```

Because the metrics truncate shares the log clear's trigger and guard, it runs exactly once per version bump and never again on that build — no separate flag needed. (The function is already called on startup; no new call site required. Verify the existing caller in the boot path is unchanged.)

## Schema evidence (preflight Rule 7)

`playback_metrics` is a standalone table — no foreign keys reference it, so a plain `DELETE FROM` fully and safely clears it. From `db_factory.dart` (migration 12/12b):

```
CREATE TABLE "playback_metrics" (
  id                        INTEGER PRIMARY KEY AUTOINCREMENT,
  session_start             INTEGER NOT NULL,
  session_minutes           REAL    NOT NULL,
  streams_opened            INTEGER NOT NULL,
  median_first_frame_ms     INTEGER NOT NULL,
  median_stable_ms          INTEGER NOT NULL,
  startup_visible_rebuffers INTEGER NOT NULL,
  total_rebuffers           INTEGER NOT NULL,
  visible_rebuffers         INTEGER NOT NULL,
  median_rebuffer_ms        INTEGER NOT NULL,

```

No migration/schema change is involved (data-only DELETE), so this is lower-risk than a migration; it was still executed in-container per Rule 8 (see Verification).

# Fix 180.1 — add `Sql.clearPlaybackMetrics()`

## `lib/backend/sql.dart` — add the method immediately after `insertPlaybackMetrics`

**Locate** the end of `insertPlaybackMetrics` (its closing `AppLog.info(...)` and `}`). Current verbatim:

```dart
    AppLog.info('Sql.insertPlaybackMetrics: session=${m.sessionStart} '
        'minutes=${m.sessionMinutes.toStringAsFixed(1)} '
        'streams=${m.streamsOpened} rebuffers=${m.totalRebuffers}');
  }
```

**Replacement** (append the new method after it):

```dart
    AppLog.info('Sql.insertPlaybackMetrics: session=${m.sessionStart} '
        'minutes=${m.sessionMinutes.toStringAsFixed(1)} '
        'streams=${m.streamsOpened} rebuffers=${m.totalRebuffers}');
  }

  /// fix180: wipe all stored Analyze/Suggest sessions. Called on the first
  /// boot of a new app version so that performance suggestions are not biased
  /// by sessions recorded before the new version's improvements.
  static Future<void> clearPlaybackMetrics() async {
    final db = await DbFactory.db;
    await db.execute('DELETE FROM playback_metrics');
    AppLog.info('Sql.clearPlaybackMetrics: playback_metrics truncated');
  }
```

**Key points:** matches the existing DAO idiom (`final db = await DbFactory.db;` then `db.execute`). A bare `DELETE FROM` (no transaction) is fine for a single statement; the table is unindexed by FK and the analyzer/suggestion reads tolerate an empty table (`getAggregatedMetrics` already returns a zero `AggregatedMetrics` when `rows.isEmpty`). Re-inserts after truncate work normally (autoincrement continues) — verified.

# Fix 180.2 — call it from the version-change hook

## `lib/backend/settings_service.dart` — truncate alongside the log clear

**Locate** the log-clear lines inside `maybeRotateLogOnVersionChange()`. Current verbatim:

```dart
    await AppLog.clearLog();
    AppLog.info(
      'Free4Me-IPTV $version — log cleared on version change',
    );
```

**Replacement:**

```dart
    await AppLog.clearLog();
    // fix180: also wipe Analyze/Suggest history so suggestions aren't biased
    // by pre-upgrade sessions. Same once-per-version trigger as the log clear.
    await Sql.clearPlaybackMetrics();
    AppLog.info(
      'Free4Me-IPTV $version — log + playback metrics cleared on version change',
    );
```

**Key points:** `Sql` is already imported in this file (`import 'package:open_tv/backend/sql.dart';`, used by `Sql.getSettings`/`updateSettings`), so no new import. The truncate sits *before* the `update[lastLogClearedVersion] = version` write, so if it throws the version isn't marked cleared and it retries next launch (consistent with the log-clear's own ordering). Placing it after `clearLog()` means a clean version boot wipes both together.

> **Ordering note:** `_capturePlaybackMetrics()` (main.dart, on app *pause*) inserts the *current* session's metrics. The version-change clear runs at *startup*, before any new session is captured, so it never races the capture — the first post-upgrade session is recorded after the wipe and is correctly retained.
