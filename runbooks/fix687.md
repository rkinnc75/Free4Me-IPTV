# fix687 — Instrument the FFI re-mux to locate the v4.0.2 failure (v4.0.3) — v2

**Builder:** desktop (Mac). **Live base:** v4.0.2. **Ships as:** v4.0.3+687.

**Why:** on v4.0.2 the re-mux fires (fix686 worked) but **both containers fail
with no visibility** (`mp4 copy failed` / `mkv copy failed` / `all containers
failed — keeping .ts`). `streamCopy` had ~15 silent `return false` points in a
background isolate. This fix is **pure diagnostics** — no logic change — so the
next capture names the failing call.

**v2 note:** v1's doc comment carried an orphaned bool-era line ("Returns true
only if…") and an angle-bracket token in a `///` comment. Both cleaned; verified
with the project's pinned **Dart 3.12.2 + repo analysis_options** (not a bare
SDK) → "No issues found!". Nothing was committed, so **v4.0.3 is not burned**.
Tag re-verified at start and before this diff — latest remote tag = **v4.0.2**.

---

## What

- `_RemuxNative.streamCopy` now returns a **`String`**: `''` on success, else a
  short `"<step> rc=<averror>"` tag at every failure point — `dlopen-threw`,
  `abi-mismatch`, `open_input rc=…`, `find_stream_info rc=…`, `no-streams`,
  `alloc_output rc=…`, `new_stream null`, `copy_params rc=…`,
  `no-av-streams-mapped`, `mp4-incompatible-codec`, `avio_open rc=…`,
  `write_header rc=…`, `packet_alloc null`, `write_frame rc=… (after N frames)`,
  `no-frames-read`, `write_trailer rc=…`, or `exception: …`.
- `_copyInIsolate` returns that string (or `isolate-threw: …`); `_processOne`
  logs it via the existing gated `[SRDBG]` channel:
  `id=<id> <ext> copy failed [<diag>]`. Also logs a null `createOutput`.
- **No change to the re-mux algorithm.** Same control flow, same success path.

The `rc` values are raw AVERROR ints; they decode to the exact ffmpeg reason,
telling us whether the failure is the `fd:` protocol, `find_stream_info`, the MP4
muxer needing AVCC extradata, a bad struct offset, etc.

## Verification

**Performed in sandbox (real):**
- **`dart analyze` with the project's Dart 3.12.2 + repo `analysis_options.yaml`
  (flutter_lints) → "No issues found!"** on `recording_remux.dart` (real `ffi`,
  stubbed Flutter/open_tv imports).
- Patch applies clean (`git apply --check`) to a **pristine v4.0.2 clone**.

**NOT performed (Mac/CI owns):** full-project `flutter analyze`, tests, native
compile, arm64 runtime.

## Gate (Mac + device)
1. CI: analyze 0 INFOs, tests green, release build succeeds.
2. Device (SM-S938U, debug logging ON): schedule a short recording with re-mux
   ON, open Recordings, then **send the log**. The `[SRDBG] id=… copy failed
   [<diag>]` line names the failing call + AVERROR — fix688 fixes the real cause
   from that. (No playback-regression risk — engine unchanged.)
