# fix669 — Scheduled Recording list: top-level Recordings destination (v3.0.13)

**Builder:** desktop. **Live base:** v3.0.12. **Ships as:** v3.0.13+669.

Third phase of the Scheduled Recording (SR) arc (667 schema → 668 capture →
**669 list/destination** → 670 creation actions → 671 optional transcode). Adds
the place recordings LIVE — a first-class destination beside Live/Movies/Series.
Split from the creation UI (fix670) so the navigation-enum change lands and can
be device-checked on its own.

## What

- **`ViewType.recordings`** APPENDED to the enum (index 5, after settings) —
  deliberately at the END so no existing index shifts (settings stays 4; all
  guards, the fixedIndex→viewIndex map, and stored `defaultView` values keep
  working). `viewTypeToString` handles it.
- **`lib/recordings_view.dart`** — the list screen. Groups by state; per-item
  actions: scheduled→Cancel (RecordingScheduler.cancel), recording→Stop
  (RecordingCapture.stop), done→Play (opens the captured file via Player
  overrideUrl), any→Delete (row only; confirms; stops first if running).
  Empty-state hint. Pull-to-refresh.
- **Phone (`bottom_nav.dart`)** — 5th fixed destination "Recordings"; the fixed
  arrays are length-driven so the render loop picks it up automatically. Routed
  like Settings (a pushed screen), NOT a Home filter — so the risky
  viewType-filter/browse path is untouched.
- **TV (`tv_shell.dart` + `tv_top_tab_bar.dart`)** — new `isRecordings` TvTab
  flag (mirrors `isSearch`); a "Recordings" tab after Search mounts
  RecordingsView via `_ensureBuilt`; `_landingIndex` maps a saved recordings
  default to Live (index 0). This runbook exposed a non-exhaustive ViewType
  switch in `_landingIndex` — now handled (the compile check caught it; that's
  exactly why the enum append gets a real analyze run).

Playing a recording builds a synthetic Channel (title only) and passes the
MediaStore path as Player.overrideUrl.

## VERIFY-FIRST — on-device

Mostly Dart/UI (low risk), but the play-a-recording path reads a real MediaStore
file, so:
1. Phone: a "Recordings" tab appears as the 5th bottom destination; opens the
   list; back returns to Live.
2. TV: a "Recordings" tab appears after Search; D-pad navigates in/out; the
   list is focusable.
3. With a done recording present (from fix668 on-device): Play opens and plays
   the captured file; Stop appears only for in-progress; Cancel only for
   scheduled; Delete removes the row (confirm dialog; Cancel autofocused).
4. Empty state shows the hint when there are no recordings.
