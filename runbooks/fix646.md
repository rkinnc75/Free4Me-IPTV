# fix646 — sub-100ms Favorites: partial favorites index + non-blocking fix418 stats

**Target tag:** `v2.2.45` (commit `9dd73490447ffe335b4ad5d24a7a87fc9540455f`)
**New version:** `2.2.46+635`
**Migration:** 43 (new partial index; additive, no data change)

## Summary

From the onn `2026-07-03T04:11` log: Movies -> Favorites took **75.1s for 3
rows** (`Sql.search sql=75135ms`). Breakdown:

- **15.3s — the query.** `favorite = 1` had no favorite-aware index; the
  planner used `idx_channels_browse_mt_safe (media_type=?)` and walked ~760k
  enabled movie rows testing `favorite = 1` per row. Row SELECTION cost — the
  ORDER BY tiers are irrelevant at 3 result rows.
- **~60s — the fix418 slow-query stats dump** (five full-catalog aggregates
  over 857k rows) ran INSIDE the caller's await chain, holding the screen.

Fixes (agreed A' + B; developer boxes keep full always-on diagnostics):

1. **A' — fix418 stats fire-and-forget** (`unawaited(_logBrowseStats())`).
   The synchronous `fix418 PLAN:` (EXPLAIN) line is untouched — it costs ~ms
   and prints with the query. The one-shot-per-process guard is unchanged.
   The dump runs on the read pool, so it does not block subsequent writes;
   its lines simply land in the log a few seconds after the PLAN line.
2. **B — migration 43**: partial index
   `idx_fav_browse ON channels(media_type, source_id, name COLLATE NOCASE)
   WHERE favorite = 1`. Favorites are a tiny set -> every favorites browse
   becomes a few index seeks. Index WHERE matches the query's `favorite = 1`
   term EXACTLY (SQLite's partial-index matcher is conservative; no COALESCE).
   `withDroppedBrowseIndexes` captures/recreates it automatically like every
   other non-unique channels index; near-zero insert-maintenance (few rows).

**Measured on a 300k-row seeded DB** (author sandbox, python sqlite3) with the
exact logged query shape: plan flips to
`SEARCH c USING INDEX idx_fav_browse (media_type=? AND source_id=?)`,
execution **0.21ms** (temp B-tree over the 3-row result is free).

No sort features removed — the fix375 validated-favorites ordering and all
BrowseOrder behaviour are unchanged.

2. Movies -> Favorites: rows should appear near-instantly; log should show
