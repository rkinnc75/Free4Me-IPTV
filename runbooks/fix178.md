# fix178 — app freezes on logo after upgrade to 1.23.11: fix174.1's unique index throws on existing data

> **Version:** 1.23.11+176 → **1.23.12+178** (phone/even build; patch bump from shipped 1.23.11+176).
> **Severity:** CRITICAL — app does not start. Cold launch hangs on the logo splash; no log can be exported because `runApp()` is never reached.
> **Evidence:** `Free4Me-IPTV-1.23.11` source zip (`version: 1.23.11+176`) + user report (freeze immediately after updating 1.23.9 → 1.23.11). Reproduced in-container against a 1.23.9-schema DB (see "Verification performed").
> **Bug:** fix174.1's migration 14 creates `CREATE UNIQUE INDEX channels_unique ON channels(source_id, media_type, COALESCE(stream_id,-1), COALESCE(series_id,''))`. The existing pre-upgrade database contains many rows that share `(source_id, media_type, -1, NULL)` — every "divider"/junk channel and every live/VOD row whose provider `stream_id` failed to parse (`int.tryParse(...) ?? -1`), plus any row with a null id. Creating a UNIQUE index over data that already violates it **throws** `UNIQUE constraint failed`. The throw propagates out of `migrations.migrate(db)` → `Sql.hasSources()` → the `await Future.wait([...])` in `main()`, so `runApp()` never executes and the splash never advances.
> **Why it also breaks fresh imports:** on a clean DB the same collision occurs the moment `insertChannelsBulk` tries to insert two sentinel-id rows in one transaction → the bulk INSERT throws mid-import.
> **Root cause:** the fix174.1 key assumed `stream_id`/`series_id` are always present and distinct. They are not: `xtreamToChannel` sets `streamId: int.tryParse(stream.streamId ?? "") ?? -1` (so unparseable → -1) and leaves `series_id` null for live/VOD. Coalescing the sentinels into the unique key makes all sentinel rows mutually conflicting. This is a design error in fix174.1, not a paste/escaping error.

## Reproduction (in-container, exact)

Old 1.23.9 channels schema + old `(name, source_id)` index, two realistic rows, then migration 14 verbatim:

```
rows before: 2
DROP INDEX IF EXISTS channels_unique;          -- ok
CREATE UNIQUE INDEX channels_unique ON channels(
  source_id, media_type, COALESCE(stream_id,-1), COALESCE(series_id,''));
  → Runtime error: UNIQUE constraint failed: index 'channels_unique'
```

Two rows named `##### DIVIDER A #####` and `##### DIVIDER B #####`, both `stream_id=-1`, `series_id=NULL`, collide under the new key. A 270k-channel provider always has such rows, so the upgrade always throws → freeze.

## The fix

Stop coalescing sentinels into the key. Enforce uniqueness **only on real ids**, using two **partial** unique indexes, and switch the upserts to a **bare `ON CONFLICT DO UPDATE`** (no explicit target) so SQLite picks whichever partial index applies — and inserts freely when none does.

- `channels_unique_stream` — UNIQUE on `(source_id, media_type, stream_id)` **WHERE `stream_id IS NOT NULL AND stream_id >= 0`** → real live/VOD streams dedupe; `-1` sentinels are exempt and coexist.
- `channels_unique_series` — UNIQUE on `(source_id, series_id)` **WHERE `series_id IS NOT NULL`** → real series dedupe; null-series rows exempt.

All four behaviours verified in-container: sentinel rows coexist (no throw), distinct real streams coexist (270k→46 bug stays fixed), duplicate real stream_ids conflict (true re-imports upsert), and bare `ON CONFLICT DO UPDATE` updates the matching row across either partial index including mixed batches.

# Fix 178.1 — migration 15: replace the throwing index with two partial indexes

## `lib/backend/db_factory.dart` — append migration 15

**Locate** the end of migration 14 (added by fix174.1) and the `await migrations.migrate(db);` line that follows. Current verbatim:

```dart
      ..add(SqliteMigration(14, (tx) async {
        await tx.execute('DROP INDEX IF EXISTS channels_unique;');
        await tx.execute('''
          CREATE UNIQUE INDEX channels_unique ON channels(
            source_id,
            media_type,
            COALESCE(stream_id, -1),
            COALESCE(series_id, '')
          );
        ''');
      }));
    await migrations.migrate(db);
```

**Replacement** (note migration 14's closing `}))` loses its `;` because the cascade continues into 15; migration 15 ends with `}));`):

```dart
      ..add(SqliteMigration(14, (tx) async {
        await tx.execute('DROP INDEX IF EXISTS channels_unique;');
        await tx.execute('''
          CREATE UNIQUE INDEX channels_unique ON channels(
            source_id,
            media_type,
            COALESCE(stream_id, -1),
            COALESCE(series_id, '')
          );
        ''');
      }))
      // fix178: migration 14's coalesced unique key threw UNIQUE-constraint on
      // upgrade because existing data has many rows sharing (-1, NULL) sentinels
      // (divider channels, unparseable stream_ids, live rows with null series_id).
      // Creating a UNIQUE index over already-duplicate data fails, which aborts
      // migrate() during cold start and freezes the app on the logo. Replace the
      // single coalesced index with two PARTIAL unique indexes scoped to REAL ids
      // only, so sentinel rows are exempt and coexist. Idempotent + self-healing:
      // drops the bad index whether or not migration 14 managed to create it.
      ..add(SqliteMigration(15, (tx) async {
        await tx.execute('DROP INDEX IF EXISTS channels_unique;');
        await tx.execute('''
          CREATE UNIQUE INDEX IF NOT EXISTS channels_unique_stream
          ON channels(source_id, media_type, stream_id)
          WHERE stream_id IS NOT NULL AND stream_id >= 0;
        ''');
        await tx.execute('''
          CREATE UNIQUE INDEX IF NOT EXISTS channels_unique_series
          ON channels(source_id, series_id)
          WHERE series_id IS NOT NULL;
        ''');
      }));
    await migrations.migrate(db);
```

**Key points:**
- Migration 15 drops `channels_unique` again (`IF EXISTS`) — this is the **self-heal**: on a phone where migration 14 *failed*, the bad index was never committed, so the drop is a no-op and the two good indexes get created. On a phone where 14 somehow succeeded (small/empty DB), the drop removes it. Either way the device ends in the correct state.
- Partial indexes use `WHERE` predicates; the `stream_id >= 0` guard excludes the `-1` sentinel explicitly (belt-and-suspenders with `IS NOT NULL`).
- **Critical caveat about the throwing migration:** because migration 14 throws on affected devices, `migrate()` aborts and the schema-version pointer is **not** advanced past 14 — so on next launch SQLite retries 14 and throws again. Migration 15 alone does not help if 14 keeps aborting first. **Therefore migration 14 must be neutralised too** — see Part B. (This is why a freeze, not just a missing index.)

## Part B — neutralise the throwing migration 14 (so migrate() can reach 15)

Migration 14 will re-run on every affected device until it stops throwing. Change its body so it no longer creates the bad index; leave it as a harmless drop (it must remain in the list at version 14 so the version sequence is unbroken for devices that already passed it).

Replace migration 14's body (the inner two `tx.execute` calls) so it only drops:

```dart

        // fix178: migration 14 originally created a coalesced UNIQUE index that
        // threw on existing data and froze the app at launch. Neutralised to a
        // no-op drop; the correct partial indexes are created in migration 15.
        await tx.execute('DROP INDEX IF EXISTS channels_unique;');
      }))
```

So the full Part A+B replacement for the migration-14 block plus new 15 is (apply as ONE atomic edit):


# Fix 178.2 — switch both upserts to bare `ON CONFLICT DO UPDATE`

The named target `ON CONFLICT (source_id, media_type, COALESCE(stream_id,-1), COALESCE(series_id,''))` references the deleted index and would error. A **bare** `ON CONFLICT DO UPDATE SET …` lets SQLite arbitrate against whichever partial index the row violates, and plain-inserts a row that violates neither (the sentinel rows). Verified in-container including mixed batches.

## Part A — `lib/backend/sql.dart`, `insertChannel` (line 69)

Current verbatim:

```dart
        ON CONFLICT (source_id, media_type, COALESCE(stream_id, -1), COALESCE(series_id, ''))
        DO UPDATE SET
```

Replacement:

```dart
        ON CONFLICT DO UPDATE SET
```

## Part B — `lib/backend/sql.dart`, `insertChannelsBulk` (line 134)

Current verbatim:

```dart
        ON CONFLICT (source_id, media_type, COALESCE(stream_id, -1), COALESCE(series_id, ''))
        DO UPDATE SET
```

Replacement:

```dart
        ON CONFLICT DO UPDATE SET
```

The `DO UPDATE SET …` body (url, group_name, media_type, stream_id, image, series_id, the COALESCE-guarded epg_channel_id, catchup_*) is unchanged in both. `excluded.*` still resolves normally with a bare conflict clause.

> **Note on the old `(name, source_id)` upsert at sql.dart:160:** that is a *different* method (an older/manual insert path) and is out of scope — it was not touched by fix174 and is not part of the import path. Leave it. (Verify it's not `insertChannel`/`insertChannelsBulk` before editing anything near it.)

# Fix 178.3 — preflight hardening: schema-line evidence (Rule 7) + migration-execution gate (Rule 8)

This is the lesson of fix174 turned into mechanical gates so it can't recur. fix174 had two assumption-based failures: (a) it wrote `COALESCE(series_id, '')` believing `series_id` was TEXT when it is `integer`, and (b) it built a UNIQUE index that throws on real data. Neither was caught by reading or by `flutter analyze`; both would have been caught by (a) pasting the schema line and (b) running the migration. Two new hard rules close exactly those gaps. **Documentation-only — no app behaviour, no compile impact.**

## Part A — `RUNBOOK-PREFLIGHT.md`: add Hard Rule 7 and Rule 8

**Locate** the end of Rule 6 and the `## Per-change checklist` header. Current verbatim:

```markdown
6. **No escaped `$` or `\u…` inside Dart string literals in code blocks (added fix176).**
   `'\$x'` / `'\${expr}'` / `'\u2022'` slip through inspection and get pasted verbatim —
   `$x` renders as literal text instead of interpolating, leaving the variable unused (fatal
   WARNING) or putting garbage in SQL. Grep every code block for `\$` and `\u`; there must
   be none. Prefer literal characters (`•`) over `\u` escapes.

## Per-change checklist (run mentally for each `# Fix N.M` block)
```

## Notes

- **Why analyze didn't catch this (and what would have):** the bug is a *runtime* migration failure against real data. `flutter analyze` passes. The only thing that catches it is executing the migration against a representative DB — now codified as preflight Rule 8 in **fix178.3**. fix174's gate checked compile but not migration-on-real-data; that gap is what bricked the app, and it (plus the schema-type assumption, now Rule 7) is closed in 178.3.
- **Numbering:** fix176 is shipped (`1.23.11+176`). This is a new runbook → next even **fix178** → **1.23.12+178**. v1.23.11 tagged/immutable.
- **`series_id` is `integer` in the schema** (not text) — fix174's `COALESCE(series_id,'')` mixed a string sentinel into an int column, compounding the bad key. The partial index sidesteps this entirely by never coalescing.
- This is a real design fix to fix174.1, owned as such — the earlier runbook's index was wrong, not merely mis-pasted.
