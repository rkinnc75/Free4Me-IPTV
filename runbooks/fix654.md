# fix654 — gzip all export files at level 9 (v2.2.54)

**Builder:** desktop. **Live base:** v2.2.53. **Ships as:** v2.2.54+643.

## Problem

Live onn testing tonight hit two crashes in the diagnostic export
(`Backup & Restore → Export settings to file → Yes`):

1. **Disk full** building the "Everything incl. databases" zip: the DB
   entries were added at `STORE` (level 0, fix594) so the zip held a
   byte-identical second copy of `db.sqlite` (954MB) + `epg.sqlite` on top
   of the originals — peak disk usage roughly doubled the already-large
   source-dump (533MB) + DB total, blowing through the box's free space.
2. **Out of Memory** building the small text-only zip on a retry: the
   `archive` package's pure-Dart `Deflate` buffered enough of a 532MB
   raw-source-dump text file to exhaust the heap, despite `addFile`
   nominally "streaming" from disk.

Separately: the existing code's `ZipFileEncoder().create(zipPath)` calls
never passed `level:`, so `ZipEncoder.startEncode` defaulted to
`Deflate.BEST_SPEED` — the fix594 comment claiming "deflate at max (level 9)"
was never actually true; the `9` passed to `addFile(...)` only toggles
STORE-vs-compress in this package version, it does not set a level.

## Root cause

The bundle builder relied entirely on the bundled `archive` package
(pure-Dart) for compression: slow, and its `Deflate` implementation isn't
memory-safe for files in the hundreds-of-MB range. Its `XZEncoder` was
briefly considered as an alternative but is a dead end — reading
`archive-3.6.1/lib/src/xz_encoder.dart` shows it only wraps bytes in a valid
`.xz` container **uncompressed** ("This encoder only currently supports
uncompressed data").

## Change

Compress with `dart:io`'s native `ZLibEncoder`/`GZipCodec` (real zlib, no new
pubspec dependency) instead of the `archive` package's Deflate, at level 9:

- **`SettingsIo.gzipBytes`** — small in-memory payloads (settings JSON, log
  text).
- **`SettingsIo.gzipFileStream`** — streams a file straight through
  `ZLibEncoder` to `<path>.gz`; never buffers the source. Used for the raw
  source dump and both SQLite DB snapshots, skipping the old plain-copy step
  entirely.
- **`SettingsIo.maybeGunzip`** — magic-byte (`1f 8b`) auto-detect on import,
  so pre-fix654 plain exports still load.

Every export item (log, settings, source dump, db.sqlite, epg.sqlite) is now
gzipped before it's offered as a standalone download or added to either zip.
Both zips (`ZipFileEncoder`) now add every entry at `STORE` — re-deflating
already-gzipped bytes would only cost CPU for no size gain.

Applies to both export paths per the ask:
- **Backup & Restore → Export settings to file**: `free4me-backup-*.json` →
  `free4me-backup-*.json.gz`.
- **Diagnostics QR/LAN export**: every bundle item gzipped; `.sqlite` →
  `.sqlite.gz`.

Import side (`importFromFile`, `importSourcesOnly`, the LAN portal's
`/import-sources` form) accepts both `.json` and `.gz` and auto-decompresses,
so existing plain-JSON backups on disk still import.

`fix572`'s `staleExportArtifactNames` backup-file match extended to also
recognize the new `.json.gz` suffix.
