# fix504 — universal arm+arm64 APK (fix 32-bit Android TV box launch crash)

**Version:** 2.0.3+503 → **2.0.4+504**
**Tag:** v2.0.4
**Type:** Build fix (regression). Partially reverses fix420's arm64-only build.
**Status:** Implemented + verified locally (analyze 0; 169 tests) AND on-device (universal APK launches on a 32-bit onn 4K box). Shipped via CI.
**Base:** fix503.

## Root cause (diagnosed via adb logcat on a Walmart onn 4K Plus / Amlogic S7D)
The box is **32-bit only** (`ro.product.cpu.abilist=armeabi-v7a`, `ro.product.cpu.abilist64` empty, `ro.zygote=zygote32`). fix420 built the APK **arm64-only** (`--target-platform android-arm64`), so on this box FlutterLoader crashed in `MainActivity.onCreate`: `Could not find 'libflutter.so'. Looked for: [armeabi-v7a], but only found: [arm64-v8a]`. The app died before any Dart ran — invisible to the in-app debug log; only logcat showed it.

## Fix
Build a **universal arm+arm64 APK** by changing `--target-platform android-arm64` → `--target-platform android-arm,android-arm64` in BOTH build paths (CI + manual). Restores 32-bit Android TV boxes (onn, many Amlogic) while keeping arm64.

## Trade-off
APK grows (media_kit's libmpv/FFmpeg native libs now ship for both ABIs). Mitigation: the audit's APK-size wins (enable R8/`shrinkResources`, drop the 428K `assets/icon.png` + unused deps) become more worthwhile to offset this. Split-per-ABI was considered but rejected (the single-asset in-app self-updater would need to publish + pick per-ABI assets).

## On-device verification
Universal profile APK installed on the onn box launched cleanly: process alive, loaded `lib/armeabi-v7a/libflutter.so`, Flutter engine + Dart VM initialized, reached the UI (Setup) — no FATAL.
