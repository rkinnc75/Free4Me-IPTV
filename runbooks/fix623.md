# fix623 ‚Äî Multi-view cells honor full livestream buffer + 30fps cap (not mini-player limits)

**Target:** v2.2.34+623 (diff against live tag **v2.2.33**, verified `git ls-remote --tags`)
**Apply check:** `git apply --check` PASSED on a fresh v2.2.33 clone.
**Analyze:** run on the build machine ‚Äî gate not asserted here. (Manual: braces balanced; mpv_engine paren delta +12/+12 over original.)

## Root cause (confirmed from the post-restart 09:24 session in the bottom-up log)
Multi-view CELLS are constructed with `previewMode: true` (multi_view_cell.dart) ‚Äî the same flag as the mini-player PiP. previewMode forces the tiny mini-player buffering: `demuxer-max-bytes = miniDemuxerMaxMB` (16 MB), half `bufferSize`, and it hard-disables the low-RAM 30 fps cap (`capFps = !previewMode && ‚Ä¶`). The device log confirms cells applying `previewMode=true demuxerMB=16 capFps30=false` regardless of the user's "Livestream cache (60 s)" / "Livestream demuxer max (512 MB)" settings. A 16 MB demux cap holds only ~40 s of a 3.5 Mbit/s stream ‚Äî exactly the observed ceiling.

The fix is NOT to turn previewMode off: the hwdec block's Tegra/Shield rainbow and onn low-RAM texture-contention paths are keyed on previewMode and MUST stay for multi-view. So the buffering behavior has to be separated from the decode behavior.

## Fix
New `multiViewMode` flag on `MpvEngine`. Multi-view cells now pass BOTH `previewMode: true` (keep the decode handling) AND `multiViewMode: true`. For buffering only, the mini-player limits now apply just to the actual PiP (`previewMode && !multiViewMode`); multi-view cells + the full player use the real values:
- livestream demuxer cap ‚Üí `liveDemuxerMaxMB` (512), not `miniDemuxerMaxMB` (16); VOD ‚Üí `vodDemuxerMaxMB`.
- `bufferSize` ‚Üí full, not half.
- 30 fps low-RAM cap ‚Üí allowed in cells (they're the juddering case), gated off only for the PiP.
The options-applied log now also prints `multiViewMode` and the true `demuxerMB`/`bufferSizeMB`. Decode/hwdec paths are untouched (still keyed on previewMode), so Tegra/Shield/onn handling is preserved. The three other MpvEngine sites (full player, tv_hero_preview, overlay mini-player) default `multiViewMode=false` ‚Äî unchanged.

## Note on applying settings
These buffer/fps options are applied when a stream OPENS. Changing the sliders and returning to an already-open multi-view won't change a running cell ‚Äî reopen multi-view (or the stream) to pick up new values. (A UI hint that "buffering changes apply to newly opened streams" is a reasonable future follow-up.)

## Files
- `lib/player/mpv_engine.dart` ‚Äî `multiViewMode` flag; buffer/demuxer/bufferSize/fps-cap use full values unless it's the PiP; richer log.
- `lib/multi_view_cell.dart` ‚Äî cell passes `multiViewMode: true`.
- `pubspec.yaml`, `lib/whats_new_modal.dart` ‚Äî version bump + changelog.
