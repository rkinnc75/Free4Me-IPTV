# fix350 — ExoPlayer removed: libmpv is the sole playback engine → v1.28.0+350

> ## WARNING BUILD MACHINE — APPLY THE PATCH, then RELEASE (EXACTLY). No questions, no DB to query.
> **FIRST:** `find . -name 'fix296-DIAGNOSE*.md' -delete 2>/dev/null || true`
> Built on live main **1.27.1+349** (fix349 must already be shipped — it is). If `origin/main` is not 1.27.1+349 — **STOP**.

## Why
Two engines meant every buffering/EOF/watchdog fix had to be reasoned twice and the paths kept diverging (fix335/342 existed solely for Exo's signalling gaps). libmpv already plays every format (MPEG-TS, HLS, DASH, MP4, RTMP) and carries all device-specific decode fixes (fix164 onn, fix314 Shield). Rich's call: go exclusive with libmpv. Net: **−2,717 lines, +44**, one dependency dropped.

## What this removes
- **Files deleted (6):** `lib/player/exo_engine.dart`, `lib/player/engine_picker.dart`, `lib/models/engine_preference.dart`, `lib/models/engine_type.dart`, `test/engine_preference_test.dart`, and **`sql.dart` at the repo root** — a stale pre-fix194 copy of `lib/backend/sql.dart` accidentally committed around fix331; it only analyzed clean because the engine classes still existed.
- **Dependency:** `video_player` dropped from `pubspec.yaml` (pubspec.lock untracked; `flutter pub get` regenerates).
- **player.dart:** engine picker/creator, `_engineType`, fix316 runtime engine-fallback (`_tryPreferenceFallback`, `_swapEngine`), Exo source-error fallback, the Exo controls overlay + fix336 VOD transport bar and its scrub state/listeners (mpv uses media_kit controls — fix348's live control panel untouched). Engine is constructed directly: `MpvEngine(...)`. fix116 engine adoption/handoff unchanged.
- **multi_view_cell.dart:** EnginePicker call + Exo ternary → always MpvEngine. fix341/342/345/346 recovery logic untouched; Exo-era comment rationale condensed.
- **Settings:** `forcedEngine` + `enginePreference` fields, props, parse/serialize, and the Settings → "Player engine" tile + picker dialog removed. Old settings JSON/backups with those keys import harmlessly (unknown keys ignored).
- **Models/SQL:** `Channel.engineOverride`, `Source.defaultEngine` removed; `Sql.setChannelEngineOverride` (zero callers) deleted; `default_engine` dropped from source INSERT/UPDATE statements with binds realigned.

## What this deliberately KEEPS
- **DB columns `channels.engine_override` and `sources.default_engine` stay in the schema, never read/written again.** Dropping them = full-table rewrite migration on a 737K-row channels table on 2GB boxes — the exact fix178 launch-freeze class. Migration ceiling stays **27**; historical migrations untouched (upgrade chain).
- **`PlayerEngine` interface** stays as the engine seam (fix116 handoff/adoption typing).
- Historical changelog entries mentioning ExoPlayer are history — untouched.
