# fix627 ‚Äî Force idx_browse_src_grp on category browse (v2.2.38)

**Builder:** desktop coder+builder (implemented + built + on-device verified).
**Live base:** v2.2.37. **Ships as:** v2.2.38+627. (Restructure item #1 of 3.)

## Symptom

Opening a Live TV **category** took **12‚Äì43 s** to show a small list (5‚Äì36
channels) on the onn 4K Plus. Private log 2026-07-01 (v2.2.36), plan:
`SEARCH c USING INDEX index_channel_source_id (source_id=?) | USE TEMP B-TREE
FOR ORDER BY`.

## Root cause

The no-query browse filters `source_id IN(...) AND group_id = ?` but the
planner seeks by **source_id alone**, reads the whole source partition
(~275k‚Äì450k rows/source, ~1.16M total), residual-filters `group_id`, then
temp-B-tree sorts. `sqlite_stat1` is **averaged** and can't model the catalog's
group-size skew (a few huge groups + thousands of tiny ones), so `ANALYZE`
(fix530) can't make the planner prefer the composite `idx_browse_src_grp`
`(source_id, group_id, tier, name)` that exists precisely for this. The team
already force the right index on the ungrouped/media-type paths (fix230/528/531)
‚Äî the **category / group_id path was never forced**, so it kept regressing.

## The fix

Force `INDEXED BY idx_browse_src_grp` on the grouped browse (groupId set),
gated on `Sql._indexExists` (fix526 pattern ‚Äî the index is dropped mid-refresh /
absent on some upgraded DBs, and a forced INDEXED BY on a missing index is a
hard crash). Applied in **both** no-query browse builders:
- single-query path (`Sql.search`, new `useSrcGrpHint`),
- per-source UNION path (`_browseMixedUnion`, extended `safeHint`, grouped takes
  priority over the safe-mode hint; mutually exclusive on groupId).

**Correctness:** `idx_browse_src_grp` is a PARTIAL index
`WHERE url IS NOT NULL AND series_id IS NULL`. Both predicates are provably
emitted for a grouped browse ‚Äî `url IS NOT NULL` in the query, `series_id IS
NULL` from `VisibilityClause` whenever groupId is set ‚Äî so results are
IDENTICAL. Verified empirically (test `browse_index_hint_guard_test.dart`,
fix627 case): a forced partial index whose WHERE is not implied is a **hard "no
query solution" error, never silent row loss**; and with both predicates the
hint is honored and returns the row. Result ORDER is unchanged ‚Äî the category
ORDER BY still uses a temp-B-tree, but now over the category's few rows instead
of the whole source.

## Scope note

This is **#1 of 3**. It eliminates the SCAN (the dominant cost). It does NOT yet
eliminate the temp-B-tree SORT ‚Äî negligible for small categories but still real
for huge ones (Movies 100k+). That's **#2** (align the index column-order to the
browse ORDER BY / precomputed sort key). **#3** trims the ~10 overlapping browse
indexes and re-ANALYZEs after the refresh index-recreate.
