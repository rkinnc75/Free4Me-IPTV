# fix236 — HOTFIX release 1.25.1: refresh crashed with "no such index: channels_unique" (fix230 was wrong)

> **Cuts a complete, ready-to-tag release 1.25.1.** Contains every edit: the real refresh fix (a new migration + corrected UPDATE), pubspec bump, changelog entry, regenerated version.json.
> **Severity:** 1.25.0's refresh is BROKEN — it throws `SqliteException(1): no such index: channels_unique` and the refresh fails entirely. This is a fast-follow hotfix.
> **Builds on:** the live **1.25.0+230** source.
> **Version:** **1.25.1+236** (patch bump; build `+236` = highest fix number). Tag `v1.25.1`.

## What went wrong (my error in fix230)

fix230 added `UPDATE channels INDEXED BY channels_unique`. I assumed `channels_unique` existed because migration 1 creates it (`CREATE UNIQUE INDEX channels_unique ON channels(name, source_id)`). **But migrations 14 and 15 (fix174/178) DROP it** — `DROP INDEX IF EXISTS channels_unique;` — and re-key uniqueness to provider stream/series ids (`channels_unique_stream`, `channels_unique_series`). So on any upgraded device, `channels_unique` does NOT exist → the hint throws.

The deeper truth this exposed: **there is no index on `channels(name, source_id)` on the device at all** anymore. restorePreserve matches on `name AND source_id`, so with only single-column indexes the planner fell back to `index_channel_source_id` (source_id only) → scan all same-source channels per row → the original ~134s. fix230's hint was trying to force a fast path via a non-existent index.

## The correct fix (verified under real device index conditions)

1. **Create the missing index** `index_channel_name_source ON channels(name, source_id)` as **migration 18** (non-unique — display names repeat across a catalog).
2. **Remove the broken `INDEXED BY channels_unique` hint** — with a real `(name, source_id)` index present, the planner picks it automatically.

Reproduced in-sandbox with the REAL sqlite_async package, simulating a post-migration-15 device (no `channels_unique`, only single-column + provider-id indexes) + migration 18:
- Plan: `SCAN p` + `SEARCH channels USING COVERING INDEX index_channel_name_source` — no scan, no hint.
- Time: **25ms** (vs ~134s with the source_id-only fallback).
- Rule-8 correctness: all rows match expected (auto COALESCE-keep-fresher, manual pin, untouched rows & other source_id preserved).

## Fix 236.6 — `lib/backend/db_factory.dart`: read-only index inventory log (diagnostic)

A missing index now only causes slowness, not a crash (after 236.2 there are NO `INDEXED BY` hints left in the code). Migration 18 deliberately does NOT re-create all indexes — blindly ensuring every index exists would risk resurrecting indexes earlier migrations intentionally dropped (e.g. `channels_unique`, removed by fix174/178), re-introducing old bugs. Instead, this adds a one-shot READ-ONLY log of which indexes actually exist on the channels/groups/sources tables, so any other gap is visible in the next log without changing the schema.

Insert immediately AFTER the existing SQLite-version log block in `_createDB` (the FIRST one, for `db.sqlite`).

### Replacement code (verbatim)
```dart
    try {
      final row = await db.get('SELECT sqlite_version();');
      AppLog.info('Sqlite: runtime version=${row.columnAt(0)}');
    } catch (_) {
      // Non-fatal; the rest of the app still works without the log line.
    }

    // fix236: one-shot read-only index inventory. Logs which indexes actually
    // exist on the channels/groups/sources tables of THIS (possibly long-
    // upgraded) database, so a bug report shows real on-device index state
    // rather than what current migration source implies. Read-only — creates
    // and drops nothing. Helps catch any index a past migration removed/renamed.
    try {
      final idx = await db.getAll(
        "SELECT name, tbl_name FROM sqlite_master WHERE type = 'index' "
        "AND tbl_name IN ('channels', 'groups', 'sources') "
        "ORDER BY tbl_name, name",
      );
      final listed =
          idx.map((r) => '${r.columnAt(1)}.${r.columnAt(0)}').join(', ');
      AppLog.info('Sqlite indexes (channels/groups/sources): $listed');
    } catch (_) {
      // Non-fatal diagnostic.
    }
```

> Produces one log line, e.g.: `Sqlite indexes (channels/groups/sources): channels.index_channel_name, channels.index_channel_name_source, channels.index_channel_source_id, groups.index_group_unique, sources.index_source_name`. Compare against expectations to spot any missing index. (After 1.25.1 confirms the device's index set, this line can stay cheaply or be removed in the fix222 cleanup.)
