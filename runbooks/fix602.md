# fix602 — v2.2.13: Live-guide channel menu also opens by touch long-press

Small follow-on to fix600/fix601. The guide's `_FocusTile` had ONLY the held-OK
key detector (fix600/fix601) to open `_showChannelMenu` — no `InkWell.onLongPress`
(unlike `channel_tile`, which has both). Consequences:
- Touchscreen users couldn't open the guide channel menu by long-press.
- The menu was NOT verifiable on a non-rooted TV box: a sustained ~450ms D-pad
  hold can't be injected via adb (`sendevent` is permission-denied; `input
  keyevent --longpress` registers as a quick press → plays), and the
  synthetic-touch-long-press trick (`input swipe x y x y 600`) that works for
  `channel_tile` couldn't reach a tile with no `onLongPress` wired. So the HIGH
  desync bug fix601 fixed had originally slipped through unverified.

**Change:** `_FocusTile`'s InkWell now wires `onLongPress: widget.onLongPress`
(when non-null — i.e. channel tiles). The remote path is unchanged (the held-OK
key detector on `node.onKeyEvent`); this just adds the touch gesture as a second,
independent trigger for the SAME menu. No conflict: key events vs touch gestures
are separate input modalities, and the key detector still pre-empts InkWell's
ActivateIntent for select keys.

**Why ship a release (not a local debug build):** the onn already holds a full
Xtream source (~149k channels) + EPG; sideloading a differently-signed debug APK
would require an uninstall (wiping that data). A release-signed v2.2.13 installs
over v2.2.12 in place, so the menu can be verified on the real box without data
loss.

## On-device verify (onn 4K Plus)
After install, in the Live guide enter a category → focus a channel → inject a
synthetic touch long-press on the focused tile (`adb shell input swipe X Y X Y
700`) → the channel menu must open with Play / Open in Multi-view / favorite.
This is the first true on-device confirmation that the guide `_showChannelMenu`
renders correctly (incl. the fix600 "Open in Multi-view" entry and the fix601
favorite handling).
