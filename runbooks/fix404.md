# fix404: 3-state zoom toggle (fit / stretch / crop) — replace the old fill toggle

**Repo:** rkinnc75/Free4Me-IPTV · **v1.35.3+404** · **APK target:** TV + phone player surface

## What this fix delivers

The aspect-ratio icon in the player controls (`Icons.aspect_ratio_outlined`, between the audio and mini-player buttons) used to be a 2-state toggle (`bool fill`) that flipped between two aspect values via media_kit's `aspectRatio` override. It had no third "fill with crop" option.

fix404 replaces the toggle with a **3-state cycle**: `fit → stretch → crop → fit`. Each state picks a distinct Material `BoxFit` for the underlying `mkvideo.Video` widget:

| State | Icon | `BoxFit` | Behavior |
|---|---|---|---|
| **fit** (default) | `Icons.aspect_ratio_outlined` | `BoxFit.contain` | Letterbox: preserve native aspect, may show bars. |
| **stretch** | `Icons.fit_screen_outlined` | `BoxFit.fill` | Stretch: force viewport aspect, may distort. |
| **crop** | `Icons.crop_landscape_outlined` | `BoxFit.cover` | Fill with crop: fill viewport, may lose content at edges. |

Each state has its own distinct icon and tooltip so the user can tell which mode is active at a glance without reading text.

## Why 3 states (not 2, not 4)

**Not 2 (pre-fix404):** The 2-state toggle couldn't fill the screen without distorting. Cinema content (1.85:1, 2.39:1, 2.76:1) on a 16:9 device either letterboxed (preserving aspect, showing bars) or stretched (distorting horizontally). `BoxFit.cover` lets the user fill the screen and crop the top/bottom (or left/right for narrower content) — useful for cinema content where the original letterbox was intentional.

**Not 4 (zoom / rotate / etc.):** Zoom (Transform.scale) and Rotate (Transform.rotate) are real features but orthogonal to "how does the frame map to the viewport." They'd add 2-3 more states for a 4- or 5-state cycle, which is too long for a TV-remote cycling pattern. Each new state also has a non-degenerate failure mode on common IPTV aspect ratios. fit/stretch/crop is the maximum useful set; adding more crosses into diminishing returns.

## Cycle order and UX

`fit → stretch → crop → fit` (3 taps to return).

The default starting state is `fit` (BoxFit.contain), matching media_kit's default — so users who never touch the icon see no change from the pre-fix404 behavior.

The cycle progresses from "least distortion" to "most distortion": fit (no distortion) → stretch (distorts) → crop (no distortion but loses content). Returning to fit (no distortion) closes the loop.

The icon button shows the icon of the current state, so a user who tapped once and walked away can come back and see `Icons.fit_screen_outlined` (stretch mode active) without remembering which mode they were in.

## Architecture

The change touches 3 layers:

1. **Model layer** — `lib/models/zoom_mode.dart` (new file). 3-value enum with `next()` cycle, `boxFit` mapping, `icon` per state, and `tooltip` per state. Mirrors the `lib/models/multi_view_decode.dart` pattern.

2. **Engine layer** — `lib/player/mpv_engine.dart`. New `BoxFit _zoomFit` field on the engine (defaults to `BoxFit.contain`, matches pre-fix404). New `setZoomMode(BoxFit)` method that calls `_videoKey.currentState?.update(fit: fit)`. The `buildVideoView` getter passes `fit: _zoomFit` to `mkvideo.Video`. The pre-fix404 `updateAspectRatio(double)` method is kept for backwards compatibility (no other call sites).

3. **Player state layer** — `lib/player.dart`. `bool fill = false` replaced with `ZoomMode _zoomMode = ZoomMode.fit`. `toggleZoom()` cycles `_zoomMode.next()` and calls `engine.setZoomMode(next.boxFit)`. The icon button uses `_zoomMode.icon` and `_zoomMode.tooltip` for live UI.

Multi-view cells inherit the engine's current fit (no per-cell override). When a user opens multi-view after setting `crop` in full-screen, the cells render with the same `crop` fit. This matches the engine's "one fit for all surfaces" routing.

## Verification commands

- **Zoom (Transform.scale):** considered and rejected as the 4th state — explained above.
- **Rotate (Transform.rotate):** considered and rejected — explained above.
- **Persistence across sessions:** the `_zoomMode` is session-only, matching the pre-fix404 `bool fill` pattern. Users who set a custom fit want it to reset to fit when they restart the app. Settings-level persistence is scope creep.
- **Per-cell override in multi-view:** the engine's fit applies to all surfaces. Adding per-cell override would require a separate API and isn't requested.
- **`BoxFit.fitWidth` / `fitHeight` / `scaleDown`:** all degenerate or collapsing for at least some common IPTV aspect ratios. See `findings-icon-aspect-matrix.md` for the full matrix.

## What I did NOT change

- **The `updateAspectRatio(double)` method on MpvEngine** — kept for backwards compatibility. No other call sites use it (the previous `toggleZoom` was the only caller). It could be removed in a future cleanup.
- **Multi-view cell rendering** — uses the engine's `buildVideoView`, which now respects `_zoomFit`. No changes needed to `multi_view_cell.dart`.
- **Other player surface aspects (DVR, multi-view decode, hwdec routing)** — unrelated to fit mode.
- **The `_helpXxx` text** for the aspect-ratio icon — there's no existing help tuple for the icon. If the user wants help text added, ship as a follow-up.

## Risks

- **First run on any device:** defaults to `fit` (BoxFit.contain), which matches pre-fix404 behavior for the never-touched case. No regression.
- **Upgrade from 1.35.2:** the `bool fill` field is gone, but it was session-only and never persisted, so no migration needed. Users get `ZoomMode.fit` on first run after upgrade.
- **The pre-fix404 `updateAspectRatio` path** — if any other code path called it (none found), behavior is unchanged because the `aspectRatio` override still works.
- **Multi-view cells** — render with the user's current fit. A user who set `crop` in full-screen will see the same crop in multi-view. This is consistent (not a bug) but could surprise some users.
