# fix341 — Manage provider connection-cycling in multi-view: quick re-open + stability buffer → v1.26.57+341

## Why (S24 log series + 20 historical logs)
Quantified across all S24 2x2 logs: genuine mid-play stalls are near zero; virtually every rebuffer sits within seconds of an engine restart, and restarts are driven by REAL provider EOF drops cycling round-robin every ~25-30s across cells — Dino (`max_connections=1`) enforcing its limit against 4 concurrent streams. Historical logs confirm: the "worked better before" sessions ran on A3000, which tolerates 4 streams (3-hour session, generations ~4, zero completed events). No app regression; the visible "stutter" is the dispose -> spinner -> reopen cycle on every drop. Live streams cannot be buffered ahead of real time, so the managed answer is (1) make each drop visually minimal and (2) optionally trade delay-behind-live for a cushion.

## What this does
**Layer 1 — same-engine quick re-open (always on):** on `stream completed`, the cell now re-opens the stream on the SAME engine instead of disposing it — the last frame stays on screen (no spinner/black), shrinking the visible artifact to the reconnect time (~1-3s freeze). Budgeted (8 per stable period, replenished after 15s stable, reset on fresh starts); any failure falls back to the existing full dispose+restart path. Headers are cached at open so the quick path needs no DB read.

**Layer 2 — optional stability buffer (Settings > Multi-view stability buffer: Off/15s/30s, default Off):** on a FRESH cell start, the engine opens then PAUSES for N seconds while mpv's cache accumulates (cells already run liveCacheSecs=45), showing "Building Ns buffer…", then plays — running N seconds behind live. mpv keeps playing buffered data after a network EOF and only fires completed when the cache drains, so brief drops play through invisibly. Drop-reconnects do NOT re-pause (they return to live edge immediately; layer 1 keeps them visually clean). The cell startup watchdog is extended by the buffer length so it cannot fire during the intentional pause. Honest limits are stated in the in-app help: the picture is delayed by the buffer length (bad for live sports), and a 30s cushion against a 25-30s drop cycle absorbs most but not necessarily all drops.

**Plumbing:** new `Settings.multiViewStabilityBufferSecs` (int, default 0) persisted like `streamCompletedDelayMs`; uses the fix336 engine `pause()`/`play()`; the picker uses fix331's `selectedId`.

## Note
- This manages the symptom; it cannot make 4 streams work on a 1-connection account. The round-robin pattern is consistent with kill-oldest-on-new-connect, meaning our reconnects may trigger the next cell's drop — the buffer masks this, the quick re-open minimises it.
