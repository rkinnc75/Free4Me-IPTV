## What this does
Wires fix315's stored engine fallback to actual runtime behaviour. When the primary engine fails to render a stream, the player now swaps to the configured fallback engine once before spending the reconnect budget on the same engine. Two trigger points:
1. **open() error** — generalized the previous hardcoded ExoPlayer→libmpv fallback to honour the user's `enginePreference.fallback` (libmpv↔ExoPlayer).
2. **startup watchdog (no frame / black screen)** — open succeeded but no frame arrived within `bufferingWatchdogSecs`; this is the classic "this engine can't render this stream" signal. Now tries the fallback engine once, then reopens, instead of immediately reconnecting on the same engine.

One-shot per playback (`_prefFallbackTried`); only fires for the two-engine preferences (libmpv→ExoPlayer, ExoPlayer→libmpv) and only when not already on the fallback engine. Single-engine modes (libmpv only / ExoPlayer only) never swap.

## Scope / limits (as agreed)
- Detects **stalls / black screen / open failure** — NOT colour corruption. Corrupted frames decode "successfully," so there is no signal to trigger on; the fix314 manual Multi-view decode toggle remains the answer for Shield colour issues.
- Reuses the existing `_swapEngine()` machinery (already used by the prior exo-fallback), so the mid-flight swap path is proven.

## Verified before delivery

- Eligibility logic tested: fires once for libmpvExo (libmpv→exo) and exoLibmpv (exo→libmpv); never for libmpvOnly/exoOnly; never twice; never when already on the fallback engine.
