# fix605 — v2.2.16: categories-mode empty EPG grid + auto-play backstop

From the user's live feedback during v2.2.15 testing.

## #1 — empty EPG grid while browsing categories
User: "when the focus is on the categories, the EPG should not have values
because there are no channels shown. At first opening, it looked like it showed
the EPG for the favorites, but no category was open."

`_init` preloads `_loadGuide(null)` (Favorites) so `_channels`/`_progByKey` are
populated, and the grid rendered those programmes even though the rail was in
CATEGORIES mode and no category had been opened — reads as "a guide for a
category that isn't open."

Fix (`tv_guide_view.dart`):
- `_grid()` returns a centered "Open a category to see its guide" hint when
  `_railMode == RailMode.categories`. The grid populates only in channels mode
  (after OK-enter).
- `_timeHeader()` returns an empty `SizedBox(height:16)` in categories mode (no
  floating timeline over an empty grid).
The preview box + hero still reflect the last-watched channel (the intended
preview behavior); only the EPG grid is blanked, per the user's ask.

## #2 — auto-play-on-enter backstop
While testing v2.2.15 I saw entering Favorites once play the first favorite
("|4K| ESPN UHD", previewMode=false). The fix603 orphan-KeyUp guard in
`_FocusTile` handles the real-remote bleed (the category-select OK's key-UP
landing on the freshly-focused first channel — caught because that node never
saw a matching key-DOWN, regardless of how long OK is held). But a very fast
category load (Favorites) can race focus such that an activation reaches the
first channel anyway.

Backstop (`tv_guide_view.dart`): `_enterChannels` stamps `_enterChannelsAtMs`;
`_play` ignores any play within `_enterPlayGuardMs` (700ms) of it, logging
"ignoring play … within enter-channels guard (bleed-through)". A deliberate
"open category → OK a channel" is always slower than this window. The orphan
guard remains the primary fix; this is a timing-agnostic safety net.

Note: adb `input keyevent` is atomic, so it cannot reproduce the real-remote
DOWN-then-UP bleed precisely — the adb play was likely the whole CENTER landing
on the channel after a fast swap. Both guards together cover the real remote;
the user confirms with the physical remote.
