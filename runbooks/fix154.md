# fix154.md — Feature: "Analyze playback & suggest settings" — a rolling-history log analyzer with one-tap Apply-all

> **Version:** Free4Me-IPTV 1.23.1 build (1.23.0 was the last shipped).
> **Goal:** A per-device, fully-LOCAL feature (no telemetry, no server) that
> reads the app's own playback logs, accumulates a rolling history across
> sessions in a small DB table, computes real buffering/startup/reconnect
> metrics, and SUGGESTS specific setting changes the user can review and
> "Apply all". This generalizes the manual tuning analysis to every user — the
> recommendations are personalized because they're based on that user's own
> device + network + providers, which `Settings.optimisedFor()` (RAM + isTV
> only) cannot capture.
>
> **Design (per product decisions):**
> - **Rolling history across sessions** (new DB table).
> - **Suggest, with "Apply all"** (never auto-applies).
> - **Lives next to "Optimise for this device" / Reset** in Settings.

---

## Architecture overview

Three pieces:
1. **Metrics extraction** — parse `AppLogger.readLog()` for known markers,
   produce a per-session `PlaybackMetrics` summary.
2. **Rolling history** — persist each session's summary in a new
   `playback_metrics` table (capped, e.g. last 50 sessions); aggregate on demand.
3. **Recommendation engine + UI** — turn aggregated metrics into setting deltas,
   shown in a review dialog with "Apply all", reusing the existing
   `_confirmAndResetSettings(builder: ...)` apply path.

**Data source:** `AppLogger.readLog()` returns the log file as a `String`
(`lib/backend/app_logger.dart`). **Important constraint: debug logging is OFF by
default** (`AppLogger._enabled = false`). The feature must detect "no/low data"
and tell the user to enable logging (Settings → Diagnostics) and watch a few
streams first. The rolling history mitigates this over time.

**Apply path:** the existing `_confirmAndResetSettings(builder: () => Settings)`
in `settings_view.dart` is the established "produce a Settings and persist it"
mechanism — the recommendation UI reuses it (build a `Settings` from current +
deltas).

**Safety ceilings:** never recommend past `DeviceMemory.maxLiveDemuxerMb` /
`maxMiniDemuxerMb` / `maxBufferSizeMb` (clamp helpers already exist in
`lib/backend/device_memory.dart`).

# Fix 154.1 — Metrics model + log parser

**New file:** `lib/backend/playback_analyzer.dart`

Define the per-session metrics and a parser over the log text. Key off these
EXACT markers (confirmed in `lib/player.dart` / `mpv_engine.dart`):

| Marker (substring) | Meaning | Metric |
|---|---|---|
| `MpvEngine: open() command sent` | stream open began | startup t0 |
| `Player: startup watchdog cancelled (buffering=false)` | first frame / playback began | time-to-first-frame (vs preceding open) |
| `Player: buffering=true` | a rebuffer began | rebuffer count + start ts |
| `Player: buffering=false` | rebuffer ended | rebuffer duration (paired with prior true) |
| `Player: overlay → "Buffering..."` (grace=false) | a VISIBLE rebuffer (past grace) | visible-rebuffer count |
| `Player: stream stable for Ns` | reached stable | time-to-stable |
| `Player: onDisconnect — attempt X/Y reason="..."` | a reconnect, with reason | reconnect count by reason |
| host in `open()` URL (`https://HOST/...`) | provider | per-host grouping (HOST ONLY — never store full URL) |

```dart
class PlaybackMetrics {
  final DateTime sessionStart;
  int streamsOpened = 0;
  // startup
  final List<int> timeToFirstFrameMs = [];   // open → first buffering=false
  final List<int> timeToStableMs = [];        // open → "stream stable"
  int startupVisibleRebuffers = 0;            // visible buffering within ~grace+3s of open
  // steady-state
  int totalRebuffers = 0;
  int visibleRebuffers = 0;                   // overlay shown (grace=false)
  final List<int> rebufferDurationsMs = [];
  double sessionMinutes = 0;                  // for rate-per-hour
  // reconnects
  int reconnectsWatchdog = 0;                 // reason="buffering watchdog"
  int reconnectsError = 0;                    // reason contains "player error"/"ffurl"
  int gaveUp = 0;                             // "max reconnects reached"
  // per-host (HOST ONLY, no credentials)
  final Map<String, int> rebuffersByHost = {};

  // derived
  double get rebuffersPerHour =>
      sessionMinutes > 0 ? totalRebuffers / (sessionMinutes / 60.0) : 0;
  int get medianFirstFrameMs => _median(timeToFirstFrameMs);
  // ... _median helper ...
}

class PlaybackAnalyzer {
  /// Parse a log string into one session's metrics. Splits on the session
  /// banner ("--- Log cleared ---" / "App started") if multiple are present;
  /// analyzes the MOST RECENT session by default.
  static PlaybackMetrics parseLatestSession(String logText) { /* ... */ }

  /// Parse ALL sessions present in the log (for first-run backfill).
  static List<PlaybackMetrics> parseAllSessions(String logText) { /* ... */ }
}
```

Parsing notes for the implementer:
- Lines are `[YYYY-MM-DD HH:MM:SS] [LEVEL] message`. Parse the timestamp prefix
  (the logger writes `DateTime.now().toLocal().toString().substring(0,19)`).
- Pair each `buffering=true` with the next `buffering=false` for duration;
  ignore unpaired trailing `true`.
- "Startup" rebuffer = a `buffering=true` whose ts is within
  (settings.startupGraceMs + 3000)ms of the most recent `open() command sent`.
- **Extract HOST ONLY** from the URL (`Uri.parse(url).host`) — NEVER store or
  log the path (it contains account/feed credentials, per the DNS report).
- `sessionMinutes` = last_ts − first_ts of the session.

**File:** `lib/backend/sql.dart` (or a new `playback_metrics_dao.dart`) — add:
- `Future<void> insertPlaybackMetrics(PlaybackMetrics m)` — INSERT one row;
  then DELETE all but the newest 50 (`DELETE FROM playback_metrics WHERE id NOT
  IN (SELECT id FROM playback_metrics ORDER BY session_start DESC LIMIT 50)`).

> Per-host data is intentionally NOT persisted to the table (keeps it simple and
> credential-free). Host grouping is used only in the live session view if you
> want a "this provider is the problem" note; do not store hosts.

# Fix 154.3 — Recommendation engine

**File:** `lib/backend/playback_analyzer.dart` (same file as 154.1)

Turn aggregated metrics + current settings + device caps into a list of
`SettingRecommendation`s. **Conservative, sample-size-gated, clamped to device
ceilings.**

```dart
class SettingRecommendation {
  final String settingKey;      // e.g. 'bufferSizeMB'
  final String label;           // 'Buffer size'
  final num currentValue;
  final num suggestedValue;
  final String rationale;       // human text shown in the dialog
  final bool requiresRestart;   // bufferSizeMB does
}

class Recommender {
  /// Returns [] if insufficient data (see minimums). Never recommends past
  /// DeviceMemory.max*Mb. Steps are small (one notch at a time).
  static List<SettingRecommendation> recommend(
      AggregatedMetrics agg, Settings s) {
    final recs = <SettingRecommendation>[];
    // Data sufficiency gate:
    if (agg.totalMinutes < 20 || agg.totalStreams < 3) return recs; // too little

    // Rule A — frequent steady-state rebuffers → bigger cache + buffer.
    if (agg.rebuffersPerHour >= 4) {
      final newCache = (s.liveCacheSecs + 15).clamp(10, 120);
      if (newCache > s.liveCacheSecs) {
        recs.add(SettingRecommendation(
          settingKey: 'liveCacheSecs', label: 'Live cache (seconds)',
          currentValue: s.liveCacheSecs, suggestedValue: newCache,
          rationale: 'About ${agg.rebuffersPerHour.toStringAsFixed(1)} rebuffers/'
              'hour across ${agg.totalMinutes.round()} min watched. A longer '
              'read-ahead reduces mid-stream rebuffering.',
          requiresRestart: false));
      }
      final newBuf = (s.bufferSizeMB + 64).clamp(16, DeviceMemory.maxBufferSizeMb);
      if (newBuf > s.bufferSizeMB) {
        recs.add(SettingRecommendation(
          settingKey: 'bufferSizeMB', label: 'Buffer size (MB)',
          currentValue: s.bufferSizeMB, suggestedValue: newBuf,
          rationale: 'More demuxer read-ahead so playback does not outrun the '
              'buffer. Capped at your device limit (${DeviceMemory.maxBufferSizeMb} MB).',
          requiresRestart: true));
      }
    }

    // Rule B — visible rebuffer right after start → widen startup grace.
    if (agg.startupVisibleRebufferRate >= 0.4 && s.startupGraceMs < 2000) {
      final newGrace = (s.startupGraceMs + 700).clamp(0, 2500);
      recs.add(SettingRecommendation(
        settingKey: 'startupGraceMs', label: 'Startup grace (ms)',
        currentValue: s.startupGraceMs, suggestedValue: newGrace,
        rationale: 'Streams often show a brief "Buffering" right after starting '
            'while the cache fills. A longer grace hides that initial settle.',
        requiresRestart: false));
    }

    // Rule C — watchdog firing then quick recovery → watchdog too aggressive.
    if (agg.reconnectsWatchdogPerHour >= 2 && agg.medianRebufferMs < 8000
        && s.bufferingWatchdogSecs < 20) {
      final newWd = (s.bufferingWatchdogSecs + 5).clamp(5, 30);
      recs.add(SettingRecommendation(
        settingKey: 'bufferingWatchdogSecs', label: 'Buffering watchdog (s)',
        currentValue: s.bufferingWatchdogSecs, suggestedValue: newWd,
        rationale: 'The watchdog is triggering reconnects on brief stalls that '
            'recover on their own. A longer watchdog avoids unnecessary reconnects.',
        requiresRestart: false));
    }

    // Rule D — slow first frame → longer open timeout (avoid premature fail).
    if (agg.medianFirstFrameMs >= 6000 && s.openTimeoutSecs < 20) {
      final newTo = (s.openTimeoutSecs + 4).clamp(5, 30);
      recs.add(SettingRecommendation(
        settingKey: 'openTimeoutSecs', label: 'Open timeout (s)',
        currentValue: s.openTimeoutSecs, suggestedValue: newTo,
        rationale: 'Streams take a median '
            '${(agg.medianFirstFrameMs/1000).toStringAsFixed(1)}s to start. A '
            'longer open timeout prevents giving up on slow-starting streams.',
        requiresRestart: false));
    }

    // Rule E — gave-up events dominated by ONE provider → informational only.
    // (Surface as a note in the UI, NOT a setting change — it's the provider.)

    return recs;
  }
}
```

> Thresholds above are starting points — tune after dogfooding. The KEY
> principles: (1) require a minimum sample (`totalMinutes`, `totalStreams`);
> (2) one small step per run (re-analyze after applying to step again);
> (3) clamp to `DeviceMemory.max*`; (4) when reconnects cluster on one host,
> say "provider issue," don't change settings.

# Fix 154.4 — Capture a session summary into history

**File:** `lib/main.dart` (or wherever app lifecycle is handled) + the analyzer.

On a natural boundary — app pause/exit, or when the user opens the analyzer —
parse the current session and insert a summary row:

```dart
// fix154: persist this session's playback metrics for rolling history.
// Cheap, local, best-effort.
Future<void> capturePlaybackMetrics() async {
  if (!AppLogger.instance.enabled) return;           // no data unless logging on
  final text = await AppLogger.instance.readLog();
  final m = PlaybackAnalyzer.parseLatestSession(text);
  if (m.streamsOpened == 0) return;                  // nothing to record
  await Sql.insertPlaybackMetrics(m);
}
```

Call it from `WidgetsBindingObserver.didChangeAppLifecycleState` on
`AppLifecycleState.paused` (add the observer in the app root if not present), so
each backgrounding snapshots the session. Best-effort; wrap in try/catch.

> Avoid double-counting: key the insert on `session_start`; if a row with the
> same `session_start` exists, UPDATE instead of INSERT (the session grows as it
> continues). Simple approach: delete any existing row with this `session_start`
> before inserting.

```dart
                      "for your device and connection. You choose what to apply.",

```

Add the handler:

```dart
  Future<void> _runPlaybackAnalysis() async {
    // 1) make sure there's data
    if (!AppLogger.instance.enabled) {
      // offer to enable logging
      final enable = await showDialog<bool>(/* "Enable debug logging and watch a
        few channels, then run this again." [Enable] [Cancel] */);
      if (enable == true) { await AppLogger.instance.setEnabled(true); /* persist */ }
      return;
    }
    // 2) snapshot current session into history, then aggregate
    await capturePlaybackMetrics();
    final agg = await Sql.getAggregatedMetrics();
    if (agg.totalMinutes < 20 || agg.totalStreams < 3) {
      // "Not enough data yet — watch a few more channels (need ~20 min) and try
      //  again." dialog
      return;
    }
    // 3) compute recommendations
    final recs = Recommender.recommend(agg, settings);
    if (recs.isEmpty) {
      // "Your playback looks healthy — no changes recommended. (Based on N min
      //  across M streams.)" dialog
      return;
    }
    // 4) review dialog with per-item current→suggested + rationale, and
    //    [Apply all] / [Cancel]. On Apply all, build a Settings from current +
    //    deltas and persist via the SAME path the optimise button uses.
    final apply = await showDialog<bool>(/* list recs; Apply all / Cancel */);
    if (apply != true) return;

    final updated = settings.copy(); // or clone current
    for (final r in recs) {
      switch (r.settingKey) {
        case 'liveCacheSecs': updated.liveCacheSecs = r.suggestedValue as int; break;
        case 'bufferSizeMB': updated.bufferSizeMB = r.suggestedValue as int; break;
        case 'startupGraceMs': updated.startupGraceMs = r.suggestedValue as int; break;
        case 'bufferingWatchdogSecs': updated.bufferingWatchdogSecs = r.suggestedValue as int; break;
        case 'openTimeoutSecs': updated.openTimeoutSecs = r.suggestedValue as int; break;
      }
    }
    await SettingsService.updateSettings(updated);   // persists + updates _cached
    if (mounted) setState(() => settings = updated);
    // toast: "Applied N changes. Some (buffer size) take effect next launch."
  }
```

> Reuse the existing confirm/apply styling from `_confirmAndResetSettings` for
> consistency. If any applied rec `requiresRestart`, include the "next launch"
> note (same wording the Optimise/Reset tiles use).

## Test plan

### No-data path
1. Fresh install, logging OFF, open the analyzer.
2. **Expected:** offered to enable logging; after enabling, told to watch a few
   channels and return. No crash, no bogus recommendations.

### Insufficient-data gate
1. Logging ON, watch one channel for 2 minutes, run analyzer.
2. **Expected:** "Not enough data yet (need ~20 min / 3 streams)." No changes.

### Healthy playback
1. Accumulate good sessions (no rebuffers).
2. **Expected:** "Playback looks healthy — no changes recommended," with the
   sample size shown.

### The real case — rebuffering
1. On a connection that rebuffers (like the YES Network HD pattern: periodic
   blips + a startup blip), accumulate ~20+ min, run analyzer.
2. **Expected:** suggests liveCacheSecs↑ and/or bufferSizeMB↑ (Rule A) and
   startupGraceMs↑ (Rule B), each with a rationale citing the measured rate.
   "Apply all" persists them; buffer-size change notes next-launch.

### Clamping
1. On a low-RAM device, force conditions that would push past
   `DeviceMemory.maxBufferSizeMb`.
2. **Expected:** suggestion is clamped to the device max; never exceeds it.

### Rolling history
1. Run across several app sessions (background/foreground between).
2. **Expected:** recommendations reflect AGGREGATE behavior (weighted by
   minutes), not just the last session; `playback_metrics` keeps ≤50 newest rows.

### Re-run after apply
1. Apply suggestions, watch more, re-run.
2. **Expected:** if still rebuffering, suggests the NEXT small step (one notch);
   if resolved, reports healthy. (Confirms the one-step-at-a-time design.)

### Privacy
1. Inspect the `playback_metrics` rows and any analyzer logging.
2. **Expected:** NO stream URLs, NO credentials, NO hosts stored in the table.
   (Host is used only transiently in-session, never persisted.)

## Notes for the implementer
- **Fully local** — reads the on-device log file, writes a local DB table. No
  network, no telemetry. This is the "same feature for every user" — each
  device analyzes its own logs.
- **Logging-off is the main caveat** — the feature must gracefully prompt to
  enable logging and gate on sample size. Rolling history makes it better over
  time once enabled.
- **Credential safety is mandatory** — parse host via `Uri.parse(url).host`;
  NEVER store/log the URL path (it carries account+feed token). The
  `playback_metrics` table stores only aggregate counts/durations.
- **Conservative by design** — sample-gated, one small step per run, clamped to
  `DeviceMemory.max*`. Suggest only; user taps Apply all.
- **Reuse existing apply path** — `SettingsService.updateSettings` (+ the
  `_confirmAndResetSettings` styling) so behavior matches Optimise/Reset.
- **Thresholds are seeds** — tune Rules A–E after dogfooding on real logs.
- **Files:** new `playback_analyzer.dart`; edits to `db_factory.dart`
  (migration 10), `sql.dart`, `main.dart`, `settings_view.dart`. Build
  `1.23.1+154`, tag `v1.23.1`, even = phone.
