# fix534 — TV tabs don't refresh after the enabled-source set changes

**Base tag:** `v2.0.33`
**Version:** `2.0.33+533` → `2.0.34+534`
**Scope:** TV shell + guide reload wiring. No schema, no query change. Full suite green.

---

## Problem (reproduced from device report)

On the TV UI, set sources to only A3000 → open Categories (shows A3000 live categories). Go back to the source list, disable A3000 + enable Trex → reopen Categories: **still shows the A3000 list.** Same staleness affects Live, Movies, and Series.

Root cause: `TvShell` builds each tab once and keeps it alive in an `IndexedStack` (`_ensureBuilt` returns early if `_built[i] != null`). Every TV content view loads its data only in `initState` (`TvGuideView`, `TvBrowseView`, `TvSearchView`, `TvCategoriesView`), so when the user changes which sources are enabled and returns to a tab, `initState` does not re-run and the view shows the previous source set. There is no implicit visibility hook with `IndexedStack` (the same constraint fix510 called out for the guide preview).

Confirmed via field log fix533: the empty-live-category data (`groups.enabled=0` on Trex/A3000 live) is genuine user-disabled state — NOT addressed here. This fix only repairs the screen staleness that made it impossible to see/correct the source set on device.

## Fix

Force the active tab to re-query when it becomes current:
- **Guide (tab 0):** add a public `reloadGuide()` (calls the existing `_loadGuide(null)`, which already reads live settings per fix524). `TvShell` calls it via the existing `_guideKey`.
- **Other tabs (Search / Movies / Series / Categories / Home-based):** fold a `_reloadGen` counter into each tab's `ValueKey`; on (re)select, bump `_reloadGen` and null the cached widget so `_ensureBuilt` rebuilds it with a fresh key → `initState`/`_load` re-runs. This mirrors the fix524 History-clear precedent (rebuild cached tab with a fresh key to force re-query).
- Trigger points: `_select(i)` (tab change) and `_openSettings()` (now `async`, reloads the current tab on return, since sources are changed in Settings).

The fix524 History long-press key scheme is updated to the combined `-$_reloadGen-$_historyGen` so the two generations don't fight.

### Deliberately NOT included

The fix298 disabled-category restore was investigated and is **correct** for the normal case (the stash is captured fresh at wipe time, so pre-refresh re-enables survive). The only latent defect — same-named cross-media categories — is mostly unreachable because the `groups` unique key is `(name, source_id)` and collapses such pairs into one row. A true fix there means adding `media_type` to the groups unique key (a migration), which belongs in its own fix + benchmark. Left out here on purpose.
