# fix678 — Scheduled Recording capture fails to start: MissingPluginException on me.free4me.iptv/recording

**Build:** `3.0.20+678` · **Base tag:** `v3.0.19` (fix677) · **Ship order:** after 677
**Files touched:** `lib/backend/recording_capture.dart`, `android/app/src/main/kotlin/me/free4me/iptv/SrStartReceiver.kt` (new), `android/app/src/main/AndroidManifest.xml`, `lib/whats_new_modal.dart`, `pubspec.yaml`
**Adds a pub dependency:** `android_intent_plus: ^5.3.0` (see build note below).

---

## Root cause (confirmed on device)

fix677 fixed the recording id; the fix676 `[SRDBG]` trace then showed the callback getting all the way to capture start and failing there:

```
SR-CB: got row id=9 rec=status=scheduled urlLen=62
SR-CB: id=9 status->recording; calling RecordingCapture.start
SR-CB: id=9 THREW MissingPluginException(No implementation found for method startCapture on channel me.free4me.iptv/recording)
```

`RecordingCapture.start` used the `me.free4me.iptv/recording` **MethodChannel**, whose native handler is registered in `MainActivity.configureFlutterEngine`. The alarm callback runs in **android_alarm_manager_plus's headless background isolate**, whose Flutter engine has **no MainActivity** — it only gets the plugins in `GeneratedPluginRegistrant` (pub packages). So that channel does not exist in the isolate and the call throws.

Verified mechanism (from plugin sources, pinned versions): the alarm plugin (v4) has no plugin-registrant hook; its background `FlutterEngine` registers only `GeneratedPluginRegistrant` plugins; app-local channels added in `MainActivity` never attach there. `GeneratedPluginRegistrant.java` is git-ignored/regenerated, so it can't be hand-edited durably.

## Fix — start the service via an explicit broadcast the isolate CAN send

`android_intent_plus` is a pub plugin (same flutter-community family as the alarm plugin), so it IS in `GeneratedPluginRegistrant` and attaches to the background engine; its Android side uses `applicationContext` (no Activity), and `IntentSender.sendBroadcast` calls `applicationContext.sendBroadcast(intent)` — verified background-isolate safe.

- `RecordingCapture.start` now sends an **explicit** (`package` + `componentName`) broadcast `me.free4me.iptv.SR_START` carrying id/url/durationMs/name, instead of the MethodChannel. Explicit + `exported=false` keeps it private and clear of Android O+ implicit-broadcast limits.
- New `SrStartReceiver` (manifest-declared, `exported=false`) reads the extras type-agnostically (the intent codec may encode a Dart int as Integer or Long) and calls the existing `RecordingCaptureService.start(applicationContext, …)`. Starting the FGS here is within the exact-alarm foreground-service-start grace window.
- `RecordingCapture.stop` / `freeBytes` keep the MethodChannel — they run only from the main (UI) isolate, where it exists.

The fix675/676 `[SRDBG]` diagnostics remain in place for one confirming pass (fix679 will gate them behind the debug-logging setting, per decision).

## Build note (IMPORTANT for the build machine)

This adds a dependency, so the build must run `flutter pub get` before building (regenerates `pubspec.lock` and `android/app/src/main/java/io/flutter/plugins/GeneratedPluginRegistrant.java`, both git-ignored). If `apply_fix.sh` / CI already runs `flutter pub get` before `flutter build` (it does for every build), no extra action is needed — the new plugin will register automatically. The diff intentionally does NOT include `pubspec.lock` or `GeneratedPluginRegistrant.java` (generated, git-ignored).
