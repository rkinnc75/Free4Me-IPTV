# fix614 — Batch refresh: targeted per-source FTS re-index (≈66min → ≈48s) + import fetch timeout setting

**Target:** v2.2.25+614 (diff against live tag **v2.2.24**, verified `git ls-remote --tags`)
**Apply check:** `git apply --check` PASSED on a fresh v2.2.24 clone.
**Analyze:** run on the build machine — no Flutter toolchain in the authoring sandbox, so the 0-issue gate is NOT asserted here. (Manual: brace balance OK on all files; code-only paren balance verified after stripping comments/strings; the apparent raw paren mismatch was prose inside comments.)

## Benchmark — MEASURED in-sandbox against the real production DB
Rich's backup (`db.sqlite`, 766 MB, **857,522 channels** across 4 sources: 273,711 / 287,950 / 149,370 / 146,491; FTS5 `unicode61`, `prefix='2 3'`, external-content) was used as the seed.

| Approach | Time |
|---|---|
| Current global `INSERT INTO channels_fts(channels_fts) VALUES('rebuild')` | exceeded the sandbox limit; matches the **~66 min** (3,983,119 ms) seen in production log `2026-06-29T21-31-58Z` |
| **fix614 targeted per-source DELETE+INSERT (all 4 sources)** | **~48–51 s** (≈15s, 15s, 10s, 11s per source) |

**Correctness** was verified, not just speed: FTS search for `"(AU)"` returned **3168 matches before and after** the targeted reindex (identical), and SQLite's FTS5 `integrity-check` PASSED. So the targeted path produces a bit-identical index to a full rebuild.

This clears Rich's "max 4 min per source" target with large headroom — FTS now adds ~10–15 s per source.

## Root cause (from the production log)
The whole 4-source refresh ran **15:46:31 → 16:57:55 ≈ 71 min**. Breakdown: fetch+load+commit ≈4 min, browse-index rebuild ≈1 min, and the **FTS global rebuild ≈66 min (93% of the runtime)**. The batch wraps the loop in one `withSuspendedFtsTriggers` with no `refreshedSourceId`, so its finally always did the full-catalog rebuild — re-tokenizing all 857K rows including the costly `prefix='2 3'` index. (The Trex failure itself cost only ~80 s; fix611 already handled it correctly — retried once, skipped, continued.)

## Fix
1. **`Sql.reindexFtsSources(List<int>)`** *(new)* — per-source DELETE-by-rowid (chunked 900) + INSERT-by-source from current `channels`. Cost ∝ refreshed rows, not whole catalog.
2. **`Sql.withSuspendedFtsTriggers`** gains `batchTargetedRebuild`. When supplied AND the body succeeded, the finally runs it (then restores triggers with `skipRebuild`) instead of the global rebuild. On the error path, or when absent, the global rebuild stays as the safe fallback.
3. **`refreshAllSources`** collects `succeededIds` (enabled minus failures) and passes `batchTargetedRebuild: () => Sql.reindexFtsSources(succeededIds)`. A fetch-failed source never wiped its channels, so its existing FTS rows stay valid and it is correctly omitted. M3U included (targeted works by source_id regardless of type).
4. **`devImportFetchTimeoutSecs`** setting (default **60s**, range **0–120**, 0 = built-in default) plumbed into `getXtreamHttpData` → `AppHttp.getWithRetry(timeout:)`. This is SEPARATE from `devNetworkTimeoutSecs` (libmpv playback). Fixes Trex failing the old hardcoded 20s (it responds in ~56 s). Includes model field + serialization (settings_io, settings_service) + a Developer-options slider with help text.

## Safety / correctness notes
- The end-of-batch targeted re-index only runs when `body` succeeded; any throw → global rebuild (unchanged).
- Per-source wipe aligns with FTS targeted delete: `wipeSource` deletes channels by `source_id`, so deleting+reinserting that source's FTS rows mirrors it exactly. Verified no orphans via FTS integrity-check.
- Search stays gated (`ftsRebuilding`) during the targeted re-index, flag always cleared.
