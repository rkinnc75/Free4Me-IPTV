# fix694 — refresh perf 1/2: FTS finalization DROP+repopulate (v4.1.2)

**Builder:** desktop. **Live base:** v4.1.1. **Ships as:** v4.1.2+694.
First of the Tier-1 refresh-perf releases (peer-analysis roadmap; owner-approved
separate-release plan). Baseline: Trex 290k rows = **18m12s** on the onn
(2026-07-08 private-repo log, v3.0.14).

## What changed (expected ≈ −3m20s)

1. **Big-source FTS finalization: fts5 `'rebuild'` → DROP+repopulate**
   (`sql.dart` withSuspendedFtsTriggers, non-targeted branch). A single-source
   refresh of a source too big for the targeted path (>50k rows or >20% of
   catalog — Trex is both) previously finalized via
   `reconcileFtsTriggers(true)` → `INSERT INTO channels_fts(channels_fts)
   VALUES('rebuild')` = **231s** observed (re-tokenizes into the existing
   segment structure, merge/tombstone overhead). It now calls
   `rebuildFtsTableFromScratch()` — the same DROP+repopulate that fix621 uses
   at end-of-batch and fix619/620 use for self-heal — **41.6s** observed for
   the identical 1.16M-row job, identical end state (fresh index, byte-identical
   sync triggers). The failed-mid-body path also uses it (strictly safer: DROP
   discards any half-state).

2. **fix620 integrity-check pre-flight relocated** (`utils.dart` refreshSource →
   `sql.dart` targeted branch). The refresh-entry
   `VALUES('integrity-check')` cost **10.9s on every refresh** but only ever
   protected the targeted delete (a corrupt index hangs it). It now runs only
   when the targeted path is about to be taken; on a malformed index the
   targeted path is skipped (no rebuild mid-suspend — the finalization
   DROP+repopulate restores consistency without touching corrupt shadow
   tables).

Not in this fix (deliberately): the index-diet work — verification showed
`idx_channels_browse_enabled` is load-bearing (fix365 test) and the peer's
"collapse to 2 indexes via tier column" does not survive our sort-mode matrix
(rebuild cost is sort-bound, not expression-bound). Any index drops come later
with per-index EXPLAIN proof. fix695 (next) = EPG ingest: ETag/304, exact
tvg-id match tier, parse-time window filter.
