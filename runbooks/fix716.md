# fix716 — TV player OSD Phase 4 unit 3: Channel Bar (v4.1.20)

**Builder:** desktop. **Live base:** v4.1.19. **Ships as:** v4.1.20+716. TV only;
phone/touch UI byte-identical. Overnight autonomous (owner: Option B + go).
Plan: `lib/player/ai/tv_osd.plan.md`. Completes the Phase 4 three-bar OSD.

## What (display-only, Option-B-compatible)
New `lib/player/tv_osd/channel_bar.dart` — `PlayerChannelBar`: a horizontal
`SizedBox(height:64)` `ListView.builder` over the current surf group
(`PlaybackPlaylist.channels`). The tuned channel (`playlist.index`) gets the
accent border + full opacity + a glass fill; neighbours are dimmed (0.55). A
`ScrollController` + post-frame `jumpTo` centers the tuned channel. Each chip:
logo (`CachedNetworkImage` 32px, errorWidget→empty) + name (2-line ellipsis).

Wired into `player.dart` `_buildTvOverlay`'s Column ABOVE the `PlayerInfoBar`,
gated `if (_canSurf && widget.playlist != null)` (matches `_canSurf`'s own
null-safety). Reuses the existing surf playlist — no new data.

## Why display-only (the Option-B design call)
The Peer2 plan triggered the Channel Bar on D-pad Center, but Option B kept
Center = play/pause + reveal, so a Center-triggered picker conflicts. Instead
the strip is **display-only**: `IgnorePointer` + `NeverScrollableScrollPhysics`
+ NO `FocusNode`/`InkWell`, so it never joins the overlay `FocusTraversalGroup`
and the D-pad / `_navMode` / trigger model is untouched. The user surfs with ▲▼
as before; a fresh Player launches at the new index (surf re-launches the route)
so the strip re-centers via `initState`. A focusable channel-picker is a possible
later refinement (would need a non-conflicting trigger, e.g. an OSD button).
