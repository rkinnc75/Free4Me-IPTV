# fix327 — TV export progress indicator + visible failure dialog (QR portal not opening) → v1.26.43+327

> ## WARNING BUILD MACHINE — APPLY THE PATCH, then RELEASE (EXACTLY). No questions, no DB to query.
> **FIRST:** `find . -name 'fix296-DIAGNOSE*.md' -delete 2>/dev/null || true`

## Why
On TV, export builds the whole bundle (gather raw source dumps, read the debug log, build the settings backup, zip everything) and starts the LAN server with NO UI in between. On a large catalogue that silent gap looks frozen/broken. Worse, when the server failed to start or no network URL was found, the only feedback was a brief SnackBar that is easy to miss on a TV — matching the report that the QR site "isn't opening." There is currently no log to confirm a specific root cause, so this fix makes the flow self-diagnosing.

## What this does
1. **Progress dialog:** TV export (both entry points — "export everything" and the source-dump long-press) now runs behind a spinner dialog that names each step: *Gathering source data → Collecting debug log → Building settings backup → Compressing files → Starting download server*.
2. **Visible failures:** server-start failure and "no network" now show a persistent, dismissible **dialog** (was a SnackBar). The user will actually see why nothing opened.
3. **Self-diagnosing:** every step and every failure is written to the debug log (`TV export: …`, `ExportServer: start failed — …`), so the cause is captured even when a log is hard to retrieve — turn on debug logging and the next attempt records it.
4. Both TV entry points routed through one helper `_runTvServerExport`; `_exportEverythingViaServer` now also uses the device-tagged stamp (fix322 consistency). No model/DB change.

## Note — not a confirmed root-cause fix
This adds instrumentation + UX so the actual cause (port already bound, no Wi-Fi/Ethernet IP, etc.) becomes visible. If after this the dialog reports a specific error, that is the lead for a targeted fix328.
