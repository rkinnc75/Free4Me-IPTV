# fix380 — "Suppressed seek probe error" log no longer fires per user-seek

**Target version:** `1.34.9+380` (from `1.34.8+379`)
**Branch:** `main`
**Toolchain (verified in sandbox):** Flutter 3.44.0 / Dart 3.12.0 (CI pin), `libsqlite3.so` symlinked.
**Written against:** `v1.34.8+379` on main.

## What & why

`lib/player.dart`'s `errorStream` handler suppressed mpv's benign
seekability-probe error ("Cannot seek in this stream." /
"force-seekable=yes") on non-seekable MPEG-TS livestreams. The
suppression was correct (the stream plays fine, the error is purely
informational), but the handler also logged `"Player: suppressed
seek probe error"` on *every* occurrence.

The startup probe fires once at `open()`. After the startup grace
window expires, the same error string comes from a different source:
**user-seek attempts that mpv rejects because `force-seekable=no` is
set** (see `lib/player/mpv_engine.dart:603`). Every rejected seek
spammed the log with a misleading "seek probe" message — the operator
reading the log couldn't tell the difference between the startup probe
(a one-time sanity check) and a user-spamming-the-seek-bar
(stream is fine, user error).

fix380 latches the suppression log to fire at most once per `open()`,
and only during startup grace. After grace, rejected seeks are still
suppressed (no reconnect, no error state) but silent in the log. The
startup probe is now correctly labelled: `"Player: startup seek
probe suppressed"`.

## Files changed
- `lib/player.dart` — added `bool _seekProbeLogged = false` to the
  Player state; reset alongside `_startupGrace` in `_startPlayback`.
  The errorStream handler now only logs the suppression once per
  open(). The pure predicate `isSeekProbeError(String)` is extracted
  as a top-level function so the detection rule is testable from
  outside the `player.dart` library (Dart's `_PlayerState` is
  library-private).
- `lib/player/mpv_engine.dart` — comment added at the `force-seekable=no`
  setup pointing to the player.dart suppression behaviour. No engine
  config change.
- `lib/whats_new_modal.dart` — `'1.34.9'` changelog entry (real UTF-8).
- `pubspec.yaml` — `1.34.8+379` → `1.34.9+380`.
- `test/player_seek_probe_log_test.dart` — **new**: 4 unit tests on
  the pure `isSeekProbeError` predicate, pinning the detection rule
  and the negative cases (real engine errors must not be misclassified).

## On-device integration check (NOT required — log volume only)

This is a **log-only** change. No playback behaviour changes (startup
grace logic, suppression, force-seekable setting are all unchanged).
Operator-side observation is the verification:

1. Install `v1.34.9+380` on any device.
2. Open a live stream (any source, any non-seekable MPEG-TS channel).
3. Watch the log during open(): expect exactly **one** line
   `Player: startup seek probe suppressed channel="…"`.
4. Seek the playhead rapidly (or just leave the channel playing with
   no user interaction). Expect: no further `seek probe` log lines.
5. Pre-fix380, each user-seek would produce one log line; after
   fix380, user-seek is silent. The startup probe is the only
   `seek probe` line that should appear per channel.
6. Trigger a real error (e.g. point at an invalid URL) to confirm
   the *non*-seek error path is unaffected: the standard
   `Player: engine error [permanent|transient]` line still fires.

If `multi_view_cell.dart` is also affected by the same flood (it has
the same suppression pattern at line 482), the fix would mirror
fix380 in that file. **Not in scope for this fix** per the summary's
file list; tracked as a follow-up if observed.

## Caveat: "mostly moot after the A3000 provider ships their VOD fix"

The summary's §9 note for fix380 calls this out: A3000's VOD
`<live>`-vs-`0s` seek bug is theirs to fix. Once they ship, A3000
seeks won't be rejected, so the user-seek log spam source dries up
on that provider. fix380 still has value for the other providers
(Dino, Z2U, Emjay) on MPEG-TS livestreams where `force-seekable=no`
is the norm. The fix is small and sandbox-provable either way.
