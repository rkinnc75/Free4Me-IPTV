# fix567 — correct the 30 fps cap filter syntax (fix566 stalled low-RAM playback)

Base: **v2.0.66** (origin/main dfa747f). Ships as **v2.0.67+567**.

## What fix566 got wrong (verified on-device, onn 4K Plus, v2.0.66)

fix566 switched the cap from `vf=fps=30` to `vf=lavfi=[fps=30]`. The property
set returns success (`-> 1`), but at filter-graph **configure** time this
libmpv's libavfilter rejects the bare positional argument:

```
Set property: vf="lavfi=[fps=30]" -> 1
ffmpeg: AVFilterGraph: No option name near '30'
lavfi: parsing the filter graph failed
user_filter_wrapper: Creating filter 'lavfi' failed.
vo/null: reconfig to 960x480 ... → lavf: deselect track 0
Player: engine error — video texture failed to attach → reconnect → STALL
```

Net effect: the video track is deselected → no first frame → **permanent
black-screen stall** on every low-RAM open. fix566's `isVfOptionError`
suppressor did NOT catch it — the set succeeded; the failure is a deferred
filter-configure error whose text ("Creating filter 'lavfi' failed", "No
option name near '30'") doesn't match, and the user-visible symptom arrives as
a legitimate "video texture failed to attach". Phones / capable TVs are
unaffected (the cap only runs on low-RAM Android).

Note: `vf=fps=30` and `vf=fps=fps=30` both parse fine in **desktop** ffmpeg —
this is a quirk of this libmpv's lavfi-bridge graph parser, not reproducible
off-device. That is why fix567 was verified on the box (CI dry-run artifact,
correctly signed, `install -r`) BEFORE any real release.

## Fix

`lib/player/mpv_engine.dart`: use the fully-explicit option form
**`vf=lavfi=[fps=fps=30]`** (filter `fps`, option `fps`=30) — no positional
shorthand for the parser to choke on. The `fps` filter exists in this build
(the v2.0.66 failure was an option-parse error, not a missing-filter error).
fix566's reconnect-suppression (`isVfOptionError`) is retained as a safety net.
