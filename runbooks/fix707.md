# fix707 — TV GUI redesign chrome pass: button/dialog/gear focus rings (v4.1.12)

**Builder:** desktop. **Live base:** v4.1.11. **Ships as:** v4.1.12+707. TV only;
phone UI byte-identical. Standing autonomous GUI mandate.

## What
`lib/main.dart` — the 4 global TV button focus themes (`filledButtonTheme`,
`iconButtonTheme`, `textButtonTheme`, `outlinedButtonTheme` — their
`ButtonStyle.side` focused resolver) change `Colors.yellow`→
`appAccentNotifier.value`. Widths kept (4 for filled, 3 for the others). All four
stay gated on `states.contains(WidgetState.focused) && !hasTouchScreen` → phone
untouched. This is the last flat-yellow focus surface (the Settings gear + dialog
buttons), so the accent-ring language now covers ALL TV chrome (tabs fix702,
tiles fix703, rails fix704, buttons fix707).

## Reactivity decision (why non-const, why not AnimatedBuilder yet)
`ThemeData` is built once in the MaterialApp `build`. Two options:
- **(chosen) non-const `appAccentNotifier.value` read** — resolves the accent at
  theme-build time. Not live-reactive, but the accent is **white today** (there
  is no accent-picker UI yet), so it is visually the intended white ring and
  cannot drift. Simplest, lowest blast radius.
- (deferred) wrap the theme/MaterialApp in an `AnimatedBuilder` on
  `appAccentNotifier` so the theme rebuilds when the accent changes. Correct once
  a picker exists — belongs to the future accent-preset unit (doc Phase 6), not
  here.
