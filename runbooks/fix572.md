# fix572 — purge stale export artifacts + diag-overlay display/audio line

**Version:** v2.1.1+572 (pending)
**Date:** 2026-06-27
**Type:** Storage leak fix (export/QR) + playback diagnostics

Two small, independent changes that ship together.

---

## Part A — Export artifacts accumulate in the temp dir (backlog #10)

### Problem

Every QR/LAN export builds a `free4me-export-<stamp>/` bundle directory in the
temp dir (`_buildExportBundle`, `lib/settings_view.dart`). When the user opted in
to credentials, that dir also holds full `db.sqlite` / `epg.sqlite` snapshots —
potentially **hundreds of MB each**. The download server is stopped when the QR
dialog closes (`server.stop()`), but the bundle dir is **never deleted** — only a
re-entry guard (`outDir.delete` at settings_view.dart:1659) removes the *current*
stamp's dir. So every export left its dir behind and the temp dir grew without
bound. The save-to-file backup (`SettingsIo.exportToFile`) deletes its own
`free4me-backup-<stamp>.json` after the file-picker save, but leaks it on a crash
between write and delete.

### Change

A reusable, self-cleaning purge in `lib/backend/settings_io.dart`:

- **Pure decision fn** `staleExportArtifactNames(names, {keepExportDir, keepBackupFile})`
  — returns the basenames matching the two timestamped families
  (`free4me-export-*` dirs/zips, `free4me-backup-*.json`) **excluding** the one the
  current session is creating. No IO → unit-testable
  (`test/fix572_export_purge_test.dart`, mirrors the `isLowRamDeviceThreshold`
  pure-split pattern). Unrelated temp files are never selected.
- **IO wrapper** `purgeStaleExportArtifacts({keepExportDir, keepBackupFile})` — lists
  `getTemporaryDirectory()`, deletes each stale entry. **Fail-soft**: a listing or
  per-entry delete error is logged (`AppLog.warn`) but never aborts the export
  (cleanup must not break the feature). Returns the count removed; logs
  `export: purged N stale export artifact(s)`.

Called at the start of each export session, keeping only the current artifact:

| Call site | Keeps | Also sweeps |
|---|---|---|
| `_buildExportBundle` (QR/LAN), before creating the dir | `free4me-export-<stamp>` | orphaned backup jsons |
| `exportToFile` (save-to-file), before writing | `free4me-backup-<stamp>.json` | orphaned QR bundle dirs |

`setup.dart:418` (the *receive* side, `ExportServer(const [])`) creates no
artifacts and is untouched.

### Why fail-soft (not fail-fast)

The repo default is fail-fast/no-fallbacks, but this is housekeeping that runs as a
side effect of the export. A leftover file we couldn't delete (locked, perms) must
not block the user's actual export. The error is surfaced in the log, not
swallowed silently.

## Part B — Diag overlay shows the active drop/sync settings

### Problem

The `PLAYBACK (debug)` overlay (`lib/player/debug_stats_overlay.dart`) and its
mirrored `STATS` logcat line showed decode path / fps / drop COUNTS / avsync /
cache / res, but not the active `framedrop` ("drop video") and `video-sync`
("sync audio") MODES. During a settings sweep you had to read the one-shot
`options applied` line at stream-open and trust it hadn't changed.

### Change

- `mpv_engine.dart` `readPlaybackStats()` — added `framedrop` and `video-sync` to
  the polled property list.
- `debug_stats_overlay.dart` — new bottom row
  **`mode   drop=<framedrop>  sync=<video-sync>`**, and the same two fields appended
  to the per-tick `STATS` log line. Shows the **effective** values libmpv is running
  (i.e. after the fix571 low-RAM `vo→decoder` auto-upgrade), so a sweep is visible
  on-screen, not just in logcat.

### On-device note (onn 4K Plus, 2026-06-27)

Validated backlog #1 with this build's logging on `SP - YES NETWORK HD` (live MLB,
1080p60 h264, software decode): `framedrop=decoder`, `video-sync=audio`, **0 VO
drops over 9.5 min**. The `vo` half of the framedrop sweep is unreachable on the
onn — it reports ~1926 MB < the 2300 MB low-RAM cutoff, so `vo` is auto-upgraded to
`decoder` (fix571). Re-measuring `vo` under the custom libmpv would need a
dev-gated bypass of that upgrade + a build; parked by decision (decoder=0 is proven
and `vo` is historically 528–849 drops/min).
