## Bug
Two long-standing `use_build_context_synchronously` INFOs in `lib/settings_view.dart`, both inside
`build(BuildContext context)` (so `context` is the build-method parameter, not the State's getter):
1. **Safe-mode switch tile** `onChanged`: after `await updateSettings()`, `ScaffoldMessenger.of(context)` was guarded only by the State's `if (!mounted)`.
2. **"Clear stream validation"** tile `onTap`: after `await Sql.clearAllStreamValidated()`, `ScaffoldMessenger.of(context)` was guarded only by the State's `if (mounted)`.

The Dart 3.12 lint treats the State's `mounted` as an "unrelated" check for a *local* `BuildContext`. These were never real defects (for a build context the element IS the State's element, so both `mounted`s flip together) — which is why they were tolerated. This retires the tolerance.

## Fix (adopted from the cold-eyes agent's patch, verified here)
Guard the **build context's own** `mounted` at each site — exactly what the lint recommends:
- Site 1: `if (!mounted) return;` → `if (!context.mounted) return;` (stale "capture messenger" comment dropped).
- Site 2: inner `if (mounted) {` → `if (context.mounted) {`. The outer `if (confirmed == true && mounted)` stays the State's `mounted` (it gates whether to run the DB clear — a liveness question, correctly).

Behaviour identical — pure static-analysis cleanup, no functional change.
