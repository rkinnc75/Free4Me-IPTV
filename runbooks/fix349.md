## Why (pending item 2: "Background mode for other long processes")
fix318 added a foreground-service wrapper (`BackgroundTaskService.run`) so a source refresh survives the user switching away on Android — but only the source refresh used it. EPG refresh, the full channel re-match, and the stream scanner are equally long-running and were still silently suspended (and could be killed) the moment the app left the foreground. All three now run through the same wrapper, gated on the same existing `backgroundProcessing` setting. Work stays on the main isolate; the service only promotes the process and mirrors progress to its notification — identical architecture to fix318, no new permissions, no new settings.

## What this does
- `lib/settings_view.dart` — `_runEpgRefresh` and `_runEpgRematch` per-source loops wrapped in `BackgroundTaskService.run` (titles "Refreshing EPG" / "Re-matching channels"); every existing `_updateRefreshDialog(status)` site also mirrors `status` to the service notification via `update(status)`. Settings toggle relabelled **"Keep long tasks running in background"**; help text lists all four covered tasks. The diff for this file is large only because the two loops gained one indent level — logic inside them is byte-identical apart from the added `update(status);` lines.
- `lib/home.dart` — `_startScan`'s `StreamScanner.scan` call wrapped the same way (title "Scanning streams", notification shows "N / M streams tested"). Cancel button still works in-app; a backgrounded scan simply runs to its existing cap.
- `lib/whats_new_modal.dart` — 1.27.1 changelog.
- `pubspec.yaml` — 1.27.1+349.
- No engine, DB, or migration changes. Notification update cadence is batch-driven (XMLTV insert batches / match chunks / per-probe), same order as the fix318 row-progress precedent.
