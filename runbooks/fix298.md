## Root cause (proven from the on-device fix296 DIAG log + real SQLite)
The fix296 diagnostic log answered it. For "espn", `ChannelSearchCache.search` returned exactly **36 ids** (`limit: pageSize`) every time, and the post-fetch SQL filter dropped **all 36** → `matched=0`.

**Two distinct bugs:**

1. **Filter at the wrong layer (the search bug).** The cache caps results at `limit` (36) using only name/media/source matching — it is NOT divider- or enabled-aware. The divider + disabled-category filters were applied *afterward*, in the SQL fetch (fix294). On a provider like Z2U, the first 36 "espn" name-matches are all dividers (`####`) and disabled-category channels (AU/NL/CRB ESPN PPV), so they consume every page slot; the real enabled channels (US ESPN PLAY/PLUS) sort past slot 36 and are never fetched. Filtering after pagination cannot recover them. **Fix: move the divider + disabled-category exclusion INTO the cache, before the limit.**

2. **Category toggles reset on refresh (the persistence bug).** A source refresh runs `wipeSource`, which `DELETE FROM groups WHERE source_id=?` then re-imports. `updateGroups` re-INSERTs the groups, so every category comes back at `enabled DEFAULT 1` — the user's disabled categories silently turn back on every refresh. **Fix: capture the disabled category names before the wipe and re-apply them after the groups are recreated.**
