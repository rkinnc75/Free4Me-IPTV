## The FAIL this fixes (VERIFY_643-647 §4, onn, 2026-07-03)

fix646/v2.2.46 added the partial favorites index (migration 43,
`idx_fav_browse ON channels(media_type, source_id, name COLLATE NOCASE)
WHERE favorite = 1`) — but shipped it WITHOUT a forced hint. On the onn the
gate "favorites render near-instantly, NO SLOW SQL" failed hard:

- Movies → Favorites: **SLOW SQL 64288ms** (rows=3), immediate retry 30140ms.
- Plan (fix418): `SEARCH c USING INDEX idx_channel_src_media_url (source_id=?
  AND media_type=? AND url>?) | USE TEMP B-TREE FOR ORDER BY` — the planner
  walks the whole (source, media_type) range (~760k enabled movie rows)
  residual-testing `favorite = 1` per row, never touching idx_fav_browse.

This is the documented fix531/fix627 planner-mis-pick class: averaged
`sqlite_stat1` cannot see that `favorite = 1` selects ~a dozen rows, so
ANALYZE can never fix the pick. The established cure is a **provably-safe
forced `INDEXED BY`, gated live on `_indexExists`** (fix526 — forcing a
missing index is a hard crash).

## Change (lib/backend/sql.dart)

1. **No-query browse (`Sql.search`)**: new `useFavHint` — condition IDENTICAL
   to the branch that appends `AND favorite = 1` (`viewType == favorites &&
   seriesId == null`), so the query always implies the index's partial WHERE →
   forcing returns the exact same rows, just via ~a-dozen index seeks. Wired
   first in the existing hint chain (mutually exclusive with
   `useSrcMtHint`/`useSrcGrpHint`, which both exclude the favorites view).
2. **`_searchLike`** (favorites + text query, likeSubstring method): same
   gated force — bounds the LIKE scan to the favorites rows instead of the
   catalog. Same identical-condition proof.
3. **Self-heal gap closed:** `idx_fav_browse` added to
   `_canonicalChannelIndexes` (17 entries now). It was missing — an
   interrupted refresh would have dropped it permanently, and the fix628
   self-heal would never rebuild it, silently disabling this hint via the
   `_indexExists` gate.
4. **Guard test** (`test/browse_index_hint_guard_test.dart`): forced hint
   honored with `favorite = 1` present (favorites-only rows returned); WITHOUT
   it SQLite throws "no query solution" (hard error — the force can never
   silently drop rows); the LIKE shape is honored too.

Favorites never routes through `_browseMixedUnion` (excluded at the dispatch),
and the FTS path drives off the MATCH join — no other sites.
