# fix505 — device perf: low-RAM search→FTS + advanced force-HW-decode

**Version:** 2.0.4+504 → **2.0.5+505**
**Tag:** v2.0.5
**Type:** Perf / device-tuning (low-RAM TV boxes). Phone unaffected.
**Status:** Implemented + verified locally (analyze 0; 171 tests incl. 2 new routing cases); shipped via CI.
**Base:** fix504.

## What (2 of the 3 fix505 items; the 1080p render cap → fix506)
1. **Low-RAM search default `likeSubstring` → `ftsTrigram`.** On a low-RAM box (<2300 MB, e.g. onn 4K Plus) the in-memory cache is skipped; fix390 then defaulted to a leading-wildcard `%LIKE%` full-table scan — slow on ~1.15M channels. Now the first-run default is `ftsTrigram` (index-backed `channels_fts`, fast + on-disk). `main.dart` already reconciles the FTS triggers for `ftsTrigram`, so the index stays fresh (accepted minor refresh write-cost). Touches `settings_service.resolveSearchMethod` + its log/doc; `search_method_low_ram_autoset_test` updated.
2. **Advanced "Force hardware decode" override.** New `forceHwDecode` setting (default off) + a Settings → Playback toggle. `androidFullscreenHwdec` gains `forceHardware`: when on, returns `mediacodec-copy` even on a low-RAM box, bypassing the deliberate low-RAM→software fallback (which exists because hardware-copy desyncs A/V on Amlogic+Mali-G310). Lets the user A/B hardware decode on their own streams. `shield_hwdec_routing_test` gains 2 cases.

## Deferred to fix506
The 1080p render cap on low-RAM 4K boxes (auto-on + Appearance toggle) — a native FlutterSurfaceView resize, verified via `dumpsys SurfaceFlinger` buffer size. Split out so these Dart-only wins ship unblocked.

## Phone-mode safety
Search autoset changes only the low-RAM first-run default (phones ≥2300 MB unchanged; a persisted choice always wins). `forceHwDecode` defaults off and only affects the Android full-screen hwdec branch. Both settings use the null-guarded serialization; the new routing param defaults false (no behavior change without opt-in).
