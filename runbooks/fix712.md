# fix712 — serialize the multi-source EPG refresh (v4.1.16)

**Builder:** desktop. **Live base:** v4.1.15. **Ships as:** v4.1.16+712. Backend
(EPG refresh) — all platforms. Overnight autonomous run (owner approved).

## Why fix709 wasn't enough
fix709 serialized the channel-MATCH phase. But the fix709 **device-verify caught
that it's insufficient**: with Trex + A3000 both enabled and "Refresh anyway"
(forcing a concurrent multi-source refresh), the completion dialog showed:
- ⚠️ `Trex: refresh completed but 0 programs loaded` (was 185516 single-source)
- ✗ `A3000: SqliteException(5) … database is locked` on `INSERT INTO
  epg_refresh_log`, plus `PathNotFoundException: … epg_src_4.xml (No such file)`

So the **download/parse** phase races too, not just the match: concurrent
downloads starve each other's per-source temp-XML fetch (→ 0 programs / file not
found) and the concurrent epg.sqlite writes (`epg_refresh_log`,
`insertProgramsBatch`) exhaust the SQLITE_BUSY retries. Non-destructive (existing
EPG survived — search "FOX" still showed "On now (13)"), but multi-source refresh
could not reliably load/match.

## The fix
`epg_service.dart` `refreshAllSources` — `const maxConcurrent = 2` → **`1`**.
Sources now download→parse→insert→match **one at a time**, matching the
proven-good single-source path (single-source Trex matched 35851/35851 cleanly).
The chunked loop is structurally unchanged (chunk size 1). fix709's
`_serializeMatch` gate stays (redundant under serial, but a cheap invariant that
protects if concurrency is ever raised, and the manual Re-match path — task #39).
The disproven "HTTP fetches don't fight each other / the writer naturally
serializes" doc rationale is corrected.

Trade-off: a full N-source refresh is now serial (slower). Acceptable — it's a
nightly background op (correctness >> speed) and serial is gentler on providers
(no burst traffic).

## Note on review vs device
fix709 had two clean adversarial reviews, but both reasoned the download phase was
"contention-tolerant" (my scoping). The **device** disproved that. Lesson: for
concurrency under real resource limits (2GB box), the on-device multi-source
re-verify is the arbiter, not another code review. This change is strictly *less*
concurrent, so it can only reduce contention.
