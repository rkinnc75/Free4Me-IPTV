# fix691 — Zero-base recording timestamps (correct duration + seek) (v4.0.7)

**Builder:** desktop (Mac). **Live base:** v4.0.6. **Ships as:** v4.0.7+691.

**Why:** after fix690 fixed audio, converted recordings showed a huge bogus
duration (8:46:42 for a 1-minute clip) and the seek bar was dead. Tag re-verified
at start and before the diff — latest remote tag = **v4.0.6**.

---

## Root cause (proven)

Live MPEG-TS packets carry the broadcast-clock PTS/DTS (a large value, not 0).
Copied straight into MP4, the container reports an ~8-hour start offset, so
players miscompute length and the seek bar spans the phantom offset. Reproduced
against libavformat: remuxing a TS whose timestamps start at ~30000s yields an
MP4 with `start_time=30000s` for 3s of content.

## Fix
Capture the first packet's **DTS** (or PTS if no DTS) as a single global
reference and subtract it from every packet's pts/dts before muxing, so the
recording starts at ~0. One offset for all streams preserves A/V sync;
`AV_NOPTS_VALUE` packets are left untouched. Only touches `pkt->pts`@8 /
`pkt->dts`@16 — shallow, already-verified offsets (no deep `AVFormatContext`
field). Video/audio/BSF paths otherwise unchanged; `[SRDBG]` retained.

## Verification

**Performed in sandbox (real):**
- Reproduced the wrong duration and **proved the fix** against libavformat: with
  zero-basing, `start_time` went from ~30000s to ~0, duration intact, audio
  still decodes cleanly.
- **`dart analyze` (project Dart 3.12.2 + repo `analysis_options`) → "No issues
  found!"**
- Patch applies clean to a **pristine v4.0.6 clone**.

**NOT performed (Mac/CI owns):** full `flutter analyze`, tests, native compile,
arm64 runtime.
