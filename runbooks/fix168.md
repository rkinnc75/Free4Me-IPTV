# fix168 — correct the analyze-breaking `ListTileThemeData.focusColor` on main

> **Version:** 1.23.7+168 (phone/even build; patch bump — `v1.23.5` and `v1.23.6` are both already tagged and spent)
> **Evidence:** CI `flutter analyze --no-fatal-infos` on the tagged build:
> `error • The named parameter 'focusColor' isn't defined • lib/main.dart:236:11 • undefined_named_parameter`
> **Bug:** fix164.3 added `listTileTheme: ListTileThemeData(focusColor: …)`. `ListTileThemeData` has **no** `focusColor` parameter — `focusColor` is a property of the `ListTile` widget, not its theme-data. This is a compile ERROR (fatal under `--no-fatal-infos`), so every build since has failed analyze.
> **Root cause:** the symbol was written from memory, not verified. Confirmed against api.flutter.dev: `ListTileThemeData`'s constructor exposes `dense/shape/style/selectedColor/iconColor/textColor/titleTextStyle/subtitleTextStyle/tileColor/selectedTileColor/…` — **no `focusColor`**. The correct global lever for focus highlight is the top-level **`ThemeData.focusColor`**, which is the documented fallback that `ListTile`, `InkWell`/`InkResponse`, `IconButton`, `Switch`, `Radio` etc. read when their own `focusColor` is null.

## State of the repo (verified against main, not the zip)

- `main` pubspec: `1.23.6+166`; `version.json` regenerated for `v1.23.6`.
- Tags present: `v1.23.6`, `v1.23.5`, `v1.23.4`, … — **both 1.23.5 and 1.23.6 exist and are immutable.**
- `main` **still contains the broken block** (`lib/main.dart:235-236`): both tags were cut on top of it, so both releases fail CI identically. The timestamp work (fix166) is already committed on `main`; it just never built because this error sits in the same tree.
- The three sibling blocks added alongside it — `iconButtonTheme` / `textButtonTheme` / `outlinedButtonTheme` — **are valid** (those `*ThemeData` classes do take `style: ButtonStyle(...)` with `side`). They compile and give buttons a yellow focus ring. **Leave them.** Only the `listTileTheme` block is broken.

# Fix 168.1 — replace the broken `listTileTheme` with `ThemeData.focusColor`

**File:** `lib/main.dart`

Locate the broken block inside `theme: ThemeData(...)` (it sits between the `filledButtonTheme` block and the `iconButtonTheme` block). Current code on `main`:

```dart
        // fix164: TV/remote focus must be visible on the dark M3 surface.
        // Gated on !hasTouchScreen — touch UI unchanged.
        listTileTheme: ListTileThemeData(
          focusColor: hasTouchScreen
              ? null
              : Colors.blueAccent.withValues(alpha: 0.32),
        ),
```

Replace with:

```dart
        // fix164/168: TV/remote focus must be visible on the dark M3 surface.
        // focusColor is the top-level ThemeData fallback that ListTile,
        // InkWell/InkResponse, IconButton, Switch, Radio etc. use when their
        // own focusColor is null — so one field lights up focus app-wide.
        // (fix168: ListTileThemeData has NO focusColor param — that was the
        // analyze error; the correct lever is ThemeData.focusColor.)
        // Gated on !hasTouchScreen — touch UI unchanged.
        focusColor: hasTouchScreen
            ? null
            : Colors.lightBlueAccent.withValues(alpha: 0.40),
```

**Key changes**
- `listTileTheme: ListTileThemeData(focusColor: …)` (invalid) → top-level `focusColor:` on `ThemeData` (valid; verified on api.flutter.dev as a real `ThemeData` field).
- `focusColor` is a **sibling** of `brightness`, `colorScheme`, `filledButtonTheme` — it sits directly in the `ThemeData(...)` arg list, not nested in any sub-theme.
- Covers more widgets than the original intent (every focusable Material widget reading the theme fallback), not just `ListTile`.
- Alpha bumped 0.32 → 0.40 and `blueAccent` → `lightBlueAccent` for a touch more contrast on black; both are `Colors` constants in `material.dart`.
- The `iconButtonTheme` / `textButtonTheme` / `outlinedButtonTheme` blocks are **untouched** — they're valid and still give buttons their ring.

> Net effect on the theme block: one invalid named-param block becomes one valid top-level field. Nothing else in `main.dart` changes.
