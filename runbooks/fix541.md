# fix541 — EPG cluster: fix Live now/next (stale window) + last-load date + 24h re-download prompt

**Base tag:** `v2.0.40`
**Version:** `2.0.40+540` → `2.0.41+541`
**Scope:** TV guide window recompute + Settings "Refresh EPG now" (last-load subtitle + 24h confirm) + one `Sql` helper. No schema change (`epg_refresh_log` already exists). Full suite green.

---

## Item 6 — Live TV not showing now/next (ROOT CAUSE)

The guide's `_windowStart` / `_windowEnd` were computed **once in `initState`** and never recomputed. The shell keeps the guide alive via `IndexedStack`, so as time passed (and across an app resume the next day) the window went stale — `getGridPrograms` then fetched a **fully-elapsed** range and the hero's NOW/NEXT details silently disappeared.

**Verified against the real field data** (`epg.sqlite`, 234,981 programmes; `db.sqlite`, 1.2M channels): the data, the channel↔programme join (68.9% of EPG-equipped channels match a programme), the bounded `getGridPrograms` SQL, and the window math are all correct — 4,468 programmes cover "now". The bug is purely the stale window. The player top-bar label (`PlayerEpgNowLabel` → `Sql.getNowNext`) was always fine (it uses live `now` each call).

**Fix:** re-anchor `_windowStart`/`_windowEnd` to the current time at the start of `_loadGuide` (runs on every load, including `reloadGuide()`), so the grid and the hero now/next always track real now.

## Item 7 — Last EPG load date

"Refresh EPG now" now shows `Last loaded …` (relative, falling back to a date >7 days) from the most recent `epg_refresh_log.last_refreshed_utc` across sources, via the new `Sql.getLatestEpgRefresh()`.

## Item 8 — 24h re-download prompt

If the most recent EPG refresh was < 24h ago, tapping "Refresh EPG now" asks for confirmation ("EPG is recent … loaded N hours ago … Refresh anyway?") before the heavy re-download. Older than 24h (or never loaded) proceeds as before.
