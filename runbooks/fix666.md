**Builder:** desktop. **Live base:** v3.0.9. **Ships as:** v3.0.10+654.
**Severity: HIGH ŌĆö regression fix.**

## What was broken
fix663 (v3.0.8) set `HttpClient.connectionFactory` on the shared AppHttp client
and returned a plain `Socket.startConnect(...)` in every branch. When a
`connectionFactory` is set, the Dart SDK uses the returned socket **as-is** for
direct (non-proxy) connections and **never upgrades it to TLS** ŌĆö verified in
`dart:_http` `_HttpClientConnection.connect`
(dart-sdk/lib/_http/http_impl.dart ~2682ŌĆō2744): `SecureSocket.startConnect` is
only used on the DEFAULT (no-factory) path, or via `createProxyTunnel` when
going through a proxy. So for a direct HTTPS request the plaintext socket was
used raw.

Result on v3.0.8/v3.0.9: **every HTTPS request through AppHttp returned an
instant null** ŌĆö update check (`update_checker.dart`), Xtream login/refresh
(`xtream.dart`), XMLTV EPG (`xmltv_parser.dart`), M3U import (`m3u.dart`),
channel prewarm (`channel_tile.dart`). Code paths that build their own
`http.Client()` (stream scanner, EPG discovery variants) were unaffected, which
is why stream scan / discovery / local browse still worked. The bug shipped
silently because no test round-tripped HTTPS through AppHttp.

## The fix (Approach A ŌĆö minimal revert)
`lib/backend/http_client.dart`:
- Removed the `_inner!.connectionFactory = _dohConnectionFactory;` wiring.
- Deleted the now-dead `_dohConnectionFactory` method and the unused
  `doh_resolver.dart` import.
- Left a comment explaining why no `connectionFactory` may be set here.

Leaving `connectionFactory` unset restores the SDK's default `SecureSocket` TLS
path ŌĆö the exact pre-fix663 behaviour. `DohResolver` itself and its settings UI
remain (inert) so nothing else fails to compile.

## Why not fix the factory in place (Approach B)?
DoH-over-HTTPS through `connectionFactory` is architecturally infeasible with
correct cert validation: `SecureSocket.startConnect(ip, ...)` validates SNI/cert
against the IP (fails ŌĆö cert is for the hostname), and the API that can separate
them, `SecureSocket.secure(raw, host:)`, returns a `Socket`, not the
`ConnectionTask` a factory must return (`ConnectionTask` has no public ctor).
DoH needs a different mechanism (e.g. `HttpOverrides`) ŌĆö a deliberate redesign,
not a hotfix. **DoH is now a no-op**; the 3.0.10 changelog says so, and the DoH
settings entry should be hidden/relabelled in that redesign.
