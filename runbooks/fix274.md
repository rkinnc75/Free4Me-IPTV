> **This is the FOURTH identical failure** (1.26.2, 1.26.4-attempt, 1.26.5). Same cause every time: the release process runs `update_version_json.py` (you see "version.json updated to 1.26.5" in the log) but the regenerated `version.json` is **not in the commit that gets tagged**. The CI guard re-runs the generator on the tagged commit and sees the OLD version (1.26.4) still committed → "version.json on the tagged commit is stale."

# PART B — PERMANENT FIX (do once, so this never recurs)

The root cause is that the tag is created on a commit that doesn't contain the regenerated version.json. The fix is to **guarantee version.json is committed before the tag is created**, in whatever script cuts releases.

## Option 1 (preferred) — only ever release via `scripts/build_and_release.sh`
That script already does the right order: it regenerates version.json (step 3b), then `git add -A` + commit (lines ~167-170), THEN pushes and creates the tag/release. If every release goes through it, the bug cannot happen.
**Action:** stop cutting releases by any other path — no manual `git commit-tree`, no partial `git add pubspec.yaml lib/...` that omits version.json, no hand-built tag. If a wrapper or Cowork automation cuts the tag, it MUST call `build_and_release.sh` rather than re-implementing the steps.

## Option 2 — add a self-check to the release script (belt-and-suspenders)
Insert this guard in the release script AFTER version.json is regenerated and BEFORE the tag/release is created. It mirrors CI and fails locally (cheap) instead of after pushing a bad tag (expensive):
```bash
# --- version.json freshness gate (must run before tagging) ---
python3 scripts/update_version_json.py
git add version.json
if ! git diff --cached --quiet version.json; then
  git commit -m "${TAG}: sync version.json"
fi
# hard stop if for any reason version.json still differs from the generator
python3 scripts/update_version_json.py
if ! git diff --quiet version.json; then
  echo "ABORT: version.json does not match generator output — not tagging." >&2
  git --no-pager diff version.json >&2
  exit 1
fi
# only now create the tag, on the commit that includes version.json
```

## RULE (also in RELEASE-PROCEDURE.md and BUILD-MACHINE-RULES.md)
version.json MUST be regenerated AND committed in the SAME commit that gets tagged. The CI guard compares the tagged commit's version.json against `update_version_json.py` output — they must be byte-identical. Printing "version.json updated to X" during a build is NOT the same as committing it.

No `.dart` files touched. After Part A the tag is correct; after Part B the failure cannot recur.
