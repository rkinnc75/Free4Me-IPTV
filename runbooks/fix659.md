# fix659 — rotate-before-open: first-stream buffering fix (v3.0.4)

**Builder:** desktop. **Live base:** v3.0.3. **Ships as:** v3.0.4+648.

## Problem (from `free4me_log-sms938u-20260706-210908.txt`)

The FIRST stream played after app launch on the phone buffers heavily; every
later stream is clean — including the SAME URL played again:

| Play | Channel | Enter-buffering stalls | pausedForCache=yes |
|---|---|---|---|
| 1 | Yankees 668200.ts | **20** | **78** |
| 2 | Crime 1920324.ts | 0 | 0 |
| 3 | Yankees 668200.ts (same URL) | 0 | 0 |

Engine options were byte-identical across all three opens. The unique factor on
play 1: `Player: ROTATE init → portrait` then `portrait → landscape` fire
DURING open — the phone's orientation is still settling when mpv opens the
decoder. Sequence: decoder opens against a NULL/portrait surface
(`h264_mediacodec: Both surface and native_window are NULL`), the landscape
surface arrives (`android-surface-size=1920x1080`), the decoder RE-OPENS,
discarding the initial cache fill — then the session thrashes
`Enter buffering (100% -> 0%)` cycles. Plays 2/3 start with orientation already
settled (zero ROTATE events) and bind the surface once.

Prewarm note from the same log: prewarm fired on tile focus (Crime, 21:01:08 and
21:07:55) and Crime's play logged `prewarmed=true`; both Yankees plays were
`prewarmed=false` (never the focused tile), so play 1 was cold AND rotated.
Prewarm behaved as designed; it just didn't cover the first-played channel.

## Fix

Rotate FIRST. In `_startPlayback`, before the first `open()` (one-shot per
Player via `_preOpenFullscreenDone`):
- `await _enterSystemFullscreen()` (the existing landscape-lock helper), then
- `await WidgetsBinding.instance.endOfFrame` so the relayout lands and the
  video surface attaches at its final landscape size before the decoder opens.

Guards mirror the existing post-open condition: skipped in PiP (can't rotate)
and for `handlesOwnFullscreen` engines. Reconnect-loop iterations skip it (flag
already set; orientation already locked). The existing post-open
`_enterSystemFullscreen()` at the success site is left in place as an
idempotent re-assert, so the reconnect/retry path is byte-for-byte unchanged.
The adopt/swap path never runs `_startPlayback` and is unaffected. Failure is
caught and non-fatal (playback proceeds as before the fix).

The buffering behavior is device-only; sandbox gates cannot prove it. On-device
acceptance:
1. Cold-launch the app in portrait (phone in hand, natural grip).
2. Play any live channel as the FIRST stream.
3. Expected log shape: `ROTATE init → landscape` (or portrait→landscape)
   BEFORE `MpvEngine: open()`, exactly ONE `Opening decoder h264`, no
   `Both surface and native_window are NULL`, and zero/near-zero
   `Enter buffering` cycles after the initial fill.
4. Regression sweep: play a second/third stream (unchanged), background→PiP→
   return (unchanged), rotate mid-playback (layout-only, unchanged), exit
   restores orientation (onExit path untouched).
