# fix735 — A/V-desync watchdog → silent resync (Peer2, inverted for mpv) — v4.1.38

Owner rule: for stability/perf, consult Peer2 first ([[peer2-first-for-stability-perf]]).

## Bug (onn, root-caused + device-verified)
A live channel left playing ~3.5h drifted to avsync≈36s (video out of sync).
A FRESH tune on ANY channel opens at avsync≈0 → it's SLOW drift accumulating over
long uninterrupted playback on the onn's software-decode/bandwidth-bound path,
NOT a latched PTS offset (genpts unneeded) and NOT a stall (position advances,
paused-for-cache=no → invisible to the buffering/startup watchdogs). A reopen
resets avsync to ~0.

## Fix (Peer2 buffering-watchdog → silent-resync, INVERTED: mpv exposes avsync,
## ExoPlayer doesn't — so we monitor it directly)
- `mpv_engine.dart`: always-on (NOT debug-gated) `_startAvsyncWatchdog()`, started
  in the `!previewMode` open() block, self-gated to livestream. Every 6s on a
  PLAYING + ADVANCING + non-buffering + not-paused-for-cache stream, reads
  `avsync`; |avsync| > 3.0s for 3 consecutive ticks (~18s) → emit `desyncStream`.
- `player_engine.dart`: `desyncStream` default = empty.
- `player.dart` `_onAvsyncDesync`: reopen via the proven `onDisconnect('avsync
  watchdog')` → `_startPlayback(null)` → fresh live `open()` (resets avsync).
  Backoff: 30s debounce; a resync that HOLDS >3min resets the strike count (legit
  slow-drift correction, keep doing it); resyncs recurring <3min = strikes → after
  3, give up (keep playing, log only) so a broken-PTS feed can't reopen-loop.
