# fix696 — defer non-critical browse-index rebuild off the refresh path (v4.1.4)

**Builder:** desktop. **Live base:** v4.1.3. **Ships as:** v4.1.4+696.
Third refresh-perf release. On the onn (v4.1.3) the channel-side **16-index
rebuild is ~5m50s** — the dominant remaining cost of a full source refresh
(fix694+695 addressed FTS + EPG). The 16 CREATE INDEXes over ~290k rows can't
be made cheaper (each is an external merge-sort), but most of them are only
needed once the user BROWSES — which happens after the refresh dialog closes.

## What changed
`Sql.withDroppedBrowseIndexes` (the drop-for-bulk-insert wrapper) now, in its
finally, **rebuilds only the 2 match-critical indexes synchronously and defers
the other ~14** (pure-browse) to a background pass:

- **Critical (sync):** `idx_epg_unmatched`, `idx_channel_src_media_url` — the
  EPG match step runs immediately after this wrapper exits and its feeders seek
  by them (index map §2). Kept synchronous so match stays fast.
- **Deferred (background):** the ~14 browse/secondary composites. Left dropped;
  their DDL is written back to `app_meta['pending_browse_index_ddl']` so the
  rebuild can replay exactly the still-missing set.
- **The kick:** `Sql.ensureBrowseIndexesPresent()` (the existing fix628
  self-heal engine — replays pending DDL + fills missing canonical, memory-
  pragma'd, idempotent) is called **unawaited after channels+EPG+match finish**
  from the two refresh completion points (`settings_view` single-source;
  `sources_refresh_dialog` global). It MUST run after match, not concurrently —
  both share the single sqlite writer. A new `_ensuringBrowseIndexes` guard
  stops the startup call and this post-refresh kick from overlapping.
- **Backstop:** if a refresh path never kicks (or the process dies), the
  fix628 startup `ensureBrowseIndexesPresent()` rebuilds the deferred set on
  next launch — same as an interrupted refresh today.

**Tradeoff (owner-approved):** browse is index-degraded (temp b-tree, 10–45s
queries) for the short window until the background pass completes — the SAME
tolerated state that already exists mid-refresh (the `_indexExists` gates
prevent crashes). In exchange the "refreshing…" dialog dismisses after 2
indexes instead of 16.

Expected: perceived (dialog) refresh time drops by roughly the deferred index
cost (~4–5 min on the onn); the DB writer then rebuilds the browse indexes on
the idle box.

## Tests
- `test/fix696_deferred_index_test.dart` (6): partition mirror (exactly 2
  critical / 14 deferred); source invariants — critical set is the 2 feeders,
  deferred DDL is persisted (nothing lost), the re-entrancy guard exists, both
  completion paths kick the rebuild.
- Full suite 346 green, analyze clean (Flutter 3.44.5).
