# fix170 — analyzer "does nothing" (missing table), multi-view D-pad audio focus, log-string cleanup

> **Version:** 1.23.8+170 (phone/even build; patch bump from shipped 1.23.7+168)
> **Evidence:** `Free4Me-IPTV-1.23.7` source + `free4me_log-20260530-134945.txt` (onn 4K Plus, version=1.23.7 build=168).
> **Bugs:**
> 1. **Playback analyzer does nothing, no error** (TV *and* phone) — `playback_metrics` table is missing on upgraded devices because its migration is registered **out of order** (v12 added before v11), so `migrate()` skips it. The analyzer then throws on `SELECT * FROM playback_metrics`, the throw is unhandled, and the future is silently dropped → no dialog, no error.
> 2. **Multi-view 2×2: can't move audio focus once all cells play** — audio follows `_focusedCell`, but the only way to change it is `onFocusTap` on a non-D-pad-focusable `GestureDetector`. On a remote there's no path to reassign audio.
> 3. **Analyzer log lines use escaped `\${…}`** — they print literal `${agg.totalMinutes...}` text instead of values (cosmetic, but it's in the same handler).

## What the log proves

- `App started — version=1.23.7 build=168` — corrected build is running.
- Multi-view `focus → FOCUSED cell=N` events occur **only** when a channel is assigned to a cell (each follows a pick). After all cells are populated there are **no further** focus changes — confirming no D-pad route to move audio between playing cells.
- **No `PlaybackAnalysis:` line appears at all**, even though `_runPlaybackAnalysis` logs one unconditionally right after `Sql.getAggregatedMetrics()`. The function therefore throws **before** that line — i.e. at `getAggregatedMetrics()`'s `SELECT * FROM playback_metrics` against a non-existent table. (The earlier `insertPlaybackMetrics` is inside a `try{}catch(_){}` that swallows its identical failure.)

## Root cause — Bug 1 (the important one)

`lib/backend/db_factory.dart` registers migrations on `SqliteMigrations()` via chained `..add(SqliteMigration(<version>, …))`. The order is:

```
… ..add(SqliteMigration(10, …))   // stream_validated column
   ..add(SqliteMigration(12, …))   // CREATE TABLE playback_metrics   ← out of order
   ..add(SqliteMigration(11, …))   // channels_au trigger + browse index
```

`sqlite_async`'s `migrate()` tracks the database's applied version and runs migrations whose version is greater than the stored one. Because **12 is registered before 11**, devices that progress through this list end up with the stored version advanced past one of the two, and **migration 12 does not run** on already-upgraded installs. The result: `playback_metrics` is never created. (Confirmed package: `sqlite_async: ^0.13.0`; `SqliteMigrations.migrate` applies by tracked version.)

Rather than re-order 11/12 — which is risky to reason about for devices that have *already* recorded an intermediate version (re-ordering can't retroactively run a skipped version on them) — the robust, package-agnostic fix is to add a **new highest-versioned migration** that creates the table idempotently. A new max version is guaranteed to run on every existing device, and `IF NOT EXISTS` makes it a safe no-op on fresh installs that already created it via v12.

# Fix 170.3 — fix escaped interpolation in analyzer log lines

**File:** `lib/settings_view.dart`

In `_runPlaybackAnalysis`, two `AppLog.info` calls use `\${…}`, which Dart treats as a literal `$` — so they log the source text, not values. Fix the interpolation.

Current:

```dart
    AppLog.info('PlaybackAnalysis: aggregate '
        'minutes=\${agg.totalMinutes.round()} streams=\${agg.totalStreams} '
        'sessions=\${agg.sessionCount} sufficient=\${agg.hasSufficientData}');
```

Replace with:

```dart
    AppLog.info('PlaybackAnalysis: aggregate '
        'minutes=${agg.totalMinutes.round()} streams=${agg.totalStreams} '
        'sessions=${agg.sessionCount} sufficient=${agg.hasSufficientData}');
```

And:

```dart
    AppLog.info('PlaybackAnalysis: \${recs.length} recommendation(s)');
```

Replace with:

```dart
    AppLog.info('PlaybackAnalysis: ${recs.length} recommendation(s)');
```

**Key changes**
- `\${…}` → `${…}` so the analyzer's diagnostic lines log real numbers — which is exactly what you need to debug the analyzer going forward (it'll now log `minutes=… streams=… sufficient=…` after 170.1 makes it run at all).

> Verified against `lib/backend/playback_analyzer.dart`: `AggregatedMetrics` exposes `sessionCount`, `totalMinutes`, `totalStreams`, and `hasSufficientData` (getter) — all four tokens are valid. No invented symbols.
