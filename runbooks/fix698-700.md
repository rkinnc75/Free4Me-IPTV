## Bugs → fixes

### fix698 — Recording red dot "never blinks" (`recordings_view.dart`)
Two compounding causes (both confirmed):
- **Too-subtle pulse:** the dot faded only 1.0↔0.25 over a ~1.4s cycle → read as
  a faint breathe. Now **1.0↔0.15, 450ms, easeInOut**, built ONCE via
  `_c.drive(Tween(1,0.15).chain(CurveTween(easeInOut)))` (a `late final` field —
  no per-build `CurvedAnimation` listener leak).
- **No live refresh (device-confirmed):** the screen only reloaded on open +
  manual refresh, so a background `scheduled→recording→done` transition (and the
  blinking state) was rarely rendered while open — the row jumped Scheduled→Done
  and the "recording" red dot was never seen. Added a **quiet 3s poll**
  (`_syncPoll` → `Timer.periodic` → `_load(quiet:true)`, which skips the
  full-screen spinner) that runs while a row is **recording/compressing or an
  IMMINENT scheduled** (start within 2 min — a far-future schedule does not keep
  the poll running), self-cancels when all rows are terminal, and is cancelled in
  `dispose` (mounted-guarded setState — no setState-after-dispose).

### fix699 — Slow first channel open (`mpv_engine.dart` `open()`)
`open()` bound the video texture too late (the locked 1.5s wait usually expires
before the texture attaches ~2–2.7s in), so mediacodec bound a NULL surface, then
the arriving texture forced a `vo=null→vo=gpu` reconfigure + ~2s replay. Added a
**second, un-locked, bounded** `_waitForTextureId(_controller, 1500ms)` after the
existing locked concurrency wait + `finally`, gated `!previewMode`, so the
full-screen `open()` binds the real surface on the first init. Runs un-locked so
it never stalls a concurrent engine create; falls through if the texture is
genuinely slow (fix352: ~25% attach >4s — those get today's null-surface path,
~1.5s later — a latency-only edge, not a correctness break).

### fix700 — Live stutter + watch-while-recording contention (opt-in setting)
The A3000/Trex YES stutter is **upstream** (server/Wi-Fi delivers below the live
bitrate during game action — same URL was clean at 1.9 Mbit/s, stuttered at
4.5–5.8) — not app-fixable at root. The Trex ~7s open was amplified by watching a
channel that was **also being recorded** (two full pulls of the same stream from
the same slow server starved each other). A full playback-from-recording tee is
**infeasible** (mpv can't tail a growing MediaStore file → stops at EOF). So:
- New **opt-in** setting `livePrebufferSecs` (**default 0 = OFF**, preserving the
  deliberate fix354 "live keeps the default behaviour"). When >0, the mpv live
  branch sets `cache-pause=yes` (forces it on even under the low-latency profile,
  which sets `cache-pause=no`) + `cache-pause-initial=yes` +
  `cache-pause-wait=livePrebufferSecs`, skipped under DVR. Converts sub-second
  rebuffer thrashing (46 events/2min observed) into fewer, longer refills, and
  rides through the watch-while-recording contention. Mirrors the `vodPrebufferSecs`
  plumbing across `settings.dart` / `settings_io.dart` / `settings_service.dart` /
  `settings_view.dart` (new "Live pre-buffer (seconds)" slider, 0–15).
