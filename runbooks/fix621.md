# fix621 — Refresh-all rebuilds the FTS index ONCE at end-of-batch (v2.2.32)

**Author/builder:** desktop coder+builder (implemented + built + on-device
verified directly; not a phone-coder runbook).
**Live base:** v2.2.31 (5de8ca9). **Ships as:** v2.2.32+621.

## Symptom

A full "refresh all sources" on the onn 4K Plus ran **20:16→21:58 (~102 min)**
and appeared to hang on "Media4u — Saving to database…". The last **~86 min**
was **78 back-to-back `DELETE FROM channels_fts WHERE rowid IN (...)` chunks**
(avg 66 s, worst 141 s), the durations RAMPING 1 s → 140 s. Media4u (the largest
source, processed last) never committed; the user cancelled.
(private-repo report `2026-07-01T01-59-27…onn4kplusstreami-wfwaihn.log`.)

## Root cause (corrected after adversarial review)

The first read blamed the `rowid IN` delete for full-scanning the FTS index.
That was **wrong** — verified two ways:
- The *identical* statement ran in **1–6 s/chunk** in the same log's midnight
  run (v2.2.26), and ~3 s for 450k rows on a clean 1.2M-row index in a bench.
- Run B's integrity-checks all PASSED (zero code-267) — the index was **not**
  corrupt.

The real cause is **fix619's per-source, delete-before-wipe, INTERLEAVED
ordering**. Each source's `getXtream` did a chunked
`DELETE FROM channels_fts WHERE rowid IN (...)` before its wipe. Across the
interleaved batch the external-content + `prefix='2 3'` FTS5 index accumulates
segments/tombstones, so each successive delete degrades — a monotonic ramp,
worst on the largest/last source. fix619 fixed fix614's code-267 corruption but
traded ~9 min for ~86 min.

Baseline for comparison: the v2.2.26 midnight run did MORE FTS work (delete +
full reindex, all 4 sources) **correctly in 9.05 min** (`reindexFtsSources …
543287ms`).

## The change

`lib/backend/sql.dart` — `withSuspendedFtsTriggers`:
1. **Pass-through branch** (inner per-source call during a batch): do NO
   per-source FTS work — just `await body(); return;`. Triggers are already
   suspended by the outer wrapper, so the wipe+insert don't touch the index;
   it simply goes stale for the loop.
2. **Batch-success branch**: rebuild the whole index ONCE via
   `rebuildFtsTableFromScratch()` (DROP TABLE + recreate + a single
   `INSERT … SELECT id,name FROM channels`, ~9–11 min). **267-proof** — DROP
   discards the old shadow tables, so there is NO delete-after-content-change
   window (fix614's corruption cause).
3. **fix621 gate (adversarial-review WARNING):** set `ftsRebuilding=true` for
   the ENTIRE batch window (from trigger-drop, not just the rebuild) so a query
   can never hit the stale external-content index. Cleared in branch-2 / else
   finally.

`lib/widgets/sources_refresh_dialog.dart` — Cancel path (review WARNING): the
Cancel button now only FLAGS cancellation (it no longer calls
`ensureFtsHealthy`, which would race the wrapper's in-flight DROP+rebuild, and
no longer marks "done" early). The completion handler shows "Cancelled" once the
end-of-batch rebuild has actually finished — the dialog stays honest.

`lib/backend/utils.dart` — comments only (the `batchTargetedRebuild` closure is
now a "batch succeeded" sentinel).

## Expected result

Full 4-source refresh: ~16 min fetch/commit + ~3 min browse-index recreate +
~9–11 min single FTS rebuild ≈ **~25–30 min total** (vs ~102 min), no per-source
delete storm, no 267.
