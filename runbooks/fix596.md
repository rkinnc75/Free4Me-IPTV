## fix596 — settings in the diagnostic report (`_submitIssueReport`)

The in-app issue report previously uploaded only the scrubbed log. It now also
sends a **scrubbed settings export** so settings are available for review
alongside the log — using the SAME builder the backup/restore export uses
(`SettingsIo.buildBackupPayload(includeCredentials: false)`), then
`AppLog.scrubSecrets` on top (belt-and-suspenders; strips any residual
host/user/pass). New POST field: `settings` (base64).

> **WORKER DEPENDENCY:** the report worker (`_issueWorkerUrl`) must be updated
> to read the new `settings` field and commit it as a SECOND file next to the
> log (e.g. `reports/<ts>.settings.json`). The app side is done; the worker is
> separate infra and needs that change before the settings file appears in the
> -private repo. Until then the field is accepted/ignored harmlessly.

## fix595 — searchPrograms On-now diagnostic (`Sql.searchPrograms`)

When the windowed EPG result is EMPTY, log whether the trigram FTS matched any
programme TITLE at all (ignoring the now→window filter):
`Sql.searchPrograms: q="…" match=… ftsTitleMatches=N windowed=0`. This
attributes an empty "On now"/"Coming up" to (a) no programme is *titled* like
the query — EXPECTED for a channel/network name like "fox" (confirmed
working-as-intended 2026-06-28) — vs (b) titles matched but none are live now.
Only runs on the empty path, so normal searches pay nothing. fix592 already logs
the per-shelf counts (`shelves[onNow= … series=]`).
