# fix312 — Publish version.json AFTER the APK upload to close the updater 404 window → v1.26.28+312

> ## ⚠️ BUILD MACHINE — APPLY THE PATCH, then RELEASE (EXACTLY). No questions, no DB to query.
> **FIRST:** `find . -name 'fix296-DIAGNOSE*.md' -delete 2>/dev/null || true`
> **This modifies `.github/workflows/release.yml` (fix290-governed CI).** A one-line rollback is built in — see the ROLLBACK box below.

## Root cause (confirmed)
The in-app updater fetched HTTP 200 on `version.json` (saw 1.26.26 → 1.26.27) but the APK download returned **HTTP 404**. The asset name/URL were correct; the APK simply hadn't finished uploading when the device checked. Retrying minutes later worked. The pipeline committed `version.json` to `main` early (in the `analyze` job) — advertising the new version — while the APK upload happens later in the `build-and-release` job. That gap is the 404 window.

## The fix (Option A — reorder)
`version.json` is now committed to `main` (and the tag moved onto that commit) **after** the APK is uploaded to the release AND confirmed downloadable (HTTP 200, retried up to 6×). So the update feed never advertises a version whose APK isn't live yet.

- New env toggle `APK_BEFORE_VERSIONJSON: 'true'` (top of release.yml).
- When `'true'`: the early `analyze`-job version.json sync is skipped (`if:` guard); a new end-of-`build-and-release` step verifies the APK then commits version.json + moves the tag to main's tip (preserves fix290: version.json on main, tag at tip).
- `build-and-release` checkout now persists credentials (it pushes now).

## 🔄 ROLLBACK (do this if a release misbehaves)
Set `APK_BEFORE_VERSIONJSON: 'false'` in **both** workflow files (the toggle now
lives in each, since GitHub Actions env does not cross workflow files):
```yaml
  APK_BEFORE_VERSIONJSON: 'false'   # .github/workflows/release.yml
  APK_BEFORE_VERSIONJSON: 'false'   # .github/workflows/analyze.yml
```
That fully restores the original fix290 ordering (early version.json sync in
release.yml's `analyze` job; analyze.yml's "Verify version.json is up to date"
hard check re-enabled) with no other change. Commit + push both edits and re-run
the release.

> **fix312 follow-up:** `analyze.yml` originally had an UNGUARDED
> "Verify version.json is up to date" step that hard-failed on every push to main
> while version.json was intentionally stale during the publish-late window. It is
> now gated behind the same toggle, so both files must be set together on rollback.

## Risk vs fix290 (be aware)
This intentionally reorders fix290's "version.json on main before the tag" sequencing. It still finalizes with version.json on main and the tag at main's tip (the ancestor invariant the pre-push hook checks is preserved), but the *timing* moves into the build job. The verify-APK-200 gate + final version.json-freshness guard protect against a half-published release. If anything looks off, use the rollback above.
