# fix723 — Re-match/refresh survives leaving Settings (v4.1.27)

**Builder:** desktop. **Ships as:** v4.1.27+723. TV + phone. Autonomous ("keep
going, pick recommended"). Fixes a KNOWN long-standing bug the owner had
deferred flag-only (2026-07-02); surfaced again when the owner asked "is that all
that's left?" — it's a real user-facing bug, higher value + more verifiable than
the remaining cosmetic tail.

## Bug
Leaving Settings while "Re-match all channels" OR a source refresh runs ABORTS it
partway (device test: only 1 of 4 sources completed; app survives, no crash).
Users commonly navigate away during the ~20-min op, so it fires in normal use.

## Root cause
The work runs on `BackgroundTaskService` (fix349) which OUTLIVES the Settings
screen. On navigate-away, `_SettingsViewState` + the progress dialog's
`StatefulBuilder` State are disposed, but the loop keeps calling
`_updateRefreshDialog(status)` → `_refreshSetState?.call(() {})`. `?.` guards a
null REFERENCE, not a DISPOSED State → the captured `setState` ran
`markNeedsBuild()` on an unmounted State → threw `Null check operator used on a
null value` (framework.dart) UNHANDLED inside the loop → abort. (The
`_refreshSetState = null` after each dialog closes — settings_view ~1086/1282 —
only covered the dismiss-while-on-Settings case, not navigate-away.)

## Fix (`lib/settings_view.dart` `_updateRefreshDialog`, ~1403)
```
_refreshStatus = status;   // latched BEFORE the guard → re-opened dialog repaints
if (!mounted) return;      // fix723: skip UI update once this State is disposed
try { _refreshSetState?.call(() {}); } catch (_) { /* dialog State torn down */ }
```
Work continues headless to completion; the fix697 native completion notification
still fires. Per-source try/catch (re-match loop + refreshAllSources finding-39)
was already present, so one source's failure never aborts the rest. The
completion-path pop/result-dialog already guards on `ctx.mounted` (~1197/1370).

## Second guard (folded in, from the review)
Adversarial review confirmed fix723 fully resolves the reported abort (both hosts
route-push Settings → dispose on leave; work is main-isolate + captures `this`;
the guard makes the loop throw-safe). It ALSO found a distinct pre-existing
setState-after-dispose in the same method's tail: `_runEpgRefresh` checked
`mounted` BEFORE `await Sql.getLatestEpgRefresh()` but called `setState` AFTER it
with no re-check → same throw if the user leaves during that await. Folded in:
`if (mounted) { final ts = await ...; if (mounted) setState(() => _latestEpgRefreshTs = ts); }`.
(`_runEpgRematch` has no trailing post-await setState — already clean.)
