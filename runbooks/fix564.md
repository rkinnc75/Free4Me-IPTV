# fix564 — developer playback-stats overlay (single-cell full-screen) + logging

Base: **v2.0.63** (CI latest, re-verified). Ships as **v2.0.64+564**.

## What this adds

A compact, semi-transparent **playback-stats panel pinned to the top-right** of
single-cell full-screen video, shown **only when debug logging is on**. It polls
libmpv once a second and shows the live decode path, frame rate, dropped-frame
rate, A/V sync, bitrate and buffer — and writes the **same snapshot to the report
log every tick** (tagged `STATS`) so a report upload during a stutter gives a
reviewable time-series.

Rows (amber = warning):
- **hwdec** `requested -> current` — amber when current is empty/`no` (software).
- **fps** `rendered / source` (`estimated-vf-fps` / `container-fps`) — amber when
  rendered falls >2 fps below source (the headline stutter signal).
- **drop** `vo N (+delta)  dec N (+delta)` — amber while either delta > 0
  (`decoder-frame-drop-count` rising = the decoder can't keep up).
- **avsync** in ms — amber when |avsync| > 80 ms.
- **bitrate**, **cache** (amber while `paused-for-cache`), **res + codec**.

## Why (the audit behind it)

On the Onn 4K Plus (low-RAM Amlogic) full-screen decode is routed to `hwdec=no`
(software) by `androidFullscreenHwdec` — chosen because `mediacodec-copy`'s
GPU→CPU readback was memory-bandwidth-bound and desynced A/V on this SoC. But
software decode on a 2 GB Amlogic stutters on real-bitrate streams, while the
reference app stays smooth using surface-mode hardware decode (ExoPlayer →
MediaCodec, no CPU readback) — a path media_kit's render API can't take (no
native Surface), which is why surface-mode `mediacodec` falls back to software.
This overlay lets you confirm live what `hwdec-current` actually is and whether
decoder drops are the cause, and A/B the existing `forceHwDecode` override while
watching `avsync` + drops.

## Implementation

- `MpvEngine.readPlaybackStats()` — pure, guarded read returning a
  `Map<String,String>` of raw libmpv properties (reuses the proven getProperty
  pattern from `logDecodeState`; returns `{}` if not a ready `NativePlayer`).
- New `lib/player/debug_stats_overlay.dart` — `DebugStatsOverlay` polls the
  engine every 1 s, renders the `Positioned` top-right panel (wrapped in
  `IgnorePointer`, so it never takes D-pad focus), and `AppLog.info`s each
  snapshot.
- `Player.build` adds it to the video `Stack`, gated on
  `settings.debugLogging && multiViewLayout == none && _engine is MpvEngine`
  (PiP mode already returns earlier, so it's excluded). No new settings field.

Verified in-sandbox: `git apply --check` passes on a fresh `v2.0.63` clone;
brace/bracket balance OK; `withValues(alpha:)` is already used elsewhere (valid
on the project's Flutter 3.44). `flutter analyze` (0 INFOs) and `flutter test` to
be confirmed on the build machine.

Files: `lib/player/mpv_engine.dart`, `lib/player/debug_stats_overlay.dart` (new),
`lib/player.dart`, `lib/whats_new_modal.dart`, `pubspec.yaml`.

+  /// (top-right, single-cell full-screen only) and the report log. Pure read —
+  /// every getProperty is guarded; returns {} if the platform player isn't a
+  /// ready NativePlayer. Keys are raw libmpv property names so the caller can
+  /// format/colour them. `estimated-vf-fps` falling below `container-fps`, a
+  /// climbing `decoder-frame-drop-count`, or a large `avsync` are the headline
+  /// stutter signals; `hwdec-current` empty/"no" means software decode.
+  Future<Map<String, String>> readPlaybackStats() async {
+    if (_disposed || _player.platform is! mk.NativePlayer) return const {};
+    final np = _player.platform as mk.NativePlayer;
+    Future<String> g(String p) async {
+      try {
+        return await np.getProperty(p);
+      } catch (_) {
+        return '';
+      }
+    }
+
+    const keys = <String>[
+      'hwdec', // requested
+      'hwdec-current', // actually engaged ('' / 'no' = software)

+      'container-fps', // source fps
