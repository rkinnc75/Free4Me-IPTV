# fix573 — backup/restore was silently dropping 3 settings (backlog #11)

**Version:** v2.1.2+575 (shipped with fix574 + fix575)
**Date:** 2026-06-27
**Type:** Data-loss fix (backup/restore)

## Problem

Three settings were persisted normally (SQLite key-value via `settings_service.dart`)
but were **missing from the backup payload** (`SettingsIo._settingsToMap` /
`_settingsFromMap`), so exporting a backup and restoring it silently reset them to
their constructor defaults:

- `multiViewDecode` (Tegra/Shield decode mode, fix314)
- `devControlsHideSecs` (control-bar auto-hide seconds, fix409)
- `playerZoomMode` (last full-screen fit/stretch/crop, fix422)

## Change

`lib/backend/settings_io.dart`:
- `_settingsToMap` now serializes all three (`multiViewDecode.toJson()`,
  `devControlsHideSecs`, `playerZoomMode.name`).
- `_settingsFromMap` restores them with the established `is String`/`is int`
  guards (a missing field keeps the constructor default — older backups are safe).
  `ZoomMode` has no `fromJson`, so it is parsed by `ZoomMode.values.firstWhere(name,
  orElse: fit)`; `MultiViewDecode` uses its existing `fromJson`.
- Added `@visibleForTesting static Settings roundTripForTest(Settings)` =
  `_settingsFromMap(_settingsToMap(s))` so the round-trip is unit-testable without
  the DB / file-picker / platform channels.

## Verification

- `test/fix573_settings_roundtrip_test.dart` sets each field to a non-default value,
  round-trips, and asserts survival (2 cases). `flutter analyze` clean, `flutter test`
  green.
- Regression guard: if a future Settings field is added to `settings_service` but not
  to `settings_io`, this same class of bug returns — the round-trip test is the place
  to pin new fields.
