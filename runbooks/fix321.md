# fix321 — Per-type partial-fetch guard: retry once, preserve a content type that comes back empty → v1.26.37+321

> ## ⚠️ BUILD MACHINE — APPLY THE PATCH, then RELEASE (EXACTLY). No questions, no DB to query.
> **FIRST:** `find . -name 'fix296-DIAGNOSE*.md' -delete 2>/dev/null || true`

## Why
Observed: Emjay refreshed in background returned `live=55357 movies=172101 series=0` — the slow tail call (`get_series`, on a max_connections=1 provider) came back empty with no error, and the refresh wiped+committed `series=0`, destroying ~44k series rows. fix318 (foreground service) reduces but cannot eliminate background network throttling at the tail of a long refresh; the real defense is to never overwrite good data with a partial/empty fetch.

## What this does (per-type, with one retry)
For each content type (live / movies / series) whose fresh fetch returns **0 rows** but the source **previously had rows** for it (`source.lastLiveCount/lastMovieCount/lastSeriesCount > 0`):
1. **Retry that one fetch once.** Transient empties are common at the tail of a long refresh.
2. If still empty, **preserve the existing rows for that type** — that media type is excluded from the wipe and not re-inserted — while the other types refresh and commit normally.

The fresh count for a preserved type is not recorded (the old `last*Count` is kept, since the rows remain). The other types behave exactly as before.

## How it's built
- `Sql.wipeSource(sourceId, {keepMediaTypes})`: when `keepMediaTypes` is non-empty, the channel DELETE becomes `... AND media_type NOT IN (…)`, and groups are deleted only if they have no surviving (kept) channels — so preserved-type categories stay intact. Empty set = previous behaviour (full wipe).
- `getXtream`: content insert closures are collected into a separate list; after the fetch, empty-but-previously-present types are retried once, the still-empty ones added to `keepMediaTypes`; then `wipeSource(keepMediaTypes:)` is spliced in front of the content inserts. `last*Count` is only overwritten for non-preserved types.

## Verified before delivery
- `flutter analyze --no-fatal-infos` → 2 tolerated `settings_view.dart` INFOs, no warnings/errors.
- Built against live main (1.26.36+320); patch applies cleanly to a fresh clone.
- **Rule 8 (real sqlite):** seeded a source with live=3/movies=2/series=4; simulated series empty → conditional wipe kept exactly the 4 series rows + their group and deleted live/movies; re-inserting fresh live/movies yielded live=2(new)/movies=1(new)/series=4(preserved). Groups orphan-cleanup left only the preserved category.

## Note
- This is independent of and complementary to fix318/fix320. It guards every refresh path (foreground or background, manual or import-triggered).
- Only triggers when a type goes to 0 against a known-nonzero history — a source that legitimately has 0 of a type (e.g. a3000’s tiny catalogue) is unaffected, and a brand-new source (no prior counts) wipes normally.
