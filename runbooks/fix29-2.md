# fix29.md — Search feels slower as the query gets longer

> Symptom: typing "mia" in the search box shows results almost
> immediately; continuing to "miami dolphins" feels like results take
> much longer to appear.
>
> Most likely cause (per benchmarking — see "What I'm confident in"
> below): the 500 ms debounce timer in `lib/home.dart:382`. Every
> keystroke cancels the in-flight timer and starts a new 500 ms one,
> so the search fires 500 ms after the user's *last* keystroke. The
> user perceives this as "longer queries are slower."
>
> **However**, I haven't measured the symptom on the actual device
> with the actual data. The SQL query is sub-millisecond on
> synthetic 200 000-row data, but the user's database could be
> larger, fragmented, on slower flash storage, or have other
> contention. Before committing to a single-number fix, this runbook
> adds **diagnostic logging at every stage** of the search pipeline
> so the next test session produces measurements that conclusively
> pin the bottleneck.
>
> Apply both parts. Run a test session. Read the log. Then decide
> whether the 200 ms debounce alone is sufficient, or whether the
> query / mapping / setState stages need their own fix.

---

## Apply order

1. **Part 29.1** — add timing logs to the search pipeline (2 files).
2. **Part 29.2** — drop the debounce from 500 ms to 200 ms (one line
   in one file, included in 29.1's `onChanged` replacement).
3. **User runs a search test** with debug logging enabled.
4. **Read the log** — the new lines tell you exactly where time goes.
5. If the debounce was the bottleneck, you're done. If something
   else dominates, follow up with a fix targeted at that stage.

# Part 29.1 — Diagnostic logging across the search pipeline

The pipeline has five distinguishable stages, each of which could
contribute to perceived slowness:

| Stage | Where it lives | What we log |
|---|---|---|
| 1. Keystroke received | `home.dart` `onChanged` | char count, time since previous keystroke, time since first keystroke in current burst |
| 2. Debounce fires | `home.dart` `Timer` callback | how long the timer actually waited (could exceed schedule under load), final query string |
| 3. SQL search executes | `sql.dart:193` `search()` | branch taken (FTS / FTS+LIKE / LIKE-only / no-query), execution time, parameter count |
| 4. Row mapping | `sql.dart:281` `results.map(rowToChannel)` | rows returned, mapping time |
| 5. setState + grid rebuild | `home.dart` `setState` block + post-frame callback | time from query-start to first frame the user actually sees |

Each log line is one short, parseable record. All gated on
`AppLog.enabled` so they're inert when logging is off.

A monotonic **invocation id** ties them together: each `load()`
call gets a fresh `[N]` tag that propagates to `Sql.search[N]` and
the post-frame `Home.load[N]: rendered` line. Reading the log, you
can follow one search end-to-end.

## Step 1 — keystroke / debounce instrumentation

**File:** `lib/home.dart`

### Add the state fields

Insert next to `Timer? _debounce;` (line 43):

```dart

// keystroke rate with perceived search latency. Reset whenever the

DateTime? _firstKeystrokeAt;
DateTime? _lastKeystrokeAt;
int _keystrokeCountInBurst = 0;
int _searchInvocation = 0; // monotonic; correlates debounce → SQL → setState lines
```

### Replace `onChanged`

**Current code (lines 380–390):**

```dart

  _debounce?.cancel();
  _debounce = Timer(
    const Duration(milliseconds: 500),
    () {
      if (!mounted) return;
      widget.home.filters.query = query;

```

**Replace with:**

```dart
onChanged: (query) {
  // fix29 diagnostics — record keystroke arrival relative to the
  // burst so we can see how long the user has been typing before
  // the debounce fires, and how stale the on-screen results are.
  final now = DateTime.now();
  if (AppLog.enabled) {
    final firstAt = _firstKeystrokeAt ??= now;
    final sinceFirst = now.difference(firstAt).inMilliseconds;
    final sincePrev = _lastKeystrokeAt == null
        ? 0
        : now.difference(_lastKeystrokeAt!).inMilliseconds;
    _keystrokeCountInBurst++;
    AppLog.info(
      'Search.keystroke: chars=${query.length} burst=$_keystrokeCountInBurst'
      ' sinceFirst=${sinceFirst}ms sincePrev=${sincePrev}ms'
      ' query="$query"',
    );
  }
  _lastKeystrokeAt = now;

  _debounce?.cancel();
  // 200 ms (was 500 ms — see fix29 part 29.2). Short enough that a
  // fast typist sees results update mid-word; long enough that a
  // single typing burst doesn't fire ~4 queries per character.
  final scheduledFor = now.add(const Duration(milliseconds: 200));
  _debounce = Timer(
    const Duration(milliseconds: 200),
    () {
      if (!mounted) return;
      if (AppLog.enabled) {
        final firedAt = DateTime.now();
        final scheduledLatency =
            firedAt.difference(scheduledFor).inMilliseconds;
        final burstDuration = _firstKeystrokeAt == null
            ? 0
            : firedAt.difference(_firstKeystrokeAt!).inMilliseconds;
        AppLog.info(
          'Search.debounce-fired: keystrokes=$_keystrokeCountInBurst'
          ' burstDuration=${burstDuration}ms'
          ' scheduledLatency=${scheduledLatency}ms'
          ' query="$query"',
        );
      }
      // Reset burst tracking now that we're firing.
      _firstKeystrokeAt = null;
      _keystrokeCountInBurst = 0;
      widget.home.filters.query = query;
      load(false);
    },
  );
},
```

`scheduledLatency` is the key diagnostic: it measures how long
*after* the 200 ms target the Timer actually fired. If this number
is consistently >50 ms, the Dart event loop is jammed and the
debounce isn't the real problem — something else upstream
(probably a prior `setState`) is blocking the main thread.

## Step 2 — instrument `load()` with end-to-end timing

**File:** `lib/home.dart`

**Current code (lines 130–163):**

```dart

  if (more) {
    widget.home.filters.page++;
  } else {
    widget.home.filters.page = 1;
  }
  if (AppLog.enabled) {

    if (AppLog.enabled) {
      AppLog.info('Home.load: got ${results.length} results '

    }
    if (!mounted) return;
    setState(() {
      if (!more) {
        channels = results;

```

**Replace with:**

```dart
Future<void> load([bool more = false]) async {
  if (more) {
    widget.home.filters.page++;
  } else {
    widget.home.filters.page = 1;
  }

  // fix29 diagnostics — every load gets a monotonic id so the
  // keystroke / debounce / SQL / setState lines can be correlated.
  final inv = ++_searchInvocation;
  final loadStart = DateTime.now();

  if (AppLog.enabled) {
    final f = widget.home.filters;
    AppLog.info(
      'Home.load[$inv]: view=${viewTypeToString(f.viewType)} '
      'page=${f.page} more=$more '
      'sources=${f.sourceIds?.length ?? "null"} '
      'mediaTypes=${f.mediaTypes?.map((m) => m.name).join(",") ?? "null"} '
      'groupId=${f.groupId} seriesId=${f.seriesId} '
      'query=${f.query ?? "none"}',
    );
  }
  await Error.tryAsyncNoLoading(() async {
    final searchStart = DateTime.now();
    final results = await Sql.search(widget.home.filters, invocation: inv);
    final searchElapsed =
        DateTime.now().difference(searchStart).inMilliseconds;
    if (AppLog.enabled) {
      AppLog.info(
        'Home.load[$inv]: got ${results.length} results in ${searchElapsed}ms'
        ' for ${viewTypeToString(widget.home.filters.viewType)}',
      );
    }
    if (!mounted) return;

    // Drop late results from a superseded query. If a newer load()
    // has started, don't clobber its results with ours.
    if (inv != _searchInvocation && !more) {
      if (AppLog.enabled) {
        AppLog.info(
          'Home.load[$inv]: SUPERSEDED — current=$_searchInvocation,'
          ' dropping ${results.length} stale results',
        );
      }
      return;
    }

    final setStateStart = DateTime.now();
    setState(() {
      if (!more) {
        channels = results;
      } else {
        channels.addAll(results);
      }
      reachedMax = results.length < pageSize;
    });

    if (AppLog.enabled) {
      // Use a post-frame callback to measure the time from setState
      // to the first frame after the new tiles are actually painted.
      // This catches any heavy work in build/layout/paint that the
      // user perceives as "search is slow."
      final setStateSync =
          DateTime.now().difference(setStateStart).inMilliseconds;
      WidgetsBinding.instance.addPostFrameCallback((_) {
        final totalElapsed =
            DateTime.now().difference(loadStart).inMilliseconds;
        AppLog.info(
          'Home.load[$inv]: rendered'
          ' setState=${setStateSync}ms'
          ' total=${totalElapsed}ms'
          ' (search=${searchElapsed}ms,'
          ' rest=${totalElapsed - searchElapsed}ms)',
        );
      });
    }
  }, context);
}
```

The post-frame callback is the key new measurement — it tells you
how long from `setState()` to the first frame on which the user
actually sees the new tiles. If `total - search` is e.g. 600 ms,
that points to SliverGrid rebuild or image preload as the
bottleneck, not SQL or debounce.

Add the import at the top of `home.dart` if not already present:

```dart
import 'package:flutter/scheduler.dart' show WidgetsBinding;
```

(Likely already imported transitively through `flutter/material.dart`.)

### Reading the timing breakdown

For each search invocation `[N]` look at four numbers:

1. **`burstDuration`** — how long the user was typing before the
   search fired. If this number is what dominates wall-clock time
   from "first character" to "results visible", **the debounce is
   the perceived slowness** and Part 29.2 fixes it.

2. **`scheduledLatency`** — how late the timer fired vs schedule.
   Normal: 0–10 ms. Bad: 100+ ms. If consistently high, the main
   thread is jammed; investigate prior frames.

3. **`sql`** — pure SQLite execution. Normal on phone: <10 ms even
   for FTS phrase queries. Bad: 50+ ms. If high, the SQL is the
   real bottleneck and we need a different fix (different index,
   query restructure, or move to a background isolate).

4. **`rest` = `total − search`** — everything else: setState +
   layout + paint + post-frame callback overhead. Normal: 30–150
   ms. Bad: 300+ ms. If high, the SliverGrid rebuild or image
   preload is the bottleneck.

# Part 29.2 — Drop the debounce from 500 ms to 200 ms

Already incorporated into the `onChanged` replacement in Part 29.1
step 1 (the `Timer` literal is `Duration(milliseconds: 200)` not
`500`).

If you want to test with the original 500 ms first to confirm the
diagnosis, swap both numerals (`scheduledFor` and the `Timer`
literal) back to 500 — both lines should match.

200 ms is the conventional default for search-as-you-type in
material-design apps. Short enough that a fast typist sees results
update mid-word; long enough that a single typing burst doesn't
fire ~4 queries per character.

The two parts are split conceptually because:

- **Part 29.1** is a low-risk diagnostic patch that helps anyone
  understand search performance. Worth keeping in the codebase even
  after the symptom is resolved.
- **Part 29.2** is the conjectured fix for *this specific symptom*.
  If the log evidence backs it up, ship; otherwise tune.

## What the log should NOT show (sanity checks)

After applying Part 29.1, watch for any of these red flags:

1. **`scheduledLatency` > 200 ms repeatedly.** Means the Dart event
   loop is jammed for hundreds of ms. Almost certainly a prior
   `setState` is doing heavy work. Could be image loading or
   another sync I/O op on the main thread.

2. **`Home.load[N]: SUPERSEDED`** firing on most loads. Means the
   debounce isn't actually deduplicating — successive loads race.
   The generation guard drops the stale ones, but the SQL work
   was already done. Tighten the debounce or restructure the
   call flow.

3. **`sql=` time growing linearly with the query length.** Would
   mean my benchmark was wrong about FTS5 trigram phrase queries.
   Possible if the user's data is dramatically different from my
   synthetic corpus (e.g. very long names, many channels matching
   the early trigrams). The `branch` tag tells you which path is
   slow.

4. **`map=` time approaching `sql=` time.** Would mean
   `rowToChannel` is significant. With 36 rows max per page this
   should be negligible (microseconds), but if it's 10+ ms on a
   slow phone we might want to skip optional columns.

5. **`rest` time exceeding 500 ms.** Would mean the widget tree
   rebuild + image preload is the bottleneck. Likely fix: stop
   including `$index` in the ChannelTile's `ValueKey` (line 468),
   which forces a fresh widget for every result change.

## Notes for the implementer

- **All logs are gated on `AppLog.enabled`.** No production
  overhead when logging is off. Even when on, each log call is one
  short `info()` write (no expensive string formatting outside the
  guard except where `${...}` interpolation is unavoidable; the
  truncation step in `Sql.search` avoids dumping very long queries).
- **No SQL schema changes.** No new dependencies.
- **The `invocation` parameter on `Sql.search` defaults to 0**, so
  any other callers that don't bother to pass one still work.
  Only `Home.load` is affected.
- **The generation-counter guard** (`if (inv != _searchInvocation`)
  is the same pattern used in `MultiViewCell._openGeneration`. It
  protects against `setState` from a stale load clobbering a newer
  one. Independent of the debounce — safe to keep even after
  Part 29.2 makes it rare to fire.
- **Total code change:** ~80 lines across 2 files, almost all of
  it new log lines. Five lines of actual logic change (the
  generation guard + the 200 ms literal + the `branch` variable
  + its three assignments).
