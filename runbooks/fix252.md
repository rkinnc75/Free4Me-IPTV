# fix252 — TV: channel selector focus starts on the first tile; UP enters the search bar (item 3)

> **Version:** build = highest fix number; patch = next available. Tag = next immutable.
> **Builds on:** current source **1.25.4+246**. Touches `lib/channel_picker_screen.dart` only.
> **Goal (item 3):** on TV, opening a channel selector should land focus on the FIRST channel tile (so the user can scroll immediately with the D-pad), and pressing UP from the top row should move focus into the search bar. This should be the standard behavior for channel selectors.
> **Root cause:** the channel picker's search field had `autofocus: true`, so it grabbed initial focus — the user landed in the text box instead of the list. The D-pad up/down traversal plumbing already exists (`DpadTextField` yields ArrowUp→previousFocus / ArrowDown→nextFocus), so once the first tile holds focus, UP naturally reaches the search bar and DOWN from the search bar returns to the list.

## What was actually wrong vs. already correct

- **Channel picker (`channel_picker_screen.dart`)** — the offender. Search `autofocus: true` trapped focus. **Fixed here.**
- **TV home (`tv_home.dart`)** — already correct: it autofocuses the first `MenuTile` (not a search field), which is the desired pattern. No change needed.
- The shared `DpadTextField` already implements the "UP/DOWN yields focus out of the text field" behavior, so no widget-level change is required for the "UP enters search" half — it works once the list holds initial focus.

## The fix

1. Search field: `autofocus: true` → `autofocus: false` (stop trapping initial focus).
2. First channel tile: `ListTile.autofocus = (i == 0)` (initial D-pad target).

With the list focused, the existing `FocusTraversalGroup`/traversal moves focus UP into the search bar from the top row, and the search bar's own ArrowDown returns to the list.

## Notes / honest scope
- This fixes the primary channel selector (the picker). If any OTHER selector screen is later found to autofocus its search field, apply the same two-line pattern there (`autofocus: false` on the field + `autofocus: i == 0` on the first item). A scan of the current source shows the picker was the one trapping focus; tv_home already does the right thing.
- Interaction edge case to watch in testing: when a search drastically changes the list, `ListTile.autofocus` on the new index-0 could pull focus back to the top. In practice `autofocus` only fires on first insertion, so a filtered rebuild should not yank focus mid-typing — but verify on-device, since this is the kind of thing that's hard to prove statically. If it does misbehave, the follow-up is to gate the tile autofocus to the very first build only (a `bool _didInitialFocus` latch).
- Multi-view's empty-cell "+" initial focus is handled separately in fix250.
