# fix246 — Multi-view: self-heal dropped cells with bounded slow recovery (5 fast + 5 slow retries)

> **Version:** build = highest fix number in release; patch = next available. Tag = next immutable.
> **Builds on:** 1.25.x source. Touches `lib/multi_view_cell.dart` only.
> **Problem (from the TV 2×2 log):** all four cells started and played fine, but streams dropped mid-session (cell 0 at +22 min, cell 1 at +28 min, cells 2/3 at +56 min — each starting with "Error decoding audio" then "Failed to open"). When a cell exhausts its fast transient retries it shows the error UI and **stays dead** until manual retry — so a cell that drops at minute 22 is dead for the rest of an hour-long session.
> **Fix:** after the existing fast retries are exhausted, attempt a BOUNDED SLOW recovery — **5 attempts at 60 s apart** — then surface the permanent error UI. The slow cadence is gentle on the provider's connection budget (important with a 4-connection account and four cells). A recovered, stable stream resets the budget so it can self-heal again later.

## Behavior (exact)

For a transient mid-session drop:
1. **5 fast retries** at 3 s apart (existing — `maxReconnectAttempts`, default 5).
2. Fast budget exhausted → **slow recovery**: dispose the dead engine (frees its provider connection while waiting), wait 60 s, re-open with a fresh fast budget. Repeat up to **5 times**.
3. After the 5th slow attempt also fails its fast retries → **permanent error UI** (broken-image + manual retry button, as before).
4. If any attempt succeeds and plays stably for 15 s → both the fast and slow counters reset (fully healed; eligible for the full ladder again if it drops later).
5. Changing the cell's channel or closing multi-view cancels any pending slow recovery (generation token + timer cancel).
- Genuine permanent errors (auth/4xx/codec — `_isTransientError` false) skip slow recovery and go straight to the error UI, unchanged.

## Pre-verified facts (checked against 1.25.x, compiled)

- File: `lib/multi_view_cell.dart` only.

### Replacement code (verbatim)
```dart
  @override
  void dispose() {
    _recoveryTimer?.cancel(); // fix246
    _recoveryTimer = null;
    _disposeEngine();
    super.dispose();
  }

  /// fix246: schedule one slow recovery attempt [_recoverySlowInterval] from
  /// now. Increments the slow-retry counter and shows a waiting message. If
  /// the channel changes or the cell is disposed before it fires, the
  /// generation token / timer cancel makes it a no-op. On the attempt it
  /// re-opens via _startEngine(isRetry: true) so the fast-retry counter is
  /// preserved; if that attempt also fails the error handler re-enters this
  /// scheduler until [_recoverySlowMax] is reached, then surfaces the error.
  void _scheduleSlowRecovery(Channel ch, int generation) {
    if (!mounted || generation != _openGeneration) return;
    _recoverySlowRetries++;
    final attempt = _recoverySlowRetries;
    final secs = _recoverySlowInterval.inSeconds;
    AppLog.info(
      'MultiViewCell: slow recovery $attempt/$_recoverySlowMax'
      ' in ${secs}s cell=${widget.cellIndex}'
      ' channel="${ch.name}"',
    );
    if (mounted) {
      setState(() {
        _retryMessage = 'Reconnecting (waiting ${secs}s)…';
        _loading = true;
        _error = false;
      });
    }
    // Dispose the dead engine now so it isn't holding a provider connection
    // open while we wait (matters with a small max_connections budget).
    _disposeEngine();
    _recoveryTimer?.cancel();
    _recoveryTimer = Timer(_recoverySlowInterval, () {
      if (!mounted) return;
      AppLog.info(
        'MultiViewCell: slow recovery attempt $attempt/$_recoverySlowMax'
        ' cell=${widget.cellIndex} channel="${ch.name}"',
      );
      if (mounted) setState(() => _retryMessage = null);
      // Reset the fast-retry counter so this attempt gets a full fast budget
      // again, but keep the slow-retry counter (carried in the field).
      _transientRetries = 0;
      _startEngine(ch, isRetry: true);
    });
  }
```
