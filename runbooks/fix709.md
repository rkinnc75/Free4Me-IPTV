# fix709 — EPG match concurrency: serialize the match phase (v4.1.14)

**Builder:** desktop. **Live base:** v4.1.13. **Ships as:** v4.1.14+709. Backend
(EPG refresh) — all platforms. Overnight autonomous run (owner approved "go").

## The bug (root-caused + confirmed on-device)
TV Guide showed "No guide data" on most/all channels despite EPG having
downloaded. Root cause: `EpgService.refreshAllSources` runs sources
`maxConcurrent=2` (line ~209); each `refreshSource` = download → **match**. Two
`matchChannels` at once collide on `db.sqlite` — `setChannelEpgIds` (513) + the
WAL **TRUNCATE** checkpoint `checkpointAndTruncateWal(epg:false)` (517, TRUNCATE
needs an EXCLUSIVE lock). The SQLITE_BUSY retries (finding-130 / fix625,
maxAttempts=5, ~5s) exhaust under sustained contention → `matchChannels` throws →
the per-source `try/catch` (227) logs + continues → that source's channels stay
silently UNMATCHED. Proof: disable all sources but Trex, refresh → **35851/35851
matched**, guide fully populated (was empty on every channel). The automated
refresh (background 24h @ 3AM, and launch-if-stale ≤1/hr) is the recurring
re-trigger, since both share `refreshAllSources`.

## The fix
`epg_service.dart` — an **in-isolate chained-Future gate** serializes the match:
```
static Future<void> _matchGate = Future<void>.value();
static Future<void> _serializeMatch(Future<void> Function() body) async {
  final prev = _matchGate; final done = Completer<void>();
  _matchGate = done.future; await prev;
  try { await body(); } finally { done.complete(); }
}
```
`refreshSource` now calls `await _serializeMatch(() => matchChannels(...))`.
- **Downloads stay parallel** — already contention-tolerant: epg.sqlite batch
  inserts retry, and the download's db.sqlite checkpoint is **PASSIVE** (378, not
  an exclusive TRUNCATE). Only the match's db.sqlite TRUNCATE needed serializing.
- **In-isolate only** — the `maxConcurrent=2` concurrency lives within one
  `refreshAllSources` call in one isolate. Cross-isolate (foreground vs
  Workmanager background) stays guarded by the app_meta refresh lock
  (`tryAcquireRefreshLock` / `epg_refresh_in_progress`) + the SQLITE_BUSY retries.
- **Gate never wedges** — released in `finally`; the initial `prev` is a
  completed `Future.value()`; `prev` is only ever a no-error `Completer.future`.
- Covers **all** refresh triggers (manual / background / launch) — one place.

## Refined mechanism (from review 2)
The precise starvation: while source A runs its match's db.sqlite **TRUNCATE**
checkpoint (must clear all readers past the checkpoint), source B's concurrent
35k-row **paging reads** (`getChannelsNeedingEpgMatchPage`, pageSize=5000, line
~453) hold a WAL read-snapshot on connections in sqlite_async's read pool →
A's TRUNCATE hits SQLITE_BUSY → 5 retries exhaust → throw → swallowed by the
per-source catch → A UNMATCHED. It was a RACE on the busy-timeout×retry budget
(hence single-source=100%, multi-source=intermittently empty). The gate makes B
idle-await while A matches (B's download already produced its channelMap), so A's
TRUNCATE has no competing reader → **deterministic** success.

## Follow-up (task #39, NOT a fix709 regression)
Manual "Re-match all" (`settings_view.dart:~1318`) calls `matchChannels` DIRECTLY,
bypassing the gate. Its own loop is sequential (can't self-collide) but could
collide with an overlapping launch/auto refresh in the same isolate. Fix later by
routing it through the gated `refreshSource(forceRematch:true)`.
