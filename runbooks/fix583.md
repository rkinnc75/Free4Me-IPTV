## #2 follow-up ‚Äî force-30 cap OFF-reset

fix582 wired the force-30 `vf` cap that had been INERT since fix570. But existing
low-RAM installs had `devCapFpsLowRam` **persisted `true`** from the old (inert)
default, so they suddenly capped to 30 fps on v2.1.9 ‚Äî a downgrade from the
full-rate 0-drop that framedrop=decoder already gives. On-device confirmed:
SP-YES showed `vfFps=30` / `srcFps=60` / `voDrop=0` (the cap works) but was
unintentionally on.

Fix: **re-key the persistence** (`devCapFpsLowRamProp` "devCapFpsLowRam" ‚Üí
"forceCapFps30LowRam_v2"). No persisted value of the inert setting reflected a real
choice, so orphaning it makes every install reload at the new default (OFF). The
wired toggle still binds to `settings.devCapFpsLowRam`; only its storage key
changed. (Folded forward per the no-wasted-revert-build rule.)

## #18 ‚Äî groups unique key + insert now include media_type

The groups unique index was `(name, source_id)` (migration 1) but `media_type` was
added to groups later (migration 33) and never added to the constraint OR the
insert grouping ‚Äî so a category with the same name in different media types (e.g. a
Live "Sports" and a Movies "Sports") collided into one row, and channels could link
to the wrong one. Four coordinated changes (all SQL ‚Äî verified on-device, not by
analyze):

1. **Migration 40** (`db_factory.dart`): drop `index_group_unique`, recreate on
   `groups(name, source_id, media_type)`. The 3-col index is LESS restrictive than
   the old 2-col, so it cannot fail on existing data; runs on every install.
2. **Upsert** (`sql.dart` updateGroups): `GROUP BY group_name, media_type` and
   `ON CONFLICT(name, source_id, media_type)` (the conflict target MUST match the
   new index or the upsert throws "no unique constraint matching ON CONFLICT").
3. **Channel‚Üígroup linking** (`sql.dart`): the `UPDATE channels SET group_id`
   join gained `AND g.media_type IS channels.media_type` (NULL-safe) so each
   channel links to the group row of its OWN media type, not an arbitrary same-name
   one.
4. Read side already filters by media_type (`groupSearchWhere`), so per-media-type
   group rows show in the correct rail with no duplicates.

Known minor: the disabled-category re-stash (updateGroups `SET enabled=0 ... name
IN (...)`) is still by name, so disabling a category name affects all its
media-type variants ‚Äî acceptable (low impact); revisit if it matters.
