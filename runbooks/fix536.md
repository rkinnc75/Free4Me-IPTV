# fix536 — Fix OOM when exporting the DB snapshot; offer DB in diagnostic export

**Base tag:** `v2.0.35`
**Version:** `2.0.35+535` → `2.0.36+536`
**Scope:** export-bundle builder + diagnostic-export opt-in. No schema/query change. Full suite green.

---

## Problem (fix535 regression, reproduced on device)

1. **OOM:** "Export settings to file" → Yes (include credentials) → **"Could not prepare the export / Out of Memory"** on the 2 GB onn 4K box. fix535 added the DB to the combined zip (`toZip[destPath] = destName`), and `ZipFileEncoder.addFile` reads the whole multi-hundred-MB `db.sqlite` into the heap to compress it → OOM.
2. **No DB from the diagnostic export:** the QR/source-dump export (`_exportSourceDumps` → `_runTvServerExport`) passes `includeCredentials:false`, so it never offered the DB — which is the export the user actually runs.

## Fix

1. **Don't zip the DB.** Stream-copy each `.sqlite` to the export dir as a STANDALONE portal download (`_streamCopyFile`, chunked read→write sink) and do NOT add it to `toZip`. The zip stays small (dumps + log + settings). A SQLite file barely compresses, so nothing is lost. This removes the heap blowout.
2. **Offer the DB in the diagnostic export.** `_exportSourceDumps` now shows the same credentials-gated "Include database snapshot?" prompt (defaults to No) and passes the choice to `_runTvServerExport`.

Both DB paths remain gated on the credential opt-in (the DB carries Xtream usernames/passwords in `sources` and embedded in channel URLs).

Validated standalone: WAL-checkpointed `.sqlite` chunk-copies byte-identically (sha256 match on a 50 MB file), and the copy never holds the whole file in memory.
