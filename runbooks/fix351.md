## Why (root cause from the S24 2026-06-11 17:41 log)
The fix341 quick-reopen budget claims to replenish "after 15s of stable playback", but the reset in the bufferingStream listener requires `_lastErrorAt != null` — and only the errorStream path ever set it. Connection-cycling providers (Dino, max_connections=1, 2×2 grid) drop via CLEAN EOF (`via completedStream`, 44/44 in the log), so `_lastErrorAt` stayed null, the reset never fired, and `_quickReopens` climbed monotonically 1/8 → 8/8 in ~5 min per cell despite 25–50s stable stretches between drops. At 8/8 the next drop forced a full dispose → spinner → 30s "Building buffer…" restart, the counter reset as a side effect, and the cycle repeated forever — the visible badness that made the stability buffer feel broken.

## What this does (one file, ~12 lines)
`lib/multi_view_cell.dart`: the EOF quick-reopen path now stamps `_lastErrorAt = DateTime.now()` when it spends budget — a clean provider EOF counts as a disturbance for the 15s stability clock, exactly like an engine error. The existing reset block (which already zeroes `_transientRetries`, `_recoverySlowRetries`, and `_quickReopens` together) now actually fires between drops. Field doc and reset-block comment updated to match.

`_lastErrorAt` is read ONLY by that reset block (verified: writes at error path + this new stamp; nulled on fresh start and after reset) — no side effects on transient-retry pacing.

**Expected behaviour after fix:** on cycling sources, each cell builds its stability buffer once per session, then recovers every subsequent drop with the ~1s freeze-frame quick re-open indefinitely (drop interval 25–50s > 15s stable threshold → budget refills faster than it drains). Full restarts stop. On stable sources: no change.
