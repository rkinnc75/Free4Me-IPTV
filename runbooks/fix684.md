# fix684 — v4.0.0: bundle the muxer-enabled libmpv engine + docs refresh

**Build:** `4.0.0+684` (major bump — new media engine) · **Base tag:** `v3.0.25` (fix683) · **Ship order:** after 683
**Files touched:** `pubspec.yaml`, `lib/whats_new_modal.dart`, `README.md`, `docs/CUSTOM_LIBMPV.md`, `docs/LIBMPV_COMPONENTS.md`
**Minimal code:** one `dependency_overrides` ref line (the engine swap). Everything else is docs/version.

---

## What this does

1. **Bundles the muxer-enabled libmpv** (the actual functional change): repoints
   `dependency_overrides: media_kit_libs_android_video` from `rkalsky/media-kit`
   @ `6ae83bb8` → **`rkinnc75/media-kit` @ `00500da2`**, whose `build.gradle` pulls the
   muxer-enabled per-ABI jars (MD5-verified) from `rkinnc75/libmpv-android-video-build`
   release `vnext`. This is what makes MP4/MKV remux possible on-device.
2. **Version → 4.0.0** (major — new media engine) + changelog entry.
3. **Docs refresh:** `README.md` (adds a Scheduled Recording feature section — previously
   absent; notes the custom muxer engine; Flutter prereq 3.27→3.44; credits),
   `docs/CUSTOM_LIBMPV.md` (muxers 0→9; `rkalsky`→`rkinnc75` fork rename; fork-ownership
   note; rebuild procedure now lists the muxer flags + symbol check),
   `docs/LIBMPV_COMPONENTS.md` (flavor line + muxer list).

The muxer-enabled engine was verified upstream: ffmpeg configure log shows all muxers
"Enabled" across 4 ABIs; the built `.so` exports the mux symbols; `vnext` published; media-kit
gradle MD5s match. This fix just wires that engine into the app.

## IMPORTANT — what the sandbox could NOT verify

- Dart `flutter analyze` = No issues found; `flutter test` = 330 passed; `flutter pub get`
  **successfully resolved** the new `rkinnc75/media-kit` git override (the Dart package + ref
  are valid).
- **The native jar swap is NOT sandbox-verifiable** — there's no Android build/NDK here, so
  analyze/tests do not exercise the new `libmpv.so`. This is the first app build to consume
  the muxer engine. A bad jar swap would break **all playback**, not just recording.

## VERIFY-FIRST (device smoke test — REQUIRED before trusting the release)

Because this swaps the native player engine, the CI release build + a real device pass are
the gate, not the sandbox:

1. CI release build succeeds (Gradle downloads the `vnext` jars, MD5s verify — a mismatch
   fails the build loudly).
2. On **SM-S938U** and **onn 4K Plus**: normal live playback works, multi-view works, the
   fps-cap path is unaffected. Playback must be **unregressed** — this is the priority check
   (the engine changed for everyone, not just recorders).
3. Recording still records + plays back (status → done) as in v3.0.24.
4. (Re-mux itself is wired by a LATER fix — the JNI shim. v4.0.0 only bundles the engine;
   it does not yet call the new muxers. So MP4 output is NOT expected to work yet; that's the
   next fix. This build's job is purely to ship the engine and confirm playback is unregressed.)
