# fix566 — make the fix565 30 fps cap actually engage + stop it forcing a reconnect

Base: **v2.0.65** (CI latest, re-verified — local synced to origin/main 6b63736).
Ships as **v2.0.66+566**.

## What the v2.0.65 field log proved (onn 4K Plus, 2026-06-27)

fix565 shipped the right *idea* (cap output to 30 fps so the Mali-G310 stops
missing vsync at the texture-upload stage) but the wrong *mechanism*. The
device log (`free4me_log-onn4kplusstreami-20260627-002212`) shows:

- Device on **2.0.65+565**, `capFps30=true` (gate engaged, RAM detected) — so
  this is **not** a stale install and **not** a gate miss.
- `[mpv:error] cplayer: Option vf: fps doesn't exist.` →
  `Set property: vf="fps=30" -> -4` (MPV_ERROR_INVALID_PARAMETER). The cap
  **never applied**: `vfFps` stayed `59.940060` the entire session; `voDrop`
  climbed to ~558.
- Worse than inert: that error-level message reached `errorStream`, was classed
  *transient*, and fired `onDisconnect(reason: "player error: Option vf: fps
  doesn't exist.")` → a **spurious reconnect** when tuning a channel.
- The fix564 premise is re-confirmed: `decDrop=0` (decoder idle) while `voDrop`
  climbs — the bottleneck is the VO upload, so halving output fps is the right
  lever. Software decode on the onn is intentional (`androidFullscreenHwdec`
  low-RAM → `no`), not a reconnect side effect.

**Root cause:** mpv resolves a bare `vf` filter name against its builtin set;
`fps` is a *libavfilter* filter and does not resolve without the `lavfi`
bridge. libavfilter IS compiled into the shipped `libmpv.so` (verified:
`avfilter_get_by_name`, and the binary's own `--vf=lavfi=[...]` help text), so
the bridge form is available — the bare name was simply the wrong syntax.

## Fix (two independent pieces)

1. **`lib/player/mpv_engine.dart`** — switch the cap filter from `'fps=30'` to
   the documented bridge form **`'lavfi=[fps=30]'`** (clear with `''` unchanged).

2. **`lib/player.dart`** — new pure predicate `isVfOptionError(err)` matching
   `"Option vf…"` / `"could not create filter"`. The `errorStream` handler now
   suppresses it (log-once-per-open via `_vfErrorLogged`, mirroring the
   seek-probe latch) and **returns before any reconnect**. `vf` is set ONLY by
   this cap, so any `vf` error is benign-to-playback by construction — worst
   case the stream plays uncapped. This is the safety net: even if a future
   build lacks the `fps` filter entirely, it can never reconnect again.

Changelog (`whats_new_modal.dart`) + version bump (`pubspec.yaml`) included.
New unit test `test/fix566_vf_option_error_test.dart` pins the predicate.

## How to verify on-device (adb, onn 4K Plus)

After CI publishes v2.0.66 and the APK is installed:
1. Tune a 60 fps channel (e.g. "SP - YES NETWORK HD").
2. Pull the log; confirm:
   - `Set property: vf="lavfi=[fps=30]"` returns success (not `-4`), and
     **no** `Option vf … doesn't exist` line.
   - `vfFps` drops to **~30** and `voDrop` stops climbing (steady).
   - **No** `onDisconnect (reason: "player error: …vf…")` on open.
3. If the cap still fails (e.g. `could not create filter`): piece 2 guarantees
   no reconnect (strictly better than v2.0.65); the cap is then infeasible via
   mpv filters on this build and should be left inert / disabled.
