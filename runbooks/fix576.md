## Problem

In the full-screen player the remote D-pad did nothing. `_onChannelKey`
(player.dart) was a bubble-up observer (`canRequestFocus:false`) that handled
ONLY the dedicated `channelUp`/`channelDown` keys; D-pad arrows + OK/center
returned `ignored`, and media_kit's controls (which held focus) did not map them.
This also blocked on-device testing of anything reachable only by tuning a
channel from the remote (e.g. the fix575 stream-info label).

## Change (direct-map core)

The player `Focus` now **holds focus** (`autofocus:true`) so it gets the keys
first, and `_onPlayerKey` maps them directly:

- The mapping is a pure function `playerKeyAction()` in
  `lib/player/player_key_action.dart` (testable; pinned by
  `test/fix576_player_key_action_test.dart`).
- **Revealing the bars:** media_kit 2.0.1 has no public show-controls API

  force-show). The bars then auto-hide via the existing `controlsHoverDuration`.

## Scope / next

This is the **direct-map** scheme: arrows always drive transport, so the bar
buttons (Cast / PiP / Subtitles / Audio / Aspect) are NOT yet D-pad-reachable.
The agreed full model ("direct-map + focusable bars" â€” after OK reveals the
bars, arrows navigate the buttons and OK activates them, reverting on auto-hide)
is **fix577 / v2.1.4**, layered on this core. It needs on-device iteration of
media_kit focus traversal, hence shipped separately.
