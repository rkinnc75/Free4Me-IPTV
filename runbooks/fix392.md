# fix392 — Fast first paint of the Live/Movies/Series lists (multi-source): stop the unconditional series index from shadowing the browse index

**Target version:** `1.34.20+392` (from `1.34.19+390`)
**Branch:** `main`
**Toolchain (verified):** Flutter 3.44.0 / Dart 3.12.0 (CI pin), `libsqlite3.so` symlinked.
**Written against:** `v1.34.19+390` on main (commit `dc45c5b`). **Migration ceiling 31 → 32.**

## What & why

On an S24 with 3 sources, the first display of the **All ▸ Movies** list took **~4.3 s** (no-text browse SQL: 2000 ms warm-up + 4297 ms actual to show 36 rows; subsequent loads ~657 ms; Categories ~10 ms).

**Root cause (measured via `EXPLAIN QUERY PLAN` on a seeded ~675k-row catalogue, not theory):** the no-text browse filters `series_id IS NULL` (via `VisibilityClause` — episodes are stored as `media_type=movie` *with* a `series_id`, so this separates standalone titles from episodes; it is **not** redundant). The base-schema index `index_channel_series_id` was **unconditional**, so it *matched* `series_id IS NULL`; the planner chose it and then `USE TEMP B-TREE FOR ORDER BY` — a full sort of the whole media-type set. That silently defeated **both** browse-tier indexes that were supposed to serve this sort: migration 27's `idx_channels_browse_tier` (source-led) and migration 30's `idx_channels_browse_mt` (media_type-led). Both were effectively unused at runtime.

**The fix is one line.** Make `index_channel_series_id` **partial on `series_id IS NOT NULL`**. It still serves the series drilldown (`series_id = ?`) but no longer matches the browse's `series_id IS NULL`, so the planner uses the existing migration-30 `idx_channels_browse_mt` — which leads with `media_type` (a single constant in the query) then `(tier, name)`, serving the **global** tier+name order across all sources straight from the index, no sort, for any source count.

Measured (3 sources, ~675k rows incl. 300k episodes, warm — relative):

| | plan | time |
|---|---|---|
| before (unconditional series index) | `index_channel_series_id` + TEMP B-TREE | ~68 ms (≈4.3 s cold on device) |
| after (partial series index) | `idx_channels_browse_mt`, no sort | ~0.04 ms (1/3/5 sources alike) |

This is an **index-only** change: query *results and order are unchanged* — only the plan changes. It supersedes the earlier `UNION ALL` proposal, which is unnecessary once the shadowing index is removed (the existing mig-30 index already does the job).

Not addressed (unchanged, not a regression): a true *multi-media-type* browse (`media_type IN (0,1,2)`) still temp-sorts — the media_type-led index can't pin a single leading value. The Live/Movies/Series tabs each query a single media_type (the logged case), so this covers them.

## Why it shipped undetected (and the test fix)

`test/browse_order_test.dart`'s EXPLAIN test had **drifted from the emitted SQL** — the exact fix330/fix344 failure its own header warns against. It EXPLAINed a query with **no `series_id IS NULL`** and the pre-fix365 correlated `groups.enabled` subquery, so it stayed green (using `idx_channels_browse_tier`) while the real `VisibilityClause` query temp-sorted. This runbook rebuilds that test to EXPLAIN the **real** query, assembled from the live `VisibilityClause` + `BrowseOrder` builders against the real index set, and asserts: `idx_channels_browse_mt` is used, **no `TEMP B-TREE`**, and the partial series index does not shadow it.
