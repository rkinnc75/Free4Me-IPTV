# fix611 — Source refresh: continue-on-error + retry-twice; FTS rebuild no longer blocks search

**Target:** v2.2.22+611 (diff against live tag **v2.2.21**, verified `git ls-remote --tags`)
**Apply check:** `git apply --check` PASSED on a fresh v2.2.21 clone.
**Analyze:** run on the build machine — no Flutter toolchain in the authoring sandbox, so the 0-issue gate is NOT asserted here.

## What this fixes (from report `2026-06-29T16-10-50-907Z-onn4kplusstreami`)

1. **One source failure aborted the whole refresh.** Trex's fetch failed
   ("all content types failed (3/3)") and the exception propagated out of
   `refreshAllSources`, so Media4u and Dino never refreshed. Now: each source
   is retried **once** (2 attempts total — separate from xtream.dart's existing
   per-content-type retry), and on final failure it is recorded and the loop
   **continues**. The dialog reports a partial-success summary
   ("Partly loaded — N of M ready. Could not refresh: Trex.").
2. **The ~36-minute FTS rebuild blocked everything.** The end-of-batch global
   `INSERT INTO channels_fts(channels_fts) VALUES('rebuild')` over the ~1M+ row
   catalog is inherent to a multi-source batch (fix514's targeted re-index is
   single-source-only by construction and does not apply). It is **not** moved
   off-thread (SQLite serializes the write regardless); instead the search UI
   now disables FTS-backed search during the rebuild — search field shows
   **"Search updating…"** and re-enables the instant the rebuild finishes **or
   errors** (cleared in a `finally`). Only `ftsPhrase` / `ftsAnd` are gated;
   `likeSubstring` and `inMemory` do not touch `channels_fts` and stay usable.

## Files
- `lib/memory.dart` — new `ValueNotifier<bool> ftsRebuilding`.
- `lib/backend/sql.dart` — set/clear `ftsRebuilding` around the global rebuild branch (try/finally).
- `lib/backend/utils.dart` — retry-twice + continue-on-error loop + `onSourceFailed` callback + failure summary log.
- `lib/widgets/sources_refresh_dialog.dart` — collect failures, partial-success completion message.
- `lib/home.dart` — gate search field enabled/hint on `ftsRebuilding` when the method is FTS-backed.
- `pubspec.yaml`, `lib/whats_new_modal.dart` — version bump + changelog.

## Notes / risk
- Retry on a **pre-existing** source is safe: `processSource` only deletes a leaked source row when `!namePreExisted` (a brand-new add), never on a refresh.
- The FTS rebuild itself is unchanged in cost — this fix removes the *user-facing block*, not the rebuild time. A future fix could investigate chunked/incremental FTS for the batch path.

## Validation plan (build machine)
1. `flutter analyze` → expect "No issues found!" (0 issues).
2. Multi-source refresh with one deliberately-bad source (e.g. wrong Trex creds): confirm it retries once, then continues; other sources load; dialog shows "Partly loaded … Could not refresh: <name>."
3. During the end-of-batch rebuild on the onn 4K (FTS method active): confirm search field reads "Search updating…" and is disabled, then re-enables when the rebuild completes.
4. Repeat (3) but kill the refresh mid-rebuild (force an error): confirm search re-enables (flag cleared in finally).
5. Switch search method to inMemory: confirm search field is NEVER disabled by a refresh rebuild.
