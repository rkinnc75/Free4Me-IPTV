# fix680 — DIAGNOSTIC: full native trace of the Scheduled Recording capture lifecycle

**Build:** `3.0.22+680` · **Base tag:** `v3.0.21` (fix679) · **Ship order:** after 679
**Files touched:** `android/app/src/main/kotlin/me/free4me/iptv/RecordingCaptureService.kt`, `android/app/src/main/kotlin/me/free4me/iptv/SrStartReceiver.kt`, `lib/whats_new_modal.dart`, `pubspec.yaml`
**Behavior change:** none — instrumentation only.

---

## Why

On 3.0.21+679 two "Record now" recordings are **stuck on `recording`** past their 2-minute duration (screenshot shows the red-dot "Recording" chip + stop button, not the orange "Processing"/`compressing` chip). The Dart trace shows `RecordingCapture.start returned OK` for both, then nothing — because everything after that is in the **native** `RecordingCaptureService`, whose logs only went to logcat (`Log.i`/`Log.w`), invisible to the Samsung in-app report. So we can't see *where* `runCapture` stalls: copy loop, `finalizeMediaStore`, remux, or the `updateStatus(done)` write-back.

(Prime suspect: `updateStatus` opens the DB with Android's framework `SQLiteDatabase` while the app writes via `sqlite_async`; a `SQLITE_BUSY`/lock on the `done` write would leave the row on `recording`, and the exception is currently only logged to logcat. But that's a hypothesis — this runbook makes the next capture conclusive rather than guessing.)

## What it does

Adds a gated, plugin-free native `srDebug` that **appends `[SRDBG]` lines to the same `app_log.txt` the reporter transmits** (readable with no adb), mirroring the Dart-side breadcrumbs. It's **gated by the `debugLogging` setting** (read once per process from the `Settings` table, same mechanism as the existing remux-flag read, cached) — off ⇒ no-op. `Log.i` to logcat stays for adb.

Per the "trace every step, don't piecemeal" decision, breadcrumbs cover the **whole lifecycle**, not just the current failure point:
- `SrStartReceiver`: entry (first native hop).
- `onStartCommand ACTION_START`.
- `runCapture`: start, locks acquired, `status=recording`, MediaStore entry (+uri), HTTP connect + response code, copy-loop start, a byte **heartbeat every ~5s**, loop exit (+reason: deadline/stream-end/cancel, +total bytes), `finalizeMediaStore`, the too-little-data / remux / plain-done branch decisions, `runCapture COMPLETE`, any thrown exception, and cleanup.
- `remuxToMp4`: start, track count, tracks mapped + `muxer.start`, `muxer.stop`, success/throw.
- `updateStatus`: **rowsAffected** for every write (this is what will show a failing/zero-row `done` write) and any throw.

## How to read the result (Samsung, no adb)

Ensure **Debug logging is ON**, run a 2-min "Record now", wait ~30s past the end, export the log. The `[SRDBG]` lines show exactly how far capture gets. Likely tells:
- Loop `EXIT (deadline) total=…` then `updateStatus id=… status=done rowsAffected=0` (or a `THREW`) ⇒ the write-back is the bug (framework-SQLite vs sqlite_async lock) → real fix is to route the native status write differently.
- Heartbeats stop long before deadline and no `EXIT` ⇒ the copy `read()` is hanging → connection/read-timeout fix.
- `remux: …THREW` / stops after `muxer.start` ⇒ remux is the culprit.

Send the log and it pins the next real fix.
