# fix570 — disable the fps cap (no usable libavfilter filter in this libmpv)

Base: **v2.0.66** (fix567/568/569 were never-released dry-run iterations).
Ships as **v2.0.70+570**. **This is the build that un-breaks the live v2.0.66
regression.**

## Conclusion of the fix565–569 arc (all confirmed on-device, onn 4K Plus)

The 60→30 fps OUTPUT cap (sound idea — fix564 proved the Mali-G310 judders at
the texture-upload stage, decoder idle) required an mpv `vf` frame-rate filter.
**media_kit's bundled libmpv ships a near-empty libavfilter filter set** — every
candidate is absent, each confirmed at runtime:

| filter | device result |
|---|---|
| `fps` | "Option vf: fps doesn't exist" → "No such filter: 'fps'" |
| `select` | "No such filter: 'select'" |
| `framestep` | absent |

A missing lavfi filter makes mpv **deselect the video track** → permanent
black-screen stall (surfaces as "video texture failed to attach"), so a missing
filter is catastrophic. v2.0.66 (`lavfi=[fps=30]`) shipped this stall to
low-RAM users. (Lesson: the binary string-probe gave false positives — "select"
appears in the .so for unrelated reasons; only the runtime "No such filter" is
authoritative. And local mpv 0.41 / ffmpeg 8.1 are newer/lenient than the
bundled libmpv, so they couldn't reproduce the device's parser strictness.)

## Fix

Disable the cap: **always `setProperty('vf', '')`** (guaranteed safe — no filter
is ever created, nothing can fail; this is the playback path that worked in the
v2.0.65 field log). `devCapFpsLowRam` toggle retained but inert. fix566's
`isVfOptionError` reconnect-suppression kept as defensive cover. Playback is
restored; the judder is a pre-existing condition, not a regression.

## Follow-up (separate, non-blocking)

The smoothness cap is only deliverable if (a) a libmpv build with a frame-rate
filter is adopted, or (b) a non-filter mechanism is verified on-device
(`--override-display-fps` + display-resample is a candidate, unproven). Until
then the toggle should arguably be hidden from Settings → Developer.
