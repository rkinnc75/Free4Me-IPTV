# fix683 — DIAGNOSTIC: log the remux/debugLogging values the alarm callback reads

**Build:** `3.0.25+683` · **Base tag:** `v3.0.24` (fix682) · **Ship order:** after 682
**Files touched:** `lib/backend/recording_scheduler.dart`, `lib/whats_new_modal.dart`, `pubspec.yaml`
**Behavior change:** none — one gated log line.

---

## Why

On v3.0.24 with the **MP4 toggle ON**, all three recordings (id 12/13/14) logged `onStartCommand … remux=false` natively, so re-mux never ran and outputs stayed `.ts`. The full flag chain reads correctly in static analysis: the UI toggle sets `settings.remuxRecordings` and calls `updateSettings()` → `SettingsService.updateSettings` writes `remuxRecordings` to the `Settings` table (line ~596); `SettingsService.getSettings()` reads it back (line ~388); the fix681 callback passes it as a broadcast extra. Every link looks correct, so the break can't be pinned from code alone.

The `[SRDBG]` trace shows what native *received* (`remux=false`) but not what Dart *read*. This adds one gated breadcrumb logging the values `getSettings()` returned in the alarm isolate, right before `RecordingCapture.start`. That single line disambiguates the two candidates:

- **`settings read remux=false`** → the setting isn't persisted/read as expected (settings storage/read bug, possibly WAL-timing in the background isolate).
- **`settings read remux=true`** while native still logs `remux=false` → the bool is lost across the `android_intent_plus` broadcast extras (codec/type bug — the same class we hardened the *int* extras against in fix678; booleans were never verified).

## How to read

Debug logging ON, MP4 toggle ON, Record now 2 min, export log. Compare the new `SR-CB: id=X settings read remux=… debugLogging=…` line against the native `onStartCommand … remux=…`. Send the log; the real fix follows the split above.
