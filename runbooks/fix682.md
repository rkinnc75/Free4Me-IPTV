# fix682 — compile fix: remove dead startCapture MethodChannel case (broke :app:compileReleaseKotlin on fix681)

**Build:** `3.0.24+682` · **Base tag:** `v3.0.23` (fix681) · **Ship order:** after 681
**Files touched:** `android/app/src/main/kotlin/me/free4me/iptv/MainActivity.kt`, `lib/whats_new_modal.dart`, `pubspec.yaml`

> **Why not re-cut fix681?** fix681's `v3.0.23` tag was pushed before the APK step ran, so the tag exists but produced **no release artifact** (build failed). Fix numbers and tags are forward-only and never reused — so this is **fix682 / v3.0.24**, carrying fix681's fix forward.

---

## What failed

fix681 (v3.0.23) passed `flutter analyze` and all 330 tests, but the APK build failed at `:app:compileReleaseKotlin`:

```
e: MainActivity.kt:231:70 No value passed for parameter 'remux'.
e: MainActivity.kt:231:70 No value passed for parameter 'debugLogging'.
```

fix681 added required `remux` + `debugLogging` parameters to `RecordingCaptureService.start(...)`. There were **two** native callers; fix681 updated `SrStartReceiver` but missed the second one — `MainActivity`'s `startCapture` MethodChannel handler, which still called the old 5-arg signature. (Analyze/tests are Dart-only and don't compile Kotlin, and the sandbox Kotlin check was single-file syntax only — it couldn't resolve a cross-file signature mismatch. Lesson logged: after changing a native signature, grep every caller across the Kotlin tree.)

## The fix

That `startCapture` MethodChannel case has been **dead code since fix678**, which switched Dart to start capture via the `SR_START` broadcast (so it works from the alarm background isolate, where this Activity-scoped channel doesn't exist). So the case is removed outright rather than patched. `stopCapture` and `getFreeBytes` on the same channel remain — they're still invoked from the main isolate.

Also bumps the version and carries fix681's changelog note forward to 3.0.24 (v3.0.23 shipped no artifact, so users first see it here).

## Verify (build machine)

The previously-failing `:app:compileReleaseKotlin` should now succeed; the rest of the SR fix (single-writer status journal from fix681) is unchanged and ships with this build. On device, verify per fix681: a finished recording shows **Done** on opening Recordings; MP4-on produces an `.mp4` (first real re-mux exercise).

## Build / verification note

Sandbox has no Android toolchain: Kotlin was syntax-checked with kotlinc (clean) and all callers of `RecordingCaptureService.start` were grepped — only the correct 7-arg `SrStartReceiver` call remains. Dart `flutter analyze` = No issues found; `flutter test` = 330 passed. Diff verified to apply cleanly to a pristine `v3.0.23` checkout.
