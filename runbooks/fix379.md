# fix379 — Low-RAM phones get the same multi-view mitigation as low-RAM TVs

**Target version:** `1.34.8+379` (from `1.34.7+378`)
**Branch:** `main`
**Toolchain (verified in sandbox):** Flutter 3.44.0 / Dart 3.12.0 (CI pin), `libsqlite3.so` symlinked.
**Written against:** `v1.34.7+378` on main.

## ⚠ Device-gated — on-device test REQUIRED before tagging

This fix changes a real runtime behaviour (multi-view `hwdec` mode) on a
class of device the sandbox cannot reach. The OPPO PGEM10 (~1999 MB
phone) is the primary target. SW decode may help OR hurt on phone GPUs
that weren't represented in the fix361 data; if it hurts, the rollback
is a one-line revert (re-add the `isTV()` check in
`DeviceDetector.isLowRamDevice`). See "On-device integration check"
below for the exact steps before tagging.

## What & why

`DeviceDetector.isLowRamTv()` (fix361) was introduced to route multi-view
preview tiles to software decode (`hwdec=no`) on low-RAM Android TV
boxes (e.g. onn 4K Plus, ~1.9 GB Amlogic). The reason is shared-GPU
texture memory: 4 concurrent `mediacodec-copy` pipelines exhaust it on
those SoCs, producing `TEXTURE-ATTACH-FAILED`.

The function gated on `isTV()`, which excluded phones. Low-RAM **phones**
(e.g. OPPO PGEM10, ~2 GB) hit the same shared-GPU exhaustion when
running several multi-view previews at once — but the gate returned
`false` so they got `mediacodec-copy` and crashed.

The 2300 MB threshold is empirical and was already used by
`ChannelSearchCache` to skip the in-memory search cache on low-RAM
devices. It is a property of the *hardware* (shared GPU memory), not of
the form factor — so the `isTV()` gate was a proxy that happened to
correlate with low-RAM boxes but excluded low-RAM phones. Removing the
gate (and renaming for accuracy) extends the same mitigation to
phones.

## Files changed
- `lib/models/device_detector.dart` — `isLowRamTv()` renamed to
  `isLowRamDevice()`. The `isTV()` gate removed. Added a new pure
  static helper `isLowRamDeviceThreshold(int totalMb)` so the 2300 MB
  cutoff is testable without platform mocks. Doc comment updated to
  cover both TVs and phones.
- `lib/player/mpv_engine.dart` — one-line call site update
  (`isLowRamTv` → `isLowRamDevice`).
- `lib/whats_new_modal.dart` — `'1.34.8'` changelog entry (real UTF-8).
- `pubspec.yaml` — `1.34.7+378` → `1.34.8+379`.
- `test/device_lowram_threshold_test.dart` — **new**: 5 tests pinning
  the threshold (0, 1999, 2300, 2301, 4096) + a regression guard
  naming the OPPO PGEM10 as the canonical low-RAM phone.

## On-device integration check (REQUIRED — sandbox cannot verify)

**Primary target: OPPO PGEM10** (the device that motivated the fix).
Secondary sanity: any low-RAM phone and the onn 4K Plus (regression
check that the existing TV behaviour still works).

### A. Low-RAM phone (OPPO PGEM10) — the new behaviour

1. Install `v1.34.8+379` on the OPPO.
2. Open the app, navigate to a multi-view grid (e.g. 2×2 preview).
3. Open a few live streams in the preview tiles (4 concurrent).
4. Watch the log for the new line:
   `Player: hwdec=no (preview, multiViewDecode=auto) channel="…"`
   (vs. the pre-fix379 line `hwdec=mediacodec-copy`).
5. Expected:
   - **No `TEXTURE-ATTACH-FAILED`** errors when 4+ previews are open
     for >30s.
   - Playback in the preview tiles is fluid (no stutter > 2s).
6. Optional: capture the log line and a screenshot of the multi-view
   grid for the runbook annex.

### A-rollback. If the OPPO regresses (SW decode is *slower* than
mediacodec-copy on the OPPO's GPU), there are **three escape paths** in
priority order — no code revert required for the first two:

1. **Per-device (no app change):** Settings → Multi-view decode → set to
   `Hardware (copy)`. This overrides the auto detection and forces
   `mediacodec-copy` (the pre-fix379 behaviour) for the OPPO. Persists
   across sessions (`MultiViewDecode` is in `Settings.multiViewDecode`,
   see `lib/models/settings.dart:155`).
2. **Per-device, global (no app change):** Settings → "Hardware
   decoding" toggle → turn OFF. Forces `hwdec=no` for *every* player
   (preview + fullscreen), not just previews. Sledgehammer but works
   (`s.hwDecode` is checked at `lib/player/mpv_engine.dart:525`).
3. **Code revert (last resort):** re-add `if (!await isTV()) return false;`
   to `isLowRamDevice()` in `lib/models/device_detector.dart`. This
   restores the pre-fix379 behaviour. Document the regression in
   `whats_new_modal.dart` and don't ship until the threshold or
   per-device carve-out is fixed.

If the OPPO regression is reproducible, also consider: per-device
carve-out (e.g. skip the mitigation for `manufacturer == "OPPO"` and
rely on the user's `multiViewDecode` setting), or bumping the
threshold upward. Both are out of scope for fix379.

### B. Low-RAM TV (onn 4K Plus) — regression check

1. Install `v1.34.8+379` on the onn 4K Plus.
2. Same multi-view flow as above.
3. Expected: same behaviour as pre-fix379 (SW decode on auto,
   `Player: hwdec=no (preview, multiViewDecode=auto)`).
4. ✓ if the existing TV behaviour is unchanged.

### C. High-RAM phone (S24/S25) — regression check

1. Install on the S24.
2. Open multi-view, watch for the log line.
3. Expected: `Player: hwdec=mediacodec-copy` (NOT `hwdec=no`).
4. ✓ if high-RAM phones still get hardware decode. (Threshold is
   `totalMb < 2300`; S24 has 12 GB → far above.)

### D. Pre-fix379 fallout check

If A, B, or C fails:
- **A fails (OPPO regression):** roll back per step A.5; document
  and consider fix380-style device-list gating (e.g. only specific
  manufacturer/model strings, not a generic threshold).
- **B fails (onn regression):** investigate the rename first; the
  `isTegra() || isLowRamDevice()` logic should be equivalent. If
  not, that's a real bug.
- **C fails (S24 regression):** the threshold is wrong; bump it
  upward (e.g. 3000) or add a per-manufacturer carve-out.

+import 'package:open_tv/models/device_detector.dart';

+void main() {
+  group('DeviceDetector.isLowRamDeviceThreshold (fix379)', () {
+    test('below 1 MB is uninitialised → false', () {
+      // 0 means DeviceMemory.init() never ran or /proc/meminfo was
+      // unreadable. Must NOT trigger software decode on a device that
+      // we couldn't read.
+      expect(DeviceDetector.isLowRamDeviceThreshold(0), isFalse,
+          reason: 'totalMb=0 means uninitialised; not a low-RAM signal');
+    });
+
+    test('just under threshold is true', () {
+      // OPPO PGEM10 reports ~1999 MB; onn 4K Plus ~1925 MB.
+      // 1999 < 2300 → true.
+      expect(DeviceDetector.isLowRamDeviceThreshold(1999), isTrue,
+          reason: 'OPPO-class device totalMb must trigger the mitigation');
+      expect(DeviceDetector.isLowRamDeviceThreshold(1925), isTrue,
+          reason: 'onn 4K Plus class totalMb must trigger the mitigation');
+    });
+
+    test('at the threshold is false (exclusive upper bound)', () {
+      // The cutoff is strictly < 2300, so 2300 itself is "not low-RAM".
+      // 2300 MB devices have been observed to handle 4 mediacodec-copy
+      // pipelines without TEXTURE-ATTACH-FAILED; only below 2300 triggers
+      // the bug.
+      expect(DeviceDetector.isLowRamDeviceThreshold(2300), isFalse,
+          reason: '2300 is the documented cutoff; inclusive above');
+    });
+
+    test('just above threshold is false', () {
+      expect(DeviceDetector.isLowRamDeviceThreshold(2301), isFalse);
