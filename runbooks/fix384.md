# fix384 — Edit and Delete stay enabled when a source is disabled

**Target version:** `1.34.13+384` (from `1.34.12+383`)
**Branch:** `main`
**Toolchain (verified in sandbox):** Flutter 3.44.0 / Dart 3.12.0 (CI pin), `libsqlite3.so` symlinked.
**Written against:** `v1.34.12+383` on main.

## What & why

In Settings → Sources, when a source was disabled, every action
button on its row went dim: Edit, Delete, Refresh, and the
color-picker (fix307 policy). Only the enable/disable Switch stayed
usable.

This is too restrictive. A user disabling a source because its URL
is broken can't fix the URL without re-enabling first — defeating
the "disable to silence a broken source" workflow. And a source
disabled for removal can't be deleted without the same
re-enable-then-delete dance.

fix384 removes the gate from **Edit** and **Delete** only. The
color-picker and Refresh stay gated: the user doesn't see / interact
with a disabled source, so changing its color or re-fetching its
channels is wasted work.

## Files changed
- `lib/settings_view.dart` — Edit and Delete `IconButton`s no
  longer gate `onPressed` on `!source.enabled`. Color-picker and
  Refresh unchanged.
- `lib/whats_new_modal.dart` — `'1.34.13'` changelog entry (real UTF-8).
- `pubspec.yaml` — `1.34.12+383` → `1.34.13+384`.
- `test/source_actions_enabled_test.dart` — **new**: 3 regression
  guards. Grep-based (the gate lives inside a private
  `_SettingsState.getSource` build method, not directly
  constructable from a widget test without mounting the full
  SettingsView).

+// always be enabled, regardless of the source's enabled/disabled
+// state. (Color picker and Refresh stay gated on source.enabled —
+// the user only sees / interacts with those when the source is
+// actually being used.)

+// `!source.enabled` guard in front of the Edit or Delete `onPressed`
+// callbacks. The check is grep-based because the gated logic lives
+// inside the build method of a private State class
+// (`_SettingsState.getSource` in lib/settings_view.dart), which is
+// not directly constructable from a widget test without mounting the
+// full SettingsView (heavy — pulls in platform channels, the
+// SettingsService cache, etc.). The grep is the cheapest reliable
+// signal that the gate is gone.

+void main() {
