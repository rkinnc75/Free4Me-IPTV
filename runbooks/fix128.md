# fix128.md — Show What's New dialog immediately; build search cache in parallel

> **Version:** Free4Me-IPTV 1.22.9 build=124 (verified against actual source
> in the uploaded zip).
> **Evidence:** User observation + source review of `lib/home.dart`
> `initializeAsync()`.
>
> **Change:** When a What's New dialog is due (version upgrade), show it
> BEFORE the `ChannelSearchCache` warmup completes, so the user can read
> the release notes while the cache builds in the background. Currently
> the cache build (`await ChannelSearchCache.ensureBuilt()`) fully
> completes before `showWhatsNew()` is called — the user sees a disabled
> search box and nothing to read while they wait.

---

## Current sequence

```
initializeAsync():
  1. log rotate / source & filter setup
  2. versionFuture = shouldShowWhatsNew()        ← kick off check (async)
  3. await load()                                ← channel list renders
  4. await ChannelSearchCache.ensureBuilt()      ← search DISABLED, user waits
  5. _searchReady = true
  6. version = await versionFuture
  7. await showWhatsNew(version)                 ← dialog shown AFTER cache done
  8. showSourcesRefreshDialog (if refresh=true)
```

Steps 4 and 7 are in the wrong order. The cache warmup can take several
seconds on a large source. The user sits watching a disabled search box with
nothing to read. Meanwhile the What's New dialog — which would give them
something to do — is queued behind it.

---

## Fix — start cache build, show dialog, then wait for cache to finish

The cache build (`ChannelSearchCache.ensureBuilt()`) and the What's New
dialog are independent. Run them concurrently: fire the cache build without
`await`, show the What's New dialog immediately (the user reads it), then
`await` the cache build future before re-enabling search. The user gets
the dialog right away; by the time they tap "Don't show again" the cache is
likely already done.

# Fix 128.1 — Reorder initializeAsync to show What's New before awaiting cache

**File:** `lib/home.dart`

**Current (lines 127–149):**

```dart
    final versionFuture = widget.firstLaunch
        ? SettingsService.shouldShowWhatsNew()
        : Future<String?>.value(null);

    await load();

    // main.dart hasn't finished yet, disable the search box until it's ready.
    final cachedSettings = SettingsService.cached;
    if (cachedSettings != null &&
        cachedSettings.searchMethod == SearchMethod.inMemory &&
        ChannelSearchCache.needsRebuild()) {
      if (mounted) setState(() => _searchReady = false);
      await ChannelSearchCache.ensureBuilt();
      AppLog.info('Home: ChannelSearchCache warmup completed');
      if (mounted) setState(() => _searchReady = true);
    }

    final version = await versionFuture;
    if (!mounted) return;
    if (version != null) {
      await showWhatsNew(version);
    }
```

**Replace with:**

```dart
    final versionFuture = widget.firstLaunch
        ? SettingsService.shouldShowWhatsNew()
        : Future<String?>.value(null);

    await load();

    // fix128: start the cache build WITHOUT awaiting it so the What's New
    // dialog can show immediately. The user reads release notes while the
    // cache builds in the background. We await the cache future AFTER the
    // dialog closes (or if there's no dialog, we await it inline as before).
    Future<void>? cacheFuture;
    final cachedSettings = SettingsService.cached;
    if (cachedSettings != null &&
        cachedSettings.searchMethod == SearchMethod.inMemory &&
        ChannelSearchCache.needsRebuild()) {
      if (mounted) setState(() => _searchReady = false);
      cacheFuture = ChannelSearchCache.ensureBuilt();
    }

    // Show What's New while the cache builds (if needed).
    final version = await versionFuture;
    if (!mounted) return;
    if (version != null) {
      await showWhatsNew(version);
    }

    // Now wait for the cache to finish (no-op if it wasn't needed or is
    // already done by the time the dialog closed).
    if (cacheFuture != null) {
      if (mounted && !_searchReady) {
        await cacheFuture;
        AppLog.info('Home: ChannelSearchCache warmup completed');
        if (mounted) setState(() => _searchReady = true);
      }
    }
```

### Key changes

1. **Cache build fires immediately** (`cacheFuture = ChannelSearchCache
   .ensureBuilt()`) — no `await`, so execution continues to the dialog.
2. **`_searchReady = false`** is still set before the build starts (same as
   before) so the search box is correctly disabled while the cache builds.
3. **What's New dialog shows right away** — the user gets something to read
   while the cache builds behind it.
4. **Cache is awaited AFTER the dialog** — if the cache finishes before the
   user dismisses the dialog, the `await` is instantaneous. If the user is a
   fast reader, the search box stays disabled the remaining few seconds.
5. **No What's New case is unchanged** — `cacheFuture` is awaited inline as
   before (just after the version check, skipping the dialog branch), so
   non-upgrade launches are unaffected.
6. **`!_searchReady` guard** — if somehow `_searchReady` got reset to true
   (shouldn't happen, but defensive), we skip the redundant await.

## Test plan

### What's New shows before search is ready

1. On a fresh version upgrade (or manually clear the `lastSeenVersion` stored
   value so the dialog triggers), launch the app with In-Memory search enabled.
2. **Expected:** What's New dialog appears promptly after the channel list
   loads — NOT after the search bar becomes enabled. The search box is disabled
   (shows "Preparing search…") WHILE the dialog is visible.
3. Dismiss the dialog. The search box enables within moments.
4. Log shows: `Home: ChannelSearchCache warmup completed` appearing AFTER the
   dialog was shown (not before it).

### No What's New — cache still warms normally

1. Launch on the same version (no dialog due). In-Memory search enabled.
2. **Expected:** `_searchReady` disables search, cache builds, search
   re-enables. Identical to pre-fix behaviour. No regression.

### What's New with fast cache

1. On a small source (few channels), the cache may finish before the dialog
   is dismissed.
2. **Expected:** search enables while the dialog is still open — that is fine
   and desirable. The `!_searchReady` guard on the post-dialog await is a no-op.

### What's New without In-Memory search

1. Source refresh / What's New with FTS or LIKE search method selected.
2. **Expected:** `cacheFuture` is null, dialog shows immediately, no cache
   await at all. Unchanged from current behaviour.

## Notes for the implementer

- **One logical change:** fire-then-await the cache around the dialog. The
  `_searchReady` state machine is fully preserved — it goes false before the
  build starts and true after it finishes, exactly as before.
- **`ChannelSearchCache.ensureBuilt()` is safe to call without awaiting** — it
  returns a `Future<void>` and is already designed to be called once; the
  `needsRebuild()` guard prevents double-build.
- **Even build number (+128)** — phone session per convention.
- **One file:** `lib/home.dart` (+ `pubspec.yaml` version, + `fix128.md`).
