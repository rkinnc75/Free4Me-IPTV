# fix395 — Shield/Tegra full-screen hardware decode (black-screen fix)

**v1.34.23+395** · `lib/player/mpv_engine.dart`, `lib/player/hwdec_routing.dart` (new), `test/shield_hwdec_routing_test.dart` (new)

## Symptom
NVIDIA Shield (Tegra X1, isTV=true, ~2.9 GB): full-screen playback shows a
black screen with audio alive — no video frames. See
`runbooks/shield-blackscreen-findings.md` and the device log
(`free4me_log-shieldandroidtv-20260617`): 12 s with no `Player: position=`
lines = playhead not advancing = video pipeline starved.

## Root cause
`lib/player/mpv_engine.dart` full-screen Android branch (fix164):
```dart
final hwdecMode = isTV ? 'no' : 'mediacodec';   // every Android TV → software
```
fix164's *intent* (and comment) was only the low-RAM onn 4K Plus (~2 GB,
Amlogic/Mali-G310), where the mediacodec-copy GPU→CPU readback is
bandwidth-bound and desyncs. But the code keyed on `isTV`, so it forced
**software decode on every Android TV**, including the Shield. On Tegra,
software H.264 decode (4× Cortex-A57) — especially alongside a 90-min DVR
writing ~5.4 GB cache-on-disk — starves the video pipeline. Audio plays; no
frames render.

## Why the comparison app works (and what it told us)
The user supplied a working reference app (`IPTV-Mj-main`, native Kotlin). It
uses **ExoPlayer (AndroidX Media3)** — Google's player — with default settings:
hardware **MediaCodec** decode into a native **SurfaceView**, no tuning. That
is precisely **`mediacodec-copy`** in libmpv terms. So the reference proves the
Shield's hardware decoder is the right path; Free4Me was just forcing software.

(Free4Me removed ExoPlayer entirely — libmpv-only — so this routing is the only
lever. The Flutter-Texture vs native-SurfaceView difference remains an
architectural gap, but it is not what caused this black screen.)

## Fix
Extract the routing into a pure, unit-tested function
(`lib/player/hwdec_routing.dart`), keyed on device class, **Tegra first**:

| Device | isTegra | isLowRam | isTV | hwdec |
|---|---|---|---|---|
| **Shield / Tegra** | true | any | true | **`mediacodec-copy`** |
| onn 4K Plus (low-RAM TV) | false | true | true | `no` (software) |
| Fire TV / capable TV | false | false | true | `mediacodec-copy` |
| Phone | false | false | false | `mediacodec` (surface) |
| Low-RAM phone | false | true | false | `no` |

Tegra is matched **first** so a RAM misdetection can never force a capable
Shield onto software, and TVs never get surface-mode `mediacodec` (which binds
a SurfaceTexture and fails silently on Tegra — fix108).

Preview / multi-view path is **unchanged**: Tegra keeps software there (fix314 —
4 concurrent mediacodec-copy pipelines corrupt colour on Tegra). This fix is
the single full-screen decoder only.

## Deliberately NOT done (with rationale)
- **No runtime "no video frame → fall back to software" watchdog.** It would
  loop-reconnect-then-give-up on legitimate **audio-only IPTV channels**
  (`videoWidth` stays 0 forever) — a real regression. The hardware-decode fix
  already matches the proven-working ExoPlayer behaviour on Shield.
- **No orientation/surface change.** The `ROTATE init → landscape` +
  `textureId=null` in the log is diagnostic-only (fix136 `didChangeMetrics`);
  the 1346 ms is normal media_kit texture-init latency, not a teardown.
- **DVR I/O untouched** — it's the user's 90-min DVR feature; hardware decode
  frees the CPU that software decode was starving, so the disk write now has
  headroom.
