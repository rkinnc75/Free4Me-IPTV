# fix599 — Live-guide focus: force-focus the target in a post-frame

**Version:** v2.2.10+599
**Date:** 2026-06-28
**Type:** Focus bug fix (on-device-found, 3rd iteration)

## Problem (v2.2.9 / fix598)

On-device: OK on a category loaded the correct channels, but focus **escaped to
the top nav** instead of the first channel. autofocus-on-remount loses the race:
when the focused category tile unmounts on the rail swap, focus escapes UP to the
nav before the new first-channel autofocus claims it.

## Fix

Keep autofocus (best-effort) but ALSO force focus in a post-frame after the swap,
in BOTH directions:
- `_loadGuide(enterChannels)`: post-frame `_channelNodes[firstId].requestFocus()`.
- `_onChannelLeftKey`: post-frame `_railNodes[selectedGroup].requestFocus()`.

The post-frame fires reliably (the `setState` that swaps `_railMode` schedules
the frame — fix585 condition met) and runs AFTER the unmount/escape settles, so
it overrides the nav grab. Targets attached nodes (channels ListView item 0 is
visible; categories use a non-lazy ListView so all category nodes are attached).

## Verification (rigorous, multiple cycles)

- analyze clean; tests green (272).
- On-device: OK several different categories → focus lands on the FIRST channel
  each time (not the nav); LEFT each time → focus returns to the SELECTED
  category. (EPG grid stays empty until an EPG refresh — known, separate.)
