# fix242 — What's New showed "See the GitHub releases page for details" (auto-generate the changelog summary from commits)

> **Version:** build = highest fix number in release; patch = next available. Tag = next immutable.
> **Builds on:** the live 1.25.x source.
> **Bug:** the in-app "What's New in X" dialog showed only "• See the GitHub releases page for details." for releases 1.25.2 / 1.25.3. Cause: those releases were cut WITHOUT a `_changelog` entry in `lib/whats_new_modal.dart`, and the dialog (and version.json notes) fall back to a placeholder when no entry exists.
> **Insight (from how this has always worked):** the What's New message is meant to be an AUTOMATED SUMMARY of the release's changes. The release scripts EXTRACT the `_changelog` entry — they never generated it — so when the entry was missing, there was nothing to show. The fix makes the summary genuinely automatic: when no hand-written entry exists, generate one from the commit subjects since the previous tag, BEFORE building. Hand-written entries still win.

## How the changelog flows (confirmed in the source)

- `lib/whats_new_modal.dart` holds `const _changelog = <String, List<String>>{ ... }`. The dialog reads it via `_entriesForVersion`; if empty → placeholder.
- `scripts/build_and_release.sh` (step "Extract changelog") and `scripts/update_version_json.py` both READ `_changelog` to build `version.json` `releaseNotes` and the GitHub release body. Missing entry → "Version X" / placeholder.
- So one missing `_changelog` entry produces the bad dialog AND bad version.json notes AND bad GitHub notes.

## The fix (two parts)

1. **NEW `scripts/gen_changelog.py`** — generates a `_changelog` entry from `git log <prev-tag>..HEAD` commit subjects (cleaned: drops release-build/chore/merge/"move fixNNN.md" noise, strips `fixNNN:` prefixes, dedupes, caps at 8 bullets). Idempotent: if an entry for the version already exists, it does nothing (hand-written wins).
2. **`scripts/build_and_release.sh`** calls it (step 3a) BEFORE the version.json update, passing the version and the previous `v*` tag. So by the time version.json + GitHub notes are built, `_changelog` always has a real entry.
3. **App fallback message improved** — the last-resort text (now essentially unreachable) reads sensibly instead of pointing at GitHub.

## Pre-verified facts (checked against 1.25.x; tested with a real git repo)

- Generator tested: from commits `fix240: clamp…`, `fix238: app-wide slow-query logging`, `refactor: move fix240.md to runbooks/`, `v1.25.3: release build` it produced a clean 2-bullet entry (`App-wide slow-query logging`, `Clamp analyze suggestions to slider bounds`) — noise dropped, prefixes stripped, inserted at the top of the map. Generated file compiles ("No issues found"). Idempotent re-run leaves it alone.
- `build_and_release.sh` passes `bash -n`. App compiles → 2 tolerated INFOs. Rule 6 clean.
- The generator inserts after the `const _changelog = <String, List<String>>{` opener (confirmed present at line 8).

## Fix 242.1 — NEW file `scripts/gen_changelog.py`

Create verbatim:
```python
#!/usr/bin/env python3
"""fix242: auto-generate a _changelog entry from git commits when one is missing.

Called by build_and_release.sh BEFORE version.json extraction. If
lib/whats_new_modal.dart already has an entry for the release version, does
nothing (hand-written entries win). Otherwise it builds a bulleted summary from
the commit subjects since the previous tag and injects it at the top of the
_changelog map, so the in-app What's New dialog, version.json, and the GitHub
release notes all show a real summary instead of a placeholder.
"""
import re, subprocess, sys

def commit_subjects(prev_tag):
    rng = f"{prev_tag}..HEAD" if prev_tag else "HEAD"
    out = subprocess.run(["git", "log", rng, "--pretty=format:%s"],
                         capture_output=True, text=True).stdout
    return [l.strip() for l in out.splitlines() if l.strip()]

def clean(subjects):
    """Turn commit subjects into user-facing bullets. Drops noise; dedupes."""
    bullets, seen = [], set()
    for s in subjects:
        # drop release/chore/merge noise
        if re.match(r'^(v?\d+\.\d+\.\d+|merge|chore|refactor: move fix|.*: release build)', s, re.I):
            continue
        # strip a leading "fixNNN: " or "fix: " prefix
        s = re.sub(r'^fix\w*:\s*', '', s, flags=re.I).strip()
        if not s:
            continue
        s = s[0].upper() + s[1:]
        key = s.lower()
        if key in seen:
            continue
        seen.add(key)
        bullets.append(s)
    return bullets

def dart_escape(s):
    return s.replace('\\', r'\\').replace("'", r"\'")

def main():
    version = sys.argv[1]
    prev_tag = sys.argv[2] if len(sys.argv) > 2 else ""
    path = "lib/whats_new_modal.dart"
    text = open(path).read()
    if f"  '{version}': [" in text:
        print(f"changelog: entry for {version} already present — leaving as-is")
        return
    bullets = clean(commit_subjects(prev_tag))
    if not bullets:
        bullets = [f"Maintenance and stability improvements."]
    # cap to a sensible number for the dialog
    bullets = bullets[:8]
    entry_lines = [f"  '{version}': ["]
    for b in bullets:
        entry_lines.append(f"    '{dart_escape(b)}',")
    entry_lines.append("  ],")
    entry = "\n".join(entry_lines) + "\n"
    # insert right after the map opener
    m = re.search(r"(const _changelog = <String, List<String>>\{\n|_changelog = <String, List<String>>\{\n|_changelog = \{\n)", text)
    if not m:
        print("changelog: ERROR could not find _changelog map opener", file=sys.stderr)
        sys.exit(1)
    text = text[:m.end()] + entry + text[m.end():]
    open(path, "w").write(text)
    print(f"changelog: auto-generated {len(bullets)}-bullet entry for {version} from commits since {prev_tag or 'start'}")

if __name__ == "__main__":
    main()
```

## Important: the version.json freshness check

`update_version_json.py` runs in CI to verify version.json matches what it would generate from `_changelog`. Because 242.2 now MODIFIES `_changelog` (adds the auto entry) during the release build BEFORE version.json is written, the committed version.json will reflect the auto entry — consistent. **Order matters:** the generator must run before the version.json write (it does — 3a before 3b). If you cut a release through a path that does NOT run build_and_release.sh, add the `_changelog` entry by hand (or run `python3 scripts/gen_changelog.py <version> <prev-tag>` then `update_version_json.py`) so the freshness check passes.
