# fix184.md â€” Multi-view: detect each source's connection limit and gate layouts (don't offer 2Ă—2 / multi-view a 1-connection plan can't support)

> **Version:** Free4Me-IPTV 1.23.14 build â†’ ships **1.23.15+184**.
> **Background:** Multi-view opens ONE provider connection per cell
> (confirmed: each cell creates its own `MpvEngine`/`ExoEngine` and `open()`s
> its own URL â€” `multi_view_cell.dart:261`). So **1Ă—2 = 2 connections, 2Ă—2 = 4**.
> A 1-connection provider (e.g. Emjay) CANNOT do multi-view at all; a
> 4-connection provider (Aniel3000) can do up to 2Ă—2 but with zero headroom.
> Today the user just hits a silent dead cell / "instant â€” possible connection
> limit". This fix surfaces the limit and offers only layouts the source can
> actually support, with a clear message otherwise.

---

## Design

Two sources of truth for a source's connection limit:
1. **Xtream** â†’ auto-detect from the `user_info.max_connections` field returned
   by the base `player_api.php` call. (`getXtreamHttpData` already does GET +
   jsonDecode; we just need the no-action/auth response, which includes
   `user_info`.) Stored on the source so it's available offline.
2. **M3U / unknown** â†’ no API; fall back to a **manual per-source field**
   ("Max connections", default unset = unknown). When unknown, do NOT block â€”
   warn instead (the user may know their plan; never hard-block on a guess).

Then **gate the multi-view layout picker** by the limit of the source(s) whose
channels are being used: hide/disable `2Ă—2` if limit < 4, and disable multi-view
entirely (only `none`) if limit < 2, with an explanatory message.

> Note: multi-view cells can mix channels from DIFFERENT sources. The safe gate
> is based on the MOST RESTRICTIVE enabled source the user could pick from, OR
> evaluated per-assignment. Simplest robust v1: gate by the limit of the
> source that owns the channels currently being assigned; if cells span
> sources, the limit is per-source (each source counts its own cells). See
> 184.4 for the pragmatic rule.

# Fix 184.2 â€” Auto-detect max_connections for Xtream on add/refresh

**File:** `lib/backend/xtream.dart`

The base `player_api.php?username=&password=` response (no `action`) returns:
```json
{ "user_info": { "max_connections": "2", "active_cons": "0", ... },
  "server_info": { ... } }
```
Add a fetch that reads it, and call it during source add + refresh.

```dart
/// fix184: fetch the Xtream account's max_connections. Returns null on any
/// failure (unknown â€” caller leaves the limit unset). max_connections comes
/// back as a STRING in the user_info block; parse defensively.
Future<int?> fetchXtreamMaxConnections(Source source) async {
  try {
    // base auth call: no action param â†’ user_info + server_info
    final data = await getXtreamHttpData('', source); // empty action = base url
    if (data is Map && data['user_info'] is Map) {
      final raw = data['user_info']['max_connections'];
      final n = raw is int ? raw : int.tryParse(raw?.toString() ?? '');
      if (n != null && n > 0) {
        AppLog.info('Xtream: source "${source.name}" max_connections=$n');
        return n;
      }
    }
  } catch (e) {
    AppLog.warn('Xtream: max_connections fetch failed for "${source.name}" â€” $e');
  }
  return null;
}
```
> Verify `buildXtreamUrl(source, '', null)` yields the base
> `player_api.php?username=...&password=...` with no `&action=` (or pass a param
> the server treats as the auth call). If the helper requires an action, call
> the lightest one that still returns `user_info` â€” most panels include
> `user_info` on EVERY `player_api.php` response, so any existing call (e.g. the
> live-streams fetch) could also be mined for it. Easiest: parse `user_info`
> opportunistically from the FIRST response during refresh and store it.

**Wire it in:** in the Xtream source add and refresh flows, after a successful
auth/first fetch:
```dart
    final mc = await fetchXtreamMaxConnections(source);
    if (mc != null) {
      source.maxConnections = mc;
      await Sql.updateSource(source);   // persist
    }
```
> Only overwrite when a value is found; never clear a known limit on a transient
> failure.

# Fix 184.4 â€” Gate the multi-view layout picker by connection limit

**File:** `lib/multi_view_picker_dialog.dart`

Compute the applicable limit and disable layouts that exceed it.

**Rule (pragmatic v1):** use the **minimum `maxConnections` across all ENABLED
sources** that have a known limit. Rationale: cells can draw from any enabled
source, so the safe ceiling is the most restrictive known one. Unknown limits
(null) are treated as "no constraint" (don't block on a guess â€” warn instead).

```dart
// fix184: determine how many cells the user's providers can support.
int? _multiViewConnectionCeiling(List<Source> enabledSources) {
  final known = enabledSources
      .map((s) => s.maxConnections)
      .whereType<int>()
      .toList();
  if (known.isEmpty) return null;        // unknown â†’ no hard gate
  return known.reduce((a, b) => a < b ? a : b);  // most restrictive
}
```

In the picker build:
- `ceiling == null` (unknown): offer all layouts, but show a subtle note â€”
  *"Multi-view needs 2 connections (1Ă—2) or 4 (2Ă—2). If your provider allows
  fewer, cells may fail to load."*
- `ceiling >= 4`: offer 1Ă—2 and 2Ă—2 normally.
- `ceiling == 2 or 3`: offer **1Ă—2 only**; show 2Ă—2 disabled/greyed with note
  *"2Ă—2 needs 4 connections; your provider allows $ceiling."*
- `ceiling < 2` (i.e. 1): **disable multi-view entirely** â€” show only "Disable
  multi-view" and a message: *"This provider allows only $ceiling connection.
  Multi-view needs at least 2 simultaneous streams, so it can't be used here."*

```dart
    final ceiling = _multiViewConnectionCeiling(enabledSources);
    final allow2x2 = ceiling == null || ceiling >= 4;
    final allow1x2 = ceiling == null || ceiling >= 2;
    // ... render the 1Ă—2 card always but disabled (greyed, non-focusable for
    //     select) when !allow1x2; same for 2Ă—2 with allow2x2; show the relevant
    //     note text. When !allow1x2, only the "Disable" action is active.
```
> Keep the fix156 D-pad focus behavior: disabled cards should NOT autofocus and
> should be skipped/greyed; the active option(s) + "Disable" remain D-pad
> reachable. If only `none` is allowed, autofocus "Disable".

> **`enabledSources`**: the picker should load enabled sources (or accept them).
> If it doesn't already have them, fetch via the existing sources query
> (`Sql.getSources()` / cached). Keep it cheap â€” sources are few.
