# fix533 — Diagnostic: groups.enabled + group_id join health (empty-live-category triage)

**Base tag:** `v2.0.32`
**Version:** `2.0.32+532` → `2.0.33+533`
**Scope:** diagnostic logging only. No schema, no query-path, no behavior change. No test changes.

---

## Why

Field export (onn4kplus, 2.0.31) showed Live TV categories present in the rail but empty when opened, on **Trex** and **A3000** only (Dino fine). The existing `fix418 stats` log proved every live channel on those sources has `channels.cat_enabled=0` (`en=0/53587`, `en=0/35805`), while movies/series are fine. That is *why* the grids are empty (the grid filters `cat_enabled=1`), but not the **root cause** of the `cat_enabled=0`.

Two candidate causes, indistinguishable from the current log:
1. **Genuine disable** — `groups.enabled=0` for those live categories (a prior user "disable / Unselect-all", which fix298 faithfully restores across refresh). If so, `cat_enabled=0` is *correct* and this is user state, not a bug.
2. **Broken propagation** — `groups.enabled=1` but the refresh `cat_enabled` denormalization UPDATE failed to set the channels, leaving them 0. That would be a real bug.

This build adds the one measurement that separates them, plus a join-health counter.

## What it logs (once, on the first browse query — same trigger as `fix418 stats`)

- `fix533 groups src=<id> mt=<0|1|2> g_n=<count> g_en=<enabled count>`
  Per source/media_type: total groups and how many are `enabled=1`.
- `fix533 orphan src=<id> null_gid=<n> dangling_gid=<n>`
  Channels with `group_id IS NULL`, and channels whose `group_id` points at a missing groups row (the `group_name`→`group_id` join-failure signal).

### How to read the next log

- Live line `fix533 groups src=2 mt=0 g_en=0/g_n` → **categories disabled at groups level** → cause #1 (prior disable). Fix path = corrected disable/restore semantics (or just re-enable on device).
- Live `g_en=N/N` but `fix418 stats … mt=0 en=0` → **propagation bug** → cause #2. Fix path = the refresh `cat_enabled` UPDATE.
- `orphan … dangling_gid` or `null_gid` large → group_id assignment didn't resolve (A3000 nonstandard `group_name` / whitespace names). Those rows default to `cat_enabled=1`, so they are NOT the empty cause, but the number quantifies join health.

Validated in a seeded mirror: a "Trex live disabled" seed produced `fix533 groups src=2 mt=0 g_n=2 g_en=0` and `fix533 orphan src=2 null_gid=1 dangling_gid=1`, exactly disambiguating the two cases.

+      // channels `en` (cat_enabled) line above to disambiguate the
+      // empty-live-category symptom: if a source's live (mt=0) shows
+      // groups_en=0/N here, the categories are genuinely disabled at the

+            ' g_n=${r['g_n']} g_en=${r['g_en']}');
+      }
+      // fix533: channels whose group_id is NULL or points at a missing groups
+      // row (the group_name->group_id join-failure signal). A nonzero unmatched
+      // count on a source means the refresh group_id assignment did not resolve
+      // every channel — those rows fall through to the `group_id IS NULL`
+      // cat_enabled=1 default, so they would NOT be the empty-category cause,
+      // but the number is the direct measure of join health.
+      final orphan = await _inner.getAll(
+        'SELECT c.source_id,'
+        ' SUM(CASE WHEN c.group_id IS NULL THEN 1 ELSE 0 END) null_gid,'
+        ' SUM(CASE WHEN c.group_id IS NOT NULL AND g.id IS NULL'
+        '   THEN 1 ELSE 0 END) dangling_gid'
+        ' FROM channels c LEFT JOIN groups g ON g.id = c.group_id'
+        ' GROUP BY c.source_id',
+      );
+      for (final r in orphan) {
+        AppLog.info('fix533 orphan src=${r['source_id']}'
+            ' null_gid=${r['null_gid']} dangling_gid=${r['dangling_gid']}');
+      }
