## Why (cold-eyes review HIGH-1 — confirmed by EXPLAIN, correcting fix330)
**Honest correction:** fix330 shipped the expression index `idx_channels_browse_tier` claiming "SQLite matches the existing alpha-mode ORDER BY to the index — no SQL changes needed." That claim was wrong, and the verification that "proved" it tested a hand-simplified ORDER BY instead of the one the app emits. The real runtime ORDER BY wraps the 6-tier CASE inside an outer CASE gated by a per-row correlated `(select sort_mode from sources …)` subquery — which the planner can never match to the index. EXPLAIN against a seeded 270k-row DB with the EXACT runtime SQL: `SCAN c … USE TEMP B-TREE FOR ORDER BY`, 323ms warmed (≈11s cold on the Shield). The fix330 win never existed at runtime.

## What this does
1. **`lib/backend/browse_order.dart` (new, pure):** single source of the browse ORDER BY. `BrowseOrder.tier` is the 6-tier CASE (structurally identical to the mig-27 index expression); `orderBy(uniformMode)` emits mode-specific clauses with NO subqueries for 'alpha' (index-served), 'provider', and 'category'; the legacy correlated form is emitted only for `null` (mixed modes across in-scope sources — behaviour preserved).
2. **`Sql.search`:** resolves the in-scope sources' sort modes ONCE in Dart (`SELECT DISTINCT sort_mode FROM sources WHERE id IN (…)`, normalised so anything not provider/category is alpha — same semantics as the old SQL's `IN ('provider','category')` test), then appends `BrowseOrder.orderBy(uniform-or-null)`. History view ordering unchanged.
3. **Migration 27 comment** now points at BrowseOrder as the counterpart that must stay structurally identical.
4. **`test/browse_order_test.dart` (new, 5 tests):** pure string assertions for every mode, plus a real-sqlite test that seeds the VERBATIM mig-27 DDL and `EXPLAIN`s the EXACT emitted query — asserting `idx_channels_browse_tier` is used and no `TEMP B-TREE` for alpha, and that the legacy mixed form DOES temp-sort (documenting why this fix exists). Because Sql.search and the test share BrowseOrder, tested == emitted by construction — the discipline failure that let fix330 ship cannot recur.

## Measured result (270k-row seeded DB, emitted query)
- Before: `SCAN c … USE TEMP B-TREE FOR ORDER BY` — 323ms warmed, ~11s cold on Shield-class storage.
- After: `SEARCH c USING INDEX … (source_id=?)` — **0.18ms warmed** (~1800x), cold path eliminated (index walk, stops after one page).
