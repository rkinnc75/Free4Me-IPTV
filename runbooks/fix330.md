# fix330 — All-view browse stall on huge catalogues: covering sort index → v1.26.46+330

> ## WARNING BUILD MACHINE — APPLY THE PATCH, then RELEASE (EXACTLY). No questions, no DB to query.
> **FIRST:** `find . -name 'fix296-DIAGNOSE*.md' -delete 2>/dev/null || true`

## Why (from the Shield log + EXPLAIN on a seeded 270k-row catalogue)
Browsing froze ~11s on the Shield ("video doesn't work" — actually the channel list, not the player; libmpv playback was fine). `Home.load` hit `SLOW SQL 11062ms`. EXPLAIN QUERY PLAN on a reproduced 269k-row DB showed **`USE TEMP B-TREE FOR ORDER BY`**: the existing `(source_id, media_type, …)` index filters but cannot satisfy the ORDER BY, because the All view spans `media_type IN (0,1,2)` AND the sort leads with a computed 6-tier CASE. So SQLite materialized and sorted the entire catalogue to return 36 rows.

## What this does
Migration **27** adds an expression index `idx_channels_browse_tier` on `(source_id, <the exact 6-tier CASE>, name COLLATE NOCASE) WHERE url IS NOT NULL`. The planner now walks the index in sort order and stops after one page. No SQL/query changes were needed — SQLite matches the existing alpha-mode ORDER BY to the index expression (verified the `c.` alias still matches).

Measured on the seeded 269k-row catalogue (same shape as the Shield: 54k live / 170k movie / 44k series):
- All view, first page: ~340ms warm (≈11s cold on Shield) → **~0.10ms**, temp B-tree gone.
- Deep page (offset 1800): **~1ms**.
- Single media_type views: also use the index.

## Notes
- **Index expression must stay byte-identical to the alpha-mode ORDER BY tier CASE in `Sql.search`.** If that CASE ever changes, update this index in lockstep or it silently stops being used.
- Provider/category sort modes don't use this index (different ORDER BY) — acceptable; the reported slow case is the default alpha All view. A follow-up could add per-mode indexes if needed.
- Built against current live main (1.26.44+328). If fix326/327/329 land first, the pubspec/changelog hunks rebase trivially (Rich reassigns the number/version).
- First migration to bump the highest DB version to 27.
