# fix339 — Live on ExoPlayer restart loop: false "stream completed" every second → v1.26.55+339

## Why (S24 2x2 on 1.26.54 — the "went backwards" report; 164 completed events in 5 min)
The smoking gun is one line pair: `ExoEngine: initialised duration=0s` immediately followed by `ExoEngine: stream completed`, repeating ~1/s on every cell (generations to 22 in a minute). Mechanism: on raw `.ts` live, video_player reports a TINY non-zero duration (sub-second / segment window — printed as `0s` because the log shows `inSeconds`). The old VOD-completed guard `duration > Duration.zero` therefore PASSED, `position >= duration` was instantly true, and "stream completed" fired in a loop; the cell's legitimate completed-retry restarted the cell each time. This was masked before fix335 (Exo on live always fell back to libmpv at 15s); once fix335 kept Exo alive on live, the false completed surfaced — an indirect regression of 335. Single-view live on Exo was equally affected.

## What this does
1. **`isLive` on `open()`** (interface + both engines): callers pass the channel's mediaType. `ExoEngine` NEVER emits completed when `isLive` — deterministic, not a duration heuristic. `MpvEngine` accepts and ignores it (mpv's completed is EOF-based and already correct).
2. **Call sites:** `Player` passes `mediaType == livestream` (both open paths, incl. the cast-resume path); `MultiViewCell` passes `isLive: true` (multi-view is live-TV-only by design).
3. **Second layer:** for unset callers, the VOD completed guard now also requires `duration >= 5s`, so a tiny bogus duration can never qualify.
