# fix164 — TV: software decode default, visible D-pad focus, multi-view picker focus trap

> **Version:** 1.23.5+164 (phone/even build; patch bump from shipped 1.23.4+162)
> **Evidence:** `free4me_log_1780155704975.txt` (onn 4K Plus, isTV=true) + on-device report.
> **Bugs:**
> 1. **A/V desync on TV** — video lags behind audio under `hwdec=mediacodec-copy`; setting Hardware Decoder OFF fixes it.
> 2. **Focus is invisible / "starts off a control"** — only `FilledButton` has a bright focus ring; `ListTile`/`IconButton`/dialog buttons fall back to the near-invisible default highlight on the black M3 surface, and several screens have no autofocus target.
> 3. **Multi-view channel picker is a focus trap** — D-pad can't leave the search bar or reach the channel list.
> **Root causes:** (1) `mpv_engine.dart` forces `mediacodec-copy` on every TV full-screen open; the GPU→CPU frame copy can't keep pace with audio on 2GB / Mali-G310. (2) The app's focus ring is themed only for `FilledButton` (`main.dart` `filledButtonTheme`); nothing else. (3) `channel_picker_screen.dart` uses a raw `TextField(autofocus:true)` instead of the repo's purpose-built `DpadTextField`, so arrow-down moves the caret instead of escaping the field.

---

## What the log proves

- Every open logs `Player: hwdec=mediacodec-copy isTV=true` (e.g. lines 57, 184, 281, 447). This is the full-screen TV branch in `mpv_engine.dart:326-329`.
- Streams themselves are fine: after the provider connection-limit cooldown clears, the same channel plays and logs repeated `stream stable for 15s` (lines 466-476). The desync is a *decode-path* problem, not a network one.
- `DeviceMemory: totalMb=1925` (line 88) — 2GB. The `mediacodec-copy` readback path is memory-bandwidth bound here; software decode keeps the clocks in sync, which matches the user's "Hardware Decoder Off is much better."

## Why software decode (not surface mediacodec) on TV

fix108 deliberately moved TV to `mediacodec-copy` because **surface-mode `mediacodec`** binds to a SurfaceTexture and silently produces audio-only black screen on Tegra-class Android TV SoCs. So we must **not** switch TV to plain `mediacodec` — that re-introduces the fix108 black-screen failure. Software decode (`hwdec=no`) is the safe universal fallback: it can't hit the surface-binding failure and it's what the user confirmed works. On a quad-A55 at 1080p MPEG-TS this is acceptable; if a future 4K-heavy stream struggles, the per-user Hardware Decoding toggle is still available.

**Scope guard — do NOT touch the preview/multi-view decode path.** fix108 also established that preview cells need `mediacodec-copy` or they render black + spinner (software stalled). fix164 changes **only the full-screen TV branch**; the `previewMode` branches stay exactly as they are.

# Fix 164.1 — TV full-screen: default to software decode

**File:** `lib/player/mpv_engine.dart`

Locate the full-screen Android hardware branch by symbol (the `else if (s.hwDecode && Platform.isAndroid)` arm that currently computes `hwdecMode`). Current code:

```dart
    } else if (s.hwDecode && Platform.isAndroid) {
      // Android TV devices (Shield, Fire TV, Onn 4K, etc.) require
      // mediacodec-copy rather than mediacodec surface mode. In surface
      // mode, mediacodec binds directly to a SurfaceTexture — this fails
      // silently on Tegra X1 and similar Android TV SoCs, producing audio
      // with a black screen. mediacodec-copy decodes in hardware but copies
      // frames to CPU memory, bypassing the surface binding. Overhead is
      // negligible on TV-class hardware.
      final isTV = await DeviceDetector.isTV();
      final hwdecMode = isTV ? 'mediacodec-copy' : 'mediacodec';
      await np.setProperty('hwdec', hwdecMode);
      AppLog.info('Player: hwdec=$hwdecMode isTV=$isTV');
    } else if (s.hwDecode && Platform.isIOS) {
```

Replace with:

```dart
    } else if (s.hwDecode && Platform.isAndroid) {
      // Phone: mediacodec surface mode (hardware, zero-copy).
      // TV: software decode. fix164 — on low-RAM TV boxes (onn 4K Plus,
      // 2GB / Mali-G310) the mediacodec-copy GPU→CPU readback fix108 used
      // for TV falls behind the audio clock, so video lags audio. Software
      // decode (hwdec=no) keeps A/V in sync and cannot hit the surface-mode
      // black-screen failure that fix108 was avoiding. The per-user Hardware
      // Decoding toggle still forces hardware off entirely on either device.
      // NOTE: this changes ONLY full-screen TV decode. Preview/multi-view

      // decode stalled to black there per fix108.
      final isTV = await DeviceDetector.isTV();
      final hwdecMode = isTV ? 'no' : 'mediacodec';
      await np.setProperty('hwdec', hwdecMode);
      AppLog.info('Player: hwdec=$hwdecMode isTV=$isTV (fix164)');
    } else if (s.hwDecode && Platform.isIOS) {
```

**Key changes**
- TV full-screen `hwdec` goes from `mediacodec-copy` → `no` (software).
- Phone behaviour (`mediacodec` surface) unchanged.
- Preview / iOS / `hwDecode`-off branches untouched.
- Log line tagged `(fix164)` so a fresh capture confirms the new path.

- The `ListTile`s in `_buildTile` are already focusable (`onTap`), so once focus can leave the field the list is reachable and selectable.

# Fix 164.3 — Make D-pad focus visible app-wide

**File:** `lib/main.dart`

The `ThemeData` only paints a focus ring for `FilledButton`. Add focus theming for the other common interactive widgets so focus is obvious on the black surface, mirroring the existing `!hasTouchScreen` gate (TV/remote only — never alters touch behaviour). Channel tiles (green border) and menu tiles (white border) already self-style and are unaffected.

Locate the `theme: ThemeData(...)` block. After the existing `filledButtonTheme: FilledButtonThemeData(...)` entry and before `useMaterial3: true`, add:

```dart
        // fix164: TV/remote focus must be clearly visible on the black M3
        // surface. Default highlight is near-invisible; only FilledButton had
        // a ring. Gate on !hasTouchScreen so touch UI is unchanged.
        listTileTheme: ListTileThemeData(
          focusColor: hasTouchScreen
              ? null
              : Colors.blueAccent.withValues(alpha: 0.32),
        ),
        iconButtonTheme: IconButtonThemeData(
          style: ButtonStyle(
            side: WidgetStateProperty.resolveWith((states) {
              if (states.contains(WidgetState.focused) && !hasTouchScreen) {
                return const BorderSide(color: Colors.yellow, width: 3);
              }
              return null;
            }),
          ),
        ),
        textButtonTheme: TextButtonThemeData(
          style: ButtonStyle(
            side: WidgetStateProperty.resolveWith((states) {
              if (states.contains(WidgetState.focused) && !hasTouchScreen) {
                return const BorderSide(color: Colors.yellow, width: 3);
              }
              return null;
            }),
          ),
        ),
        outlinedButtonTheme: OutlinedButtonThemeData(
          style: ButtonStyle(
            side: WidgetStateProperty.resolveWith((states) {
              if (states.contains(WidgetState.focused) && !hasTouchScreen) {
                return const BorderSide(color: Colors.yellow, width: 3);
              }
              return null;
            }),
          ),
        ),
```

**Key changes**
- `ListTileThemeData.focusColor` — a solid blue wash (alpha 0.32) makes the focused row in the picker / settings list obvious. The picker's `ListTile`s pick this up automatically; no per-tile edits.
- `IconButtonTheme` / `TextButtonTheme` / `OutlinedButtonTheme` — yellow 3px focus border, matching the existing `FilledButton` ring so dialog buttons and the settings help icons are visibly focusable.
- All gated on `!hasTouchScreen`, identical to the existing `filledButtonTheme` pattern; touch builds get `null` (default).
- `withValues(alpha:)` is the current non-deprecated API (the repo already uses it, e.g. `multi_view_picker_dialog.dart:93`).

> **Note on "focus starts off a control":** with 164.2 the picker autofocuses the search bar, and 164.3 makes every landing spot visible. If any *other* screen still opens with no focused control, that's a missing `autofocus` on that screen's first target — call it out as a follow-up (fix166) with the specific screen; it's out of scope here since the report was specific to the picker.

# Fix 164.4 — Setup wizard: same focus-trap class on the first-run screen

**File:** `lib/setup.dart`

A tree-wide sweep for the picker's bug class found the **first-run setup wizard** has the same problem, and it's higher-impact: a TV user must complete it before using the app. Each wizard step is one text field plus a Next button; if the field eats arrow-down (raw field / unwrapped `FormBuilderTextField`), the user can't move from the field to Next and is stuck. `edit_dialog.dart` already solves this exact case by wrapping every `FormBuilderTextField` in `DpadFocusEscape` inside a `FocusTraversalGroup` — `setup.dart` has the `FocusTraversalGroup` (line ~422) but is missing the per-field `DpadFocusEscape` wrappers, and its EPG step uses a raw `TextField`.

> Verified clean here (do NOT change): `edit_dialog.dart` (already wraps all three fields), `home.dart` / `settings_view.dart` / `epg_channel_mapping.dart` (use `DpadTextField`). The only other text-input file with the trap is `setup.dart`.

**(a) Add the import** alongside the other `open_tv/widgets/...` imports:

```dart
import 'package:open_tv/widgets/dpad_text_field.dart';
```

**(b) Wrap each of the four `FormBuilderTextField`s** (the Name, URL, Username, Password steps) in `DpadFocusEscape`, matching the `edit_dialog.dart` idiom. For each, the field is the `child:` of that step's `FormBuilder`. Change:

```dart
            key: _formKeys[Steps.name],
            child: FormBuilderTextField(
```
to:
```dart
            key: _formKeys[Steps.name],
            child: DpadFocusEscape(
              child: FormBuilderTextField(
```
and add the matching closing `)` after that field's existing closing `),` (i.e. the `FormBuilderTextField(...)` call gains one wrapping paren). Repeat identically for `Steps.url`, `Steps.username`, `Steps.password`. **Locate each by its `key: _formKeys[Steps.<x>]` line** — there are exactly four, one per step.

> Implementer note: these `FormBuilderTextField`s set `textInputAction: TextInputAction.next` and a `focusNode` — leave both as-is. `DpadFocusEscape` only intercepts D-pad up/down at the `Focus` layer; it does not interfere with the form's own next-field action or validation.

**(c) Convert the raw EPG-URL `TextField`** (the `Steps.epgUrl` step) to `DpadTextField`. Current:

```dart
            TextField(
              controller: _epgUrlController,
              decoration: const InputDecoration(
                labelText: "EPG URL (optional)",
                border: OutlineInputBorder(),
                prefixIcon: Icon(Icons.tv),
                helperText: "Leave blank to skip",
              ),
              autocorrect: false,
              keyboardType: TextInputType.url,
              textInputAction: TextInputAction.done,
              onSubmitted: (_) => handleNext(),
            ),
```

Replace with:

```dart
            DpadTextField(
              controller: _epgUrlController,
              decoration: const InputDecoration(
                labelText: "EPG URL (optional)",
                border: OutlineInputBorder(),
                prefixIcon: Icon(Icons.tv),
                helperText: "Leave blank to skip",
              ),
              keyboardType: TextInputType.url,
              onSubmitted: (_) => handleNext(),
            ),
```

**Key changes**
- All five setup inputs now yield D-pad focus, so the wizard is completable with a remote.
- The EPG field drops `autocorrect`/`textInputAction` (not exposed by `DpadTextField`); `DpadTextField` already treats Enter/Select as submit-and-advance, and calls `onSubmitted` — so `handleNext()` still fires on Select. Behaviour preserved.
- Matches the existing `edit_dialog.dart` pattern exactly — no new mechanism introduced.
