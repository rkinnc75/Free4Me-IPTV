# fix188 — Multi-view audio lingers ~5s after exit (mute + stop before MpvEngine teardown)

> **Version:** build `+188`; patch = next available at ship time (read live pubspec — do NOT assume `1.23.17`; patch and fix numbers drift). Tag = next available `v1.23.X` (immutable; never reuse/delete/recreate).
> **Evidence:** On the onn 4K Plus, exiting a multi-view session leaves audio playing for ~5 seconds after the UI is gone.
> **Bug:** `MpvEngine.dispose()` calls `await _player.dispose()` without first stopping or muting playback. libmpv's audio output keeps running while the async native teardown completes, so sound outlives the visible player.
> **Root cause:** `dispose()` goes straight from logging to cancelling stream subs, closing controllers, and `_player.dispose()`. `Player.dispose()` tears down asynchronously; until the native instance is fully released, the audio track is still being rendered. Nothing silences or halts decode first.

## What the evidence proves

The lingering audio is heard *after* the widget tree is gone, which means the Dart-side `dispose()` has been called but the native libmpv instance is still producing audio during its async teardown. Muting (`setVolume(0)`) silences output instantly; `stop()` halts decode and unloads the current media. Doing both before `Player.dispose()` removes the audio tail.
