# fix234 — CI: title workflow runs by tag/version, not by the last commit message

> **Version:** CI-only change. No app code / version bump required to land it (apply on a normal push to main).
> **Problem:** In the Actions list, recent runs are titled `refactor: move fix224.md to runbooks/` and `chore: move fix226.md to runbooks/` instead of something meaningful like `Release v1.24.1`. The TAGS and the RELEASED CODE are correct — only the run's DISPLAY TITLE is uninformative.
> **Cause:** a workflow run's default name is the triggering commit's message. Older releases were tagged on the fix commit (so the run showed "fix206: …"), but recent releases were tagged on top of a later docs/chore commit ("move fixNNN.md to runbooks/"), so GitHub titled the run after the chore.
> **Fix:** add a top-level `run-name:` to both workflows so the run is always titled by the tag/version (release) or branch+SHA (analyze), regardless of the last commit message.

## Pre-verified facts (checked against the 1.24.1 workflow files; YAML validated)

- **Files:** `.github/workflows/release.yml` and `.github/workflows/analyze.yml`. One added line each (plus a comment).
- `run-name` is a documented top-level GitHub Actions key; it sits as a sibling of `name:` and `on:`.
- For release.yml, `github.ref_name` on a tag push is the tag (e.g. `v1.25.0`). For a `workflow_dispatch` run there is no tag, so the expression falls back to a literal label.
- For analyze.yml (branch push), `github.ref_name` is the branch (`main`); appending the SHA makes each run uniquely identifiable.
- Both files parse as valid YAML after the edit (verified). The release `run-name` is a single-line double-quoted string (avoids folded-scalar pitfalls that would split the `${{ }}` expression).
- This is independent of fix232/the 1.25.0 release (different lines; fix232 does not touch the workflow headers). Apply order does not matter.

## Fix 234.1 — `.github/workflows/release.yml`: add `run-name`

### Current code (verbatim — match exactly)
```yaml
name: Release

on:
```
### Replacement code (verbatim)
```yaml
name: Release
# fix234: title the run by the tag/version (and dispatch) instead of the
# last commit message, which was often a 'move fixNNN.md to runbooks/' chore.
run-name: "Release ${{ github.event_name == 'workflow_dispatch' && 'manual dispatch' || github.ref_name }}"

on:
```

## Fix 234.2 — `.github/workflows/analyze.yml`: add `run-name`

### Current code (verbatim — match exactly)
```yaml
name: Analyze

on:
```
### Replacement code (verbatim)
```yaml
name: Analyze
# fix234: title the run by branch + short SHA so it is identifiable even
# when the last commit is a docs/chore commit.
run-name: 'Analyze ${{ github.ref_name }} @ ${{ github.sha }}'

on:
```

## Notes
- Tags remain immutable and correct; this only changes how runs are LABELED in the Actions UI. Past runs keep their old titles; new runs get the new naming.
- Optional complementary habit (not required with this fix): tag the version-bump commit rather than pushing a docs/chore commit after it — but `run-name` makes the title correct regardless, so this is no longer necessary.
- Independent of all app fixes and of fix232's 1.25.0 release.
