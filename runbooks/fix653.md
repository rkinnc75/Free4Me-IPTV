# fix653 — Single-cell player: stop hiding diag/PiP/mini-player behind the persisted multi-view layout (v2.2.53)

**Builder:** desktop. **Live base:** v2.2.52. **Ships as:** v2.2.53+642.

## Problem (reported from the S25 Ultra, private log 2026-07-04T00-26 sms938u)

With debug logging ON, the multi-view grid showed its compact per-cell stats
overlays, but the full-screen ("single cell") player showed NO diagnostics
panel. Log evidence: v2.2.52, MpvEngine (`hwdec=mediacodec-copy`), 294 STATS
lines (all from the multi-view cells), and `multiViewLayout=twoByTwo` on every
settings line.

## Root cause

Three player.dart gates keyed off `widget.settings.multiViewLayout ==
MultiViewLayout.none` — the **persisted grid-size setting**, not whether a
grid is actually on screen:

- the fix564 `DebugStatsOverlay` mount (player.dart ~1891),
- the PiP button (~2099),
- the mini-player button (~2151, comment claimed "hidden when multi-view is
  active").

Merely CONFIGURING a 2x2 layout (which the phone keeps persisted) made all
three vanish from every single-cell play — both a normal from-tile play and a
cell maximized from multi-view (`multi_view_cell.dart:826` pushes the same
`Player` with identical args). `home.dart` only uses the layout setting to
show the multi-view launcher button; from-tile playback still opens the
single Player, so the clause never distinguished anything meaningful. Not a
regression: the clauses date to fix564/fix580-era code, byte-identical since
before v2.2.47.

## Change (lib/player.dart) — TWO of three gates flipped

Dropped the `multiViewLayout == none` conjunct from the two SAFE gates:
- **Stats panel**: now `debugLogging && _engine is MpvEngine`. Pure UI (no
  engine); a pushed Player covers the grid, whose compact overlays live in
  `MultiViewCell`. Safe even for a maximized-from-grid play.
- **PiP button**: now just `_pipSupported`. Auto-PiP on Home press was
  already unconditional (player.dart:883), so this only surfaces an existing
  capability — no new state class.

**Mini-player button: held, NOT flipped.** Adversarial review (below) found a
bare flip opens a newly-reachable degraded state: multi-view and the
mini-player are mutually exclusive by design (`multi_view_screen.dart:55`
`stopOverlay` on entry), so minimizing a cell maximized from the grid and
returning restarts the cell on the SAME `.ts` URL the overlay now holds →
"Failed to open" churn or double audio (the app documents this provider
behavior at `multi_view_cell.dart:810-815`). The button stays gated on
`multiViewLayout == none`; the play-from-tile common case still can't be
distinguished from a maximized cell by that gate, so a correct fix needs a
UX decision (mini-player wins + cell idle, vs grid wins + overlay stops) and
phone-touch verification — queued as fix654.

The TV-only gates at ~1616/~1958 (overlay routing) are intentionally
untouched — but note they carry the SAME persisted-setting confusion (on TV
with a layout configured, OK opens media_kit's non-D-pad-reachable bars
instead of the focusable overlay). Pre-existing; flag for a follow-up.

## Adversarial review outcome

DO-NOT-SHIP for the 3-gate version → SHIP for this 2-gate version. Verdicts:
stats-panel double-panel SAFE (grid not painted under the opaque route);
PiP SAFE (auto-PiP already unconditional); bottom-bar overflow SAFE (10-button
max was already shipping for layout-none users); mini-player flip =
REGRESSION → reverted here.
