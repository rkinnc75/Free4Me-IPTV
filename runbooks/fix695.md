# fix695 — refresh perf 2/2: EPG ingest cost-cutters (v4.1.3)

**Builder:** desktop. **Live base:** v4.1.2. **Ships as:** v4.1.3+695.
Second Tier-1 refresh-perf release. The EPG post-parse tail on the onn is the
WAL checkpoint of inserted programmes (~3–3.5 min; baseline WAL ~197 MB). Every
lever here reduces or eliminates inserts.

## Three mechanisms (owner-approved: all of them)

All in `lib/backend/xmltv_parser.dart` + `epg_service.dart` + one `Sql` helper.

1. **Body-hash skip (provider-independent).** The download now streams to a
   temp file while computing SHA-256. If the hash equals the previous refresh's
   (`app_meta['epg_val_<sourceId>']`), the parse+insert+checkpoint+match tail is
   skipped entirely — the whole ~3.5 min tail vanishes when the guide didn't
   change. Memory stays bounded (buffered to disk, not RAM).
2. **Conditional GET (ETag / If-Modified-Since).** Stored validators are sent;
   a **304** short-circuits before any download. Likely inert on dynamic Xtream
   `xmltv.php` (verify on-device), but free when a provider honors it; the
   body-hash is the fallback that always works.
3. **Bound-channel parse filter.** `Sql.getBoundEpgIds(sourceId)` = distinct
   non-null `epg_channel_id` on the source's channels. Programmes whose xmltv
   `channel` isn't in that set are dropped at `<programme>` START, before any
   allocation. On the onn Trex only 1657 of 5583 feed channels are matched, so
   this cuts inserts hard (unmatched channels have no epg id to join anyway →
   lossless for them). Empty set (first-ever EPG for a source) ⇒ no filter, so
   the initial ingest is complete and the first match sees everything.
   - **Caveat:** a channel matched for the FIRST time on refresh N gets its
     programmes on N+1 (it wasn't in the bound set when N parsed). Self-heals
     next refresh; negligible since most matching happens on the first EPG.

Also folded in: the trivial window check moved to `<programme>` start (was at
end), and text accumulation gated so skipped programmes don't build `<desc>`
strings — CPU/alloc win, zero behaviour change.

Validators (`{etag,lastModified,hash}`) are persisted after every refresh
(changed or not) so a 304/hash-skip can still carry a refreshed validator.
Unchanged refreshes still mark the source cleanly-fresh (`upsertEpgRefreshLog`,
0 inserted, no error) so the launch-refresh debounce (finding 37) counts them
and there's no re-download loop.
