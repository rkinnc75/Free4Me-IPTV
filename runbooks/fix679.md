# fix679 — SR cleanup: gate diagnostic logging + stop orphan alarms

**Build:** `3.0.21+679` · **Base tag:** `v3.0.20` (fix678) · **Ship order:** after 678
**Files touched:** `lib/backend/recording_scheduler.dart`, `lib/recordings_view.dart`, `lib/whats_new_modal.dart`, `pubspec.yaml`
**No native/manifest/dependency changes.**

---

## Context

fix678 verified: Scheduled Recording now records and plays back end-to-end (device log 3.0.20+678 shows `SR-CB: … RecordingCapture.start returned OK` and `.ts` playback via `fd://`). This is cleanup of two things that surfaced in that same log.

## A — gate the `[SRDBG]` diagnostics behind the debug-logging setting

The fix675/676 `[SRDBG]` breadcrumbs write to `app_log.txt` unconditionally, even with debug logging off. Now they're gated: `_srDebug` reads the user's `debugLogging` setting **once per isolate run** (cached in `_srDebugEnabled`; the callback already reads the DB, so this is isolate-safe) and skips the file write when off. `debugPrint` (logcat only, adb) stays unconditional and harmless. The logs are kept, just tied to the setting — as requested. (The full SR diagnostic set can be removed later once the feature soaks.)

## B — stop orphan alarms (they re-fire on every boot)

The 3.0.20 log showed stray callbacks for `id=9` and `id=0` hitting `rec=NULL`. Alarms are scheduled with `rescheduleOnReboot: true`, and `android_alarm_manager_plus` has **no API to enumerate alarms**, so an orphan can't be found and cancelled in bulk — it just keeps firing. Two fixes close both the source and the existing orphans:

1. **Source (the leak):** `recordings_view._delete` removed the DB row via `Sql.deleteRecording` but never cancelled the alarm — so deleting a scheduled recording orphaned its alarm. It now calls a new `RecordingScheduler.cancelAlarm(id)` first. `cancelAlarm` cancels the alarm **only** (no status write, since the row is about to be deleted — using the existing `cancel`, which writes `cancelled`, would touch a row that's immediately removed).
2. **Existing orphans (self-heal):** when `recordingAlarmCallback` finds `rec == null`, it now cancels its own alarm before returning. So the already-leaked `id=0`/`id=9` ghosts cancel themselves on their next (harmless) fire and stop rescheduling.

Both are Android-guarded and best-effort (swallow cancel errors); off-Android they're no-ops.

- **Delete** a *scheduled* (not-yet-fired) recording, reboot, confirm no ghost `SR-CB … rec=NULL` for that id in the log.
- The old `id=0`/`id=9` ghosts: after their next fire post-update, they should stop appearing on subsequent boots.
