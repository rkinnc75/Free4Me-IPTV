# fix303 — Multi-view channel picker loads only page 1; add scroll pagination → v1.26.19+303

> ## ⚠️ BUILD MACHINE — APPLY THE PATCH, then RELEASE (EXACTLY). No questions, no DB to query.
> **FIRST:** `find . -name 'fix296-DIAGNOSE*.md' -delete 2>/dev/null || true`

## Root cause
The multi-view "Select channel" picker's empty-query browse (`_loadInitialBrowse`) fetched only `Sql.search(... page: 1)` — one page of `pageSize` (36) — by design for speed. Favorites/history are pinned to the top, leaving room for only a handful of "All channels", and there was no scroll-load, so the rest of the catalogue was unreachable unless the user typed a search. (Search already paginates fully in `_loadSearch`; only browse was capped.)

## The fix
Add scroll pagination to the empty-query browse, mirroring the search path: keep the fast page-1 initial load, then load subsequent pages as the user nears the bottom. Pages are merged (dedup by id) and re-sorted so favorites/history stay pinned above the growing tail. Search behavior is unchanged; clearing the search box restores the now multi-page cached browse and scrolling resumes.

## Verified before delivery
- `flutter analyze --no-fatal-infos` → 2 tolerated `settings_view.dart` INFOs, no warnings/errors.
- Patch applies cleanly to fresh main.
- UI/pagination logic (no SQL change): invocation guard prevents a stale browse page from landing after a query change; `_browseHasMore` stops at the last partial page; ScrollController disposed.
