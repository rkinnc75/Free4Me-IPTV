# fix662 — finding 58: defer the heavy migration-tail full-scans (v3.0.7)

**Builder:** desktop. **Live base:** v3.0.6. **Ships as:** v3.0.7+651.

Resolves the last deferred code-review item (finding 58, P3, VERIFY-FIRST).
Migrations 35 (fix519 channels_fts unicode61 rebuild) and 38 (fix530 `ANALYZE`)
ran their DATA-heavy full-scans INSIDE the awaited migration chain — before
`runApp()`. On an upgrade over a populated Shield-scale catalog (~1.2M rows) that
blocked first frame long enough to risk an ANR ("not responding" / black screen).
The memory pragmas (fix608, temp_store=FILE + 32 MiB cache) already bounded the
OOM angle; the remaining risk was wall-time. This moves the two largest
full-scans off the cold-start critical path.

## Change (minimal-risk path from IMPLEMENTATION_SPEC.md finding 58)
- **db_factory.dart mig35**: removed the inline `INSERT INTO channels_fts SELECT
  id, name FROM channels` backfill. The migration now creates the fts virtual
  table + triggers EMPTY only (fast DDL). Comment states the deferral.
- **db_factory.dart mig38**: removed the inline `ANALYZE;` — migration is now a
  documented no-op kept for history (version chain head stays 43).
- **sql.dart**: new `Sql.runPendingFtsAndAnalyze()` — runs the two operations
  unawaited after first frame:
  - **channels_fts backfill**: condition-driven self-heal (no marker). Runs
    `INSERT INTO channels_fts(channels_fts) VALUES('rebuild')` when channels is
    non-empty AND the index is empty. Emptiness is probed via the
    `channels_fts_docsize` shadow table — a bare `SELECT FROM channels_fts`
    reads the CONTENT table on an external-content fts5 and would never report
    empty. This also adds the channels_fts self-heal that did not exist before
    (only programmes_fts had one).
  - **ANALYZE**: marker-gated (`mig38_analyze_done`), runs once over a populated
    catalog. A device that upgraded through the OLD inline-ANALYZE mig38 runs one
    extra harmless ANALYZE off the critical path.
  - Both skip (unmarked) when channels is empty (fresh install) → they run on a
    later launch once a source is loaded; the first refresh's own fts rebuild
    (fix621) covers fresh-install search in the meantime.
- **main.dart**: chained `runPendingFtsAndAnalyze()` AFTER
  `runPendingIndexMaintenance()` (unawaited warm-up) so ANALYZE reflects the
  final fix537 index shapes.
