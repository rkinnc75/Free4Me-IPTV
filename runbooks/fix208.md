# fix208 — "What's New" shows stale notes from an old version (string-prefix match bug)

> **Version:** build `+208` (or higher if bundled); patch = next available at ship time (read live pubspec). Tag = next available immutable `v1.23.X`.
> **Evidence:** On 1.23.22, the "What's New" dialog shows the correct fix206 note (refresh ~2.6× faster) AND a stale note: "Fix (TV): the multi-view layout picker cards (1x2, 2x2) are now reachable with the D-pad. Settings rows no longer trap focus on the help icon…" — which is the **1.23.2** changelog entry (fix156-era), not a 1.23.22 change.
> **Bug:** `_entriesForVersion` in `lib/whats_new_modal.dart` selects changelog entries with `version.startsWith(e.key)` — a raw STRING prefix. `"1.23.22".startsWith("1.23.2")` is true, so the `1.23.2` entry leaks into the 1.23.22 dialog.
> **Root cause:** version keys are dot-delimited components, but the match treats them as plain strings. Any running patch whose string contains an older key as a prefix pulls that older entry in.

## Why this recurs (and worsens)

The bug fires whenever a newer patch number string-contains an older key:
- 1.23.22 → wrongly includes 1.23.2 (observed).
- 1.23.23 → would include 1.23.2.
- 1.23.30 → would include 1.23.3; 1.23.100+ → would include 1.23.1 / 1.23.10, etc.

This matches the user's report that "sometimes an old fix still shows."

## Pre-verified facts (checked against pristine `v1.23.21+204`, compiled)

- **File:** `lib/whats_new_modal.dart`, function `_entriesForVersion`.
- The changelog (`_changelog`) is keyed by version strings; the doc comment states a key may be `major.minor.patch` OR `major.minor` ("any patch in that minor"). The intended semantics are **component-based**, which the string-prefix match violates.
- The stale note's text is exactly the `'1.23.2'` entry (confirmed in `_changelog`).
- The running version passed in comes from `PackageInfo.version` (no `+build` suffix), so `key == version` works; the fix also defensively strips any `+build` suffix.
- **Verified matching behavior (modeled):** running `1.23.22` → old match = `[1.23.22, 1.23.2, (1.23 if present)]`; new match = `[1.23.22, (1.23 if present)]`. Running `1.23.2` itself → both correctly match `1.23.2` (the fix does not break the legitimate case). Minor-only keys like `1.23` still match any `1.23.x` (feature preserved).
- **Compile status:** applied to the `1.23.21` tree; `flutter analyze --no-fatal-infos` → exactly the 2 tolerated `settings_view.dart` INFOs. Rule 6 grep clean.

## Fix 208.1 — match on version components, not string prefix

**File:** `lib/whats_new_modal.dart`

### Current code (verbatim — match exactly)
```dart
/// Returns all changelog entries whose version key is a prefix of [version].
List<MapEntry<String, List<String>>> _entriesForVersion(String version) {
  return _changelog.entries
      .where((e) => version.startsWith(e.key))
      .toList();
}
```
### Replacement code (verbatim)
```dart
/// Returns the changelog entries that apply to [version].
/// fix208: match on dot-delimited version COMPONENTS, not a raw string prefix.
/// The old `version.startsWith(key)` wrongly matched e.g. key "1.23.2" against
/// running "1.23.22" (string prefix), leaking an old release's notes into a
/// newer version's "What's New". A key matches only if it equals the version
/// exactly, OR it is a minor-only key (major.minor) and the version is a patch
/// within that minor (starts with key + ".").
List<MapEntry<String, List<String>>> _entriesForVersion(String version) {
  // Defensive: drop any build suffix (e.g. "1.23.22+208" -> "1.23.22").
  final v = version.split('+').first;
  return _changelog.entries.where((e) {
    final key = e.key;
    if (key == v) return true;
    final parts = key.split('.');
    if (parts.length == 2 && v.startsWith('$key.')) return true;
    return false;
  }).toList();
}
```

### Key changes
- Replaces the raw `startsWith` with component-aware matching: exact equality, plus a `major.minor` key matching any patch within that minor.
- Strips a `+build` suffix defensively.
- No other change; the selection's callers and sort are untouched.
