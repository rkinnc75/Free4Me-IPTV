# fix515 — stream-info in the single-cell top bar ("720p H.264")

**Version:** 2.0.14+514 → **2.0.15+515**
**Type:** Feature. Single-cell full-screen player only (multi-view cells untouched, per scope).
**Tag:** v2.0.15
**Base:** v2.0.14 (2.0.14+514).

## What
The top bar (channel name + EPG) now also shows the stream's actual resolution and video codec — e.g. **"720p H.264"** — once the first real frame decodes. As decided: resolution + codec only (no fps), sampled once at first frame (not live-updating mid-stream), single-cell full-screen only.

This also surfaces provider mislabeling for free: a channel literally named "4K ESPN" that's actually delivering 720p (a real case seen in earlier log review) will show "720p H.264" right in the bar.

## How
No new mpv polling — this rides the engine's existing first-frame detection. `MpvEngine._observeFirstFrame()` already `observeProperty('dwidth', ...)` and only proceeds once `dwidth` is confirmed non-zero/non-empty/non-`'?'` (guards against sampling mid-glitch garbage on a stream with decode errors, same guard the existing decode-state log already relies on). fix515 piggybacks on that exact moment to also read `dheight` + `video-codec` and publish a formatted label.

- **`MpvEngine.formatStreamInfo(dheight, rawCodec)`** (static, pure) — height→tier mapping (2160→"4K", 1080→"1080p", 720→"720p", else literal "{h}p") + codec normalize (h264→H.264, hevc/h265→H.265, else uppercased raw token). Returns `null` when there's no real frame yet, so the bar stays blank rather than flashing "0x0".
- **`MpvEngine.streamInfoStream`** (new, added to the `PlayerEngine` interface — `MpvEngine` is the only implementation) — emits the formatted label once per play, right after `_observeFirstFrame` confirms a real size.
- **`PlayerStreamInfoLabel`** (new widget, `lib/widgets/player_stream_info_label.dart`) — a small `StatefulWidget` that subscribes to `streamInfoStream` itself and rebuilds its own subtree on each event. This sidesteps a real landmine already documented in `player.dart` (fix367/fix405): media_kit's `MaterialVideoControlsTheme` reads `topButtonBar`'s widget **list** once at mount and never re-reads it on parent rebuilds — a plain `Text` built from `State` that changes after mount would never update. Placing a self-contained stateful widget in that frozen list slot works because the widget *occupying* the slot is free to redraw its own subtree; this is the exact same pattern `PlayerEpgNowLabel` already uses for the EPG "now playing" text beside it.
- **`player.dart`** — added the label to `topButtonBar`, gated `if (widget.settings.multiViewLayout == MultiViewLayout.none)` (the same gating already used for the PiP button), keeping it single-cell-only per scope.

## Testing
New `test/format_stream_info_test.dart` — **24 tests** on the pure formatter logic: the not-yet-a-real-frame guard (0/empty/`'?'`/negative/garbage height all correctly return `null`, never throw), every resolution-tier boundary (699/700, 999/1000, 1999/2000), and codec normalization (h264/hevc/h265, profile-annotated forms like `"h264 (High)"`, case-insensitivity, unrecognized codecs, empty/`'?'` codec falling back to resolution-only).

`flutter analyze` clean (0 infos), **205 tests pass** (181 + 24 new).
