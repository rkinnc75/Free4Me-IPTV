# fix585 — #4 rail LEFT-re-expand: focus the rail immediately (v2.2.1 regression)

**Version:** v2.2.2+585
**Date:** 2026-06-27
**Type:** TV focus regression fix (on-device-found)

## Problem (fix584 / v2.2.1)

On-device on the onn: pressing D-pad LEFT from a channel in the Live guide did
NOT move focus to the category rail — focus stayed on the channel. fix584's
`_onChannelLeftKey` returned `handled` (consuming LEFT) but ran the
`requestFocus` inside `addPostFrameCallback`. When the rail was already expanded,
the handler did NOT call `setState`, so no frame was scheduled — and
`addPostFrameCallback` does not itself schedule a frame, so the callback never
ran. Net: LEFT was consumed but focus never moved → the previously-working
directional LEFT-to-rail was regressed (focus stranded on the channel). This is
the exact CI-green/device-fail focus class the design review flagged.

## Fix

`requestFocus` the stored rail node **immediately** (no post-frame). The rail
items stay mounted even when collapsed (OverflowBox keeps `_rail()` laid out at
210 px), so the node is always attached + focusable; the post-frame was both
unnecessary and the bug. Also guarded: only consume LEFT (`handled`) when the
rail node exists; otherwise return `ignored` so it can never strand focus.
