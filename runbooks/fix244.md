# fix244 — inMemory is already the default (doc fix) + partial index for the EPG auto-match scan

> **Version:** build = highest fix number in release; patch = next available. Tag = next immutable.
> **Builds on:** 1.25.x source (migrations currently top out at 18). Two small, independent changes.
> **Context:** the TV log showed `searchMethod=1` (ftsAnd) and a 1,338ms EPG-match scan. Two queue items from that:
>   1. "Make inMemory the default" — **already done** (fix212). The default in code is `SearchMethod.inMemory`; the TV showed ftsAnd only because it had an explicitly-saved value from before fix212 (a stored setting overrides the default). The only loose end is a stale doc comment. No behavior change.
>   2. The EPG auto-match scan (`getUnmatchedLiveChannels`) does a `media_type` scan; a partial index makes it a targeted lookup.

## Part 1 — correct the stale default comment

The `searchMethod` field doc still claims the default is `ftsAnd`; the actual default (and all fallback sites: settings_service, settings_io, channel_picker_screen, filters) is `inMemory`. This is a comment-only fix so the source isn't misleading. (No code/default change is needed — inMemory is already the default for any new or unset install.)

### Fix 244.1 — `lib/models/settings.dart`
Current code (verbatim — match exactly):
```dart
  /// Default: [SearchMethod.ftsAnd] — faster than phrase for multi-word
```
Replacement code (verbatim):
```dart
  /// Default: [SearchMethod.inMemory] — fastest; served by ChannelSearchCache
```
> Note: this does NOT change any device that already has a saved `searchMethod`. Those keep their stored value (which is why the TV was on ftsAnd). To move an existing device to inMemory, change it in Settings (as already done on the TV). There is intentionally no forced migration of a user-chosen setting.

## Part 2 — partial index for the EPG auto-match scan

The query (in `getUnmatchedLiveChannels`, sql.dart):
```sql
SELECT * FROM channels
WHERE source_id = ? AND media_type = 0
  AND epg_manual_override IS NULL AND epg_channel_id IS NULL
```
runs during EPG refresh to find live channels still needing an EPG match. On the 2GB TV with a ~320k catalog it took **1,338ms** (the planner used the `media_type` index and scanned). A PARTIAL index keyed on `source_id` but restricted to exactly the `IS NULL` predicate indexes only the small set of still-unmatched live channels — tiny on disk, and it turns the scan into a targeted lookup.

### Pre-verified (real sqlite_async, file DB)
- 45k live channels, ~2k unmatched: query went **9.05ms → 2.25ms**; plan switched from `SEARCH ... USING INDEX index_channel_media_type` to `SEARCH ... USING INDEX idx_epg_unmatched (source_id=?)`.
- Partial index only covers unmatched-live rows, so storage and insert overhead are negligible.
- Compiles clean (2 tolerated INFOs). Migrations currently top at 18 → new = 19.

### Fix 244.2 — `lib/backend/db_factory.dart`: add migration 19
Current code (verbatim — match exactly):
```dart
      ..add(SqliteMigration(18, (tx) async {
```

Replacement code (verbatim):
```dart
      ..add(SqliteMigration(18, (tx) async {
        await tx.execute(
            'CREATE INDEX IF NOT EXISTS index_channel_name_source '
            'ON channels(name, source_id);');
      }))
      // fix244: partial index for the EPG auto-match scan
      // (getUnmatchedLiveChannels): WHERE source_id=? AND media_type=0
      // AND epg_manual_override IS NULL AND epg_channel_id IS NULL. The
      // partial WHERE indexes only the small set of still-unmatched live
      // channels, so it is tiny and turns a media_type scan (~1.3s on a 2GB
      // TV with a 320k catalog) into a targeted lookup.
      ..add(SqliteMigration(19, (tx) async {
        await tx.execute(
            'CREATE INDEX IF NOT EXISTS idx_epg_unmatched '
            'ON channels(source_id) '
            'WHERE media_type = 0 AND epg_manual_override IS NULL '
            'AND epg_channel_id IS NULL;');
      }));
    await migrations.migrate(db);
```
