# fix752 — A FAILED fetch is committed as "empty" and wipes the catalogue

**Base:** main @ `6cf79d7c9b24a1c737cfb8258a4a714d77eae1ba` (v4.1.52+751)
**Version:** 4.1.52+751 → **4.1.53+752**

**Self-contained — apply exactly as written, make NO ship-time edits. Any gate failure = STOP and report.**

> **SEVERITY: data loss.** This is the bug that left the Trex source with zero live channels on 2026-07-14. It ranks ahead of the in-flight silent-stall work (fix749 remains withdrawn).

## 1. What happened (S938U logs 20260714-142732 / -145335)

The Trex source showed **no channels, no categories, nothing**. A refresh did not fix it.

Then, minutes later with nothing playing, the *same* refresh worked: `live=55325`.

## 2. Root cause — proven, not inferred

The failing refresh:

```
Xtream: fetched source="Trex" live=0 movies=192940 series=42344
Sql.wipeSource: sourceId=2 deleted 0 channels        ← source was ALREADY empty
Sql.restorePreserve: sourceId=2 total=0              ← nothing to preserve
Sql.ensureBrowseIndexesPresent: an interrupted refresh left 19 browse index(es) un-rebuilt
```

The action dumps show `get_live_categories` (60 KB) but **no `get_live_streams` dump at all** — and `getXtreamHttpData` returns `null` *before* writing its dump on HTTP failure. So the live fetch **failed** (null), `fetchLive()` hit `res[0] == null` → `failCount++` → `return 0`.

**Why the fetch failed:** the provider reports `max_connections=1`, and a stream was playing during the refresh. The largest call (`get_live_streams`, 18.5 MB) lost the connection race; the smaller VOD/series calls succeeded. Proof: the clean retry with nothing playing fetched all six actions in **11 seconds** vs **3.5 minutes** for the failing one.

**Why nothing protected the data — all three existing guards key off PRIOR STATE, and an earlier interrupted refresh had already emptied the source:**

| Guard | Requires | Actual | Result |
|---|---|---|---|
| fix321 retry | `lastLiveCount > 0` | 0 | skipped |
| finding-63 preserve | existing rows in DB | none | skipped |
| fix322 throw | `failCount >= 3` | 1 (only live failed) | no throw |

…so the refresh committed `live=0` as a legitimate result and reported **"Refresh complete."**

The defect in one line: **a failed fetch and a genuinely-empty result are both represented as `count = 0`**, and only prior state distinguishes them — which is exactly what is unavailable after an interrupted refresh.

## 3. Fix

- **Track WHICH types failed** (`failedMediaTypes`), not just how many. All **six** `failCount++` sites (null-fetch and decode-catch, for live/movies/series) are tagged.
- **`XtreamRefreshLogic.typesToKeepOnFetchFailure()`** — a failed type is added to `keepMediaTypes` **unconditionally**, so the wipe cannot touch it. This decision uses only THIS refresh's outcome, so it protects a source whose prior state is empty, stale, or unknown. Existing rows survive; if there are none, nothing is written — but nothing is destroyed.
- **Free repair of the state corruption:** because failed types are now in `keepMediaTypes`, `source.lastLiveCount` is no longer clobbered to 0 (existing fix321 logic already skips preserved types) — so fix321's retry guard stays armed for the next refresh instead of being permanently disarmed.
- **`XtreamRefreshLogic.isPartialRefresh()` + `XtreamPartialRefreshException`** — thrown **after** a successful commit. The fetched types ARE written; this is a warning, not a failure.
- **UI (settings_view):** the partial exception is caught around `Utils.refreshSource` **only**, so the EPG refresh and — critically — `ensureBrowseIndexesPresent()` still run. (Letting it propagate would leave the browse indexes dropped, recreating the very "interrupted refresh" state that caused this incident.) The dialog then shows `⚠ Live channels could not be fetched — existing content kept…` instead of "Refresh complete."

**Not fixed here (deliberate):** the underlying non-atomic wipe+reinsert (`finding 64` already notes "process death mid-commit can leave a partial catalog"). That needs a shadow/generation column and is its own arc. fix752 makes that state *survivable* rather than destructive.
