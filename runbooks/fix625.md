# fix625 — EPG "database is locked (code 5)" on re-match / refresh (v2.2.36)

**Builder:** desktop coder+builder (implemented + built + on-device verified).
**Live base:** v2.2.35. **Ships as:** v2.2.36+625.
(fix622–624 were the phone coder's multi-view/fps work — this is the next free number.)

## Symptom

On the onn 4K Plus, **"Re-match all channels"** (Settings → EPG) processed 4
sources; the FIRST (Emjay) succeeded, the other three (Trex/Dino/Media4u) ALL
failed with `SqliteException(5): while selecting from statement, database is
locked, database is locked (code 5)` (causing statement `BEGIN IMMEDIATE`), on
`epg.sqlite`. "Refresh EPG now" minutes earlier succeeded for all 4 — the race
just didn't happen to overlap that run.

## Root cause (two independent analyses + Python repro + on-device timing)

`epg.sqlite` had **two independent SQLite write connections in two different
Dart isolates**:
1. the **foreground** re-match/refresh (main isolate), and
2. the **background Workmanager EPG task** (`callbackDispatcher` →
   `EpgService.refreshAllSources(background:true)`, which processes 2 sources
   concurrently), running in a **separate headless isolate** — its own
   `EpgDbFactory._db` singleton (statics are per-isolate) → its own write
   connection.

`sqlite_async`'s write mutex only serializes writers **within one isolate**, so
when the two overlap, one isolate holds SQLite's single writer lock and the
other's `BEGIN IMMEDIATE` waits the full 30s `busy_timeout` and then throws
`SQLITE_BUSY`. Fingerprint confirmed in the log: Media4u's two errors are
**exactly 30s apart** (the busy_timeout elapsing). A POSIX file lock can't fix
this — `fcntl` locks are per-**process**, and the headless isolate shares the
process, so it wouldn't serialize the two isolates.

## The fix

`SQLITE_BUSY` from concurrent access is a **transient, expected, retryable**
condition — the same thing `sqlite_async` itself retries for its setup pragmas.
Added `Sql._epgWriteWithRetry(label, op, {maxAttempts:5})` (500ms·attempt
backoff) and wrapped every `epg.sqlite` **write**:
- `insertProgramsBatch` (the whole `writeTransaction` — idempotent upsert on the
  `(source_id, epg_channel_id, start_utc)` unique index),
- `deleteStalePrograms`, `rebuildProgrammesFts` (FTS `'rebuild'`),
  `upsertEpgRefreshLog`, `deleteEpgForSource`,
- the `PRAGMA wal_checkpoint(TRUNCATE)` in `checkpointAndTruncateWal` (retry then
  **rethrow** — shared with import/export, which must not ship a WAL-stale
  snapshot; not swallowed here).

Background write transactions are short (per parse batch) with gaps, so a retry
grabs the lock within a few attempts. All wrapped ops are idempotent, so
re-running a partially-applied attempt is safe.

Adversarial-review fix: `_isDbBusy` matches `SqliteException.resultCode == 5`
(the SQLITE_BUSY family 5/261/517/773), **not** `e.toString()` — the string form
appends untrusted XMLTV programme titles, which could contain "database is
locked" and mask a real error.

Also fixed the stale `_runEpgRematch` docstring (it claimed it does not
re-download; it does).

`lib/backend/sql.dart` (`_isDbBusy` + `_epgWriteWithRetry` + wraps),

## Known follow-ups (not in this fix)
- The retry is the correct handling but not the deeper structural fix
  (foreground and background EPG refreshes still *can* overlap). A cross-isolate
  guard (IsolateNameServer, or the background task deferring while the app is
  foregrounded) would eliminate the overlap; deferred — retry resolves the
  user-facing failure.
- Worst-case latency under *sustained* contention is ~155s/op (5×30s +
  backoff); acceptable vs the old hard-fail, but a shorter per-attempt
  busy-timeout could bound it tighter.
