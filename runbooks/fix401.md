# fix401 — Categories search works from 2 characters (finish the fix400 gate)

**Target version:** `1.35.0+401` (from `1.34.28+400`)
**Branch:** `main` · **Base:** `cd23d7b` (v1.34.28+400)
**Toolchain (verified):** Flutter 3.44.0 / Dart 3.12.0 (CI pin), `libsqlite3.so` symlinked.

## What & why

A device log (S24, v1.34.28+400, `searchMethod=likeSubstring`) showed fix400 working for the channel search — a single character no longer scans — but the **Categories** search still effectively required **3** characters:

```
chars=2 query="en"  → Home.load: got 0 results in 0ms   ← short-circuited, not queried
chars=2 query="us"  → Home.load: got 0 results in 0ms
chars=3 query="us|" → Home.load: got 36 results in 4ms   ← real query ran
```

The `0ms` is the tell: `Sql.searchGroup` has its **own** short-skip — `if (every term length < 3) return []` — independent of fix400's UI gate. fix400 lowered the keystroke gate to ≥2, but a 2-char query then reached `searchGroup` and was dropped by this `< 3` guard, so the Categories list stayed empty until the 3rd character.

**Fix:** lower the Categories short-skip from `< 3` to `< 2`, matching fix400. The groups table is small and LIKE-searched (`%query%`), so a 2-char scan is cheap (the 3-char one was 4ms) — this guard had no trigram constraint, only a stale performance threshold. The decision is extracted to a one-line pure helper (`groupSearchAllTermsTooShort`) so it's unit-testable, mirroring fix400's `searchQueryShouldLoad`.

**Deliberately *not* changed:** the channel-search FTS path (`searchMethod == ftsAnd`/keywords) keeps its own `< 3` handling — FTS5 trigram genuinely can't match terms under 3 chars, and a bare 2-char `LIKE` over a 90k-channel source is the multi-second scan that guard exists to prevent. On `likeSubstring` (this device) the channel search already works at 2 chars; only Categories was affected.

+/// fix401: whether a Categories (group) search should skip the scan because
+/// every term is too short to be meaningful.
+///
+/// Aligned with fix400's ≥2-char UI gate: a 2-char term is allowed (the groups
+/// table is small and LIKE-searched, so a 2-char scan costs ~ms). The old
+/// threshold was < 3, which made the Categories search silently require three
+/// characters even on `likeSubstring`, where a 2-char query matches fine.
+///   • empty / whitespace-only → false (don't skip; restores the full list)
+///   • every term exactly 1 char → true (skip)
+///   • any term ≥2 chars → false (run the search)
+bool groupSearchAllTermsTooShort(String rawQuery, {bool useKeywords = false}) {
+  final q = rawQuery.trim();
+  if (q.isEmpty) return false;
+  final terms = useKeywords
+      ? q.split(RegExp(r'\s+')).where((t) => t.isNotEmpty)
+      : [q];
+  return terms.every((t) => t.length < 2);
+}
