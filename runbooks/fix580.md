# fix580 — TV player Mode B: custom focusable D-pad control overlay

**Version:** v2.1.7+580
**Date:** 2026-06-27
**Type:** TV/remote UX (player) — the working Mode B (third attempt)

## Context

fix576/577 (v2.1.3–4) shipped the verified direct-map D-pad (▲▼ channel, ◀▶ seek,
OK play-pause + reveal). fix578 (v2.1.5) tried to make media_kit's OWN bars
D-pad-navigable and FAILED on-device (their auto-hide ignores key/focus; buttons
aren't focusable) + regressed the D-pad → reverted in fix579 (v2.1.6).

fix580 does Mode B the way that actually works: a **custom, app-owned focusable
overlay** using native Flutter focus — the same path the TV browse UI already uses
(`tv_top_tab_bar.dart`). Designed + adversarially reviewed by two multi-agent
workflows; the review caught two on-device-failure bugs before ship (see below).

## Change (all `lib/player.dart` unless noted)

- **TV gating, runtime + synchronous:** `_tvMode` seeded in `initState` from
  `forceTVMode || DeviceDetector.isTvCached || !Utils.hasTouchScreenCached`.
  `isTV()`/`hasTouchScreen()` are now **memoized** (`device_detector.dart`,
  `backend/utils.dart`); main.dart resolves both at startup so the cache is ready
  before any Player — no async window, no per-call-site flag, works on
  auto-detected leanback boxes regardless of launch screen.
- **Suppress media_kit controls on TV+live:** `buildVideoView(context,
  {suppressControls})` added to the `PlayerEngine` interface + `MpvEngine` → passes
  media_kit `NoVideoControls`. `_buildVideoArea` on TV+live returns that (no
  `MaterialVideoControlsTheme`); phone/touch + VOD keep the existing path verbatim.
- **Custom overlay** (`_buildTvOverlay`): `Positioned.fill` scrim +
  `FocusTraversalGroup` of focusable `IconButton`s (top: back, name+stream-info,
  EPG, cast, pip; bottom: subtitles/audio, channel ▲▼, rewind/play-pause/fwd/live
  when DVR, aspect, mini). Play/Pause rebuilds from `engine.playingStream`. Arrows
  traverse geometrically (DirectionalFocusIntent), OK activates — no custom key code.
- **State machine:** `_navMode`. `_onPlayerKey` first line `if (_navMode) {
  _resetOverlayHideTimer(); return ignored; }`. OK (`playPauseReveal`): TV+live →
  `_openOverlay()`; phone or TV+VOD → existing `_revealControls()`. `_closeOverlay()`
  is the **SOLE writer** of `_navMode=false` and ALWAYS `_surfKeyFocus.requestFocus()`
  — the keystone vs the fix578 dead-D-pad regression. Auto-hide 8 s, reset on every
  key. Back (PopScope) closes the overlay before leaving the player.

## Two bugs the pre-ship review caught (would have repeated the on-device failure)

1. **autofocus is dropped** — `_surfKeyFocus` already holds focus in the same
   FocusScope, so the overlay's `autofocus` Play/Pause never gets focus → D-pad dead
   until auto-hide (exactly fix577/578). **Fix:** dedicated `_overlayFirstFocus`
   FocusNode + explicit `requestFocus()` in a post-frame callback in `_openOverlay`.
2. **async TV-detection race** — `isTV()` is an async platform call; `_tvMode` was
   false during a sub-second window (recurring every channel-surf via
   pushReplacement) → OK briefly took the media_kit path. **Fix:** memoize +
   synchronous cached read (above).

- **On-device (post-ship, mandatory — CI green is NOT sufficient here):**
