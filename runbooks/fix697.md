# fix697 — SR completion notification (item 2) + in-progress-delete orphan fix (item 1)

**Builder:** desktop (Mac). **Live base:** v4.1.4. **Ships as:** v4.1.5+697.

Closes the two open items from the fix693 SR-UX review (14-agent verified
workflow, 2026-07-11): item 2 (completion toast) was never built, and deleting a
still-recording row orphaned the partial clip.

## What changed

### Item 2 — recording completion notification
A recording that finishes/fails must tell the user. A Flutter SnackBar can't:
scheduled recordings complete in the foreground **service** when the app is
often backgrounded or killed. So the signal is a **native notification**.

- **`RecordingCaptureService.kt`**: new `postCompletion(id, name, ok, detail)` on
  a SEPARATE channel `free4me_recording_done` (`IMPORTANCE_DEFAULT`, audible/
  heads-up; the ongoing-capture channel is `LOW`/silent) with a distinct id
  `DONE_NOTI_ID_BASE(48000)+id`. The distinct id is load-bearing:
  `stopSelfIfIdle()` → `stopForeground(STOP_FOREGROUND_REMOVE)` clears only the
  ongoing FGS notification (`NOTI_ID_BASE 47000 +id`), so this one survives.
  Auto-cancel; tapping launches the app.
- Fired on the **done** branch and **both failed** branches (too-little-data +
  exception) and the create-failed early return — but **suppressed when
  `userCancelled`** (`cancelFlags[id]==true`): a user-initiated stop means
  they're already in the app.
- **`MainActivity.kt`**: one-time `POST_NOTIFICATIONS` runtime request in
  `onCreate` (API 33+ only; `runCatching`, non-blocking). The manifest already
  declared the permission; without the runtime grant an independent
  `mgr.notify()` is silently dropped on API 33+ (the Oppo emulator is API 36).
- **In-app twin**: `RecordingStatusJournal.drain()` now returns the terminal
  (`done`/`failed`) transitions it applied; `recordings_view._load` shows a
  SnackBar for them — but **only after the first (opening) drain** (`_firstLoad`),
  so a stale journal on screen-open doesn't spam (the native notification
  already covered those).

### Item 1 — deleting an in-progress recording no longer orphans the partial
Two root causes, both fixed:
1. **No remove option while recording**: the row's `output_path` was journaled
   `null` at `status=recording` (the MediaStore URI was only written on `done`),
   so `hasFile=false` and the 3-option dialog hid "remove file". **Fix**:
   `runCapture` now creates the MediaStore entry BEFORE the status write and
   journals `updateStatus(id, "recording", mediaUri.toString(), null)` — the row
   is trackable throughout capture.
2. **Fire-and-forget stop orphaned the file**: `stop()` returned immediately, so
   Dart deleted the row while the native loop later finalized a partial into
   MediaStore (a no-op status write on a dead row → orphan). **Fix**: native
   `deleteOnCancel` map + `EXTRA_DELETE_FILE`. `stop(id, deleteFile=true)` sets
   it; when the copy loop exits cancelled with the flag, the service deletes the
   MediaStore entry ITSELF **after** the `use{}` output stream has closed (no
   cross-isolate open-fd race), skips finalize + notification, and lets Dart drop
   the row. Dart (`recordings_view._delete`) routes a **running-row** remove
   through `RecordingCapture.stop(deleteFile: choice=='remove')` and no longer
   calls `remuxDeleteTs` for it (that would race the still-open fd); finalized
   rows still use `remuxDeleteTs`.

## Adversarial review (two passes, all findings resolved)
Two Opus-ultracode review workflows (find → adversarially verify each finding)
ran against the implementation before build:

- **Pass 1** caught a **blocker** — a duplicate `onCreate` override in
  MainActivity (a second `onCreate` already existed for the fix506 render-cap),
  which would have failed the Kotlin compile and shipped nothing. Merged the
  POST_NOTIFICATIONS request into the single existing `onCreate`. Also fixed:
  the HTTP-non-2xx failure branch was silent (no `postCompletion`) and orphaned
  its pending entry — the most common unattended failure (offline channel /
  expired token); a stale-`recording` delete (capture finished but journal not
  drained) orphaned the finalized file, and the killed-process case likewise —
  both closed by passing the URI to `stop()` so `ACTION_STOP` deletes directly
  when no live capture thread exists; a spurious completion SnackBar for a
  just-deleted row; a nit id-range aliasing; and `deleteOnCancel` reset hygiene.
- **Pass 2** (re-review of the fixes) caught a **regression I introduced**:
  persisting the URI at `status=recording` + the `COALESCE(?, output_path)`
  update meant a `failed`-after-HTTP row kept pointing at the deleted URI
  (`hasFile=true` → misleading "remove file"). Fixed by an empty-string CLEAR
  sentinel in `Sql.updateRecordingStatus` + journaling `""` on that branch. Also
  bulletproofed a deadline-vs-delete TOCTOU (the `finally` now honors a
  late-arriving delete flag) and suppressed the Stop-button completion snack
  (`_userEndedIds`).
