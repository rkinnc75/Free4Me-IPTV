**Builder:** desktop. **Live base:** v3.0.10. **Ships as:** v3.0.11+667.

**Numbering note:** originally drafted as "fix666a" before the owner corrected
the scheme — fix numbers are plain numeric, no letter suffixes. The Mac's

## What (foundation phase — no capture yet, deliberately)

Foundation only, so it's fully verifiable before the device-only capture work:
- **Migration 44**: `recordings` table (id, channel_id, channel_name, url,
  scheduled_start_utc, duration_ms, pad_before_min, pad_after_min, status,
  output_path, error, created_utc) + `idx_recordings_start`. The row stores the
  RESOLVED padded window, so changing the global pad defaults later never
  shifts an already-scheduled recording.
- **`lib/models/recording.dart`**: Recording model + RecordingStatus
  (scheduled → recording → compressing → done / failed / cancelled).
- **Sql CRUD**: insertRecording / getRecordings / getRecordingById /
  updateRecordingStatus / deleteRecording.
- **`lib/backend/recording_scheduler.dart`**: pure, unit-tested
  `computeWindow(start, stop, padBefore, padAfter)`; `scheduleForProgramme`
  (EPG path, pads applied, per-recording override params); `scheduleNow`
  (manual path, default 60 min, no after-pad); exact wall-clock alarm per
  recording via `android_alarm_manager_plus`
  (`oneShotAt(..., exact, wakeup, allowWhileIdle, rescheduleOnReboot)`, alarm
  id = recording id); `cancel`. The alarm CALLBACK IS A STUB — it flips status
  to `recording` and logs; the actual HTTP-stream capture is fix668.
- **Pad settings** (per owner decisions): `recordPadBeforeMin` default **1**
  (0–15) and `recordPadAfterMin` default **1** (0–240), threaded through the 5
  layers + backup round-trip. Per-recording overrides are stored on the row
  (e.g. after=90 for a live event). Manual "record now" defaults 60 min.
- **Manifest**: SCHEDULE_EXACT_ALARM / USE_EXACT_ALARM /
  RECEIVE_BOOT_COMPLETED.
- **main.dart**: `RecordingScheduler.init()` after settings load (Android-only
  no-op elsewhere, failure-safe).

No user-facing UI yet (fix669); the changelog entry says capture + UI arrive in
the next updates.

+/// fix667: a DVR recording — a scheduled, active, or finished capture of a
+/// channel's HTTP stream to a file.
+///
+/// The stored window ([scheduledStartUtc] + [durationMs]) is ALREADY padded:
+/// pads are folded in at schedule time so later changing the global pad
+/// defaults never retroactively shifts an existing recording. [padBeforeMin] /
+/// [padAfterMin] are retained for display and per-recording editing.
+enum RecordingStatus {
+  scheduled,
+  recording,
+  compressing,
+  done,
+  failed,
+  cancelled;
+
+  static RecordingStatus fromName(String? s) {
+    for (final v in RecordingStatus.values) {
+      if (v.name == s) return v;
+    }
+    return RecordingStatus.scheduled;
+  }
+}

+class Recording {
+  final int? id;
+  final int? channelId;
+  final String channelName;
+  final String url;
+
+  /// Padded start (epoch seconds) — programme start minus the before-pad.
+  final int scheduledStartUtc;
+
+  /// Padded total duration in milliseconds — programme length plus both pads
+  /// (or the manual duration for a "record now").
+  final int durationMs;
+
+  final int padBeforeMin;
+  final int padAfterMin;
+
+  final RecordingStatus status;
+  final String? outputPath;
+  final String? error;
+  final int createdUtc;
+
+  const Recording({
+    this.id,
+    this.channelId,
+    required this.channelName,
+    required this.url,
+    required this.scheduledStartUtc,
+    required this.durationMs,
+    this.padBeforeMin = 0,
+    this.padAfterMin = 0,
+    this.status = RecordingStatus.scheduled,
+    this.outputPath,
+    this.error,
+    required this.createdUtc,
+  });
+
+  DateTime get startTime =>
+      DateTime.fromMillisecondsSinceEpoch(scheduledStartUtc * 1000);
+  DateTime get endTime =>
+      startTime.add(Duration(milliseconds: durationMs));

+  Recording copyWith({
