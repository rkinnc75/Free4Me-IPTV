# fix641 — Show Xtream subscription expiry on the edit screen + auto-disable expired sources

**Target:** v2.2.41+630 (diff against live tag **v2.2.40**, verified `git ls-remote --tags`)
**Apply check:** `git apply --check` PASSED on a fresh v2.2.40 clone.
**Analyze:** run on the build machine — gate not asserted here. (Manual: braces balanced; xtream/sql paren additions balanced over each file's pre-existing 1-off comment artifact.)

> Re-targeted from an earlier draft: a build shipped in between (v2.2.40 = fix629, which added its own **migration 41** dropping dead channel indexes). This fix now targets v2.2.40 and its schema migration is **42** (not 41). No functional change from the earlier draft — same feature, renumbered.

## What this adds
`player_api.php` user_info (already fetched every Xtream refresh for `max_connections`) also carries `exp_date` and `status`. This captures them, stores them, shows the expiry on the source edit screen, and auto-disables an expired/banned line on refresh. No new network call.

## Behavior
- **Capture:** `fetchXtreamAccountInfo` reads `max_connections` + `exp_date` + `status` from one user_info response (replaces `fetchXtreamMaxConnections` at the call site; the old fn is retained, still referenced elsewhere).
- **Store:** new `sources.exp_date` (INTEGER epoch) + `sources.status` (TEXT), **migration 42**. Persisted via `updateSource` (refresh) and the import/create upsert. `rowToSource` maps cols 17/18 (source column order unchanged by fix629, which only dropped channel indexes).
- **Auto-disable:** after the fetch, `isSourceExpired(source)` → disable if `status ∈ {expired, banned, disabled}` OR (`exp_date` present, non-zero, `< now UTC`). A **null/0 exp_date is never expired** (lifetime lines). Persisted via `setSourceEnabled(false, id)` (updateSource intentionally doesn't write `enabled`). **Just-fetched channels are kept** — only the source is disabled — so the user can still see stale content and re-enable after renewing. A renewed+re-enabled source reports a future date on next refresh and stays enabled.
- **Display:** edit screen "Expires" row (Xtream only; hidden for M3U), red when expired, "Unknown" when not reported.

## Scope notes
- Export/import (`settings_io`) intentionally NOT extended — exp_date/status are refresh-derived and repopulate on next refresh; carrying a stale "expired" through import could wrongly disable a re-imported source.
- Timezone: comparison is UTC; display converts to local.

## Files
- `lib/models/source.dart` — `expDate` + `status` fields.
- `lib/backend/db_factory.dart` — **migration 42** (exp_date, status columns).
- `lib/backend/xtream.dart` — `fetchXtreamAccountInfo` + `isSourceExpired`; capture + auto-disable in getXtream.
- `lib/backend/sql.dart` — upsert + `updateSourceSql` + `rowToSource` cols 17/18.
- `lib/edit_dialog.dart` — "Expires" info row (Xtream only) + `_expiryText`; `_infoRow` gains optional color.
- `pubspec.yaml`, `lib/whats_new_modal.dart` — version bump + changelog.

## Validation plan (build machine)
1. `flutter analyze` → expect "No issues found!".
2. Refresh an Xtream source with a known expiry: log shows `account info … exp_date=… status=…`, edit screen "Expires" shows the correct local date.
3. M3U source: NO "Expires" row.
4. Expired account (or set a past `exp_date` in the DB and refresh, or a provider returning status=Expired): source auto-disables, log shows `auto-disabled — … expired subscription`, channels remain.
5. Lifetime/no-date account (exp_date null/0, status Active): NOT disabled, shows "Unknown".
6. Re-enable a renewed source + refresh: stays enabled, shows the new date.
7. Migration: upgrade an existing install (which already ran shipped migration 41); confirm 42 runs, no crash, columns populate on next refresh.
