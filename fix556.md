# fix556 — EPG "what's on" search: force FTS-first plan (CROSS JOIN), kills 30–120s lag

**Version:** 2.0.56 (`2.0.56+556`)
**Base:** v2.0.55 (live CI tag). Single ```diff block, byte-exact against v2.0.55.
Verified clean on a fresh v2.0.55 clone.
**Branch from:** v2.0.55.

---

## Why (diagnosed via fix554 instrumentation + real EPG seed)

fix554's wall-clock log isolated the lag to `Sql.searchPrograms`, not channel
search:

| query | total | name | **epgProg** |
|---|---|---|---|
| YES Network | 119,655ms | 15ms | **119,639ms** |
| World Cup | 85,654ms | 51ms | **85,584ms** |
| family | 68,067ms | 41ms | **68,015ms** |

`EXPLAIN QUERY PLAN` on the 1.1M-row EPG seed showed the planner driving off
`idx_programs_time_range (source_id, start_utc)` and doing **one random FTS probe
per in-window programme** (hundreds of them). On the Onn's eMMC that's hundreds
of random seeks into the 1.1M-row FTS index — 30–120s of random I/O.

## Fix

Change the join in `searchPrograms` from `INNER JOIN` to **`CROSS JOIN`**, which
forces SQLite to make the **FTS virtual table the outer driver**: scan one term's
posting list sequentially, resolve rowids by PK, filter the time window, sort
≤200. Measured on the real EPG seed:

| query | before plan | after (CROSS JOIN) |
|---|---|---|
| family | 0.39s (SSD; far worse on eMMC) | **15ms** |
| World Cup | — | **14ms** |
| news | — | **40ms** |
| Yankees | — | **4ms** |

Result rows verified **identical** to the original query across all test terms.
Pure plan change — no schema, no migration, no new index.

Run the same searches (YES Network, World Cup, family). The fix554 log line
