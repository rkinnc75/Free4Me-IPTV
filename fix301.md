## Root cause (confirmed from 3 provider dumps + the in-app counts)
trex's Xtream API caps the unfiltered `get_vod_streams` call at 3775 movies even though it lists 364 VOD categories holding ~190K movies. The app loaded exactly what was returned (in-app counts matched the dump byte-for-byte: Live 49036 / Movies 3775 / Series 40204). emjay (172009 movies) and dino (102790) return their full catalogs in one call, so they were never affected. This is a provider-side limit, worked around app-side.

## The fix
After the bulk VOD fetch, if the movie count looks truncated relative to the VOD category count, refetch `get_vod_streams&category_id=N` for each category and merge, deduped by `stream_id`. Single-call providers skip the fallback (heuristic doesn't fire).

- Heuristic `_vodLooksTruncated(movies, cats)`: fires only when `cats > 20` AND `movies < cats * 15` (avg < 15/category).
- `getXtreamHttpData` already accepts `extraQueryParams`, so the per-category call is `{'category_id': id}`.
- Best-effort: a failed/ malformed category fetch is skipped; whatever was gathered is kept.

## Verified before delivery

- Heuristic tested on the real numbers: trex (3775/364) → fires; emjay (172009/432) → no; dino (102790) → no; small provider → no. Dedup by stream_id verified.

## Note
- The fallback issues up to ~N category requests (364 for trex) on refresh for affected providers only — slower refresh for them, unchanged for others.
- No schema change, no migration.
