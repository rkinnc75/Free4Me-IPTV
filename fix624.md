# fix624 — "Cap 60→30 fps" setting works on any device (drop the low-RAM gate)

**Target:** v2.2.35+624 (diff against live tag **v2.2.34**, verified `git ls-remote --tags`)
**Apply check:** `git apply --check` PASSED on a fresh v2.2.34 clone.
**Analyze:** run on the build machine — gate not asserted here. (Manual: braces balanced; help-text newline escaping matches the original's existing form.)

## Why
The `devCapFpsLowRam` toggle required BOTH the flag AND `DeviceDetector.isLowRamDevice()`, so on a phone / higher-RAM device enabling it did nothing (confirmed: `capFps30=false` on the 12GB SM-S938U even with the option on). The `fps` filter works on any device; the low-RAM restriction was a policy choice, not a technical need. Per request, the flag should force 30 fps whenever enabled, regardless of RAM.

## Fix
- `capFps` now = `!(previewMode && !multiViewMode) && s.devCapFpsLowRam` — the `isLowRamDevice()` condition is removed. The mini-player PiP is still excluded (unchanged); full player + multi-view cells honor it on any device.
- Updated the setting's title ("Cap 60→30 fps") and help text to say it applies to any device when enabled (was "low-RAM only").

## Note
Applied at stream open — reopen the stream / multi-view after toggling to see `capFps30=true` and `vfFps≈30`.

## Files
- `lib/player/mpv_engine.dart` — drop the low-RAM gate on the cap.
- `lib/settings_view.dart` — title + help text updated.
- `pubspec.yaml`, `lib/whats_new_modal.dart` — version bump + changelog.

## Validation plan (build machine)
1. `flutter analyze` → expect "No issues found!".
2. On the phone (non-low-RAM): enable "Cap 60→30 fps", reopen a 60 fps stream, confirm the log shows `capFps30=true` and the stats overlay shows `vfFps≈30` (was 59.94).
3. Toggle OFF → confirm `capFps30=false` and full 59.94 fps returns.
4. Regression: mini-player PiP still does NOT cap (`previewMode=true multiViewMode=false` → no fps filter).
5. onn still behaves as before when enabled (now via the flag alone, not the RAM check).
