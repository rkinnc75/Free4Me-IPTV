# fix620 — Pre-flight FTS heal (stops the corrupt-index hang) + Cancel button

**Target:** v2.2.31+620 (diff against live tag **v2.2.30**, verified `git ls-remote --tags`)
**Apply check:** `git apply --check` PASSED on a fresh v2.2.30 clone.
**Analyze:** run on the build machine — gate not asserted here. (Manual: braces balanced; dialog paren delta +14/+14 over original.)

## Why fix619 wasn't enough (from log `…125858`)
fix619 added code-267 recovery in the refresh dialog's CATCH block. But the corrupt FTS makes the refresh **HANG, not throw** — `DELETE FROM channels_fts` blocks inside the synchronous SQLite call on the MAIN isolate (the work runs there even under the foreground service). A hang never reaches the catch, so recovery never fired, and a Cancel button can't be tapped while the isolate is blocked. The single-source "Emjay" refresh also hung — and it never went through the dialog's recovery at all.

## Fix
1. **Pre-flight FTS heal — the real fix.** Both `refreshAllSources` and the single-source `refreshSource` now call `Sql.ensureFtsHealthy()` FIRST: run FTS5's `integrity-check` (cheap — ~1s on the 857K-row catalog, measured), and if it fails, `rebuildFtsTableFromScratch()` BEFORE anything deletes from the index. A healthy index can't hang the delete, so the refresh proceeds normally. This repairs the existing corrupted install on the next refresh — no clear-data needed.
2. **Cancel button** (requested). Shown while refreshing. On tap it flags cooperative cancellation (checked between sources — the loop is responsive at its await points now that the hang is gone), repairs the FTS index so a half-done refresh doesn't leave a broken search index, and closes the dialog. Honest limit: cooperative cancellation can't interrupt a truly synchronous call, but the pre-flight heal removes the one operation that was blocking.

## Files
- `lib/backend/sql.dart` — `ensureFtsHealthy()` (integrity-check → rebuild on failure).
- `lib/backend/utils.dart` — pre-flight heal in `refreshAllSources` + `refreshSource`; `shouldCancel` param + between-source check.
- `lib/widgets/sources_refresh_dialog.dart` — Cancel button (flags cancel, repairs FTS, closes); passes `shouldCancel`.
- `pubspec.yaml`, `lib/whats_new_modal.dart` — version bump + changelog.

## Validation plan (build machine)
1. `flutter analyze` → expect "No issues found!".
2. On the corrupted sms938u: tap the all-sources ↻. Confirm the log shows `Sql.ensureFtsHealthy: FTS integrity-check failed … rebuilding`, then the refresh PROCEEDS (no hang at "Preparing…"), channels populate, search works.
3. Single-source refresh on the corrupted DB: same pre-flight heal, no hang.
4. Cancel button: start a refresh, tap Cancel mid-run → confirm it stops, status shows "repairing search index", dialog closes via OK, and search still works (index intact).
5. Clean install: confirm pre-flight integrity-check passes silently (no rebuild) and refresh is normal.
