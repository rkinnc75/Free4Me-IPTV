# fix198 — Multi-view ceiling should sum across enabled sources, not take the minimum

> **Version:** build `+198`; patch = next available at ship time (read live pubspec — patch and fix numbers drift; do NOT assume). Tag = next available `v1.23.X` (immutable; never reuse/delete/recreate).
> **Evidence:** With Aniel3000 (4 connections) and Emjay (1 connection) both enabled, the multi-view layout dialog shows "This provider allows only 1 connection. Multi-view needs at least 2 simultaneous streams." and blocks all layouts — even though 5 connections are available in total.
> **Bug:** `_connectionCeiling` returns the MINIMUM `maxConnections` across enabled sources, so the smallest source (Emjay = 1) caps the whole multi-view gate. The user has 4 + 1 = 5 total and should be allowed 1×2 (needs 2) and 2×2 (needs 4).
> **Root cause:** Multi-view cells can each be filled from a different source, so the true ceiling on simultaneous streams is the SUM of per-source connection limits, not the minimum. `_connectionCeiling` used `reduce((a, b) => a < b ? a : b)` (min).

## What the evidence proves

Each multi-view cell consumes one connection on its source; cells can draw from different sources. So total simultaneous capacity = sum of per-source limits. With Aniel3000=4 and Emjay=1, the total is 5 — enough for both 1×2 (2 cells) and 2×2 (4 cells). The min-based gate incorrectly let Emjay's single connection veto multi-view entirely. (Per-source over-allocation — e.g. putting two Emjay channels in a 2×2 — is a separate concern handled by hiding at-budget sources in the channel picker; see Notes. This runbook only fixes the layout gate.)

## Fix 198.1 — sum the per-source limits

**File:** `lib/multi_view_picker_dialog.dart`

### Current code (verbatim — match exactly)
```dart
  /// Minimum known maxConnections across enabled sources (null = unknown).
  static int? _connectionCeiling(List<Source> enabled) {
    final known = enabled.map((s) => s.maxConnections).whereType<int>().toList();
    if (known.isEmpty) return null;
    return known.reduce((a, b) => a < b ? a : b);
  }
```
### Replacement code (verbatim)
```dart
  /// fix198: SUM of known maxConnections across enabled sources (null =
  /// none known). Multi-view cells can be filled from different sources, so
  /// the total simultaneous-stream ceiling is the sum of per-source limits,
  /// not the minimum. (e.g. a 4-connection source + a 1-connection source =
  /// 5 total, enough for 1×2 or 2×2.) Per-source over-allocation is prevented
  /// separately in the channel picker, not here.
  static int? _connectionCeiling(List<Source> enabled) {
    final known = enabled.map((s) => s.maxConnections).whereType<int>().toList();
    if (known.isEmpty) return null;
    return known.reduce((a, b) => a + b);
  }
```

### Key changes
- `reduce((a, b) => a < b ? a : b)` (min) → `reduce((a, b) => a + b)` (sum). Null/unknown handling unchanged.

## Notes
- **Per-source over-allocation is NOT addressed here.** With the gate opened to the sum, a user could still assign more cells to one source than it allows (e.g. two Emjay cells in a 2×2), and the extra cell would fail to load (handled by the cell's existing retry budget). The agreed follow-up is: in the multi-view channel picker (`ChannelPickerScreen`), as cells are assigned, track per-source usage and **make that source's channels unselectable — hidden entirely — once its connection budget is used up**. The source itself is never disabled (it stays enabled everywhere else); only its channels drop out of the picker, and they reappear if a cell using that source is removed (live running count). That is a larger multi-file change (`multi_view_screen.dart` must pass per-source remaining budget through `multi_view_cell.dart` to `channel_picker_screen.dart`) and is scoped as its own future runbook.
- Independent of fixes 188/190/192/194/196. Touches only `lib/multi_view_picker_dialog.dart`.
