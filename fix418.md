# fix418 — diagnostic: log query plan + data shape for slow queries

**Version:** 1.35.16+417 → **1.35.17+418**
**Type:** Diagnostic only (no behaviour change). Text patch, 1 file (+ version/changelog).
**Tag:** v1.35.17
**Base:** current origin/main (1.35.16+417). Applies directly.

## Why
Issue #2 (Snapdragon 765G) and Rich's S24 repro both show category / VOD / Series browse running 400–3485 ms for ~36 rows, while Live browse is 8–11 ms. To fix it correctly we need the **actual query plan** for the slow statements on a real large playlist — and that can't be captured with `adb run-as` on a release build. This adds the missing visibility to the **existing** SLOW-SQL logger so it can be captured from a normal exported log.

## What it does (all gated behind debug logging, like every other diagnostic)
In `lib/backend/timed_db.dart`, whenever a statement exceeds the slow threshold (1000 ms) and is already being logged as `SLOW SQL …`:
1. **`fix418 PLAN: …`** — logs `EXPLAIN QUERY PLAN` for that exact statement + params (reusing the fix222 extraction). EXPLAIN runs the planner only — **no row execution**, so it never re-runs a slow DELETE/UPDATE.
2. **`fix418 stats …`** — the first time a slow **browse** query (`FROM channels c`) is seen in a session, dumps a one-time per-source row shape: counts by `media_type`, plus `cat_enabled` / `favorite` / `stream_validated` / `is_adult` / `is_divider` totals, category count, and the 5 biggest categories. This is exactly what's needed to build a faithful benchmark seed.

No production code path changes; when debug logging is off, the wrapper is unchanged (single Stopwatch, no logging). `flutter analyze` clean, 166 tests pass.

## How to capture (Rich)
1. Apply fix418, build, install.
2. **Diagnostic logging ON.**
3. Isolate the heavy provider (disable other sources), **force-stop**, reopen.
4. Open **VOD (movie)** and a **category** — reproduce the slow load (you already did this).
5. **Export the log** and send it. I'll have the `fix418 PLAN:` lines (which index the planner picks for the slow cases) and the `fix418 stats` lines (data shape) — enough to design the real fix and benchmark before/after on a matching seed.
