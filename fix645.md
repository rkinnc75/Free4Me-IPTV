# fix645 — Live TV rail refreshes after category toggles (Option A dirty-flag)

**Target tag:** `v2.2.44` (commit `5110590975be672af41a88ede6560279c5938e51`)
**New version:** `2.2.45+634`
**Migration:** none

## Summary

Toggling categories (Categories tab -> back to Live TV) never refreshed the
guide's category rail. Root cause: fix610's keep-your-place optimization —
`reloadGuide()` only re-ran `_init()` when the enabled-SOURCE set changed;
category enable/disable doesn't touch sources, so the kept-alive rail stayed
stale. (Movies/Series/Categories tabs rebuild under a fresh `_reloadGen` key on
every tab select, so only the guide was affected.)

Fix (agreed Option A — generation counter at the Sql choke point, so EVERY
toggle surface is covered with no widget coupling):

- `Sql.groupsGen` — a static monotonic counter bumped by `setGroupEnabled`
  (single toggle), `setAllGroupsEnabled` (bulk Select/Unselect-All, which
  `setAllGroupsEnabledForSearch` routes through), and `favoriteGroup`
  (favorites sort to the top of the rail, fix308 — order is part of the list).
- The guide snapshots the counter at the START of `_init()` (a toggle landing
  mid-init differs from the snapshot and triggers another rebuild next switch,
  instead of being lost) and `reloadGuide()` now rebuilds when the snapshot is
  stale — alongside the existing source-set check. The fix610 keep-your-place
  path is unchanged for plain tab switches with no category changes.

## On-device test notes (onn)

1. Live TV -> note the rail. Categories tab -> disable a live category ->
   back to Live TV: the category is gone from the rail (and re-appears after
   re-enabling). Same for Select/Unselect All.
2. Favorite a category in the phone Home / Categories view -> Live TV rail
   re-sorts it to the top.
3. Tab-switch WITHOUT touching categories: guide keeps your place exactly as
   before (fix610 behaviour unchanged).
