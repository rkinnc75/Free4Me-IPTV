# fix186.md — fix184 build break: `fetchXtreamMaxConnections` is CALLED but never DEFINED

> **Version:** Free4Me-IPTV 1.23.15+184 → ships **1.23.16+186**.
> **Evidence:** `flutter analyze --no-fatal-infos`, exit 1:
> ```
> error • The function 'fetchXtreamMaxConnections' isn't defined
>   • lib/backend/xtream.dart:131:20 • undefined_function
> ```
> **Cause:** fix184.2 had the build machine add the CALL to
> `fetchXtreamMaxConnections(source)` (xtream.dart:131) but the function
> DEFINITION never landed — `grep` confirms the name appears EXACTLY ONCE in the
> whole codebase, at the call site. The fix184 runbook wrote the definition with
> a "verify the base URL / if not, do Y" hedge, so the definition block was not
> applied cleanly. This adds the missing definition, hedge resolved against the
> actual `buildXtreamUrl`/`getXtreamHttpData` signatures.
>
> The two `use_build_context_synchronously` INFOs (~2143 / ~2731) are the
> long-standing tolerated ones — leave them.

## Verified facts (from actual source)

- **The call exists, the definition does not** (`xtream.dart:131`):
  ```dart
  final mc = await fetchXtreamMaxConnections(source);
  if (mc != null && source.id != null) {
    source.maxConnections = mc;
    await Sql.updateSource(source);
  }
  ```
  So `source.maxConnections` (fix184.1) and `Sql.updateSource` are already wired
  — ONLY the function body is missing.

- **Base auth call resolves the fix184 hedge:** `buildXtreamUrl(source, action,
  …)` ALWAYS sets `'action': action`. Passing `action: ''` yields `…&action=`
  (empty), which Xtream `player_api.php` treats as the auth call and returns the
  `user_info` block (with `max_connections`). So `getXtreamHttpData('', source)`
  is correct — no need for any special base-URL helper.

- `AppLog` and `Source` are already imported in `xtream.dart` (lines 3, 8).
  No new imports.

## fix186.1 — Add the missing `fetchXtreamMaxConnections` definition

**File:** `lib/backend/xtream.dart`

Add this **top-level function** (place it immediately AFTER the
`getXtreamHttpData` function so it sits beside the related HTTP helpers — any
top-level position in the file works, as long as it's not nested inside another
function):

```dart
/// fix184/186: fetch the Xtream account's max_connections from the user_info
/// block returned by the base player_api.php auth call. Returns null on any
/// failure (unknown → caller leaves the source's limit unset).
/// max_connections is typically a STRING in user_info, so parse defensively.
Future<int?> fetchXtreamMaxConnections(Source source) async {
  try {
    // Empty action => base player_api.php?username=&password=&action= , which
    // Xtream panels answer with { user_info: {...}, server_info: {...} }.
    final data = await getXtreamHttpData('', source);
    if (data is Map && data['user_info'] is Map) {
      final raw = data['user_info']['max_connections'];
      final n = raw is int ? raw : int.tryParse(raw?.toString() ?? '');
      if (n != null && n > 0) {
        AppLog.info(
            'Xtream: source "${source.name}" max_connections=$n');
        return n;
      }
    }
  } catch (e) {
    AppLog.warn(
        'Xtream: max_connections fetch failed for "${source.name}" — $e');
  }
  return null;
}
```

### Why this is exact (no hedge)
- `getXtreamHttpData('', source)` — confirmed signature `(String action, Source
  source, [Map?])`; empty `action` produces the base auth URL via
  `buildXtreamUrl` (which always injects `action`). Returns decoded JSON or null.
- `data['user_info']['max_connections']` — standard Xtream field; comes back as
  a String on most panels, int on some → `int.tryParse(raw?.toString())` handles
  both.
- Null/failure → returns null; the existing call site already guards
  `if (mc != null && source.id != null)`, so a null is safely ignored.
