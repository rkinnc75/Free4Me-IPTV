# fix130.md — Post-swap black screen: ROOT CAUSE = media_kit's hidden fullscreen ROUTE desyncs with the app's navigator on swap. Confirmed from media_kit + media_kit_video source.

> **Version:** Free4Me-IPTV 1.22.10 build=126 (verified against actual source).
> **Evidence:** `free4me_log_1780074094173.txt` (1.22.10+126) **+ direct reading
> of the media_kit 1.2.6 AND media_kit_video 2.0.1 source.**
>
> **The root cause is now confirmed end-to-end from upstream source — and it is
> NOT primarily the texture (the previous theory). It is a hidden ROUTE.**
>
> `MpvEngine.handlesOwnFullscreen == true`, so the app delegates fullscreen to
> media_kit_video. `MpvEngine.open()` with `fullscreenOnOpen == true` (the
> DEFAULT for full-screen players) calls media_kit's `enterFullscreen()`, which
> **pushes its OWN route onto the ROOT navigator**
> (`Navigator.of(context, rootNavigator: true).push(PageRouteBuilder(... a
> second Video ...))`) — the SAME root navigator the app's Player routes and the
> swap's `pop+push` operate on. **media_kit's push/pop is never logged by
> AppLog, which is why three prior fixes never saw it.**
>
> So a normally-opened full-screen mpv channel is TWO routes on the stack:
> `[Home, Player, mediaKitFullscreen]`, each with its own `PopScope`. The app's
> `onExit` and the swap's `pop+push` assume ONE route per Player. On swap the
> counts desync: `pop+push` removes one route, leaving either a stale Player or
> an **orphaned media_kit fullscreen route whose Video keeps rendering the
> engine's texture → a black layer over Home, dead back button, force-close.**

## What the source proves (both packages, quoted)

**media_kit_video 2.0.1 — `enterFullscreen` pushes a root-navigator route:**
```
// media_kit_video_controls/src/controls/methods/fullscreen.dart
Future<void> enterFullscreen(BuildContext context) {
  …
  Navigator.of(context, rootNavigator: true).push(        // ← ROOT navigator push
    PageRouteBuilder(pageBuilder: (_, __, ___) => Material(
      child: … Video(controller: controllerValue, …)       // ← a SECOND Video on the same controller
    ), transitionDuration: Duration.zero, …),
  );
}
Future<void> exitFullscreen(BuildContext context) {
  …
  await Navigator.of(context).maybePop();                  // ← pops that route
}
```
And the fullscreen route wraps itself in a `PopScope`:
```
// fullscreen_inherited_widget.dart
PopScope(onPopInvokedWithResult: (_, __) { onExitFullscreen(context)?.call(); }, child: child)
```

**media_kit 1.2.6 — `Player.dispose()` DOES release the texture (so the texture
is NOT the leak):**
```
// native/player/real.dart dispose():
await super.dispose();                       // drains release[] (below), THEN…
Future.delayed(const Duration(seconds: 5), () => mpv.mpv_terminate_destroy(ctx));  // native lingers 5s

// platform_player.dart base dispose():
for (final callback in release) { await callback.call(); }   // ← runs VideoOutputManager.Dispose
```
The Android texture teardown is registered on `release` and therefore runs
during `_player.dispose()`. **The texture is freed by dispose.** Combined with
the log (fix126 logged `released video + player`, yet still black), this
*rules the texture out* as the primary cause and points squarely at the
orphaned fullscreen ROUTE's still-mounted `Video`.

**media_kit 1.2.6 — `setProperty` is a safe direct mpv call (efficiency
review):**
```
// native/player/real.dart setProperty(): after init-wait,
mpv.mpv_set_property_string(ctx, name, data);   // plain, no internal lock vs option mgr
```
→ The app's `np.setProperty(...)` tuning calls (`hwdec`, `cache`, `cache-secs`,
`demuxer-max-bytes`, `force-seekable`, …) are the intended pattern and are
fine. **Only fix126's `vid=no` is problematic** — it's not a tuning option, it
disables video output and collides with media_kit's own `--vid/--vo/--wid`
surface state machine (driven in `android_video_controller`'s `widListener`).

## Why it's the swap specifically (the desync, step by step)

```
Open Disney full-screen:  ChannelTile.play → Navigator.push(Player(Disney))
                          → MpvEngine.open(fullscreenOnOpen=true) → enterFullscreen()
                          → root.push(mediaKitFullscreen)
   STACK: [Home, Player(Disney), mkFS(Disney)]        ← TWO routes (mkFS invisible in logs)

Swap (Disney↔ESPN):       _swap pop+push:
                            _nav.pop();   ← removes TOP route = mkFS(Disney)
                            _nav.push(Player(ESPN, adopt));
                          ESPN adopted → open() SKIPPED → NO enterFullscreen → NO mkFS(ESPN)
   STACK: [Home, Player(Disney), Player(ESPN)]         ← STALE Player(Disney) trapped underneath!
   (or, if pop targets Player(Disney) first depending on timing, mkFS(Disney) is orphaned —
    its Video keeps rendering the controller texture as a black full-screen layer)

Close mini, then back on full-screen ESPN:
                          onExit pops Player(ESPN) → reveals the STALE Player(Disney)
                          (whose engine was detached/disposed) OR the orphaned mkFS layer
   RESULT: black screen, dead back (the trapped route's PopScope eats it), force-close.
```

This matches every symptom and every log: clean `onExit`, repaints firing,
texture released — yet black, because an **extra route** (that the app never
accounted for) is on the stack.

# Fix 130.1 — Stop using media_kit's fullscreen route; let the app own fullscreen (THE fix)

**File:** `lib/player/mpv_engine.dart`

The app already has its own system-fullscreen mechanism (`_enterSystemFullscreen`
in player.dart, used for the ExoPlayer path where `handlesOwnFullscreen ==
false`). The mpv path delegating to media_kit's route-pushing fullscreen is what
creates the hidden second route. **Make the mpv engine NOT handle its own
fullscreen**, so the app drives fullscreen the same single-route way it does for
ExoPlayer — one route per Player, no hidden media_kit route, swap math correct.

**Change `handlesOwnFullscreen` to false and make enter/exit/​isFullscreen
no-ops** (mirroring `ExoEngine`):

**Current (lines 133, 254–265):**
```dart
    if (fullscreenOnOpen) await _videoKey.currentState?.enterFullscreen();
…
  @override
  bool get handlesOwnFullscreen => true;

  @override
  Future<void> enterFullscreen() async {
    await _videoKey.currentState?.enterFullscreen();
  }

  @override
  Future<void> exitFullscreen() async {
    await _videoKey.currentState?.exitFullscreen();
  }

  @override
  bool get isFullscreen => _videoKey.currentState?.isFullscreen() ?? false;
```

**Replace with:**
```dart
    // fix130: do NOT call media_kit's enterFullscreen() — it pushes a SECOND
    // route onto the ROOT navigator (a whole extra Video on this controller),
    // which the app's own navigation (Player routes, swap pop+push, onExit)
    // does not account for. On swap the route counts desync and an orphaned
    // fullscreen route's Video renders the texture as a black layer over Home.
    // The app drives fullscreen itself (immersive system UI + landscape) via
    // _enterSystemFullscreen(), exactly as it does for the ExoPlayer path.
    // (fullscreenOnOpen retained on the engine for API compatibility but no
    // longer triggers a media_kit route.)
…
  // fix130: the mpv path no longer uses media_kit's route-based fullscreen.
  // handlesOwnFullscreen=false makes the Player call _enterSystemFullscreen()
  // (immersive + orientation) — one route per Player, swap math stays correct.
  @override
  bool get handlesOwnFullscreen => false;

  @override
  Future<void> enterFullscreen() async {}

  @override
  Future<void> exitFullscreen() async {}

  @override
  bool get isFullscreen => false;
```

> Delete the `if (fullscreenOnOpen) await _videoKey.currentState?.enterFullscreen();`
> line at 133 entirely (the engine no longer enters media_kit fullscreen on
> open). `fullscreenOnOpen` may remain as a field; it simply has no media_kit
> effect now. (Optional follow-up: remove the field in a later cleanup pass.)

### What this changes in the app (already wired)

`player.dart` already branches on `handlesOwnFullscreen`:
- line 261 / 685: `if (!_engine.handlesOwnFullscreen) await _enterSystemFullscreen();`
  → now RUNS for mpv (immersive + landscape), giving real fullscreen with ONE
  route.
- line 1100: `if (_engine.handlesOwnFullscreen && _engine.isFullscreen) await
  _engine.exitFullscreen();` → now SKIPPED for mpv (no media_kit route to exit).

No other call sites. `_videoKey` is no longer used for fullscreen (it remains
only as the `Video` widget key).

# Fix 130.2 — Remove fix126's surface pokes + revert the unsafe controller/key churn

**File:** `lib/player/mpv_engine.dart`

Source proof: (a) `vid=no` fights media_kit's surface machine; (b) one texture
per Player handle (`_controllers[handle]` in android_video_controller), so
recreating the controller orphans the native texture; (c) the texture is freed
by `Player.dispose()` anyway. Remove the fix126 manual teardown and the
key/controller recreation.

**In `dispose()`, replace the fix126 block (exitFullscreen + vid=no + ordering
comments) with the minimal correct teardown:**
```dart
    // fix130: no manual exitFullscreen()/vid=no here. media_kit releases the
    // texture via Player.dispose()'s release[] callbacks (VideoOutputManager.
    // Dispose, keyed by player handle — confirmed in media_kit source). The
    // fix126 pokes fought media_kit's --vid/--vo/--wid machine. Just cancel
    // our mirror-stream subs and dispose the player.
    for (final s in _subs) {
      await s.cancel();
    }
    await _bufferingCtrl.close();
    await _completedCtrl.close();
    await _errorCtrl.close();
    await _positionCtrl.close();
    await _player.dispose();
    AppLog.info('MpvEngine: dispose() player disposed (surface released by media_kit)'
        ' eid=${identityHashCode(this)}');
```

**Restore `_controller` to `late final` and `_videoKey` to a stable final key;
make `onAdopt()` a logged no-op:**
```dart
  late final mkvideo.VideoController _controller =
      mkvideo.VideoController(_player);
  // fix130: STABLE key (fix126's per-adopt recreation didn't help — texture is
  // owned by the controller/handle, not the VideoState).
  final GlobalKey<mkvideo.VideoState> _videoKey =
      GlobalKey<mkvideo.VideoState>();
…
  /// fix130: NO-OP (was fix126.2). One texture per Player handle; recreating
  /// the controller orphans the native texture. Adoption keeps the same
  /// engine/player/controller/texture. Retained for the call site + tracing.
  void onAdopt() {
    AppLog.info('MpvEngine: onAdopt (no-op; same controller/texture)'
        ' eid=${identityHashCode(this)}');
  }
```

# Fix 130.3 — Defensive: unmount the Texture on exit (covers any residual second-Video case)

**File:** `lib/player.dart`

Even with 130.1 removing the hidden route, unmount the `Video` (Texture) the
instant exit begins, so no stale frame can ever composite during teardown. This
is the cheap belt-and-suspenders matching what a normal route-unmount gives.

**In `_buildVideoArea()` (first line):**

      return const ColoredBox(color: Colors.black, child: SizedBox.expand());
    }

  }

**In `onExit()`, as the FIRST action after the guards + START log (before
`final navigator = Navigator.of(context)`):**
```dart

```

Everything else in `onExit` (fix118 pop-first, fix124 forced repaint, unawaited
engine dispose) stays exactly as-is.

## Notes for the implementer

- **Confirmed from BOTH packages' source, not inferred.** media_kit_video:
  `enterFullscreen` pushes a root-navigator route with a second Video.
  media_kit: `Player.dispose()` drains `release[]` (frees the texture) and
  `setProperty` is a safe direct mpv call. The hidden route is the cause; the
  texture is not.
- **130.1 is the fix.** Switching mpv to `handlesOwnFullscreen=false` removes
  the hidden second route entirely and routes fullscreen through the app's own
  one-route `_enterSystemFullscreen`, which the codebase already implements and
  branches on. **Verify fullscreen still looks right** (immersive + landscape)
  — that's the one behavioural change to watch.
- **130.2 reverts fix126** (surface pokes + controller/key churn). Expected and
  correct — those fought media_kit / orphan the per-handle texture.
- **Do NOT recreate VideoController on the same `_player`** — one texture per
  handle (android_video_controller `_controllers[handle]`); recreation orphans
  the native surface.
- **The `np.setProperty(...)` tuning calls are fine** (efficiency review):
  source shows `setProperty` is a plain `mpv_set_property_string` after
  init-wait. Only `vid=no` was problematic (removed in 130.2).
- **Note the 5s mpv terminate delay** (`Future.delayed(5s,
  mpv_terminate_destroy)`): native context lingers 5s after dispose. Not a bug,
  but explains why rapid open/close of the SAME url can still collide — the
  existing reconnect/backoff logic already handles that; no change here.
- **`flutter analyze --no-fatal-infos` clean.** Files: mpv_engine.dart,
  player.dart, overlay_player_widget.dart (+ pubspec + fix130.md). **Version
  1.22.11+130, tag v1.22.11, even = phone.**
