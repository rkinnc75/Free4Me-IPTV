# fix337 — Shield 2x2 multi-view black cells: texture-attach race + auto-recovery → v1.26.53+337

> ## WARNING BUILD MACHINE — APPLY THE PATCH, then RELEASE (EXACTLY). No questions, no DB to query.

## Why (Shield log, fix331 SURFACE probes — the definitive data)
The fix331 mid-playback probes finally captured the Shield black screen, and it is in **2x2 multi-view**, not single view:
- Single view: healthy every time (`SURFACE[mid-playback+2s] textureId=0 rect=1920x1080`).
- 2x2 multi-view: in EVERY session, **two of the four cells had `textureId=null` at mid-playback** — one even with `playerWH=1920x1080` (mpv DECODING, no texture => black cell with audio). Allocated texture ids skip numbers (1, null, null, 4 / 1, null, 3, null): the `VideoController` platform registration races when four cells create controllers in the same frame, and ~two never complete.
- The recovery is also proven in the log: when a provider "stream completed" forced a cell engine RESTART, the new engine always obtained a texture (ids 5, 6).

## What this does (both halves derived from the log)
1. **Prevent the race:** `MpvEngine.open()` now forces controller creation one engine at a time via a static init chain, giving each platform registration a head start (lock held max 1.5s per engine; `_waitForTextureId` helper resolves early once the id arrives).
2. **Auto-recover the stragglers:** 4s after open, if the texture id is still null (the exact black-cell signature), the engine logs `TEXTURE-ATTACH-FAILED` and emits an engine error — which routes into the EXISTING retry machinery (MultiViewCell transient retry / Player reconnect) and restarts the engine, the recovery the log proves works. The error string does not match the seek-probe suppression filters.
