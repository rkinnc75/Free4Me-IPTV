## Why fix360/364/366 all failed — the actual root cause
On-device at 1.33.1+366 (2026-06-13 20:52 log + screenshot) the DVR transport row STILL did not show, despite: the engine getter present (fix364), DVR fully active on the SAME engine instance (log: `created eid=198898874` → `DVR enabled` → `open() succeeded`, no adoption), and a post-open `setState` firing (fix366). The control bar rendered the play/pause-only layout.

The reason: **media_kit_video 2.0.1's `_MaterialVideoControlsState` reads `_theme(context).primaryButtonBar` at its own mount and does not rebuild when the `MaterialVideoControlsTheme` InheritedWidget above it changes.** DVR activates INSIDE `open()` — i.e. AFTER the controls first mount with `dvrActive == false` — so the controls keep the play/pause-only `primaryButtonBar` forever. Parent `setState`/theme swaps (fix366) don't help because the controls state isn't listening for theme changes. This is why three prior attempts, each targeting timing or the getter, couldn't fix it: the problem is media_kit caching the bar at mount.

## Fix
Key the controls subtree on `dvrActive`:
```dart
MaterialVideoControlsTheme(
  key: ValueKey('mvct-dvr-${_engine.dvrActive}'),
  …)
```
When `dvrActive` flips false→true (once, when DVR turns on during open), the key changes, Flutter discards and **remounts** media_kit's controls state, which re-reads `primaryButtonBar` and renders the rewind / play-pause / forward / back-to-live row. One remount per playback; no effect when DVR is off (key stays constant) or for VOD.
