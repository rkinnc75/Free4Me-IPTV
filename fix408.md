# fix408 — Merge playback transport into a single, evenly-spread bottom control row

**Target version:** `1.35.7+408` (from `1.35.6+407`)
**Branch:** `main` · **Base:** `f1fb912` (v1.35.6+407)
**Toolchain (verified):** Flutter 3.44.0 / Dart 3.12.0 (CI pin), `libsqlite3.so` symlinked.
**Layout reviewed and approved via preview (even spread). Ready to tag.**

## What & why

The transport controls used to sit in media_kit's centred `primaryButtonBar` (across the middle of the picture). fix406 tried to lower them with a `Transform` and broke taps + rotation (reverted in fix407). The robust placement is media_kit's **bottom** bar — correctly positioned and tappable — so this moves them there.

**Layout-only, so hit-testing is correct by construction** (no `Transform`): the buttons are real flex `Row` children, exactly like the existing subtitles/audio/aspect icons that already work.

**Tiered (Option 2)** to avoid crowding:
- **Always:** channel ▲/▼ (when surfable), play/pause — alongside subtitles / audio / aspect / mini-player.
- **DVR only:** rewind 10s, forward 10s, back-to-live.
- **VOD:** prev / play / next.

**Even-spread distribution:** all bottom-bar icons are spread evenly across the row via `_spreadEvenly` (a `Spacer` between each item — first hard-left, last hard-right, equal gaps). This was chosen over a centred-transport island so no single gap stands out. The centre bar (`primaryButtonBar`) is now empty.

### Why the DVR tiering re-reads correctly
media_kit reads `bottomButtonBar` once at mount (the fix405 gotcha). But the controls theme is keyed `ValueKey('mvct-dvr-${_engine.dvrActive}')` (fix367), so toggling DVR remounts the subtree and re-reads the bottom bar — so the ±10s/live buttons appear/disappear correctly with DVR.
