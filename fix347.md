## Why (cold-eyes review HIGH-4)
The export server binds `anyIPv4` with no auth, and its `/import-sources` endpoint WRITES to the source list and then auto-triggered a network refresh against the imported URLs. While the export dialog is open (up to the 10-min idle window), anyone on the same LAN (guest Wi-Fi, dorm, coffee shop) could: (1) browse/download the exports, (2) POST a crafted settings JSON that injects sources which the device then immediately fetches over cleartext HTTP — an SSRF-flavoured vector plus persistent source pollution — and (3) OOM a 2GB box with an unbounded multipart body (`_collectBytes` buffered the whole request; the same class of bug fix328 fixed on the export side).

## What this does
1. **Access token on every route (`export_server.dart`):** the server generates a 12-hex-char `Random.secure()` token at start; `?t=<token>` is required on `/`, `/file/…`, and `/import-sources` (403 otherwise, with the remote address logged). The displayed URLs and the QR carry the token automatically, and the portal's file links and import form propagate it — so possession of the QR/dialog IS the authorisation; LAN strangers get 403.
2. **Body size cap:** `_collectBytes` rejects bodies over 5 MB (a settings export is tens of KB) — the OOM vector is closed before buffering.
3. **Confirm-before-refresh (`settings_view.dart`):** a successful portal import no longer auto-fires the refresh. The device shows "Sources imported — Refresh now / Later"; network fetches run only on explicit on-device confirmation. Portal text updated to match ("The device will ask to confirm the refresh." / "Confirm the refresh on the device.").
