## Source
Cold-eyes review of 1.33.3+368. Three actionable findings (MED-2, the 3-path category-visibility consolidation, is deferred to fix371).

## HIGH-1 — restored disabled categories leak into browse (mine, fix365 follow-up)
`Sql.applyGroupState` (backup restore) updated `groups.enabled` but NOT the denormalized `channels.cat_enabled` (fix365). It runs during restore AFTER the refresh that sets `cat_enabled`, so a category the backup marked disabled stayed visible in Live/VOD/Series until the next manual refresh — the fix355 hide-category feature silently broken on its main surface (restore onto a new device).
**Fix:** `applyGroupState` now also `UPDATE channels SET cat_enabled = <enabled> WHERE source_id = ? AND group_id = (the group's resolved id)`. **Rule 8:** restore of a disabled category sets all its channels' `cat_enabled=0` (0 leak), stays consistent with `groups.enabled` (0 mismatch), and re-enable via restore propagates back — all pass.

## MED-1 — DVR controls present but dead when Low Latency is on
With both Low Latency and Live DVR enabled, the `if (s.lowLatency)` branch set `demuxer-max-back-bytes=0`, but `_dvrActive` was already true — so the transport bar appeared yet rewind/back-to-live had no buffer to seek in.
**Fix:** DVR wins. The low-latency back-buffer-zeroing branch now only runs when `dvrBackMB <= 0`; when DVR is active the back buffer is kept and the low-latency profile is skipped (a multi-minute disk buffer is fundamentally at odds with low-latency), with a log line noting the suppression. DVR off → low-latency behaves exactly as before.

## LOW-1 — QR receiver server leaked on dialog throw (mine, fix368)
`_receiveViaQr` called `server.stop()` after the QR dialog; if `showDialog` threw, port 9479 leaked.
**Fix:** the post-`start()` dialog is wrapped in `try { … } finally { await server.stop(); }`.
