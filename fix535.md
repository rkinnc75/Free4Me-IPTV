# fix535 — Include SQLite DB snapshot in the diagnostic export

**Base tag:** `v2.0.34`
**Version:** `2.0.34+534` → `2.0.35+535`
**Scope:** export-bundle builder + one dialog string. No schema, no query change. Full suite green.

---

## Why

The QR/LAN diagnostic export ships source dumps + settings + log, which let us mirror schema and row *shape* but NOT the real `cat_enabled` / `favorite` / `stream_validated` / EPG distribution. Faithfully benchmarking the open performance issues (enable-source delay, Movies first-load, the Favorites browse hang) needs the actual on-device database. This adds the DB to the export so a real benchmark seed is available.

## What it does

In `_buildExportBundle` (settings_view.dart), after the settings backup and before the zip:

1. **Gated on `includeCredentials`.** `db.sqlite` stores Xtream usernames/passwords in the `sources` table and embeds them in every channel `url`, so the DB is included ONLY when the user explicitly opts into credentials (the same gate already used for the settings backup). When creds are excluded, the DB is omitted rather than shipped with secrets.
2. **Checkpoint+truncate the WAL first** via the existing `Sql.checkpointAndTruncateWal()`, so the `.sqlite` files are self-consistent (recent writes otherwise live only in the `-wal` sidecar).
3. **Copies each DB file-to-file** (`db.sqlite`, `epg.sqlite`) into the export dir — no in-memory buffering, preserving the fix328 OOM-avoidance design on the 2 GB TV box. Each is added as an `ExportItem` (so it appears on the LAN portal) and into the combined zip.

The credential-prompt dialog text is updated so the user knows "Yes" now also includes the database snapshot.

Validated standalone: a WAL-checkpointed `.sqlite` copies cleanly and is fully queryable with intact `cat_enabled`/credentials data, confirming both that the snapshot is benchmark-usable and why the credential gate is required.
