# fix212 — Default to in-memory search + maintain FTS triggers only when an FTS search method is active

> **⚠️ DEPENDENCY: apply fix210 (revert of fix206) FIRST.** fix212 assumes fix206 is gone (the refresh path is back to a bare `wipeSource`, no drop/rebuild helpers). Ship/apply order: **fix210 → fix212**.
>
> **Version:** build = highest fix number in release; patch = next available (read live pubspec). Tag = next immutable `v1.23.X`.
> **Motivation:** The user's `searchMethod` is `inMemory` (the app's "fastest" option, served by `ChannelSearchCache`), NOT FTS. But the FTS5 sync triggers (`channels_ai/au/ad`) are created by the DB migration on every install and fire on every channel insert regardless of search method — so a large refresh pays per-row FTS (trigram) cost for an index that non-FTS users never query. Also, the default search method was `ftsAnd`, despite `inMemory` being labeled "fastest".
> **Two parts:** (A) make `inMemory` the default search method; (B) the FTS triggers exist ONLY when an FTS search method (`ftsTrigram`/`ftsAnd`) is active — dropped otherwise. Reconciled at app boot (once) and on every settings change. When switching ONTO an FTS method, the index is rebuilt once (it was stale while triggers were absent).
> **Why this is clean (and not fix206):** the refresh path gets ZERO FTS special-casing — if you're not on FTS, the triggers simply don't exist, so inserts are naturally fast. No whole-catalog rebuild at refresh time (fix206's fatal flaw), no per-refresh logic, no process-static flags. Trigger state is driven by the setting, a rare deliberate action.

## Pre-verified facts (checked against `v1.23.22+206` with fix210 applied; compiled; Rule-8 tested)

- **FTS triggers** `channels_ai/au/ad` and the external-content `channels_fts` (`content='channels'`, supports `'rebuild'`) are created in `db_factory.dart` and exist by default on every install.
- **Search dispatch** (`sql.dart` `search`): `inMemory`->`_searchInMemory`, `likeSubstring`->`_searchLike` (both return before the FTS block); only `ftsTrigram`/`ftsAnd` reach `channels_fts MATCH`. So when triggers are absent, only non-FTS methods are in use and never query FTS.
- **Default sites** (currently `SearchMethod.ftsAnd`): `models/settings.dart`, `models/filters.dart`, `backend/settings_service.dart` (parse fallback), `backend/settings_io.dart` (parse fallback), `channel_picker_screen.dart` (null fallback).
- **Hook points:** `main.dart` (after `SettingsService.reload()`, already branches on `searchMethod`) for the one-time boot reconcile; `SettingsService.updateSettings` (the single settings persist path, called by the search-method picker at settings_view.dart:1633) for the on-change reconcile.
- **Rule-8 test (lifecycle):** migration creates 3 triggers -> boot reconcile(inMemory) drops to 0 -> non-FTS refresh inserts 3000 rows unsynced (FTS MATCH=0, stale) -> switch-to-FTS reconcile rebuilds + recreates (MATCH=3000, healed; triggers=3) -> new insert syncs via restored trigger (MATCH=1) -> switch back to inMemory drops to 0. All correct.
- **Compile status:** all edits (on top of fix210) -> `flutter analyze --no-fatal-infos` = exactly the 2 tolerated `settings_view.dart` INFOs. Rule 6 grep clean.

## PART B — FTS triggers exist only when an FTS method is active

### Fix 212.6 — `lib/backend/sql.dart`: add `reconcileFtsTriggers`

Insert immediately BEFORE the `insertChannelsBulk` doc comment.

Current code (verbatim — match exactly):
```dart
  /// fix174.3: bulk upsert for large imports. Same ON CONFLICT key as
  /// insertChannel (fix174.1). Does NOT read last_insert_rowid().
```
Replacement code (verbatim):
```dart
  /// fix212: the FTS index/triggers are only needed for the FTS search methods
  /// (ftsTrigram/ftsAnd). When the active method is inMemory or likeSubstring,
  /// the per-row FTS trigram maintenance during a refresh is pure overhead.
  /// This reconciles the trigger state to [ftsActive]:
  ///   - ftsActive=false: drop the FTS sync triggers (inserts stay fast).
  ///   - ftsActive=true : (re)create the triggers; if they were absent, the FTS
  ///     index is stale, so rebuild it once from the content table first.
  /// Called from a one-time boot check (main) and on every settings change
  /// (SettingsService.updateSettings). Idempotent.
  static Future<void> reconcileFtsTriggers(bool ftsActive) async {
    final db = await DbFactory.db;
    final existing = await db.getAll(
      "SELECT name FROM sqlite_master WHERE type = 'trigger' "
      "AND name IN ('channels_ai', 'channels_au', 'channels_ad')",
    );
    final triggersPresent = existing.length == 3;
    if (ftsActive) {
      if (triggersPresent) return;
      // Triggers were absent => FTS index is stale. Rebuild once, then recreate.
      await db.execute("INSERT INTO channels_fts(channels_fts) VALUES('rebuild');");
      await db.execute('CREATE TRIGGER IF NOT EXISTS channels_ai '
          'AFTER INSERT ON channels BEGIN '
          'INSERT INTO channels_fts(rowid, name) VALUES (new.id, new.name); END;');
      await db.execute('CREATE TRIGGER IF NOT EXISTS channels_ad '
          'AFTER DELETE ON channels BEGIN '
          "INSERT INTO channels_fts(channels_fts, rowid, name) "
          "VALUES('delete', old.id, old.name); END;");
      await db.execute('CREATE TRIGGER IF NOT EXISTS channels_au '
          'AFTER UPDATE OF name ON channels BEGIN '
          "INSERT INTO channels_fts(channels_fts, rowid, name) "
          "VALUES('delete', old.id, old.name); "
          'INSERT INTO channels_fts(rowid, name) VALUES (new.id, new.name); END;');
    } else {
      if (!triggersPresent && existing.isEmpty) {
        return;
      }
      await db.execute('DROP TRIGGER IF EXISTS channels_ai;');
      await db.execute('DROP TRIGGER IF EXISTS channels_au;');
      await db.execute('DROP TRIGGER IF EXISTS channels_ad;');
    }
  }

  /// fix174.3: bulk upsert for large imports. Same ON CONFLICT key as
  /// insertChannel (fix174.1). Does NOT read last_insert_rowid().
```

## Test plan
1. **Fresh install / inMemory default.** Boot -> Settings shows search method "In-Memory". (Dev) `SELECT name FROM sqlite_master WHERE type='trigger' AND name LIKE 'channels_a%'` returns nothing (boot reconcile dropped them).
2. **Non-FTS refresh speed.** Refresh a large source on inMemory. Read fix204 `commitWriteBatched: DONE ... total Xms`: faster than pre-212 (no per-row FTS cost). inMemory search still works (ChannelSearchCache).
3. **Switch to an FTS method, then search.** Settings -> FTS AND/Phrase -> reconcile recreates triggers + rebuilds FTS once. Search a >=3-char term for a just-refreshed source -> correct results. New inserts afterward stay synced.
4. **Switch back to inMemory.** Triggers dropped again; next refresh fast.
5. **Existing FTS user (stored ftsAnd).** Boot reconcile is a no-op (triggers present, ftsActive true); search unaffected.

## Notes
- **Ordering with fix210 is mandatory.**
- The refresh path (`xtream.dart`) is intentionally untouched — correctness/perf for non-FTS users comes purely from the triggers being absent, set by the boot/settings reconcile. This is the key simplification over the earlier drop-at-refresh idea.
- Switching onto FTS does a one-time whole-table FTS `'rebuild'` (cost proportional to total catalog) — acceptable because it is a rare, explicit user action, not a per-refresh cost.
- Independent of fix208. Builds on fix210.
