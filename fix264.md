# fix264 — Release remediation: version.json never committed for 1.26.2 (CI "stale" guard)

> ## ⚠️ BUILD MACHINE — APPLY VERBATIM (read before doing anything)
> This is a GIT + version.json remediation. There are NO Dart/code changes. Apply the version.json content byte-for-byte and run the git commands exactly in order. Do not improvise. See BUILD-MACHINE-RULES.md.

> **Situation:** the 1.26.2 build regenerated version.json locally ("version.json updated to 1.26.2") but that change was **never committed to `main`**. `main`'s version.json still says `1.26.1`, so the `v1.26.2` tag points at a commit with stale version.json → CI guard: *"version.json on the tagged commit is stale."* The app CODE for 1.26.2 (the DESC fix from fix262) — confirm it is on main; this runbook ONLY fixes the version.json + tag.
> **Goal:** put the correct `1.26.2` version.json on `main`, then move the `v1.26.2` tag to that commit.

## Step 0 — sanity check what's on main
```bash
git fetch origin
git checkout main
git pull origin main
grep '"latest"' version.json   # if this already shows 1.26.2, skip to Step 2
grep '^version:' pubspec.yaml   # expect 1.26.2+262 (the fix262 bump). If it says 1.26.1, the fix262 code/pubspec edits were not committed either — STOP and report.
```
> If `pubspec.yaml` is NOT `1.26.2+262`, the fix262 changes never landed on main — do not proceed; report that first.

## Step 1 — write the correct version.json (exact bytes) and commit it
Overwrite `version.json` with EXACTLY this content:
```json
{
  "latest": "1.26.2",
  "releaseUrl": "https://github.com/rkinnc75/Free4Me-IPTV/releases/tag/v1.26.2",
  "minSupportedAndroidApi": 21,
  "criticalUpdate": false,
  "releaseNotes": "\u2022 Fix: channel lists could fail to load with a database error after the provider-order update. Browsing now works correctly again."
}
```
Then commit and push to main:
```bash
git add version.json
git commit -m "v1.26.2: fix stale version.json (was not committed during release)"
git push origin main
```

## Step 2 — move the tag to the commit that now has the correct version.json
```bash
# delete the misplaced tag locally and remotely, then recreate on current main HEAD
git tag -d v1.26.2
git push origin :refs/tags/v1.26.2
git tag v1.26.2            # points at HEAD = the commit from Step 1
git push origin v1.26.2
```

## Step 3 — verify
```bash
# the tagged commit's version.json must now say 1.26.2
git show v1.26.2:version.json | grep '"latest"'      # → "latest": "1.26.2"
```
- Re-run the CI workflow (or the release script's verification step). The "stale" guard should pass.
- The GitHub Release for v1.26.2 should reference this commit. If the release was already created against the old commit, edit the release to point at the new tag (the API release uses tag_name; if it shows the old SHA, delete and recreate the release for v1.26.2, or re-run the release job).

## Why this happened (process note)
`build_and_release.sh` regenerates version.json (step 3b) and stages with `git add -A` (line 167) before committing — so version.json SHOULD have been committed. If the release was cut a different way (the manual `commit_and_release.sh` builds a commit via `git commit-tree` from an explicit file list and may not have included version.json), the regenerated file was left uncommitted. **Rule for releases: version.json must be in the SAME commit that gets tagged.** Prefer `build_and_release.sh` (handles ordering) over hand-rolled git; if cutting manually, `git add version.json pubspec.yaml lib/whats_new_modal.dart` explicitly, commit, push, THEN tag that commit.

## No code changed
This runbook does not touch any `.dart` file. `flutter analyze` is unaffected. After Step 2, the release is consistent: main, the tag, and version.json all say 1.26.2.
