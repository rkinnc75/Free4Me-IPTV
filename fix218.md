# fix218 — CI: make the Release workflow run analyze FIRST, as a gate before the APK build

> **Version:** CI-only change (no app version bump strictly required, but follow normal release process to land it on main). No Dart/app code changes.
> **Problem reported:** Analyze and Release appear to run "at the same time." Clarification: they trigger on DIFFERENT events — `analyze.yml` on push to `main`, `release.yml` on push of a `v*` tag. Pushing a commit and its tag together fires both events at once, so they run in parallel. More importantly, inside `release.yml` the expensive `flutter build apk` and the `flutter analyze` step run in the SAME job sequentially, but if analyze is ever reordered or skipped, a broken commit could still consume a full build. The goal: guarantee analyze passes BEFORE the APK build starts within the release run, in a structurally enforced way.
> **Fix:** split `release.yml` into two jobs — an `analyze` job and a `build-and-release` job that `needs: analyze`. GitHub will not start the build job unless the analyze job succeeds.

## Why this is the right shape

- `analyze.yml` (push to main) stays as the fast pre-tag feedback loop — unchanged.
- `release.yml` (tag push) currently has analyze as one step among many in a single job. Making analyze its own job that the build job depends on means the ~10-minute build is never even scheduled if analyze fails — saving runner time and making the gate explicit.
- This also fixes the perceived "parallel" problem within the release pipeline: build now provably waits for analyze.

## Fix 218.1 — restructure `.github/workflows/release.yml` into analyze → build jobs

**File:** `.github/workflows/release.yml`

Replace the single `jobs:` block. The current structure is:
```yaml
jobs:
  build-and-release:
    name: Build APK and publish release
    runs-on: ubuntu-latest
    timeout-minutes: 45

    steps:
      - name: Checkout
        uses: actions/checkout@v4
        with:
          fetch-depth: 0
      ...
      - name: flutter analyze
        run: flutter analyze --no-fatal-infos

      - name: flutter build apk --release
      ...
```

### Change: add a separate `analyze` job and make `build-and-release` depend on it.

Insert this new job ABOVE `build-and-release:` (as the first entry under `jobs:`):
```yaml
jobs:
  analyze:
    name: flutter analyze (gate)
    runs-on: ubuntu-latest
    timeout-minutes: 15
    steps:
      - name: Checkout
        uses: actions/checkout@v4
        with:
          fetch-depth: 0

      - name: Set up Flutter
        uses: subosito/flutter-action@v2
        with:
          flutter-version: '3.44.0'
          channel: stable
          cache: true

      - name: flutter pub get
        run: flutter pub get

      - name: flutter analyze
        run: flutter analyze --no-fatal-infos

      - name: Verify version.json is up to date
        run: |
          set -euo pipefail
          python3 scripts/update_version_json.py
          if ! git diff --quiet version.json; then
            echo "::error::version.json on the tagged commit is stale."
            git --no-pager diff version.json
            exit 1
          fi
          echo "version.json is current."

  build-and-release:
    name: Build APK and publish release
    needs: analyze
    runs-on: ubuntu-latest
    timeout-minutes: 45
```

> The `needs: analyze` line is the gate — `build-and-release` will not start unless the `analyze` job succeeds.

### Then REMOVE the now-redundant inline analyze step from `build-and-release` (it's covered by the gate job):
Current code (verbatim — match exactly):
```yaml
      - name: flutter analyze
        run: flutter analyze --no-fatal-infos

      - name: flutter build apk --release
```
Replacement code (verbatim):
```yaml
      - name: flutter build apk --release
```

> Optionally also remove the duplicate "Verify version.json is up to date" step from `build-and-release` since the `analyze` gate job now performs it — but leaving it is harmless (defense in depth). Recommended: leave it.

## Notes
- This does not change the separate `analyze.yml` (push-to-main feedback). Both remain: `analyze.yml` is the fast pre-tag check; the new `analyze` job inside `release.yml` is the enforced gate at release time.
- Independent of all app fixes. CI only.
- Had this gate existed, the apostrophe build-break (fix214/216) would still have produced a failed release run — but it would have failed in the cheap analyze job in ~2 min instead of after a build attempt, and no release/APK would be published from a broken tree.
