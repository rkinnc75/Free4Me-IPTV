# fix546 — Discard divider/header rows at import; purge existing; drop the divider filter

**Base tag:** `v2.0.45`
**Version:** `2.0.45+545` → `2.0.46+546`
**Scope:** import (xtream + m3u) discard, deferred one-time purge, VisibilityClause filter removal. Full suite green.

---

## Change

"`##### HEADER #####`" divider entries are cosmetic separators with no playable content (~7.5k rows on the field DB). This removes them end-to-end:

1. **Discard at import** — both import paths now skip dividers so they never enter the DB: `xtream.dart` (`if (Channel.nameIsDivider(live.name)) continue;`) and `m3u.dart` (`if (channel.isDivider) return;` in `commitChannel`).
2. **Purge existing** — `Sql.runPendingDividerCleanup()` runs once, **deferred** (unawaited from `main()` after first frame, gated by an `app_meta` marker), `DELETE FROM channels WHERE is_divider = 1`. It is deliberately OFF the cold-start path: on the 1.43GB DB this delete touches ~7.5k rows across the indexes and took ~5.5s — the fix542 lesson (never block startup). Best-effort + idempotent; retries next launch if interrupted.
3. **Drop the filter** — the per-row `NOT (is_divider=1 AND (SELECT hide_dividers …)=1)` correlated subquery is removed from `VisibilityClause` (its backing `idx_channel_divider` was already dropped in fix537), so every browse query is slightly simpler.

The `sources.hide_dividers` column and toggle are left in place (harmless no-op now; removing a column needs a risky table rebuild). New imports are clean immediately; existing catalogs are clean within seconds of the first post-update launch.
