# fix383 — A rejected Add Source no longer leaves a broken empty source behind (roll back the up-front-committed row on a failed new add)

**Target version:** `1.34.12+383` (from `1.34.11+382`)
**Branch:** `main`
**Toolchain (verified in sandbox):** Flutter 3.44.0 / Dart 3.12.0 (CI pin), `libsqlite3.so` symlinked.
**Written against:** `v1.34.11+382` on main (commit `537338d`, i.e. fix382 already shipped).

## What & why

**Symptom (user-reported).** When adding a provider, if the source is rejected
(bad URL, wrong username/password), the app still created the source row, and
then after correcting the credentials the user hit a "name already exists" error
or ended up with a duplicate. Expected: a failed add should not leave a source
behind.

**Root cause.** The importers commit the source row **up front**, before the
network fetch:
- `getXtream` (`lib/backend/xtream.dart`): fix376 added an up-front
  `Sql.commitWrite([Sql.getOrCreateSourceByName(source)])` so `source.id` is
  known before the bulk channel insert.
- `processM3UUrl` / `processM3U` (`lib/backend/m3u.dart`): same pattern
  (`commitWrite([getOrCreateSourceByName(source)])`, then `source.id = …`).

If the subsequent fetch/auth fails, the importer throws
(`getXtream` throws once all content types fail; the M3U path rethrows on a
parse/HTTP error), but the **source row is already persisted and is never rolled
back**. The failure surfaces as a "Failed to add source" snackbar in
`lib/setup.dart`, yet an empty, broken source now exists. On the corrected
retry, `setup.dart`'s pre-submit `Sql.sourceNameExists(name)` check now returns
true → the user is blocked with a false "name already exists" (or, after
renaming to dodge it, ends up with a leftover empty source plus the working one).

**Fix.** Roll back the leaked row at the single dispatch choke point,
`Utils.processSource` (covers all three source types). Before dispatching,
record whether a source with this name already existed. If the import throws and
the source did **not** pre-exist (a brand-new add), delete the row the importer
created (`source.id` is set by reference during the up-front commit, so it is
available in the catch). `Sql.deleteSource` cascades EPG/channels/groups, so any
partial inserts are cleaned up too. A refresh / re-import of an **existing**
source (name already present — the `refreshSource` path calls
`processSource(source, true, …)`) is deliberately left untouched, so a transient
refresh failure never deletes a working source. The cleanup is itself guarded so
a delete failure can never mask the original import error.

This removes the leak at its source, which also resolves the downstream
"duplicate name" / false "name already exists": once a failed add cleans up
after itself, re-adding with the same name simply works.

## Files changed
- `lib/backend/utils.dart` — wrap the `processSource` source-type switch in
  try/catch; on failure of a brand-new add (`!namePreExisted && source.id != null`),
  `Sql.deleteSource(source.id!)` then `rethrow`. Cleanup is guarded; original
  error always propagates.
- `lib/whats_new_modal.dart` — `'1.34.12'` changelog entry (real UTF-8).
- `pubspec.yaml` — `1.34.11+382` → `1.34.12+383`.

No new unit test. The change is a guarded try/catch around the importer dispatch,
whose behaviour depends on `DbFactory` (a real DB) plus live network I/O; the
existing suite has no harness that mocks those for `processSource`/`getXtream`
(only the pure `XtreamRefreshLogic` is unit-tested). Verified by analyze + the
on-device steps below. (Flagged honestly rather than adding a contrived test.)

## On-device integration check
1. Install `v1.34.12+383`.
2. Add Source → Xtream. Enter a name (e.g. "Test"), a reachable URL, and a
   **wrong** password. Submit. Expect the "Failed to add source" snackbar.
3. Open the source list / Settings. **Pre-fix383:** a broken empty "Test" source
   was present. **fix383:** no "Test" source exists.
4. Add Source again with the **same** name "Test" and the **correct** password.
   Submit. Expect: success, no "name already exists" error, exactly **one**
   source named "Test".
5. Repeat 2–4 with M3U URL using a bad URL, to confirm the dispatcher-level fix
   covers M3U as well as Xtream.
6. Refresh-safety invariant (no behaviour change expected): with a working
   source, trigger a refresh while the provider is briefly unreachable. The
   existing source must **remain** (the `namePreExisted` guard skips deletion on
   the refresh path).
