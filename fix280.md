# fix280 — PERMANENT FIX: build_and_release.sh creates the tag on the wrong commit (root cause of ALL version.json "stale" failures)

> ## ⚠️ BUILD MACHINE — EXECUTE EXACTLY
> This patches `scripts/build_and_release.sh`. It is a one-time fix to the release script. Apply the two `Replacement` blocks verbatim, commit the script change, then use the script normally. There are NO app code changes.

## ROOT CAUSE (proven by reading the script)
The version.json "stale on tagged commit" error has fired for 1.26.2, 1.26.4, 1.26.5, and 1.26.7. It is NOT the builder improvising — it is a defect in `build_and_release.sh`:

1. The script regenerates version.json, `git add -A`, commits, and `git push origin main`.
2. Then it creates the GitHub Release via the API with `"tag_name": "v{ver}"` and **NO `target_commitish`** and **no local `git tag`**.
3. When you create a release for a tag that does not exist, GitHub creates the tag at **the default branch HEAD as the server sees it at that instant** — which can be the PREVIOUS commit (the push may not be fully processed, or the new commit may not be HEAD). The tag lands on the old commit, whose version.json is the previous version → CI guard fails.

The error diff proves it every time: the tagged commit shows the PREVIOUS version's version.json (1.26.7 attempt showed 1.26.6). The fix is to **create the tag locally on the exact commit that contains version.json, push that tag, and pin the release to it.**

## Fix 280.1 — `scripts/build_and_release.sh`: create + push the tag on the version.json commit
Find the commit/push block and replace it so the tag is created locally on HEAD (the commit that includes version.json) and pushed explicitly. Also abort if version.json is somehow not current on HEAD.
### Current code (verbatim)
```bash
# Stage everything modified (version bump in pubspec.yaml, changelog, etc.)
# APKs are in .gitignore so they won't be staged.
git add -A
if ! git diff --cached --quiet; then
  git commit -m "${TAG}: release build"
fi

echo " Pushing to GitHub"
git push origin main
```
### Replacement code (verbatim)
```bash
# Stage everything modified (version bump in pubspec.yaml, changelog, etc.)
# APKs are in .gitignore so they won't be staged.
git add -A
if ! git diff --cached --quiet; then
  git commit -m "${TAG}: release build"
fi

# fix280: HARD GUARD — version.json on HEAD must match the generator, else the
# tag would point at a commit with stale version.json (the recurring CI failure).
python3 scripts/update_version_json.py
if ! git diff --quiet version.json; then
  git add version.json
  git commit -m "${TAG}: sync version.json"
fi
python3 scripts/update_version_json.py
if ! git diff --quiet version.json; then
  echo "ERROR: version.json does not match generator after commit — aborting before tag." >&2
  git --no-pager diff version.json >&2
  exit 1
fi

echo " Pushing to GitHub"
git push origin main

# fix280: create the tag LOCALLY on the exact commit that contains version.json,
# then push it. This pins the tag to the right commit instead of letting the
# GitHub release API create it at the server's branch HEAD (which raced to the
# PREVIOUS commit and caused every "version.json on the tagged commit is stale"
# failure: 1.26.2, 1.26.4, 1.26.5, 1.26.7).
RELEASE_SHA="$(git rev-parse HEAD)"
git tag -f "${TAG}" "${RELEASE_SHA}"
git push -f origin "refs/tags/${TAG}"
echo " Tagged ${TAG} at ${RELEASE_SHA} (commit with current version.json)"
```

## Fix 280.2 — `scripts/build_and_release.sh`: pin the release payload to that commit
Add `target_commitish` so the release API references the exact SHA (defensive; the tag from 280.1 already exists, so the API attaches to it).
### Current code (verbatim)
```bash
  # Build JSON payload safely  avoid shell quoting nightmares
  JSON_PAYLOAD=$(python3 - <<PYEOF
import json
print(json.dumps({
  "tag_name": "${TAG}",
  "name": "${TAG}",
  "body": """${RELEASE_BODY}""",
  "draft": False,
  "prerelease": False
}))
PYEOF
```
### Replacement code (verbatim)
```bash
  # Build JSON payload safely  avoid shell quoting nightmares
  # fix280: target_commitish pins the release/tag to the exact commit that has
  # the current version.json, so it can never resolve to a stale commit.
  JSON_PAYLOAD=$(python3 - <<PYEOF
import json
print(json.dumps({
  "tag_name": "${TAG}",
  "target_commitish": "${RELEASE_SHA}",
  "name": "${TAG}",
  "body": """${RELEASE_BODY}""",
  "draft": False,
  "prerelease": False
}))
PYEOF
```

## Why this is the permanent fix
The failure was structural: the script never pinned the tag to a commit, so GitHub placed it at whatever the server-side branch HEAD resolved to — frequently the previous commit. After 280.1/280.2 the tag is created locally on `git rev-parse HEAD` (the commit that just got the regenerated version.json), pushed explicitly, and the release is pinned to that SHA via `target_commitish`. There is no longer any path by which the tag can land on a stale commit. The added HARD GUARD also aborts the build before tagging if version.json ever fails to match the generator, so a bad release can never be published again.
