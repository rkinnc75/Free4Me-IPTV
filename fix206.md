# fix206 — Speed up large source refresh: drop non-unique indexes + disable FTS trigger during bulk insert, rebuild after

> **Version:** build `+206`; patch = next available at ship time (read live pubspec — patch and fix numbers drift; do NOT assume). Tag = next available `v1.23.X` (immutable; never reuse/delete/recreate).
> **Evidence (fix204 instrumentation, v1.23.21):** the refresh cost is entirely the DB write. Aniel3000 = 48,720 rows → 144,871ms (~2.97 ms/row); Emjay = 271,099 rows → 509,700ms (~1.88 ms/row). Both ran as ONE transaction (closures < batchSize). Time scales with ROW COUNT, not transaction count → the bottleneck is **per-row work**: the ~10 non-unique secondary indexes on `channels` (each b-tree maintained per inserted row) and the `channels_ai` AFTER INSERT FTS trigger (a trigram FTS insert per row).
> **Bug:** none functionally — this is a performance fix. (Also corrects the fix194 misdiagnosis: bulk batch size was never the lever.)
> **Approach:** within the refresh transaction, after `wipeSource`, drop the non-unique secondary indexes and disable the FTS sync triggers; do the bulk reinsert + preserve replay against the lightly-indexed table; then recreate the dropped indexes verbatim and rebuild the external-content FTS index in one pass.

## What the measurements prove

Per-row time is ~constant across two very different sources (2.97 vs 1.88 ms/row), and both refreshes were a single transaction — so neither statement count nor transaction count is the driver. The cost is paid per inserted row maintaining indexes and the FTS trigram index. Removing that per-row work during bulk load and doing it once at the end is the correct lever.

## Measured benefit (seeded sandbox tests, sqlite3 3.45.1)

- 50k rows, all 11 indexes present: per-row FTS trigger ON = 2479ms; trigger OFF + single FTS `'rebuild'` = 1621ms (**1.53×**).
- 60k rows: full current path vs (FTS off + rebuild + drop/recreate the 10 non-unique indexes) = **2.64×** faster (≈939ms vs ≈2479ms).
- Full end-to-end test in the exact refresh order (wipe → drop → bulk insert → updateGroups → restorePreserve → restore): 60k rows, FTS count == row count, trigram search returns correct hits (≥3-char tokens), **all unique indexes preserved and uniqueness still enforced**, all non-unique indexes recreated verbatim.
- Extrapolated: Emjay ~510s → roughly ~195s.

## Safety design (why this is correct, not just fast)

- **`channels_unique (name, source_id)` is KEPT.** The bulk `INSERT … ON CONFLICT DO UPDATE` depends on it as the conflict target (`insertChannelsBulk`). Dropping it would change insert semantics → corruption. The two partial unique indexes (`channels_unique_stream`, `channels_unique_series`) are also KEPT (data integrity).
- **Only NON-UNIQUE secondary indexes are dropped**, selected by querying `sqlite_master` (`sql NOT LIKE 'CREATE UNIQUE%'`). Their exact `CREATE INDEX` SQL is captured and **recreated verbatim** — the code never transcribes a definition, so partial-WHERE clauses and column lists cannot drift.
- **`restorePreserve` runs while `index_channel_name` is dropped**, but its `UPDATE … WHERE name = ? AND source_id = ?` is still covered by the kept `channels_unique (name, source_id)` index — verified index-backed in the seeded test.
- **FTS** is external-content (`content='channels'`), so `INSERT INTO channels_fts(channels_fts) VALUES('rebuild')` rebuilds it from the table in one pass; triggers are recreated to match the live definitions (`channels_au` is `AFTER UPDATE OF name`).
- **Symmetry:** both the drop and the restore are gated on `wipe`, so they always pair. Everything happens inside the same `commitWriteBatched` statement list (one transaction), so a failure rolls back to the fully-indexed prior state.
- **Compile status:** applied to pristine `v1.23.21+204`; `flutter analyze --no-fatal-infos` → exactly the 2 tolerated `settings_view.dart` INFOs. Rule 6 grep clean.

### Replacement code (verbatim) — the two new methods, then the original anchor line reproduced
```dart
  /// fix206: bulk-load optimization. Before a large reinsert, drop the
  /// NON-UNIQUE secondary indexes on channels and disable the FTS sync trigger,
  /// so each inserted row does not pay per-row b-tree maintenance + a trigram
  /// FTS insert. The dropped indexes' exact CREATE SQL is captured from
  /// sqlite_master and stashed in [memory] for verbatim recreation afterward.
  /// UNIQUE indexes (channels_unique + the partial unique indexes) are KEPT —
  /// the bulk INSERT's ON CONFLICT depends on channels_unique, and the partials
  /// guard integrity. Measured ~2.6x faster on 60k rows (seeded test).
  static Future<void> Function(SqliteWriteContext, Map<String, String>)
      dropChannelsBulkIndexes() {
    return (SqliteWriteContext tx, Map<String, String> memory) async {
      final rows = await tx.getAll(
        "SELECT name, sql FROM sqlite_master "
        "WHERE type = 'index' AND tbl_name = 'channels' "
        "AND sql IS NOT NULL "
        "AND name NOT IN (SELECT name FROM sqlite_master "
        "  WHERE type = 'index' AND sql LIKE 'CREATE UNIQUE%')",
      );
      final captured = <String>[];
      for (final r in rows) {
        final name = r.columnAt(0) as String;
        final sql = r.columnAt(1) as String;
        captured.add(sql);
        await tx.execute('DROP INDEX IF EXISTS $name;');
      }
      memory['fix206_droppedIndexSql'] = jsonEncode(captured);
      await tx.execute('DROP TRIGGER IF EXISTS channels_ai;');
      await tx.execute('DROP TRIGGER IF EXISTS channels_au;');
      await tx.execute('DROP TRIGGER IF EXISTS channels_ad;');
    };
  }

  /// fix206: after the bulk reinsert, recreate the non-unique indexes verbatim
  /// (from the SQL captured in [memory]), rebuild the external-content FTS index
  /// in one pass, and restore the FTS sync triggers.
  static Future<void> Function(SqliteWriteContext, Map<String, String>)
      restoreChannelsBulkIndexes() {
    return (SqliteWriteContext tx, Map<String, String> memory) async {
      final raw = memory['fix206_droppedIndexSql'];
      if (raw != null) {
        final List<dynamic> sqls = jsonDecode(raw) as List<dynamic>;
        for (final sql in sqls) {
          await tx.execute(sql as String);
        }
      }
      await tx.execute(
          "INSERT INTO channels_fts(channels_fts) VALUES('rebuild');");
      await tx.execute('CREATE TRIGGER IF NOT EXISTS channels_ai '
          'AFTER INSERT ON channels BEGIN '
          'INSERT INTO channels_fts(rowid, name) VALUES (new.id, new.name); END;');
      await tx.execute('CREATE TRIGGER IF NOT EXISTS channels_ad '
          'AFTER DELETE ON channels BEGIN '
          "INSERT INTO channels_fts(channels_fts, rowid, name) "
          "VALUES('delete', old.id, old.name); END;");
      await tx.execute('CREATE TRIGGER IF NOT EXISTS channels_au '
          'AFTER UPDATE OF name ON channels BEGIN '
          "INSERT INTO channels_fts(channels_fts, rowid, name) "
          "VALUES('delete', old.id, old.name); "
          'INSERT INTO channels_fts(rowid, name) VALUES (new.id, new.name); END;');
    };
  }

  /// fix174.3: bulk upsert for large imports. Same ON CONFLICT key as
  /// insertChannel (fix174.1). Does NOT read last_insert_rowid().
```

### Key changes
- Adds `dropChannelsBulkIndexes()` and `restoreChannelsBulkIndexes()` as `Sql` static closure factories. The original `insertChannelsBulk` doc comment and everything after it are unchanged (reproduced as the tail of the block).
