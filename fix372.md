## Bug
The Categories home view (`Sql.searchGroup`) ordered groups by `COALESCE(favorite,0) DESC, name COLLATE NOCASE ASC, id ASC` — favorites first, then strictly alphabetical. `enabled` was read (col 5 → `groupEnabled`, used only to gray-out the tile) but never used in the sort. So enabled and disabled categories were intermixed alphabetically: with most categories disabled (Rich's setup — only a handful enabled), an enabled category could sit far down the list below a wall of grayed-out disabled ones. On-device 1.34.0+371: enabled categories were not grouped at the top.

## Fix (per Rich: enabled on top, disabled still visible below, grayed)
Insert an enabled-first tier into the ORDER BY:
```
ORDER BY COALESCE(favorite, 0) DESC,
         COALESCE(enabled, 1) DESC,      -- fix372: enabled before disabled
         name COLLATE NOCASE ASC, id ASC
```
Final tiering: favorites first, then enabled-before-disabled, then A–Z within each tier. Disabled categories remain visible (grayed), just sorted below the enabled ones. `enabled` defaults to 1 (`COALESCE(...,1)`), so a brand-new source with nothing toggled is unaffected. No WHERE change — nothing is hidden.

## Rule 8 (executed)
Seeded groups (favorites/enabled/disabled, deliberately anti-alphabetical) → emitted order is exactly favorites (enabled-first within), then non-fav enabled A–Z, then non-fav disabled A–Z; a NULL-`enabled` row sorts as enabled (new-source safety). All assertions pass.
