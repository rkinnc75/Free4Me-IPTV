# fix388 — Edit Source: Test Connection now reports "Login OK" when auth succeeds (was reporting "Login failed" for any user whose `user_info.max_connections` is missing or 0)

**Target version:** `1.34.17+388` (from `1.34.16+387`)
**Branch:** `main`
**Toolchain (verified in sandbox):** Flutter 3.44.0 / Dart 3.12.0 (CI pin), `libsqlite3.so` symlinked.
**Written against:** `v1.34.16+387` on main.

## What & why

In the Edit Source dialog (added in fix385), the Xtream "Test connection" button calls `fetchXtreamMaxConnections(probe)` and treats a `null` return as "login failed". The function is misnamed: it doesn't just fetch `max_connections` — it also doubles as the auth probe. When the response is `{'user_info': {'auth': 1, 'max_connections': 0, ...}}`, the auth **succeeded** but `max_connections` is 0 (or missing), so the function returns `null`, and the dialog shows:

> "Xtream login failed — check URL, username, password."

This is wrong. The login worked. The function just couldn't extract a positive `max_connections` value.

**Real-world trigger:** A3000 (Media4u, `tv.media4u.top`) with a **test user that has limited permissions**. The provider returns `user_info.auth=1, status="Active", active_cons=2, max_connections=0` — the auth clearly succeeded, but `max_connections=0` is reported (likely because the test user's role doesn't see the account-level connection budget). Refresh on the same source works fine because `getXtream` doesn't gate on `max_connections`. Test Connection reports "login failed" even though the same credentials just authenticated.

Verified on 2026-06-16 against `tv.media4u.top` (source 6, Media4u). Raw dump:
```json
{"user_info":{"auth":1,"status":"Active","active_cons":2,"max_connections":0,...}}
```
`auth: 1` + `status: Active` = login OK. `max_connections: 0` = permission-limited, not "no connection".

## Files changed
- `lib/backend/xtream.dart` — new function `checkXtreamAuth(Source) → Future<bool>` alongside the existing `fetchXtreamMaxConnections`. The new function returns `true` if and only if the server accepts the credentials (`HTTP 200` + `user_info.auth == 1`). It no longer pretends to return `max_connections` — that's a different concern. The JSON-parsing logic is extracted to a public top-level `parseXtreamAuthResponse(dynamic) → bool` so it can be unit-tested without HTTP. `fetchXtreamMaxConnections` is kept (it's still legitimately called by `getXtream` line 269 to actually persist the live max_connections to the source row after a refresh).
- `lib/edit_dialog.dart` — `_testConnection`'s Xtream branch now calls `checkXtreamAuth` and uses the bool directly. The misleading "Xtream login failed — check URL, username, password." message is unchanged in wording but now only fires on actual auth failure (HTTP error, missing `user_info`, or `auth != 1`).
- `lib/whats_new_modal.dart` — `'1.34.17'` changelog entry.
- `pubspec.yaml` — `1.34.16+387` → `1.34.17+388`.
- `test/xtream_test_connection_test.dart` — **new**: 6 unit tests on `parseXtreamAuthResponse`, including a regression guard naming the A3000/Media4u `auth=1, max_connections=0` case.

## Design choices

- **Don't try to extract `max_connections` from the test response.** The dialog doesn't display it. If we want to show "Connected (4 max)" in the future, that's a separate UX fix. The bug here is the false-failure report, not a missing piece of info.
- **Keep `fetchXtreamMaxConnections` for the legitimate caller** in `getXtream` (line 269). It is the right function for the refresh-and-persist path; the bug was only that the dialog misused it as an auth check.
- **Make the parser public** (`parseXtreamAuthResponse`, no underscore) so the test can import it directly. Document it as "exposed for unit testing; internal helper." Top-level public functions in a Dart library are importable from tests; the underscore prefix would have made it library-private.
- **The function is renamed `checkXtreamAuth`** because it actually checks auth, not just fetches `max_connections`. Future readers will see the name and the auth check inside, and not be tempted to re-introduce the `max_connections > 0` gating.

2. **Case A — A3000/Media4u test user (the bug repro):** Settings → Sources → Edit on the Media4u source. Tap **Test connection**.
   - **Pre-fix388:** dialog shows "Xtream login failed — check URL, username, password."
   - **fix388:** dialog shows "Connected" (or whatever the success wording is).
3. **Case B — Real auth failure:** edit the password to something wrong, tap Test connection, confirm the dialog **still** shows "Login failed" (the regression-guard for case A — we didn't accidentally make the function always return true).
4. **Case C — Real auth success on a different provider:** test a S24/Stalker/Emjay Xtream source (full-permissions user, max_connections > 0). Confirm Test connection still shows "Connected". (This validates that the parser is symmetric across the auth=1 case.)
5. **Refresh on a real source:** Settings → Sources → Refresh on a working Xtream source. Confirm `getXtream` still works end-to-end and `sources.max_connections` is updated from the live `user_info.max_connections` value (the `fetchXtreamMaxConnections` path through line 269 is unchanged). The log line `Xtream: source "..." max_connections=N` should still appear on a successful refresh.

## Caveat (record, not a fix)

After this fix, **`fetchXtreamMaxConnections` is the only function in the codebase that reads `user_info.max_connections` for the persistent `sources.max_connections` column**. The dialog no longer reads it. If `getXtream` is ever refactored to not call `fetchXtreamMaxConnections` (e.g. to avoid the extra HTTP call when only the live count matters), the `sources.max_connections` column would stop being updated on refresh. The column is read in `multi_view_cell.dart` for connection-budget logic, so it'd silently drift. Out of scope for fix388, but worth flagging.
