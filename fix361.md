## Source
onn 4K Plus diagnostic (2026-06-13, v1.31.0+357). Four items.

### Issue 2 — "Error decoding audio" (YES Network, onn / Amlogic)
Repeated audio-decode failures on a multichannel E-AC3 feed.
**Decision:** the cold-eyes/reviewer suggestion of `audio-spdif=eac3,ac3` passthrough was **deliberately NOT used** — passthrough requires a capable AV receiver downstream and would SILENCE audio on a plain box→TV HDMI path (most users). Implemented the safe direction instead: **downmix multichannel → stereo** (`audio-channels=stereo`, `ad-lavc-downmix=yes`), decoded in software, which plays on every output. Exposed as a setting **"Downmix audio to stereo", default ON** (off only for users with a receiver who want the surround mix). Full plumbing + backup round-trip; old backups default ON.

### Issue 4 — slow large-catalog refresh (Dino: DELETE 26s, commit 60s)
When an FTS search method is active, the per-row `channels_ai`/`channels_ad` triggers fire ~300K times during a 148K-row wipe+reinsert. **Fix:** new `Sql.withSuspendedFtsTriggers(body)` drops the three FTS triggers around the bulk commit and rebuilds the index ONCE at the end; wrapped around the Xtream refresh commit. **No-op for non-FTS users** (their triggers are already absent, fix212) and **exception-safe** (finally restores+rebuilds). Scoped to the Xtream path (where the onn pain lives); the M3U streaming-insert path is deferred (smaller catalogs, would need flow restructuring). **Rule 8:** suspend+rebuild yields a byte-identical FTS index to row-by-row triggers, with stale pre-refresh rows correctly gone — assertions pass.

### Issue 3 — multi-view TEXTURE-ATTACH-FAILED on the onn (4× mediacodec-copy)
**Fix:** `DeviceDetector.isLowRamTv()` (Android TV + `DeviceMemory.totalMb < 2300`); the `multiViewDecode.auto` branch now routes preview tiles to software decode on Tegra/Shield OR low-RAM TV. Phones and large-RAM TVs keep mediacodec-copy. fix352 watchdog and full-screen fix164 decode unchanged.

### searchMethod logging
The "App started" diagnostic now records `searchMethod=…` and `ftsTriggers=true/false`, so future slow-refresh logs are diagnosable (the onn log couldn't tell us whether FTS was active).
