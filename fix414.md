# fix414 — PiP reconnect recovery: software-decode fallback + no menu-pop while minimized

**Version:** 1.35.12+413 → **1.35.13+414**
**Type:** Logic fix (player reconnect + decode routing). Text patch, 2 files.
**Tag:** v1.35.13
**Base:** current origin/main (fix413 live at 1.35.12+413). Applies directly.

## Root cause (from sms938u 2026-06-21 log)
A live stream played fine in PiP, hit a transient buffering stall, and the watchdog reconnected — but **every reconnect attempt failed while minimized**: stuck `initializing`, `pos=0`, `frame=null`, no first frame, then `max reconnects → onExit → menu`. The **same stream reconnected successfully twice while full-screen** seconds earlier. Reconnect is a same-engine `_engine.open()`; the surface is fine (`vo=gpu`) — the **MediaCodec hardware decoder can't re-configure without a foreground surface**, which PiP doesn't grant. (3–4 prior occurrences, always after minimizing.)

## Fix (two parts)
**Part 1 — survive PiP (`mpv_engine.dart`).** New `forceSoftwareDecode` flag threaded `_startPlayback → reapplyOptions → _applyMpvOptions`, set from `_inPipMode`. When reconnecting while minimized, force `hwdec=no` — libavcodec needs no MediaCodec/foreground, so the re-open re-inits and recovers in PiP. No-op for the initial open (never in PiP).

**Part 2 — don't dump to the menu (`player.dart`).** Give-up/route-pop is gated `&& !_inPipMode`; while minimized it keeps retrying quietly ("Reconnecting…", 5 s cadence) instead of popping. On return to full-screen the attempt counter resets, so the stream gets a fresh set of *hardware* attempts and only then gives up/exits as before.

**Known minor:** after a PiP reconnect the stream stays on software decode until the next channel change / full-screen re-open (avoids a re-open stall on every PiP exit). On the S24 full-screen software decode is fine.

## Verification
`flutter analyze` clean, 162 tests pass. **On-device:** play a live channel → minimize to PiP → induce a drop (toggle Wi-Fi/data briefly) → it should show "Reconnecting…" and resume in PiP, and must NOT pop to the menu. Confirm full-screen behaviour is unchanged (still gives up + exits after max attempts).
